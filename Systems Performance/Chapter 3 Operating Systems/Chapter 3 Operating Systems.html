Chapter 3 Operating Systems

理解操作系统及其内核对系统性能分析至关重要。在分析系统行为时，您将经常需要提出假设并进行测试，例如：系统调用如何执行，内核如何在 CPU 上调度线程，有限的内存如何影响性能，或文件系统如何处理 I/O。这些活动将要求您应用操作系统和内核的相关知识。

本章的学习目标是：

  • 学习内核术语：了解上下文切换、交换（swapping）、分页（paging）、抢占（preemption）等概念。
  • 理解内核的角色和系统调用的作用。
  • 掌握内核内部的工作原理：包括中断（interrupts）、调度器（schedulers）、虚拟内存（virtual memory）和 I/O 栈（I/O stack）等内容。
  • 了解内核性能特性的演变：从 Unix 到 Linux 的性能功能变化。
  • 对扩展的 BPF（eBPF）有一个基本的理解。

本章概述了操作系统和内核，是本书其余内容的基础知识。如果您错过了操作系统课程，可以把它当作速成课程。留意自己知识的盲点，因为最后会有一个考试（开玩笑的；其实只是小测验）。有关内核内部的更多信息，请参见本章末尾的参考资料。

本章分为三个部分：

  1. 术语：列出了重要的核心术语。
  2. 背景知识：总结了关键的操作系统和内核概念。
  3. 内核实现：总结了 Linux 和其他内核的具体实现细节。

与性能相关的领域，包括 CPU 调度、内存、磁盘、文件系统、网络以及许多特定的性能工具，将在后续章节中详细讨论。

3.1 Terminology

作为参考，以下是本书中使用的核心操作系统术语。许多术语也是本章及后续章节中将更详细解释的概念。

  • 操作系统：指安装在系统上的软件和文件，使得系统能够启动并执行程序。它包括内核、管理工具和系统库。
  • 内核：内核是管理系统的程序，包括（根据内核模型的不同）硬件设备、内存和 CPU 调度。它运行在特权 CPU 模式下，允许直接访问硬件，这种模式被称为内核模式。
  • 进程：操作系统的抽象和执行程序的环境。程序在用户模式下运行，并通过系统调用或陷入内核来访问内核模式（例如执行设备 I/O）。
  • 线程：一个可调度在 CPU 上运行的可执行上下文。内核有多个线程，一个进程包含一个或多个线程。
  • 任务：Linux 可运行的实体，既可以是进程（包含一个线程），也可以是多线程进程中的线程，或内核线程。

  • BPF 程序：在 BPF 执行环境中运行的内核模式程序。

    注：BPF 最初代表伯克利数据包过滤器，但现在该技术与伯克利、数据包或过滤几乎没有关系，因此 BPF 已成为一个独立的名称，而非首字母缩略词。

  • 主内存：系统的物理内存（例如 RAM）。

  • 虚拟内存：主内存的抽象，支持多任务处理和超额订阅。实际上，它是一种近乎无限的资源。
  • 内核空间：内核的虚拟内存地址空间。
  • 用户空间：进程的虚拟内存地址空间。
  • 用户区：用户级程序和库（例如 /usr/bin, /usr/lib...）。
  • 上下文切换：从一个线程或进程切换到另一个线程或进程。这是内核 CPU 调度程序的正常功能，涉及将正在运行的 CPU 寄存器集（线程上下文）切换为新的寄存器集。
  • 模式切换：在内核模式和用户模式之间切换。
  • 系统调用（syscall）：用户程序请求内核执行特权操作的明确协议，包括设备 I/O。
  • 处理器：不应与进程混淆，处理器是一个物理芯片，包含一个或多个 CPU。
  • 陷阱：发送到内核的信号，请求执行系统例程（特权操作）。陷阱类型包括系统调用、处理器异常和中断。
  • 硬件中断：物理设备发送给内核的信号，通常是请求 I/O 服务。中断是一种陷阱。 词汇表包括本章的更多术语供参考，包括地址空间，缓冲区，CPU，文件描述符，Posix和寄存器。

3.2 Background

以下部分描述了通用的操作系统和内核概念，帮助您理解任何操作系统。为了加深理解，本节还包含了一些与 Linux 相关的实现细节。 接下来的 3.3 节（内核）和 3.4 节（Linux）将重点介绍 Unix、BSD 和 Linux 内核的具体实现细节。

3.2.1 Kernel

内核是操作系统的核心软件，其功能取决于内核模型。类 Unix 操作系统（包括 Linux 和 BSD）使用的是单体内核（monolithic kernel），它负责管理 CPU 调度、内存、文件系统、网络协议以及系统设备（如磁盘和网络接口）。单体内核模型如 图 3.1 所示。

图中还展示了系统库，它通常用于提供比系统调用（syscalls）更丰富、更易用的编程接口。应用程序包括所有运行中的用户级软件，例如数据库、Web 服务器、管理工具和操作系统 Shell。

图中的系统库被表示为一条断开的环线，说明应用程序可以直接调用系统调用（syscalls）。例如，Golang 的运行时有其自身的系统调用层，无需依赖系统库（如 libc）。传统上，这类图通常用完整的环线表示，以反映从内核中心开始的特权递减模式（这一模型起源于 Unix 的前身 Multics [Graham 68]）。

注 2：这种模型有一些例外情况。例如，内核旁路技术（Kernel Bypass Technology）有时用于网络，使用户层可以直接访问硬件（详见第 10 章《网络》，第 10.4.3 节《软件》，标题“内核旁路”）。在某些情况下，I/O 可以绕过系统调用接口直接提交给硬件（尽管初始化时仍需要系统调用），例如内存映射 I/O（Memory-Mapped I/O）、主缺页（详见第 7 章《内存》，第 7.2.3 节《需求分页》）、sendfile(2) 和 Linux 的 io_uring（详见第 5 章《应用程序》，第 5.2.6 节《非阻塞 I/O》）。

还有其他内核模型：

  • 微内核（Microkernel）：采用小型内核，将功能移到用户模式程序中。
  • 单内核（Unikernel）：将内核和应用程序代码编译为一个单独的程序。
  • 混合内核（Hybrid Kernel）：例如 Windows NT 内核，它结合了单体内核和微内核的特点。

这些模型将在第 3.5 节《其他主题》中总结。

最近，Linux 通过引入一种新的软件类型——扩展 BPF（Extended BPF），改变了其内核模型。扩展 BPF 允许安全的内核模式应用程序运行，并提供了自己的内核 API：BPF 助手函数（BPF Helpers）。这使得某些应用程序和系统功能可以用 BPF 重写，从而实现更高的安全性和性能。这一模型如 图 3.2 所示。 ![[Figure 3.2 BPF applications.png]]

扩展 BPF 的相关内容将在第 3.4.4 节《扩展 BPF》中总结。

Kernel Execution 内核是一个大型程序，通常有数百万行代码。它主要在用户级程序发出系统调用或设备发出中断时按需执行。一些内核线程以异步方式运行，用于内务管理，其中可能包括内核时钟例程和内存管理任务，但这些线程都尽量做到轻量级，只消耗极少的 CPU 资源。

频繁执行 I/O 的工作负载（如网络服务器）主要在内核上下文中执行。计算密集型工作负载通常在用户模式下运行，不受内核干扰。我们可能会认为内核不会影响这些计算密集型工作负载的性能，但在很多情况下，内核确实会影响工作负载的性能。最明显的就是 CPU 竞争，当其他线程竞争 CPU 资源时，内核调度程序需要决定哪个线程运行，哪个线程等待。内核还会选择线程在哪个 CPU 上运行，并可为进程选择硬件缓存更强或内存定位性更好的 CPU，从而显著提高性能。

3.2.2 Kernel and User Modes (内核模式与用户模式)

内核运行在一种特殊的 CPU 模式中，称为 内核模式，该模式允许完全访问设备并执行特权指令。内核负责调度设备访问，以支持多任务处理，并防止进程和用户相互访问对方的数据，除非明确允许。

用户程序（进程）运行在 用户模式 下，在该模式中，它们通过系统调用向内核请求特权操作，例如进行 I/O 操作。

内核模式和用户模式通过处理器中的 特权环（或保护环）来实现，如图 3.1 所示。例如，x86 处理器支持四个特权环，编号为 0 到 3。通常只使用其中的两到三个：分别用于用户模式、内核模式和（如果存在）虚拟机监控程序（Hypervisor）。访问设备的特权指令仅在内核模式下允许执行；在用户模式下执行这些指令会引发异常，异常由内核处理（例如生成权限拒绝错误）。

在传统的内核中，系统调用是通过切换到内核模式并执行系统调用代码来完成的。如图 3.3 所示。

![[Figure 3.3 System call execution modes.jpg]] 图 3.3 系统调用执行模式

模式切换 是指在用户模式和内核模式之间的切换。

所有系统调用都涉及模式切换。某些系统调用还会进行 上下文切换：例如，阻塞的系统调用（如磁盘和网络 I/O）会进行上下文切换，以便在第一个线程被阻塞时，其他线程可以运行。

由于模式切换和上下文切换会产生少量的开销（CPU 周期），因此有各种优化方法可以避免它们，包括：

  • 用户模式系统调用：可以仅通过用户模式库实现某些系统调用。Linux 内核通过导出一个虚拟动态共享对象（vDSO）来实现这一点，该对象被映射到进程的地址空间中，包含如 gettimeofday(2) 和 getcpu(2) 等系统调用 [Drysdale 14]。
  • 内存映射：用于需求分页（参见第 7 章《内存》，第 7.2.3 节《需求分页》），它还可以用于数据存储和其他 I/O 操作，从而避免了系统调用的开销。
  • 内核绕过：允许用户模式程序直接访问设备，绕过系统调用和典型的内核代码路径。例如，DPDK（数据平面开发工具包）用于网络。
  • 内核模式应用程序：这些包括实现于内核中的 TUX Web 服务器 [Lever 00]，以及最近扩展的 BPF 技术，如图 3.2 所示。

内核模式和用户模式有各自的软件执行上下文，包括堆栈和寄存器。一些处理器架构（例如 SPARC）使用独立的内核地址空间，这意味着模式切换时还必须更改虚拟内存上下文。

3.2.3 System Calls (系统调用)

系统调用请求内核执行特权系统例程。虽然系统调用有成百上千种，但内核维护者尽量保持系统调用的数量尽可能少，以保持内核的简洁（Unix 哲学；[Thompson 78]）。可以在用户空间基于这些系统调用构建更复杂的接口，作为系统库，这样它们更容易开发和维护。操作系统通常包括一个 C 标准库，为许多常见的系统调用提供更易用的接口（例如，libc 或 glibc 库）。

  • futex(2)：此系统调用用于处理用户空间锁的一部分：尤其是可能会阻塞的部分。

ioctl(2) 系统调用可能是最难学习的，因为它的性质模糊不清。以其用法为例，Linux 的 perf(1) 工具（在第 6 章《CPU》介绍）执行特权操作来协调性能仪器化。为了避免为每个操作增加系统调用，Linux 仅添加了一个系统调用：perf_event_open(2)，它返回一个文件描述符，用于与 ioctl(2) 配合使用。然后，使用不同的参数调用该 ioctl(2) 来执行不同的操作。例如，ioctl(fd, PERF_EVENT_IOC_ENABLE) 用于启用仪器化。此例中的参数 PERF_EVENT_IOC_ENABLE 可以由开发者更容易地添加和修改。

3.2.4 Interrupts(中断)

中断是处理器接收到的一种信号，表示某个事件发生，需要进行处理，并中断当前的处理器执行以处理该事件。通常情况下，中断会导致处理器进入内核模式（如果尚未在内核模式下），保存当前线程的状态，然后执行中断服务例程（ISR）来处理该事件。

中断可以分为 异步中断 和 同步中断。它们分别由外部硬件和软件指令产生，如图 3.4 所示。 ![[Figure 3.4 Interrupt types.jpg]]

同步中断

同步中断是由软件指令产生的。以下是几种不同类型的软件中断，使用了“陷阱”（traps）、“异常”（exceptions）和“故障”（faults）这几个术语；不过，这些术语经常可以互换使用。

  • 陷阱（Traps）：有意调用内核，如通过 int（中断）指令。系统调用的实现之一就是调用 int 指令，并传递系统调用处理程序的向量（例如，在 Linux x86 上使用 int 0x80）。int 指令会引发一个软件中断。
  • 异常（Exceptions）：发生异常条件，如执行除零操作时。
  • 故障（Faults）：通常指内存事件，例如访问没有 MMU 映射的内存位置时触发的页面故障。参见第 7 章《内存》。

对于这些中断，负责的程序和指令仍然在 CPU 上。

中断线程

中断服务例程（ISRs）设计得尽可能快速，以减少对活跃线程的干扰。如果一个中断需要执行较多的工作，尤其是可能会在锁上阻塞的工作，它可以由一个中断线程来处理，内核可以调度这个线程。图 3.6 展示了这一处理过程。 ![[Figure 3.6 Interrupt processing.jpg]] 图 3.6 中断处理

这种实现方式取决于内核的版本。在 Linux 中，设备驱动程序可以分为两个部分，上半部分快速处理中断，并调度下半部分来稍后处理工作 [Corbet 05]。快速处理中断很重要，因为上半部分在中断禁用模式下运行，以推迟新的中断到来。如果上半部分运行过长时间，会对其他线程造成延迟问题。下半部分可以是 tasklets 或工作队列；后者是可以由内核调度的线程，必要时可以进入休眠。

例如，Linux 的网络驱动程序上半部分处理入站数据包的 IRQs，然后调用下半部分将数据包推送到网络栈。下半部分通常实现为 softirq（软件中断）。

从中断到服务完成的时间被称为 中断延迟，它依赖于硬件和实现。这个问题是实时或低延迟系统研究的一个课题。

中断屏蔽

内核中的某些代码路径不能被安全地中断。例如，当内核代码在执行系统调用时获取了一个自旋锁，而该自旋锁也可能被中断需要时，若在持有该锁的情况下发生中断，可能会导致死锁。为防止这种情况发生，内核可以通过设置 CPU 的中断屏蔽寄存器来暂时屏蔽中断。中断禁用时间应该尽可能短，因为它可能会扰乱由其他中断唤醒的应用程序的及时执行。这对于实时系统尤为重要——那些有严格响应时间要求的系统。中断禁用时间也是性能分析的一个目标（这种分析可以通过 Ftrace 中的 irqsoff 跟踪器直接进行，参见第 14 章《Ftrace》）。

某些高优先级事件不应被忽略，因此实现为 不可屏蔽中断（NMI）。例如，Linux 可以使用智能平台管理接口（IPMI）看门狗定时器，检查内核是否因在一段时间内没有中断而出现死锁。如果发生这种情况，看门狗可以发出一个 NMI 中断来重启系统。
Linux 还有一个软件 NMI 看门狗，用于检测死锁 [Linux 20d]。

3.2.5 Clock and Idle (时钟与空闲)

原始 Unix 内核的一个核心组件是 clock() 例程，它是通过定时器中断执行的。历史上，它通常每秒执行 60 次、100 次或 1,000 次（通常以赫兹（Hertz，单位：每秒周期数）表示），每次执行称为 tick。其功能包括更新时间、过期定时器和线程调度的时间片、维护 CPU 统计信息，以及执行计划的内核例程。

其他常见的频率包括 Linux 2.6.13 的 250 Hz、Ultrix 的 256 Hz 和 OSF/1 的 1,024 Hz [Mills 94]。

Linux 还会跟踪 jiffies，它是类似于 ticks 的时间单位。

时钟在早期内核中存在一些性能问题，后来在内核的版本中得到了改进，主要包括：

  • Tick 延迟：对于 100 Hz 的时钟，定时器可能会遇到最多 10 毫秒的额外延迟，直到它在下一个 tick 时被处理。这个问题已通过使用高分辨率实时中断来修复，使得执行可以立即进行。
  • Tick 开销：Tick 会消耗 CPU 周期并稍微影响应用程序的性能，这是操作系统 抖动（jitter）现象的一个原因。现代处理器还具有动态电源管理功能，可以在空闲期间关闭部分硬件电路。时钟例程会中断这种空闲时间，从而可能无谓地消耗电力。

无 Tick 内核

现代内核将许多功能从时钟例程移到了按需中断，以此来创建一个 无 Tick 内核。这样做不仅减少了开销，还提高了电源效率，因为它允许处理器在空闲时长时间保持在睡眠状态。

Linux 中的时钟例程是 scheduler_tick()，并且 Linux 提供了一些方法，可以在没有 CPU 负载的情况下跳过时钟调用。时钟本身通常以 250 Hz 运行（由 CONFIG_HZ Kconfig 选项及其变种配置），并且通过 NO_HZ 功能减少时钟调用（由 CONFIG_NO_HZ 及其变种配置），这个功能现在通常是启用的 [Linux 20a]。

空闲线程

当 CPU 没有工作任务时，内核会调度一个占位线程来等待工作，这个线程叫做 空闲线程。一个简单的实现方式是，空闲线程会通过一个循环检查是否有新的工作任务。现代 Linux 中，空闲任务可以调用 hlt（停止）指令，将 CPU 关闭，直到接收到下一个中断，从而节省电力。

3.2.6 Processes (进程)

进程是执行用户级程序的环境。它由内存地址空间、文件描述符、线程栈和寄存器组成。从某种意义上说，进程就像是一个虚拟的早期计算机，其中只有一个程序在执行，并且它拥有自己的寄存器和栈。

进程由内核进行多任务调度，内核通常支持在单一系统上执行成千上万个进程。进程通过进程ID（PID）进行唯一标识，PID是一个独特的数字标识符。

一个进程包含一个或多个线程，这些线程在进程的地址空间内操作，并共享相同的文件描述符。线程是一个可执行上下文，包含栈、寄存器和指令指针（也叫程序计数器）。多个线程允许单个进程在多个CPU上并行执行。在Linux中，线程和进程都被视为任务（task）。

内核启动的第一个进程被称为“init”，位于 /sbin/init（默认位置），其PID为1，负责启动用户空间服务。在Unix系统中，这涉及到从 /etc 运行启动脚本，这种方法现在被称为SysV（来自Unix System V）。现代Linux发行版通常使用systemd软件来启动服务并跟踪它们的依赖关系。

进程创建

进程通常通过Unix系统上的 fork(2) 系统调用来创建。在Linux中，C库通常通过封装多用途的 clone(2) 系统调用来实现 fork 函数。这些系统调用会创建一个进程的副本，并赋予它自己的进程ID。然后，可以调用 exec(2) 系统调用（或变种，如 execve(2)）来开始执行一个不同的程序。

3.2.7 Stacks(栈)

栈是一个用于存储临时数据的内存区域，采用后进先出（LIFO）方式组织数据。它用于存储不如CPU寄存器集中的数据重要的信息。当一个函数被调用时，返回地址会被保存到栈中。如果某些寄存器的值在调用后需要使用，这些寄存器的值也可能被保存在栈上。函数执行完毕后，它会恢复所需的寄存器值，并通过从栈中取回返回地址，将执行权交还给调用函数。栈还可以用于向函数传递参数。与函数执行相关的数据集合被称为栈帧。

注释
调用约定由处理器的ABI（应用程序二进制接口）指定，确定哪些寄存器在函数调用后应保留其值（这些寄存器是非易失性的），并由被调用函数保存到栈中（“callee-saves”）。其他寄存器则是易失性的，可能会被被调用函数覆盖；如果调用者希望保留这些寄存器的值，它必须将它们保存到栈中（“caller-saves”）。

通过检查线程栈中所有栈帧中保存的返回地址，可以查看到当前执行函数的调用路径（这一过程称为栈遍历）。这个调用路径被称为栈回溯或栈跟踪。在性能工程中，这通常简称为“栈”。这些栈可以回答某个操作为何执行，并且是调试和性能分析中不可或缺的工具。

注释
有关栈遍历和不同技术（包括：基于帧指针的、调试信息、最后分支记录和ORC）的更多细节，请参见《BPF性能工具》一书中的第2章《技术》，第2.4节《栈跟踪遍历》[Gregg 19]。

如何读取栈

以下是一个示例内核栈（来自Linux），显示了TCP传输的路径，打印工具所显示的栈信息：

tcp_sendmsg+1
sock_sendmsg+62
SYSC_sendto+319
sys_sendto+14
do_syscall_64+115
entry_SYSCALL_64_after_hwframe+61

栈通常以叶子到根的顺序打印，所以打印的第一行是当前执行的函数，下面是它的父函数，再下方是祖父函数，依此类推。在这个示例中，tcp_sendmsg() 正在执行，由 sock_sendmsg() 调用。在这个栈示例中，函数名右侧显示的是指令偏移，表示函数中的具体位置。第一行显示 tcp_sendmsg() 的偏移量是 1（即第二条指令），由 sock_sendmsg() 偏移 62 调用。这个偏移量只有在你希望深入了解代码路径、甚至到指令级别时才有用。

通过沿栈向下读取，可以看到完整的调用路径：函数、父函数、祖父函数等。或者，反过来从栈底向上读取，可以跟踪执行路径，了解如何到达当前函数。

由于栈揭示了通过源代码执行的内部路径，通常除了代码本身外，无法找到这些函数的文档。对于这个示例栈，文档就是Linux内核的源代码。一个例外是当函数是API的一部分时，这些函数会有文档。

3.2.8 Virtual Memory(虚拟内存)

虚拟内存是主内存的一种抽象，它为进程和内核提供了各自的、几乎无限的私有视图。虚拟内存支持多任务处理，使得进程和内核能够在各自的私有地址空间中操作，而无需担心竞争问题。它还支持主内存的过度订阅，允许操作系统根据需要在主内存和辅助存储（磁盘）之间透明地映射虚拟内存。

内存管理

虚拟内存使得可以通过辅助存储扩展主内存，但内核仍然努力将最活跃的数据保持在主内存中。为此，内核有两种方案：

  • 进程交换：将整个进程在主内存和辅助存储之间移动。
  • 分页：将内存分成小单元，称为页（例如4 KB）。

进程交换是原始的Unix方法，但它会导致严重的性能损失。分页更加高效，并且随着分页虚拟内存的引入，BSD系统也开始支持它。在这两种方案中，最不常用（或最近没有使用过）的内存会被移动到辅助存储中，只有在再次需要时才会移回主内存。

在Linux中，术语“交换”（swapping）通常指的是分页。Linux内核并不支持（较旧的）Unix风格的进程交换，后者涉及整个线程或进程的交换。

有关分页和交换的更多信息，请参见第7章《内存》。

3.2.9 Schedulers(调度)

Unix及其衍生系统是时间共享系统，通过在多个进程之间分配执行时间，使它们能够同时运行。进程在处理器和各个CPU上的调度由调度程序（scheduler）执行，调度程序是操作系统内核的一个关键组件。调度程序的作用如图3.12所示，它在线程（在Linux中称为任务）上进行操作，并将它们映射到CPU上。 ![[Figure 3.12 Kernel scheduler.jpg]] 图3.12 内核调度程序

调度程序的基本目的是将CPU时间分配给活动进程和线程，并维护优先级概念，以便更重要的工作能够更早执行。调度程序跟踪所有处于准备运行状态的线程，传统上这些线程按优先级排队，称为运行队列（run queues）[Bach 86]。现代内核可能会为每个CPU实现这些队列，并且可能会使用队列以外的其他数据结构来跟踪线程。当想要运行的线程多于可用的CPU时，优先级较低的线程会等待它们的执行机会。大多数内核线程的优先级高于用户级进程。

进程的优先级可以由调度程序动态修改，以改善特定工作负载的性能。工作负载可以分为以下两类：

  • CPU绑定：执行大量计算的应用程序，例如科学和数学分析，通常需要长时间运行（几秒、几分钟、几小时、几天，甚至更长）。这些应用程序的执行通常受限于CPU资源。
  • I/O绑定：执行I/O操作而计算量较少的应用程序，例如Web服务器、文件服务器和交互式Shell等，这类应用通常对低延迟响应有较高要求。当负载增加时，它们的瓶颈通常是I/O操作，受限于存储或网络资源。

一种常用的调度策略起源于UNIX，它识别CPU绑定的工作负载并降低其优先级，从而允许I/O绑定的工作负载（需要低延迟响应的工作）更早执行。这可以通过计算最近的计算时间（在CPU上执行的时间）与实际时间（经过的时间）之比，并降低那些计算比率高的进程的优先级来实现[Thompson 78]。这种机制偏向于较短时间运行的进程，通常这些进程是执行I/O的进程，包括人机交互进程。

现代内核支持多种调度类或调度策略（如Linux中），它们为管理优先级和可运行线程应用不同的算法。这些调度策略可能包括实时调度，它的优先级高于所有非关键任务，包括内核线程。结合抢占支持（稍后会描述），实时调度为需要实时性和低延迟调度的系统提供可预测和低延迟的调度。

有关内核调度程序和其他调度算法的更多信息，请参见第6章《CPU》。

3.2.10 File Systems(文件系统)

文件系统是将数据组织为文件和目录的方式。它们提供基于文件的接口来访问数据，通常遵循POSIX标准。内核支持多种文件系统类型和实例。提供文件系统是操作系统最重要的职责之一，曾被描述为操作系统的最重要角色之一 [Ritchie 74]。

操作系统提供一个全局的文件命名空间，以根目录（“/”）为起点，按照自上而下的树形拓扑结构组织。文件系统通过挂载（mount）将它们自己的树结构附加到一个目录（挂载点）上，从而加入到这个树中。这使得最终用户可以透明地浏览文件命名空间，而无需关心底层文件系统的类型。

3.2.12 Networking(网络)

现代内核提供了一组内置的网络协议栈，使得系统能够通过网络进行通信，并参与分布式系统环境。这通常被称为网络栈或TCP/IP栈，后者来源于广泛使用的TCP和IP协议。用户级应用通过称为“套接字”（sockets）的可编程端点访问网络。

连接到网络的物理设备是网络接口，通常由网络接口卡（NIC）提供。系统管理员的一个历史性职责是将IP地址与网络接口关联，以便它可以与网络通信；如今，这些映射通常通过动态主机配置协议（DHCP）自动化。

网络协议变化不大，但有一个新的传输协议正在被越来越多的采用：QUIC（在第10章《网络》中有总结）。协议的增强和选项变化较为频繁，如更新的TCP选项和TCP拥塞控制算法。新的协议和增强通常需要内核支持（除非是用户空间协议实现）。另一个变化是对不同网络接口卡的支持，这需要为内核提供新的设备驱动程序。

更多关于网络和网络性能的内容，请参见第10章《网络》。

3.2.13 Device Drivers(设备驱动程序)

内核必须与各种物理设备进行通信。这种通信是通过设备驱动程序实现的：设备管理和I/O的内核软件。设备驱动程序通常由开发硬件设备的厂商提供。一些内核支持可插拔设备驱动程序，这些驱动程序可以在不需要重启系统的情况下加载和卸载。

设备驱动程序可以为它们的设备提供字符设备和/或块设备接口。字符设备，也称为原始设备，提供不缓冲的顺序访问，可以根据设备提供从单个字符到任意I/O大小的数据访问。这类设备包括键盘和串行端口（在原始Unix中，还包括纸带和线路打印机设备）。

块设备则以块为单位进行I/O操作，历史上这些块通常为512字节。可以基于其块偏移量随机访问这些块，块偏移量从块设备的起始位置（偏移量0）开始。在原始Unix中，块设备接口还提供了块设备缓冲区的缓存，以提高性能，这部分缓存位于主内存中的缓冲区缓存区域。在Linux中，这个缓冲区缓存现在是页面缓存的一部分。

3.2.14 Multiprocessor(多处理器)

多处理器支持允许操作系统使用多个CPU实例并行执行任务。通常以对称多处理（SMP）形式实现，其中所有CPU被视为平等的。技术上，这一实现曾非常困难，特别是在并行运行的线程之间访问和共享内存及CPU时，存在一些问题。在多处理器系统中，可能会有多个内存银行与不同的插槽（物理处理器）连接，这在非一致性内存访问（NUMA）架构中尤为常见，也会带来性能上的挑战。有关调度和线程同步的详细信息，请参见第6章《CPU》，有关内存访问和架构的详细信息，请参见第7章《内存》。

IPIs（进程间中断）

在多处理器系统中，当CPU需要协调时，例如为了保证内存翻译条目的缓存一致性（通知其他CPU某个条目如果被缓存则已过时），就需要使用进程间中断（IPI）（也称为SMP调用或CPU跨调用）。IPIs是设计为快速执行的处理器中断，以最小化对其他线程的干扰。 IPIs还可以用于抢占。

3.2.15 Preemption(抢占)

内核抢占支持允许高优先级的用户级线程中断内核并执行。这使得实时系统能够在给定的时间限制内执行任务，包括飞机和医疗设备等系统。一个支持抢占的内核被称为“完全可抢占”（fully preemptible）内核，尽管在实际操作中，它仍然会有一些不能被中断的小型关键代码路径。

Linux还支持一种方法叫做自愿内核抢占（voluntary kernel preemption），即内核代码中的逻辑停止点可以检查并执行抢占。这避免了支持完全可抢占内核的一些复杂性，同时为常见的工作负载提供低延迟抢占。自愿内核抢占通常通过Linux的CONFIG_PREEMPT_VOLUNTARY Kconfig选项启用；另外，还可以通过CONFIG_PREEMPT选项使所有内核代码（除了关键区段外）都可以抢占，或通过CONFIG_PREEMPT_NONE选项禁用抢占，从而在提高吞吐量的同时牺牲一定的延迟。

3.2.16 Resource Management(资源管理)

操作系统可以提供各种可配置的控制来细化对系统资源的访问，如CPU、内存、磁盘和网络等。这些是资源控制，可以用于管理在不同应用程序运行或托管多个租户（云计算）的系统上的性能。这些控制可能会对每个进程（或进程组）施加固定的资源使用限制，或者采取更灵活的方法，允许将剩余资源在它们之间共享。

早期版本的Unix和BSD具有基本的每进程资源控制，包括使用nice(1)设置CPU优先级，以及使用ulimit(1)设置一些资源限制。

在Linux中，控制组（cgroups）在Linux 2.6.24（2008年）中被开发并集成，之后又添加了各种额外的控制功能。这些控制在内核源代码中的Documentation/cgroups目录下有文档说明。还有一个改进的统一层次结构方案叫做cgroup v2，它在Linux 4.5（2016年）中引入，并在Documentation/admin-guide/cgroup-v2.rst中进行了文档化。

后续章节中会根据需要提到特定的资源控制。例如，在第11章《云计算》中，描述了管理操作系统虚拟化租户性能的一个用例。

3.2.17 可观察性

操作系统由内核、库和程序组成。这些程序包括用于观察系统活动和分析性能的工具，通常安装在/usr/bin和/usr/sbin目录下。第三方工具也可以安装在系统中，以提供额外的可观察性。

3.3.1 Unix

Unix 由 Ken Thompson、Dennis Ritchie 和其他人于 1969 年在 AT&T 贝尔实验室开发，后续几年不断改进。其起源在《The UNIX Time-Sharing System》中有描述 [Ritchie 74]：

第一个版本是在我们中的一人（Thompson）对现有的计算设施感到不满后编写的。他发现了一台少有人用的 PDP-7，并着手创建一个更宜人的环境。

UNIX 的开发者此前曾参与过多路信息与计算服务系统（Multics）的开发。UNIX 被设计为一个轻量级的多任务操作系统及内核，最初名为 UNiplexed Information and Computing Service（UNICS），是对 Multics 的一个双关语。关于 UNIX 的实现，《UNIX Implementation》一书中提到 [Thompson 78]：

核心是唯一不能被用户随意替换的 UNIX 代码。因此，核心应尽量少做实际决定。这并不意味着允许用户拥有一百万种方式做同一件事，而是意味着只允许一种方式做一件事，但这方式应是所有可能提供的选项中最小的公约数。

虽然内核很小，但它为高性能提供了一些特性。例如，进程具有调度优先级，从而减少高优先级工作的运行队列延迟。磁盘 I/O 以较大的块（512 字节）执行以提高效率，并在内存中缓存到每个设备的缓冲区中。空闲进程可以被交换到存储中，从而让更繁忙的进程运行在主内存中。此外，系统支持多任务处理，允许多个进程并发运行，从而提高作业吞吐量。

为了支持网络、多种文件系统、分页及其他现在被认为是标准的功能，内核不得不扩展。随着多个衍生系统的出现，包括 BSD、SunOS（Solaris）以及后来的 Linux，内核性能变得具有竞争性，推动了更多特性和代码的加入。

3.3.2 BSD

伯克利软件分发版（BSD）操作系统始于对第六版 Unix 的增强，由加州大学伯克利分校开发，首次发布于 1978 年。由于原始 Unix 代码需要 AT&T 的软件许可证，到了 1990 年代早期，这些 Unix 代码在 BSD 中被重写，采用新的 BSD 许可证，从而允许 FreeBSD 等免费发行版的出现。

BSD 内核在性能方面的主要贡献包括：

  • 分页虚拟内存：BSD 将分页虚拟内存引入 Unix，不再将整个进程交换出以释放主内存，而是移动（分页）较小的最近最少使用（LRU）内存块。详见第 7 章 内存，第 7.2.2 节 分页。
  • 按需分页：推迟物理内存到虚拟内存的映射，直到页面首次被写入，从而避免对可能永远不会使用的页面进行不必要的早期性能和内存开销。详见第 7 章 内存，第 7.2.2 节 分页。
  • FFS：伯克利快速文件系统（FFS）将磁盘分配分组到柱面组中，大大减少了碎片，并提高了旋转磁盘上的性能，同时支持更大的磁盘及其他增强功能。详见第 8 章 文件系统，第 8.4.5 节 文件系统类型。
  • TCP/IP 网络协议栈：BSD 为 Unix 开发了第一个高性能的 TCP/IP 网络协议栈，包含于 4.2BSD（1983）。BSD 仍以其高性能的网络栈而闻名。
  • 套接字：Berkeley 套接字是用于连接端点的 API，包含于 4.2BSD，已成为网络标准。详见第 10 章 网络。
  • Jails：轻量级操作系统级虚拟化，允许多个客户共享一个内核。Jails 首次在 FreeBSD 4.0 中发布。
  • 内核 TLS：将传输层安全协议（TLS）的大部分处理移到内核中，从而提高性能 [Stewart 15]。

虽然没有 Linux 那么流行，BSD 仍被用于某些性能关键的环境，如 Netflix 的内容分发网络（CDN）以及 NetApp、Isilon 等公司的文件服务器。Netflix 在 2019 年总结 FreeBSD 在其 CDN 中的性能 [Looney 19]：

“使用 FreeBSD 和商品化硬件，我们能以 ~55% 的 CPU 使用率在一台 16 核 2.6 GHz CPU 上实现 90 Gb/s 的 TLS 加密连接服务。”

关于 FreeBSD 内部机制的优秀参考书是同一出版商的《The Design and Implementation of the FreeBSD Operating System, 2nd Edition》[McKusick 15]。

3.3.3 Solaris

Solaris 是一个基于 Unix 和 BSD 的内核和操作系统，由 Sun Microsystems 于 1982 年创建。最初名为 SunOS，并针对 Sun 工作站进行了优化。到 1980 年代后期，AT&T 基于 SVR3、SunOS、BSD 和 Xenix 开发了一个新的 Unix 标准 Unix System V Release 4（SVR4）。Sun 基于 SVR4 创建了新的内核，并将操作系统重新命名为 Solaris。

Solaris 内核在性能方面的主要贡献包括：

  • VFS：虚拟文件系统（VFS）是一个抽象层和接口，使多种文件系统能够轻松共存。Sun 最初创建它是为了让 NFS 和 UFS 共存。详见第 8 章 文件系统。
  • 完全可抢占内核：为高优先级工作（包括实时工作）提供低延迟支持。
  • 多处理器支持：20 世纪 90 年代初，Sun 投入大量资金开发多处理器支持，包括非对称和对称多处理（ASMP 和 SMP）[Mauro 01]。
  • Slab 分配器：取代 SVR4 的伙伴分配器，内核 slab 内存分配器通过每个 CPU 缓存预分配缓冲区，提供了更高性能。这种分配器及其衍生技术已成为包括 Linux 在内的内核的标准。
  • DTrace：一个静态和动态跟踪框架及工具，提供对整个软件堆栈的实时、生产环境的几乎无限的可观测性。Linux 有 BPF 和 bpftrace 提供类似的可观测性。
  • Zones：一种基于操作系统的虚拟化技术，用于创建共享一个内核的操作系统实例，类似于早期的 FreeBSD Jails 技术。现在 OS 虚拟化已广泛用于 Linux 容器。详见第 11 章 云计算。
  • ZFS：一个具有企业级功能和性能的文件系统，现在也可用于其他操作系统，包括 Linux。详见第 8 章 文件系统。

2010 年，Oracle 收购了 Sun Microsystems，Solaris 被重命名为 Oracle Solaris。本书第一版对 Solaris 有更详细的讨论。

3.4 Linux

Linux 是由 Linus Torvalds 于 1991 年创建的，是为英特尔个人计算机设计的一个免费的操作系统。他在一个 Usenet 的帖子中宣布了这个项目：

我正在做一个（免费的）操作系统（只是一个业余爱好，不会像 GNU 那样大且专业），用于 386（486）AT 克隆机。这从四月份开始酝酿，现在逐渐成熟了。我希望能收到大家对 MINIX 喜欢/不喜欢之处的反馈，因为我的操作系统在某些方面与它相似（例如，文件系统的物理布局由于实际原因相同）。

这里提到的 MINIX 操作系统是一个免费的小型（mini）Unix 版本，专为小型计算机设计。当时，BSD 也在致力于提供一个免费的 Unix 版本，但受到法律纠纷的困扰。

Linux 内核的开发借鉴了许多前辈的思想，包括：

  • Unix（及 Multics）：操作系统层次结构、系统调用、多任务处理、进程、进程优先级、虚拟内存、全局文件系统、文件系统权限、设备节点、缓冲区缓存。
  • BSD：分页虚拟内存、按需分页、快速文件系统（FFS）、TCP/IP 网络协议栈、套接字。
  • Solaris：VFS（虚拟文件系统）、NFS（网络文件系统）、页面缓存、统一页面缓存、slab 分配器。
  • Plan 9：资源分叉（rfork），用于在进程和线程（任务）之间创建不同级别的共享。

如今，Linux 被广泛应用于服务器、云实例，以及包括手机在内的嵌入式设备。

3.4.1 Linux 内核的发展

Linux 内核在性能方面的发展涵盖多个方面，包括以下技术（括号内通常标明首次引入的内核版本）：

  1. CPU 调度类：引入了先进的 CPU 调度算法，例如为非一致性内存访问（NUMA）优化的调度域（2.6.7）。详见第 6 章。
  2. I/O 调度类：引入了多种块 I/O 调度算法，如 Deadline（2.5.39）、Anticipatory（2.5.75）和 CFQ（2.6.6）。这些算法在 Linux 5.0 被新的多队列 I/O 调度器取代。详见第 9 章。
  3. TCP 拥塞算法：支持多种拥塞控制算法，如 Reno 和 Cubic，增强网络性能。详见第 10 章。
  4. 内存过量分配（Overcommit）和 OOM 机制：优化内存使用效率，详见第 7 章。
  5. 快速用户空间互斥锁（Futex, 2.5.7）：提升用户级同步性能。
  6. 大页支持（Huge Pages, 2.5.36）：支持大内存页的预分配，详见第 7 章。
  7. 性能工具：如系统分析器 OProfile（2.5.43）和 Blktrace（2.6.17）用于跟踪 CPU 使用和块 I/O 事件。
  8. 调试和追踪工具：
    • RCU（读-复制-更新, 2.5.43）：允许多次并发读取操作，提高性能和扩展性。
    • Epoll（2.5.46）：用于高效处理多个文件描述符的 I/O。
    • Tracepoints（2.6.28）：静态内核追踪点，为追踪工具提供关键逻辑点。
    • Perf（2.6.31）：支持性能分析和追踪。
  9. 内存优化：
    • 动态时钟中断（DynTicks, 2.6.21）：减少空闲状态下的时钟中断，节省 CPU 资源。
    • 透明大页（Transparent Huge Pages, 2.6.38）：动态分配大页内存。
  10. 调度改进：
    • 完全公平调度器（CFS, 2.6.23）：提升调度公平性。
    • SCHED_DEADLINE（3.14）：实现基于最早截止时间优先的调度策略。
  11. 新硬件支持：
    • NUMA 负载均衡（3.8+）：自动优化多 NUMA 系统中的内存位置。
    • Intel CAT（4.10）：支持为任务分配专用 CPU 缓存空间。
  12. 网络增强：
    • TCP Fast Open（TFO, 3.6+）：减少三次握手时间。
    • TCP BBR（4.9）：一种提高延迟和吞吐量的新型拥塞控制算法。
    • eXpress Data Path（XDP, 4.8+）：基于 BPF 的高性能网络处理。
  13. 存储改进：
    • 多队列 I/O（3.13+）：通过多队列提升 SSD 性能。
    • I/O 环形缓冲区（io_uring, 5.1）：提供高效异步 I/O 接口。

      3.4.2 systemd

systemd 是 Linux 的服务管理器，取代了传统 UNIX 的 init 系统。它提供了依赖感知的服务启动和时间统计功能。

  • 优化启动时间：使用 systemd-analyze 显示启动时间和服务启动关键链：

      # systemd-analyze critical-chain

    输出显示服务启动的关键路径，有助于发现启动延迟的瓶颈。

3.4.3 KPTI（Meltdown 漏洞）

KPTI（内核页表隔离）是为修复 Intel Meltdown 漏洞而引入的补丁，但会带来性能开销。通过以下措施可以降低成本：

  • 使用大页以加快 TLB 预热。
  • 减少系统调用频率。

3.4.4 扩展 BPF（eBPF）

eBPF 是 Linux 内核中一个通用执行引擎，用于网络、性能观测和安全。其特点包括：

  • 安全性：通过验证器确保 BPF 程序不会崩溃或破坏内核。
  • 灵活性：支持多种事件源（如 tracepoints、kprobes）和数据输出方式（如 maps 和 perf 环形缓冲区）。
  • 应用场景：eBPF 支持创建高效、定制化的追踪工具。例如，通过记录 I/O 开始和结束时间，绘制出性能直方图。

本书第 15 章深入探讨 BCC 和 bpftrace 这两个 eBPF 的前端工具。

3.5 Other Topics

一些值得总结的额外内核和操作系统话题包括PGO内核、Unikernel、微内核、混合内核和分布式操作系统。

3.5.1 PGO Kernels

基于配置文件的优化（PGO），也称为反馈导向优化（FDO），使用CPU配置文件信息来改进编译器决策[Yu 14a]。这可以应用于内核构建，过程如下：

  1. 在生产环境中，采集CPU配置文件。
  2. 根据该CPU配置文件重新编译内核。
  3. 部署新内核到生产环境中。

这样就能创建一个针对特定工作负载优化的内核。像JVM这样的运行时自动执行此过程，通过运行时性能重新编译Java方法，并结合即时编译（JIT）。创建PGO内核的过程则需要手动步骤。

相关的编译优化是链接时优化（LTO），它将整个二进制文件一次性编译，以便在整个程序中进行优化。微软Windows内核大量使用LTO和PGO，PGO带来了5%到20%的性能提升[Bearman 20]。谷歌也使用LTO和PGO内核来提高性能[Tolvanen 20]。

gcc和clang编译器，以及Linux内核，都支持PGO。内核PGO通常需要运行一个专门仪表化的内核来收集配置文件数据。谷歌发布了一个名为AutoFDO的工具，能够绕过这种特殊内核的需求：AutoFDO允许使用perf(1)从普通内核收集配置文件，然后将其转换为编译器可用的正确格式[Google 20a]。

关于如何使用PGO或AutoFDO构建Linux内核的最新文档，来自2020年Linux Plumber's Conference上的两场演讲，分别由微软[Bearman 20]和谷歌[Tolvanen 20]进行介绍。

一段时间内，最新的文档来自2014年，适用于Linux 3.13[Yu 14b]，这在新版内核上妨碍了PGO的采用。

3.5.2 Unikernel

Unikernel是一种将内核、库和应用程序软件合并到一个单一应用机器映像中的技术，通常可以在硬件虚拟机或裸机上以单一地址空间运行。它可能带来性能和安全上的优势：更少的指令文本意味着更高的CPU缓存命中率和更少的安全漏洞。但它也带来了问题：可能没有SSH、Shell或性能工具可供登录和调试系统，也无法添加这些工具。

为了在生产环境中对Unikernel进行性能调优，必须构建新的性能工具和度量指标来支持它们。作为概念验证，我构建了一个基础的CPU分析器，它从Xen的dom0运行，分析domU的Unikernel客体，并构建了一个CPU火焰图，仅仅为了展示这是可行的[Gregg 16a]。

Unikernel的例子包括MirageOS[MirageOS 20]。

3.5.3 微内核和混合内核

本章大部分讨论的是类似Unix的内核，也称为单体内核，其中所有管理设备的代码作为一个大的内核程序一起运行。对于微内核模型，内核软件被保持在最小限度。微内核支持内存管理、线程管理和进程间通信（IPC）等基本功能。文件系统、网络栈和驱动程序则作为用户态软件实现，这使得这些用户态组件更容易修改和替换。想象一下，不仅可以选择安装哪个数据库或Web服务器，还可以选择安装哪个网络栈。微内核也更具容错性：驱动程序崩溃不会导致整个内核崩溃。微内核的例子包括QNX和Minix 3。

微内核的一个缺点是执行I/O和其他功能时需要额外的IPC步骤，从而降低性能。为了解决这个问题，混合内核应运而生，它结合了微内核和单体内核的优点。混合内核将性能关键的服务移回内核空间（使用直接函数调用而不是IPC），就像单体内核一样，但保留了微内核的模块化设计和容错性。混合内核的例子包括Windows NT内核和Plan 9内核。

3.5.4 分布式操作系统

分布式操作系统在一组独立的计算节点上运行一个单一的操作系统实例，这些节点通过网络连接在一起。每个节点通常使用微内核。分布式操作系统的例子包括Plan 9（来自贝尔实验室）和Inferno操作系统。

虽然这是一种创新的设计，但这种模型并没有得到广泛应用。Plan 9和Inferno的共同创始人Rob Pike描述了多种原因，其中包括[Pike 00]：

“在1970年代末和1980年代初，有人声称Unix已经杀死了操作系统研究，因为没有人再尝试其他的东西。那时，我并不相信这个说法。今天，我勉强接受这个说法可能是对的（尽管有微软的例外）。”

在云计算中，今天常见的计算节点扩展模型是通过一组相同的操作系统实例进行负载均衡，这些实例可能根据负载的变化进行扩展（见第11章，云计算，第11.1.3节，容量规划）。

哪种内核最快？

这取决于操作系统的配置、工作负载以及内核参与的程度。一般来说，由于 Linux 在性能改进、应用程序和驱动程序支持、广泛的使用以及发现并报告性能问题的大型社区方面的广泛工作，预计它会优于其他内核。自1993年以来，由 TOP500 列表追踪的全球前500台超级计算机在2017年实现了100% Linux化[TOP500 17]。当然，也有一些例外；例如，Netflix 在云平台上使用 Linux，但在其 CDN 上使用 FreeBSD。

16FreeBSD 提供了比 Linux 更高的性能，尤其是在 Netflix 的 CDN 工作负载中，部分原因是 Netflix OCA 团队对内核进行的改进。这一性能差异经过了常规测试，最近一次是在2019年，测试了 Linux 5.0 与 FreeBSD 的生产环境对比，我参与了该分析。

内核性能通常使用微基准测试进行比较，但这种做法容易出错。这样的基准测试可能会发现某个内核在特定的系统调用上要快得多，但这个系统调用并不会在生产工作负载中使用。（或者它确实被使用，但微基准测试没有测试某些标志，而这些标志会显著影响性能。）准确比较内核性能是一项需要高级性能工程师完成的任务——这个任务可能需要数周时间才能完成。有关方法，请参见本书第12章《基准测试》，第12.3.2节《主动基准测试》。

在本书的第一版中，我在这一节结束时提到，Linux 没有成熟的动态追踪器，缺乏这一工具可能会错失一些显著的性能提升。从第一版以来，我转向了全职的 Linux 性能角色，并帮助开发了 Linux 缺失的动态追踪器：BCC 和 bpftrace，基于扩展的 BPF。这些内容在第15章和我之前的书籍中有所介绍[Gregg 19]。

第3.4.1节《Linux内核发展》列出了第一版与本版之间发生的许多其他 Linux 性能发展，涵盖了内核版本 3.1 到 5.8 之间的变化。一个尚未列出但值得注意的重大进展是，OpenZFS 现在将 Linux 作为其主要内核，从而为 Linux 提供了一个高性能和成熟的文件系统选项。

然而，随着 Linux 的发展，复杂性也随之而来。Linux 上有如此多的性能特性和可调参数，以至于为每个工作负载进行配置和调整变得非常繁琐。我见过许多部署未经过调优。比较内核性能时，请牢记这一点：每个内核是否都经过了调优？本书的后续章节及其调优部分可以帮助你解决这个问题。

3.7 Exercises

进程、线程和任务的区别

1. 进程 (Process)

  • 定义：进程是一个具有独立资源分配和执行环境的程序实例。每个进程都有自己独立的内存地址空间、文件描述符和其他系统资源。
  • 特点：
    • 独立的内存地址空间
    • 进程间通信（IPC）相对复杂（如管道、消息队列、共享内存等）
    • 进程切换开销大（涉及内存管理和寄存器切换）

2. 线程 (Thread)

  • 定义：线程是运行在进程中的轻量级执行单元，线程共享进程的地址空间、全局变量和资源。
  • 特点：
    • 共享进程的内存空间和资源
    • 线程之间的通信更高效（如共享内存）
    • 线程切换开销小（只需切换寄存器、程序计数器等）

3. 任务 (Task)

  • 定义：任务的概念在不同的上下文中有所不同。在某些情况下，任务可以指代进程或线程，而在实时系统中，任务通常表示一个调度实体。
  • 特点：
    • 在现代操作系统中，任务通常表示“可调度的实体”，包括内核线程和用户线程。

模式切换 (Mode Switch) 和 上下文切换 (Context Switch) 的区别

1. 模式切换 (Mode Switch)

  • 定义：模式切换是指从用户模式切换到内核模式，或从内核模式切换回用户模式。
  • 触发条件：
    • 系统调用（如I/O操作、内存分配等）
    • 硬件中断（如定时器中断、I/O中断等）
  • 切换成本：较小的成本（不需要切换内存地址空间）

2. 上下文切换 (Context Switch)

  • 定义：上下文切换是指从一个进程或线程切换到另一个进程或线程，涉及保存和恢复CPU寄存器、程序计数器和内存地址空间等。
  • 触发条件：
    • 时间片到期（时间调度）
    • I/O等待
    • 系统调用或中断（如信号、死锁等）
  • 切换成本：成本较高，涉及寄存器状态保存、内存管理切换等操作。

分页 (Paging) 和 进程交换 (Process Swapping) 的区别

1. 分页 (Paging)

  • 定义：将进程的内存划分为固定大小的页面（page），并将页面存储在物理内存或磁盘中。
  • 特点：
    • 仅在需要时将页面从磁盘加载到内存（即按需分页，Demand Paging）
    • 避免了将整个进程加载到内存中，提高了内存利用率
    • 分页是虚拟内存实现的基础

2. 进程交换 (Swapping)

  • 定义：将整个进程从内存交换到磁盘中，并在需要时将其加载回内存。
  • 特点：
    • 整个进程被交换到磁盘上，内存完全释放
    • 进程的交换通常发生在内存不够用或内存压力较大时
    • 成本较高，因为涉及磁盘I/O

      I/O密集型 (I/O-bound) 和 CPU密集型 (CPU-bound) 工作负载的区别

1. I/O密集型 (I/O-bound)

  • 定义：I/O操作（如磁盘、网络）是限制程序性能的主要因素。
  • 特点：
    • 频繁地等待I/O操作完成
    • CPU经常处于空闲状态，等待I/O操作
    • 典型示例：文件读取、网络请求等

2. CPU密集型 (CPU-bound)

  • 定义：CPU计算是限制程序性能的主要因素。
  • 特点：
    • 大量的计算操作，CPU始终处于繁忙状态
    • I/O操作较少，CPU利用率高
    • 典型示例：科学计算、加密解密、图像处理等

内核的角色

  • 定义：内核是操作系统的核心组件，负责管理内存、进程、设备和文件系统。
  • 角色：
    • 进程管理：负责进程的创建、调度和终止。
    • 内存管理：提供虚拟内存和分页机制，管理物理内存的分配和回收。
    • 设备管理：通过设备驱动程序与硬件设备交互。
    • 文件系统：管理文件存储、访问和权限。
    • 中断和系统调用：响应硬件中断，并为用户程序提供系统调用接口。

系统调用的角色

  • 定义：系统调用是用户程序请求内核服务的接口。
  • 角色：
    • 提供受控的用户与内核交互方式，避免直接访问硬件。
    • 常见的系统调用：文件操作（open、read、write）、进程管理（fork、exec）、内存管理（mmap）等。

虚拟文件系统 (VFS) 的角色及其在I/O堆栈中的位置

  • 定义：虚拟文件系统（VFS）是一种抽象层，提供对多种文件系统（如ext4、NTFS、NFS等）的统一接口。
  • 角色：
    • 提供对多种文件系统的统一接口
    • 隐藏底层文件系统的实现细节
  • 在I/O堆栈中的位置：
    • 用户程序发出文件I/O请求
    • VFS解析路径并调用底层文件系统的具体实现（如ext4、xfs等）

线程离开当前CPU的原因

  • 1. I/O等待：线程正在等待I/O操作（如磁盘读取或网络请求）完成。
  • 2. 时间片到期：线程的时间片用完，调度器将其替换为其他就绪线程。
  • 3. 主动让出 (Yield)：线程调用yield()或类似的系统调用，主动让出CPU。
  • 4. 锁或同步：线程被阻塞，等待锁、信号量或条件变量的释放。
  • 5. 系统调用：某些系统调用（如wait()）会导致线程阻塞。
  • 6. 中断：硬件中断可能会触发内核调度器，将CPU分配给其他线程。

虚拟内存和按需分页的优势

1. 虚拟内存的优势

  • 内存抽象：为每个进程提供独立的虚拟地址空间，避免了地址冲突。
  • 内存扩展：通过磁盘提供的“虚拟内存”扩展了物理内存。
  • 内存保护：隔离不同进程的内存，提供内存保护机制。
  • 高效的内存使用：通过共享内存页（如程序的只读代码段）减少物理内存的浪费。

2. 按需分页的优势

  • 延迟加载：仅在需要时加载页面，减少内存的初始加载时间。
  • 减少I/O开销：只有在页面缺失（page fault）时才进行磁盘I/O。
  • 更高的内存利用率：未使用的页面不加载到内存，节省了物理内存。
