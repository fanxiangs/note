## 第11章

云计算

云计算的兴起解决了性能领域的一些老问题，同时也产生了新问题。可以立即创建云环境并按需扩展，而无需构建和管理本地数据中心的典型开销。云还可以实现更好的部署粒度——服务器的一部分可以根据需要由不同的客户使用。然而，这也带来了挑战：虚拟化技术的性能开销以及与邻近租户的资源争用。

本章的学习目标是：

*   了解云计算架构及其性能影响。
    
*   了解虚拟化的类型：硬件、操作系统和轻量级硬件。
    
*   熟悉虚拟化内部结构，包括 I/O 代理的使用和调优技术。
    
*   了解每种虚拟化类型下不同工作负载的预期开销。
    
*   诊断主机和来宾的性能问题，了解工具的使用如何根据所使用的虚拟化而变化。
    

虽然整本书都适用于云性能分析，但本章重点讨论云特有的性能主题：虚拟机管理程序和虚拟化如何工作、资源控制如何应用于来宾以及主机和来宾的可观察性如何工作。云供应商通常会提供自己的自定义服务和 API，此处不予介绍：请参阅每个云供应商为其自己的服务集提供的文档。

本章由四个主要部分组成：

*   背景介绍了一般云计算架构及其性能影响。
    
*   硬件虚拟化，虚拟机管理程序将多个来宾操作系统实例作为虚拟机进行管理，每个实例都使用虚拟化设备运行自己的内核。本节以 Xen、KVM 和 Amazon Nitro 虚拟机管理程序为例。
    
*   操作系统虚拟化，单个内核管理系统，创建彼此隔离的虚拟操作系统实例。本节以Linux容器为例。
    
*   轻量级硬件虚拟化提供了一种两全其美的解决方案，其中轻量级硬件虚拟化实例使用专用内核运行，具有与容器类似的启动时间和密度优势。本部分使用 AWS Firecracker 作为示例管理程序。
    

虚拟化部分按照它们在云中广泛可用的时间排序。例如，Amazon Elastic Compute Cloud (EC2) 在 2006 年提供了硬件虚拟化实例，在 2017 年提供了操作系统虚拟化容器 (Amazon Fargate)，在 2019 年提供了轻量级虚拟机 (Amazon Firecracker)。

### 11.1 背景

云计算允许将计算资源作为服务提供，从服务器的一小部分扩展到多服务器系统。云的构建块取决于安装和配置的软件堆栈的数量。本章重点介绍以下云产品，它们都提供可能是的服务器实例：

*   硬件实例：也称为基础设施即服务 (IaaS)，使用硬件虚拟化提供。每个服务器实例都是一个虚拟机。
    
*   操作系统实例：用于提供轻量级实例，通常通过操作系统虚拟化。
    

这些可以一起称为服务器实例、云实例或仅称为实例。支持这些的云提供商的示例包括 Amazon Web Services (AWS)、Microsoft Azure 和 Google Cloud Platform (GCP)。还有其他类型的云原语，包括函数即服务 (FaaS)（请参阅第 11.5 节，其他类型）。

总结关键的云术语：云计算描述了实例的动态配置框架。一个或多个实例作为物理主机系统的来宾运行。来宾也称为租户，术语“多租户”用于描述它们在同一主机上运行。主机可能由运营公共云的云提供商管理，也可能由您的公司管理，仅供内部使用，作为私有云的一部分。一些公司构建了跨越公共云和私有云的混合云。[^1]云访客（租户）由其最终用户管理。

[^1]例如，Google Anthos 是一个应用管理平台，支持具有 GCP 实例的本地 Google Kubernetes Engine (GKE) 以及其他云。

对于硬件虚拟化，一种称为虚拟机管理程序（或虚拟机监视器，VMM）的技术创建和管理虚拟机实例，这些实例显示为专用计算机并允许安装整个操作系统和内核。

实例通常可以在几分钟或几秒钟内创建（和销毁）并立即投入生产使用。通常会提供云 API，以便其他程序可以自动执行此配置。

通过讨论各种与性能相关的主题，可以进一步理解云计算：实例类型、架构、容量规划、存储和多租户。以下各节总结了这些内容。

#### 11.1.1 实例类型

云提供商通常提供不同的实例类型和大小。

某些实例类型是通用的并且在资源之间保持平衡。其他类型可能针对特定资源进行优化：内存、CPU、磁盘等。例如，AWS 将类型分组为“家族”（缩写为字母）和世代（数字），目前提供：

*   m5：通用（平衡）
    
*   c5：计算优化
    
*   i3、d2：存储优化
    
*   r4、x1：内存优化
    
*   p1、g3、f1：加速计算（GPU、FPGA 等）
    

每个家庭都有不同的规模。例如，AWS m5 系列的范围从 m5.large（2 个 vCPU 和 8 GB 主内存）到 m5.24xlarge（24 个超大：96 个 vCPU 和 384 GB 主内存）。

各种尺寸通常具有相当一致的性价比，允许客户选择最适合其工作负载的尺寸。

一些提供商（例如 Google Cloud Platform）还提供可以选择资源量的自定义机器类型。

有了如此多的选项以及重新部署实例的便利性，实例类型就变得像一个可根据需要进行修改的可调参数。与公司多年来可能无法改变的选择和订购物理硬件的传统企业模式相比，这是一个巨大的进步。

#### 11.1.2 可扩展架构

企业环境历来使用垂直可扩展性方法来处理负载：构建更大的单一系统（大型机）。这种方法有其局限性。计算机的物理尺寸存在实际限制（可能受到电梯门或集装箱尺寸的限制），并且随着 CPU 数量的增加，CPU 缓存一致性的困难也越来越大电力和冷却。解决这些限制的方法是在许多（可能是小型）系统中扩展负载；这称为水平可扩展性。在企业中，它已用于计算机场和集群，特别是高性能计算（HPC，其使用早于云）。

云计算也基于水平可扩展性。图 11.1 显示了一个示例环境，其中包括负载均衡器、Web 服务器、应用程序服务器和数据库。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/11fig01.jpg)

图11.1 云架构：水平扩展

每个环境层都由一个或多个并行运行的服务器实例组成，并添加更多服务器实例来处理负载。可以单独添加实例，也可以将架构划分为垂直分区，其中将由数据库服务器、应用程序服务器和 Web 服务器组成的组作为单个单元添加。[^2]

[^2]例如，Shopify 将这些单元称为“pod”\[Denis 18\]。

该模型的一个挑战是传统数据库的部署，其中一个数据库实例必须是主要的。这些数据库（例如 MySQL）的数据可以在逻辑上分为称为分片的组，每个组都由自己的数据库（或主/辅助对）管理。分布式数据库架构（例如 Riak）动态处理并行执行，将负载分散到可用实例上。现在有专为在云上使用而设计的云原生数据库，包括 Cassandra、CockroachDB、Amazon Aurora 和 Amazon DynamoDB。

由于每台服务器实例的大小通常较小，例如 8 GB（在具有 512 GB 或更多 DRAM 的物理主机上），因此可以使用细粒度扩展来实现最佳性价比，而不是预先投资于大型系统可能大部分仍处于闲置状态。

#### 11.1.3 容量规划

本地服务器可能是一笔巨大的基础设施成本，无论是硬件还是可能持续数年的服务合同费用。新服务器投入生产也可能需要几个月的时间：花费时间在批准、等待部件可用性、运输、上架、安装和测试上。容量规划至关重要，因此可以购买适当大小的系统：太小意味着失败，太大则成本高昂（并且，根据服务合同，未来几年的成本可能会很高）。还需要进行容量规划，以便提前预测需求的增长，以便及时完成冗长的采购程序。

本地服务器和数据中心是企业环境的常态。云计算则非常不同。服务器实例价格低廉，并且几乎可以立即创建和销毁。公司可以根据实际负载根据需要增加使用的服务器实例的数量，而不是花时间规划可能需要的内容。这可以根据性能监控软件的指标，通过云 API 自动完成。小型企业或初创公司可以从一个小型实例发展到数千个实例，而无需像企业环境中那样进行详细的容量规划研究。[^3]

[^3]当公司扩展到数十万个实例时，事情可能会变得复杂，因为云提供商可能会因需求而暂时用完给定类型的可用实例。如果达到这种规模，请与您的客户代表讨论缓解这种情况的方法（例如，购买储备容量）。

对于成长中的初创公司来说，另一个需要考虑的因素是代码更改的速度。站点通常每周、每天甚至每天多次更新其生产代码。容量规划研究可能需要数周时间，并且由于它基于性能指标的快照，因此在完成时可能已经过时。这与运行商业软件的企业环境不同，商业软件每年的变化可能不会超过几次。

在云中执行的容量规划活动包括：

*   动态调整大小：自动添加和删除服务器实例
    
*   可扩展性测试：短期购买大型云环境，以测试可扩展性与合成负载（这是一项基准测试活动）
    

考虑到时间限制，还可以对可扩展性进行建模（类似于企业研究），以估计实际可扩展性如何低于理论上的水平。

##### 动态调整大小（自动缩放）

云供应商通常支持部署可随着负载增加而自动扩展的服务器实例组（例如，AWS 自动扩展组 (ASG)）。这还支持微服务架构，其中应用程序被分成更小的网络部分，这些部分可以根据需要单独扩展。

自动缩放可以解决快速响应负载变化的需求，但也存在过度配置的风险，如图 11.2 所示。例如，DoS 攻击可能表现为负载增加，从而触发服务器实例的昂贵增加。应用程序更改也存在类似的风险，这些更改会导致性能下降，需要更多实例来处理相同的负载。监控对于验证这些增加是否有意义非常重要。

作为其他示例，2012 年 12 月，Pinterest 报告称，通过在下班后自动关闭其云系统以响应流量负载，将成本从 54 美元/小时削减至 20 美元/小时 \[Hoff 12\]，而在 2018 年，Shopify 迁移到云并看到了巨大的增长。节省基础设施：从平均空闲时间为 61% 的服务器迁移到平均空闲时间为 19% 的云实例 \[Kwiatkowski 19\]。性能调整也可以立即节省成本，减少处理负载所需的实例数量。

一些云架构（参见第 11.3 节，操作系统虚拟化）可以使用称为突发的策略立即动态分配更多 CPU 资源（如果可用）。这无需额外成本即可提供，旨在通过提供缓冲区来帮助防止过度配置，在此期间可以检查增加的负载以确定其是否真实且可能继续。如果是这样，可以配置更多实例，从而保证资源的继续使用。

这些技术中的任何一种都应该比企业环境高效得多，尤其是那些选择固定大小来处理服务器生命周期内预期峰值负载的技术：此类服务器可能大部分时间都处于空闲状态。

#### 11.1.4 存储

云实例需要存储操作系统、应用软件和临时文件。在 Linux 系统上，这是根卷和其他卷。这可以由本地物理存储或网络存储来提供。该实例存储是易失性的，并且会在实例被销毁时被销毁（称为临时驱动器）。对于持久存储，通常使用独立的服务，它为实例提供存储作为：

*   文件存储：例如，NFS 上的文件
    
*   块存储：例如 iSCSI 上的块
    
*   对象存储：通过 API，通常基于 HTTP
    

它们通过网络运行，并且网络基础设施和存储设备都与其他租户共享。由于这些原因，尽管可以通过云提供商使用资源控制来提高性能一致性，但与本地磁盘相比，性能的可预测性要低得多。

云提供商通常为此提供自己的服务。例如，Amazon 提供 Amazon Elastic File System (EFS) 作为文件存储、Amazon Elastic Block Store (EBS) 作为块存储以及 Amazon Simple Storage Service (S3) 作为对象存储。

#### 11.1.5 多租户

Unix 是一个多任务操作系统，旨在处理访问相同资源的多个用户和进程。 Linux 后来添加的功能提供了资源限制和控制，以更公平地共享这些资源，并提供可观察性，以便在出现涉及资源争用的性能问题时识别和量化。

云计算的不同之处在于整个操作系统实例可以共存于同一物理系统上。每个来宾都有自己独立的操作系统：来宾（通常为[^5]）无法观察同一主机上其他来宾的用户和进程（这将被视为信息泄漏），即使它们共享相同的物理资源。

[^5] Linux 容器可以通过命名空间和 cgroup 以不同的方式组装。应该可以创建彼此共享进程命名空间的容器，这可以用于可以调试其他容器进程的自省（“sidecar”）容器。在 Kubernetes 中，主要的抽象是 Pod，它共享一个网络命名空间。

由于资源在租户之间共享，因此吵闹的邻居可能会导致性能问题。例如，同一主机上的另一个来宾可能会在峰值负载期间执行完整数据库转储，从而干扰您的磁盘和网络 I/O。更糟糕的是，邻居可能会通过执行微基准来评估云提供商，这些微基准故意使资源饱和以找到其极限。

这个问题有一些解决方案。多租户效应可以通过资源管理来控制：设置提供性能隔离（也称为资源隔离）的操作系统资源控制。这是对系统资源的使用施加每个租户限制或优先级的地方：CPU、内存、磁盘或文件系统 I/O 以及网络吞吐量。

除了限制资源使用之外，能够观察多租户争用还可以帮助云运营商调整限制并更好地平衡可用主机上的租户。可观察性的程度取决于虚拟化类型。

#### 11.1.6 编排（Kubernetes）

许多公司使用在自己的裸机或云系统上运行的编排软件来运行自己的私有云。最流行的此类软件是 Kubernetes（缩写为 k8s），最初由 Google 创建。 Kubernetes，希腊语“Helmsman”的意思，是一个开源系统，使用容器（通常是 Docker 容器，尽管任何实现开放容器接口的运行时也可以工作，例如 containerd）管理应用程序部署 \[Kubernetes 20b\]。公共云提供商还创建了 Kubernetes 服务来简化这些提供商的部署，包括 Google Kubernetes Engine (GKE)、Amazon Elastic Kubernetes Service (Amazon EKS) 和 Microsoft Azure Kubernetes Service (AKS)。

Kubernetes 将容器部署为称为 Pod 的共置组，容器可以在其中共享资源并在本地（本地主机）相互通信。每个 Pod 都有自己的 IP 地址，可用于与其他 Pod 进行通信（通过网络）。 Kubernetes 服务是一组 Pod 提供的端点的抽象，具有包括 IP 地址的元数据，并且是这些端点的持久且稳定的接口，而 Pod 本身可以添加和删除，从而允许将它们视为一次性的。 Kubernetes 服务支持微服务架构。 Kubernetes 包括自动缩放策略，例如“Horizontal Pod Autoscaler”，可以根据目标资源利用率或其他指标缩放 Pod 副本。在 Kubernetes 中，物理机称为 Node，一组 Node 连接到同一个 Kubernetes API 服务器就属于一个 Kubernetes 集群。

Kubernetes 中的性能挑战包括调度（在集群上的何处运行容器以最大限度地提高性能）和网络性能，因为需要使用额外的组件来实现容器网络和负载平衡。

对于调度，Kubernetes 会考虑 CPU 和内存请求和限制，以及节点污点（其中节点被标记为从调度中排除）和标签选择器（自定义元数据）等元数据。 Kubernetes 目前不限制块 I/O（将来可能会使用 blkio cgroup 添加对此的支持 \[Xu 20\]），这使得磁盘争用成为性能问题的可能根源。

对于网络，Kubernetes 允许使用不同的网络组件，确定使用哪个组件是确保最大性能的重要活动。容器网络可以通过插件容器网络接口（CNI）软件来实现；示例CNI软件包括基于netfilter或iptables的Calico和基于BPF的Cilium。两者都是开源的 \[Calico 20\]\[Cilium 20b\]。为了负载平衡，Cilium 还提供了 kube-proxy 的 BPF 替代品 \[Borkmann 19\]。

### 11.2 硬件虚拟化

硬件虚拟化创建了一个可以运行整个操作系统（包括其自己的内核）的虚拟机 (VM)。 VM 由虚拟机管理程序创建，也称为虚拟机管理器 (VMM)。虚拟机管理程序的常见分类将其标识为类型 1 或类型 2 \[Goldberg 73\]，它们是：

*   类型 1 直接在处理器上执行。管理程序管理可以由可以创建和启动新来宾的特权来宾执行。类型 1 也称为本机虚拟机管理程序或裸机虚拟机管理程序。该虚拟机管理程序包含其自己的用于来宾虚拟机的 CPU 调度程序。一个流行的例子是 Xen 虚拟机管理程序。
    
*   类型 2 在主机操作系统中执行，该主机操作系统具有管理虚拟机管理程序和启动新来宾的权限。对于这种类型，系统启动传统操作系统，然后运行虚拟机管理程序。该虚拟机管理程序由主机内核 CPU 调度程序进行调度，而来宾在主机上显示为进程。
    

尽管您可能仍然会遇到术语“类型 1”和“类型 2”，但随着虚拟机管理程序技术的进步，这种分类不再严格适用 \[Liguori, 07\] — 通过使用内核模块，类型 2 已成为类型 1 型，以便虚拟机管理程序的某些部分可以直接访问硬件。图 11.5 显示了更实用的分类，说明了我命名为配置 A 和 B \[Gregg 19\] 的两种常见配置。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/11fig05.jpg)

图 11.5 常见的虚拟机管理程序配置

这些配置是：

*   配置 A：也称为本机虚拟机管理程序或裸机虚拟机管理程序。虚拟机管理程序软件直接在处理器上运行，创建用于运行来宾虚拟机的域，并将虚拟来宾 CPU 调度到真实 CPU 上。特权域（图 11.5 中的数字 0）可以管理其他域。一个流行的例子是 Xen 虚拟机管理程序。
    
*   配置B：管理程序软件由主机操作系统内核执行，并且可以由内核级模块和用户级进程组成。主机操作系统有权管理虚拟机管理程序，其内核调度虚拟机 CPU 以及主机上的其他进程。通过使用内核模块，此配置还提供对硬件的直接访问。一个流行的例子是 KVM 管理程序。
    

两种配置都可能涉及在域 0 (Xen) 或主机操作系统 (KVM) 中运行 I/O 代理（例如，使用 QEMU 软件），以服务来宾 I/O。这增加了 I/O 的开销，多年来已经通过添加共享内存传输和其他技术进行了优化。

最初的硬件虚拟机管理程序由 VMware 于 1998 年首创，使用二进制转换来执行完整的硬件虚拟化 \[VMware 07\]。这涉及在执行之前重写特权指令，例如系统调用和页表操作。非特权指令可以直接在处理器上运行。这提供了一个由虚拟化硬件组件组成的完整虚拟系统，可以在其上安装未经修改的操作系统。对于服务器整合带来的节省来说，这种高性能开销通常是可以接受的。

此后，这一问题得到了改进：

*   处理器虚拟化支持：AMD-V 和 Intel VT-x 扩展于 2005 年至 2006 年推出，旨在为处理器的 VM 操作提供更快的硬件支持。这些扩展提高了虚拟化特权指令和 MMU 的速度。
    
*   半虚拟化（paravirt 或 PV）：提供一个虚拟系统，其中包括一个供来宾操作系统使用的接口，以便高效使用主机资源（通过超级调用），而无需对所有组件进行完全虚拟化。例如，启用定时器通常涉及多个必须由管理程序模拟的特权指令。这可以简化为单个超级调用，供半虚拟化来宾使用，以便管理程序更有效地进行处理。为了进一步提高效率，Xen 虚拟机管理程序将这些超级调用批处理为多重调用。半虚拟化可以包括由访客使用半虚拟网络设备驱动程序来更有效地将数据包传递到主机中的物理网络接口。虽然性能得到了提高，但这依赖于来宾操作系统对半虚拟化的支持（Windows 历史上从未提供过）。
    
*   设备硬件支持：为了进一步优化虚拟机性能，处理器以外的硬件设备已经添加了虚拟机支持。这包括网络和存储设备的单根 I/O 虚拟化 (SR-IOV)，它允许来宾虚拟机直接访问硬件。这需要驱动程序支持（示例驱动程序包括 ixgbe、ena、hv\_netvsc 和 nvme）。
    

多年来，Xen 不断发展并提高了其性能。现代 Xen VM 通常以硬件 VM 模式 (HVM) 启动，然后使用支持 HVM 的 PV 驱动程序来提高性能：一种称为 PVHVM 的配置。通过完全依赖某些驱动程序的硬件虚拟化（例如网络和存储设备的 SR-IOV），可以进一步改进这一点。

#### 11.2.1 实施

硬件虚拟化有许多不同的实现方式，其中一些已经提到过（Xen 和 KVM）。例子有：

*   VMware ESX：VMware ESX于2001年首次发布，是一款用于服务器整合的企业产品，是VMware vSphere云计算产品的关键组件。它的虚拟机管理程序是一个运行在裸机上的微内核，第一个虚拟机称为服务控制台，它可以管理虚拟机管理程序和新的虚拟机。
    
*   Xen：Xen 于 2003 年首次发布，最初是剑桥大学的一个研究项目，后来被 Citrix 收购。 Xen 是一个 Type 1 虚拟机管理程序，可运行半虚拟化来宾以获得高性能；后来添加了对硬件辅助来宾的支持，以实现未修改的操作系统支持 (Windows)。虚拟机称为域，其中权限最高的是 dom0，从该域管理虚拟机管理程序并启动新域。 Xen 是开源的，可以从 Linux 启动。 Amazon Elastic Compute Cloud (EC2) 以前基于 Xen。
    
*   Hyper-V：Hyper-V 随 Windows Server 2008 一起发布，是一种类型 1 虚拟机管理程序，可为执行来宾操作系统创建分区。 Microsoft Azure 公共云可能正在运行 Hyper-V 的定制版本（具体细节尚未公开）。
    
*   KVM：这是由 Qumranet 开发的，该公司是一家初创公司，于 2008 年被 Red Hat 收购。KVM 是一个 2 类虚拟机管理程序，作为内核模块执行。它支持硬件辅助扩展，并且为了获得高性能，在来宾操作系统支持的某些设备上使用半虚拟化。为了创建完整的硬件辅助虚拟机实例，它与称为 QEMU（快速仿真器）的用户进程配对，QEMU 是一个可以创建和管理虚拟机的 VMM（管理程序）。 QEMU 最初是一个使用二进制翻译的高质量开源 Type 2 虚拟机管理程序，由 Fabrice Bellard 编写。 KVM 是开源的，被 Google 用于 Google 计算引擎 \[Google 20c\]。
    
*   Nitro：该虚拟机管理程序由 AWS 于 2017 年推出，使用基于 KVM 的部件，并为所有主要资源提供硬件支持：处理器、网络、存储、中断和计时器 \[Gregg 17e\]。不使用 QEMU 代理。 Nitro 为来宾虚拟机提供近乎裸机的性能。
    

以下部分描述与硬件虚拟化相关的性能主题：开销、资源控制和可观察性。这些因实现及其配置而异。

#### 11.2.2 开销

了解何时以及何时不需要虚拟化产生性能开销对于调查云性能问题非常重要。

硬件虚拟化可以通过多种方式实现。资源访问可能需要管理程序进行代理和转换，从而增加开销，或者它可以使用基于硬件的技术来避免这些开销。以下部分总结了 CPU 执行、内存映射、内存大小、执行 I/O 以及来自其他租户的争用的性能开销。

##### 中央处理器

一般来说，来宾应用程序直接在处理器上执行，而受 CPU 限制的应用程序可能会体验到与裸机系统几乎相同的性能。在进行特权处理器调用、访问硬件和映射主内存时可能会遇到 CPU 开销，具体取决于管理程序如何处理它们。下面描述了不同硬件虚拟化类型如何处理 CPU 指令：

*   二进制翻译：识别并翻译对物理资源进行操作的来宾内核指令。在硬件辅助虚拟化可用之前，就使用了二进制翻译。由于没有虚拟化的硬件支持，VMware 使用的方案涉及在处理器环 0 中运行虚拟机监视器 (VMM)，并将来宾内核移至之前未使用的环 1（应用程序在环 3 中运行，大多数处理器提供四个环；保护环在第 3 章“操作系统”、第 3.2.2 节“内核和用户模式”中介绍。由于某些来宾内核指令假定它们在环 0 中运行，因此为了从环 1 执行，它们需要进行转换，调用 VMM，以便可以应用虚拟化。此转换在运行时执行，会消耗大量 CPU 开销。
    
*   半虚拟化：来宾操作系统中必须虚拟化的指令被对虚拟机管理程序的超级调用所取代。如果修改来宾操作系统以优化超级调用，使其知道自己正在虚拟化硬件上运行，则可以提高性能。
    
*   硬件辅助：在硬件上操作的未修改的来宾内核指令由虚拟机管理程序处理，该虚拟机管理程序在低于 0 的环级别运行 VMM。来宾内核特权指令不是翻译二进制指令，而是强制捕获到更高特权的 VMM ，然后可以模拟支持虚拟化的权限 \[Adams 06\]。
    

硬件辅助虚拟化通常是首选，具体取决于实施和工作负载，而半虚拟化则用于提高某些工作负载（尤其是 I/O）的性能（如果来宾操作系统支持）。

作为实现差异的一个例子，VMware 的二进制转换模型多年来一直在大力优化，正如他们在 2007 年所写的 \[VMware 07\]：

由于管理程序到来宾转换的开销较高以及严格的编程模型，VMware 的二进制转换方法目前在大多数情况下优于第一代硬件辅助实现。第一代实现中严格的编程模型在管理虚拟机管理程序到来宾转换的频率或成本方面几乎没有留下软件灵活性的空间。

来宾和虚拟机管理程序之间的转换速率以及虚拟机管理程序中花费的时间可以作为 CPU 开销的指标进行研究。这些事件通常称为来宾退出，因为发生这种情况时虚拟 CPU 必须停止在来宾内部执行。图 11.6 显示了与 KVM 内部来宾退出相关的 CPU 开销。

##### 内存映射

如第 7 章“内存”中所述，操作系统与 MMU 一起创建从虚拟内存到物理内存的页面映射，并将它们缓存在 TLB 中以提高性能。对于虚拟化，将新的内存页面（页面错误）从客户机映射到硬件涉及两个步骤：

1.  虚拟到客户物理转换，由客户内核执行
    
2.  来宾物理到主机物理（实际）转换，由虚拟机管理程序 VMM 执行
    

然后，从客户虚拟机到主机物理机的映射可以缓存在 TLB 中，以便后续访问可以正常速度运行，而不需要额外的转换。现代处理器支持 MMU 虚拟化，因此可以仅在硬件中更快地调用已离开 TLB 的映射（页面遍历），而无需调用虚拟机管理程序。支持此功能的功能在 Intel 上称为扩展页表 (EPT)，在 AMD 上称为嵌套页表 (NPT) \[Milewski 11\]。

如果没有 EPT/NPT，提高性能的另一种方法是维护来宾虚拟机到主机物理映射的影子页表，这些映射由虚拟机管理程序管理，然后在来宾执行期间通过覆盖来宾的 CR3 寄存器进行访问。通过这种策略，来宾内核可以维护自己的页表，正常情况下，该页表从来宾虚拟映射到来宾物理。虚拟机管理程序拦截对这些页表的更改，并在影子页中创建到主机物理页的等效映射。然后，在来宾执行期间，管理程序会覆盖 CR3 寄存器以指向影子页面。

##### 内存大小

与操作系统虚拟化不同，使用硬件虚拟化时会有一些额外的内存消耗。每个来宾运行自己的内核，该内核消耗少量内存。存储架构还可能导致双重缓存，即来宾和主机都缓存相同的数据。 KVM 风格的虚拟机管理程序还为每个 VM 运行一个 VMM 进程，例如 QEMU，它本身会消耗一些主内存。

##### 输入/输出

从历史上看，I/O 是硬件虚拟化的最大开销来源。这是因为每个设备 I/O 都必须由虚拟机管理程序进行转换。对于高频 I/O，例如 10 Gbit/s 网络，每个 I/O（数据包）的少量开销可能会导致整体性能显着下降。人们已经创建了一些技术来减轻这些 I/O 开销，最终通过硬件支持完全消除这些开销。此类硬件支持包括 I/O MMU 虚拟化（AMD-Vi 和 Intel VT-d）。

提高 I/O 性能的一种方法是使用半虚拟化驱动程序，它可以合并 I/O 并执行更少的设备中断，以减少管理程序开销。

另一种技术是 PCI 直通，它将 PCI 设备直接分配给来宾，因此可以像在裸机系统上一样使用它。 PCI 直通可以提供可用选项的最佳性能，但它会降低配置具有多个租户的系统时的灵活性，因为某些设备现在由来宾拥有并且无法共享。这也可能使实时迁移变得复杂\[Xen 19\]。

有一些技术可以通过虚拟化来提高使用 PCI 设备的灵活性，包括单根 I/O 虚拟化（SR-IOV，前面提到过）和多根 I/O 虚拟化（MR-IOV）。这些术语指的是公开的根复合体 PCI 拓扑的数量，以不同的方式提供硬件虚拟化。 Amazon EC2 云一直采用这些技术来加速网络和存储 I/O，这些技术默认与 Nitro 虚拟机管理程序 \[Gregg 17e\] 一起使用。

GK 是“来宾内核”，BE 是“后端”。虚线箭头表示控制路径，其中组件以同步或异步方式相互通知更多数据已准备好传输。在某些情况下，数据路径（实线箭头）可以通过共享存储器和环形缓冲区来实现。 Nitro 未显示控制路径，因为它使用相同的数据路径直接访问硬件。

配置 Xen 和 KVM 有不同的方法，此处未显示。该图显示了它们使用 I/O 代理进程（通常是 QEMU 软件），这些进程是为每个来宾 VM 创建的。但它们也可以配置为使用 SR-IOV，允许来宾 VM 直接访问硬件（图 11.7 中未显示 Xen 或 KVM）。 Nitro 需要此类硬件支持，从而无需 I/O 代理。

Xen 使用设备通道（dom0 和来宾域 (domU) 之间的异步共享内存传输）提高其 I/O 性能。这避免了在域之间传递 I/O 数据时创建额外的 I/O 数据副本所带来的 CPU 和总线开销。它还可以使用单独的域来执行 I/O，如第 11.2.3 节“资源控制”中所述。

I/O 路径（控制和数据）中的步骤数量对于性能至关重要：越少越好。 2006 年，KVM 开发人员将 Xen 等特权客户系统与 KVM 进行了比较，发现 KVM 可以使用一半的步骤执行 I/O（五比十，尽管测试是在没有半虚拟化的情况下执行的，因此并不反映大多数现代配置） \[库姆拉网 06\]。

由于 Nitro 虚拟机管理程序消除了额外的 I/O 步骤，我希望所有寻求最高性能的大型云提供商都会效仿，使用硬件支持来消除 I/O 代理。

##### 多租户争用

根据虚拟机管理程序配置以及租户之间共享的 CPU 和 CPU 缓存数量，可能会出现其他租户造成的 CPU 被盗时间和 CPU 缓存污染，从而降低性能。对于容器来说，这通常是一个比虚拟机更大的问题，因为容器促进这种共享以支持 CPU 爆发。

执行 I/O 的其他租户可能会导致执行中断，具体取决于虚拟机管理程序配置。

资源争用可以通过资源控制来管理。

#### 11.2.3 资源控制

作为来宾配置的一部分，CPU 和主内存通常配置有资源限制。管理程序软件还可以提供对网络和磁盘 I/O 的资源控制。

对于类似 KVM 的虚拟机管理程序，主机操作系统最终控制物理资源，除了虚拟机管理程序提供的控制之外，操作系统提供的资源控制也可以应用于来宾。对于 Linux，这意味着 cgroup、任务集和其他资源控制。有关主机操作系统可用的资源控制的更多信息，请参见第 11.3 节“操作系统虚拟化”。以下部分作为示例描述了 Xen 和 KVM 虚拟机管理程序的资源控制。

##### 中央处理器

CPU 资源通常作为虚拟 CPU (vCPU) 分配给来宾。然后，这些由虚拟机管理程序进行调度。分配的 vCPU 数量粗略地限制了 CPU 资源的使用。

对于 Xen，可以通过管理程序 CPU 调度程序为来宾应用细粒度的 CPU 配额。调度程序包括\[Cherkasova 07\]\[Matthews 08\]：

*   借用虚拟时间（BVT）：基于虚拟时间分配的公平共享调度器，可以提前借用，为实时和交互式应用程序提供低延迟执行
    
*   简单最早截止时间优先（SEDF）：允许配置运行时保证的实时调度程序，调度程序优先考虑最早截止时间
    
*   基于信用：支持CPU使用的优先级（权重）和上限，以及跨多个CPU的负载平衡
    

对于 KVM，主机操作系统可以应用细粒度的 CPU 配额，例如在使用前面描述的主机内核公平共享调度程序时。在 Linux 上，可以使用 cgroup CPU 带宽控制来应用此功能。

这两种技术在尊重客人优先事项方面都存在局限性。来宾的 CPU 使用率通常对虚拟机管理程序来说是不透明的，并且来宾内核线程的优先级通常无法被看到或尊重。例如，一个来宾中的低优先级日志轮换守护进程可能与另一来宾中的关键应用程序服务器具有相同的管理程序优先级。

对于 Xen，CPU 资源使用可能会因高 I/O 工作负载而进一步复杂化，这些工作负载会消耗 dom0 中的额外 CPU 资源。仅来宾域中的后端驱动程序和 I/O 代理可能会消耗超过其 CPU 分配的资源，但不计入 \[Cherkasova 05\]。解决方案是创建隔离的驱动程序域 (IDD)，它将 I/O 服务分开以实现安全、性能隔离和计费。如图 11.8 所示。

Xen 中使用的最新技术是存根域，它运行迷你操作系统。

##### CPU缓存

除了 vCPU 的分配之外，还可以使用 Intel 缓存分配技术 (CAT) 来控制 CPU 缓存的使用。它允许 LLC 在来宾之间进行分区，并共享分区。虽然这可以防止某个来宾污染另一个来宾的缓存，但它也会通过限制缓存使用来损害性能。

##### 内存容量

内存限制是作为来宾配置的一部分施加的，来宾只能看到设置的内存量。然后，来宾内核执行其自己的操作（分页、交换）以保持在其限制内。

为了提高静态配置的灵活性，VMware 开发了一个气球驱动程序 \[Waldspurger 02\]，它能够通过“膨胀”内部的气球模块来减少正在运行的虚拟机消耗的内存，该模块会消耗虚拟机内存。然后，虚拟机管理程序会回收该内存以供其他来宾使用。气球也可以放气，将内存返回给客户内核以供使用。在此过程中，客户内核执行其正常的内存管理例程以释放内存（例如，分页）。 VMware、Xen 和 KVM 都支持气球驱动程序。

当气球驱动程序正在使用时（要从访客处检查，请在 dmesg(1) 的输出中搜索“balloon”），我会留意它们可能导致的性能问题。

##### 文件系统容量

主机为来宾提供虚拟磁盘卷。对于类似 KVM 的虚拟机管理程序，这些可能是由操作系统创建并相应调整大小的软件卷。例如，ZFS 文件系统可以创建所需大小的虚拟卷。

##### 设备输入/输出

硬件虚拟化软件的资源控制历来侧重于控制CPU使用率，这可以间接控制I/O使用率。

网络吞吐量可能会受到外部专用设备的限制，或者在类似 KVM 的虚拟机管理程序的情况下，会受到主机内核功能的限制。例如，Linux 具有来自 cgroup 以及不同 qdisc 的网络带宽控制，可应用于访客网络接口。

对 Xen 的网络性能隔离进行了研究，得出以下结论 \[Adamczyk 12\]：

...当考虑网络虚拟化时，Xen 的弱点是缺乏适当的性能隔离。

\[Adamczyk 12\]的作者还提出了一种Xen网络I/O调度的解决方案，它添加了网络I/O优先级和速率的可调参数。如果您使用的是 Xen，请检查是否已提供该技术或类似技术。

对于具有完整硬件支持的虚拟机管理程序（例如 Nitro），I/O 限制可能由硬件或外部设备支持。在 Amazon EC2 云中，网络连接设备的网络 I/O 和磁盘 I/O 使用外部系统限制在配额内。

#### 11.2.4 可观察性

虚拟化系统上可观察的内容取决于虚拟机管理程序以及可观察工具的启动位置。一般来说：

*   从特权来宾 (Xen) 或主机 (KVM)：所有物理资源都应该可以使用前面章节中介绍的标准操作系统工具进行观察。可以通过分析 I/O 代理（如果正在使用）来观察来宾 I/O。虚拟机管理程序应提供每个来宾的资源使用统计信息。来宾内部结构（包括其进程）无法直接观察。如果设备使用直通或 SR-IOV，某些 I/O 可能无法观察到。
    
*   从硬件支持的主机 (Nitro)：使用 SR-IOV 可能会使从虚拟机管理程序观察设备 I/O 变得更加困难，因为来宾直接访问硬件，而不是通过代理或主机内核。 （亚马逊实际上如何在 Nitro 上实现虚拟机管理程序可观察性并不是公开知识。）
    
*   从来宾处：可以看到虚拟化资源及其对来宾的使用情况，并推断出物理问题。由于VM有自己的专用内核，因此可以分析内核内部结构，并且内核跟踪工具（包括基于BPF的工具）都可以工作。
    

从特权来宾或主机（Xen 或 KVM 管理程序），通常可以在较高级别观察物理资源使用情况：利用率、饱和度、错误、IOPS、吞吐量、I/O 类型。这些因素通常可以按每个客人来表示，以便可以快速识别大量用户。无法直接观察哪些来宾进程正在执行 I/O 及其应用程序调用堆栈的详细信息。可以通过登录来宾操作系统（前提是已授权和配置这样做的方法，例如 SSH）并使用来宾操作系统提供的可观察性工具来观察它们。

当使用直通或 SR-IOV 时，客户机可能会直接向硬件进行 I/O 调用。这可能会绕过虚拟机管理程序中的 I/O 路径以及它们通常收集的统计信息。结果是 I/O 对虚拟机管理程序来说变得不可见，并且不会出现在 iostat(1) 或其他工具中。一种可能的解决方法是使用 PMC 检查与 I/O 相关的计数器，并以此方式推断 I/O。

为了确定来宾性能问题的根本原因，云运营商可能需要登录虚拟机管理程序和来宾，并从两者执行可观察性工具。由于涉及的步骤，跟踪 I/O 路径变得复杂，并且还可能包括对虚拟机管理程序内部结构和 I/O 代理（如果使用）的分析。

从访客看来，可能根本无法观察到物理资源的使用情况。这可能会诱使来宾客户将神秘的性能问题归咎于吵闹的邻居造成的资源争用。为了让云客户安心（并减少支持请求），有关物理资源使用情况（已编辑）的信息可以通过其他方式提供，包括 SNMP 或云 API。

为了使容器性能更易于观察和理解，有各种监控解决方案可以呈现图形、仪表板和有向图来显示您的容器环境。此类软件包括 Google cAdvisor \[Google 20d\] 和 Cilium Hubble \[Cilium 19\]（两者都是开源的）。

以下部分演示了可以在不同位置使用的原始可观察性工具，并描述了分析性能的策略。 Xen 和 KVM 用于演示虚拟化软件可能提供的信息类型（不包括 Nitro，因为它是 Amazon 专有的）。

##### 11.2.4.3 策略

前面的章节介绍了物理系统资源的分析技术，物理系统的管理员可以遵循这些技术来查找瓶颈和错误。还可以检查对客人施加的资源控制，以查看客人是否始终处于限制状态，并应通知并鼓励客人升级。在不登录访客的情况下，管理员无法识别更多内容，这对于任何认真的性能调查可能是必要的。[^6]

[^6]一位评论者指出了另一种可能的技术（请注意，这不是建议）：可以分析访客存储的快照（前提是未加密）。例如，给定先前磁盘 I/O 地址的日志，文件系统状态的快照可用于确定哪些文件可能已被访问。

对于来宾来说，可以应用前面章节中介绍的分析资源的工具和策略，请记住，本例中的资源通常是虚拟的。由于虚拟机管理程序看不见的资源控制或来自其他租户的争用，某些资源可能不会达到极限。理想情况下，云软件或供应商为客户提供一种检查编辑后的物理资源使用情况的方法，以便他们可以自行进一步调查性能问题。如果不是，则可能会因 I/O 和 CPU 调度延迟的增加而推断出争用和限制。这种延迟可以在系统调用层或来宾内核中测量。

我用来识别来自来宾的磁盘和网络资源争用的策略是仔细分析 I/O 模式。这可能涉及记录 biosnoop(8) 的输出（请参阅前面的示例），然后检查 I/O 序列以查看是否存在任何延迟异常值，以及它们是否是由其大小（大 I/O 是较慢）、它们的访问模式（例如，在写入刷新之后排队读取），或者两者都不是，在这种情况下，可能是物理争用或设备问题。

### 11.3 操作系统虚拟化

操作系统虚拟化将操作系统划分为 Linux 称为容器的实例，这些实例就像单独的访客服务器一样，可以独立于主机进行管理和重新启动。它们为云客户提供小型、高效、快速启动的实例，为云运营商提供高密度服务器。操作系统虚拟化客户如图 11.9 所示。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/11fig09.jpg)

图11.9 操作系统虚拟化

这种方法起源于 Unix chroot(8) 命令，该命令将进程隔离到 Unix 全局文件系统的子树（它更改进程所看到的顶级目录“/”，以指向其他位置） 。 1998 年，FreeBSD 将其进一步发展为 FreeBSD 监狱，提供充当自己的服务器的安全隔间。 2005 年，Solaris 10 包含一个名为 Solaris Zones 的版本，具有各种资源控制。与此同时，Linux 一直在部分添加进程隔离功能，2002 年首次为 Linux 2.4.19 添加了命名空间，2008 年首次为 Linux 2.6.24 添加了控制组 (cgroup) \[Corbet 07a\]\[Corbet 07b\]\[Linux 20m\] 。命名空间和 cgroup 组合起来创建容器，容器通常也使用 seccomp-bpf 来控制系统调用访问。

与硬件虚拟化技术的一个主要区别是只有一个内核在运行。以下是容器相对于硬件虚拟机的性能优势（第 11.2 节，硬件虚拟化）：

*   快速初始化时间：通常以毫秒为单位测量。
    
*   来宾可以将内存完全用于应用程序（无需额外的内核）。
    
*   有统一的文件系统缓存——这可以避免主机和来宾之间的双重缓存场景。
    
*   对资源共享 (cgroup) 进行更细粒度的控制。
    
*   对于主机操作员：提高了性能可观察性，因为来宾进程及其交互是直接可见的。
    
*   容器可以共享常见文件的内存页面，释放页面缓存中的空间并提高 CPU 缓存命中率。
    
*   CPU是真正的CPU；自适应互斥锁的假设仍然有效。
    

并且也有缺点：

*   对内核资源（锁、缓存、缓冲区、队列）的争用增加。
    
*   对于来宾：性能可观察性降低，因为通常无法分析内核。
    
*   任何内核恐慌都会影响所有来宾。
    
*   来宾无法运行自定义内核模块。
    
*   来宾无法运行运行时间较长的 PGO 内核（请参阅第 3.5.1 节，PGO 内核）。
    
*   来宾无法运行不同的内核版本或内核。[^7]
    

[^7] 有些技术可以模拟不同的系统调用接口，以便不同的操作系统可以在内核下运行，但这在实践中会对性能产生影响。例如，此类仿真通常仅提供一组基本的系统调用功能，其中高级性能功能返回 ENOTSUP（不支持错误）。

同时考虑前两个缺点：从虚拟机迁移到容器的来宾更有可能遇到内核争用问题，同时也失去分析这些问题的能力。他们将更加依赖主机运营商进行此类分析。

容器的一个非性能缺点是它们被认为不太安全，因为它们共享内核。

所有这些缺点都可以通过轻量级虚拟化解决，如第 11.4 节“轻量级虚拟化”中所述，尽管以牺牲一些优点为代价。

#### 11.3.2 开销

容器执行的开销应该是轻量级的：应用程序 CPU 和内存使用应该体验裸机性能，尽管由于文件系统和网络路径中的层，内核中可能存在一些额外的 I/O 调用。最大的性能问题是由多租户争用引起的，因为容器促进了内核和物理资源的大量共享。以下部分总结了 CPU 执行、内存使用、执行 I/O 以及来自其他租户的争用的性能开销。

##### 中央处理器

当容器线程在用户模式下运行时，没有直接的 CPU 开销：线程直接在 CPU 上运行，直到它们屈服或被抢占。在 Linux 上，在命名空间和 cgroup 中运行进程也不会产生额外的 CPU 开销：无论是否使用容器，所有进程都已在默认命名空间和 cgroup 集中运行。

CPU 性能很可能因与其他租户的争用而降低（请参阅后面的多租户争用部分）。

对于 Kubernetes 等编排器，额外的网络组件可能会增加一些 CPU 开销来处理网络数据包（例如，有许多服务（数千个）；由于大量的 iptables 规则集，kube-proxy 在必须处理大型 iptables 规则集时会遇到第一个数据包开销使用 Kubernetes 服务。可以通过用 BPF 替换 kube-proxy 来克服这种开销 \[Borkmann 20\]）。

##### 内存映射

内存映射、加载和存储应该在没有开销的情况下执行。

##### 内存大小

应用程序可以使用为容器分配的全部内存。与此相比，硬件虚拟机为每个租户运行一个内核，每个内核占用少量主内存。

通用容器配置（使用 OverlayFS）允许在访问同一文件的容器之间共享页面缓存。与虚拟机相比，这可以释放一些内存，虚拟机在内存中复制公共文件（例如系统库）。

##### 输入/输出

*   文件系统 I/O：例如，overlayfs
    
*   网络 I/O：例如桥接网络

这有多重要取决于工作负载及其 IOPS 速率。对于低 IOPS 服务器（例如，<1000 IOPS），其开销应该可以忽略不计。

##### 多租户争用

其他正在运行的租户的存在可能会导致资源争用和中断，从而损害性能，包括：

*   CPU 缓存的命中率可能较低，因为其他租户正在消耗和驱逐条目。对于某些处理器和内核配置，上下文切换到其他容器线程甚至可能刷新 L1 缓存。[^10]
    
    [^10]例如，2020 年 6 月，Linus Torvalds 拒绝了一个允许进程选择 L1 数据缓存刷新的内核补丁 \[Torvalds 20b\]。该补丁是云环境的安全预防措施，但由于担心不必要的情况下的性能成本而被拒绝。虽然没有包含在 Linux 主线中，但如果这个补丁在云中的某些 Linux 发行版中运行，我不会感到惊讶。
    
*   由于其他租户的使用，以及上下文切换时的刷新（如果使用 PCID，则可以避免），TLB 缓存也可能具有较低的命中率。
    
*   对于执行中断服务例程的其他租户设备（例如网络 I/O），CPU 执行可能会被短暂中断。
    
*   内核执行可能会遇到缓冲区、缓存、队列和锁的额外争用，因为多租户容器系统可能会将其负载增加一个数量级或更多。这种争用可能会稍微降低应用程序性能，具体取决于内核资源及其可扩展性特征。
    
*   由于使用 iptables 实现容器网络，网络 I/O 可能会遇到 CPU 开销。
    
*   使用系统资源（CPU、磁盘、网络接口）的其他租户可能会争用这些资源。
    

Gianluca Borello 的一篇文章描述了当系统上存在某些其他容器时，容器的性能如何随着时间的推移缓慢而稳定地恶化 \[Borello 17\]。他追踪发现 lstat(2) 延迟较高，这是由其他容器的工作负载及其对 dcache 的影响造成的。

Maxim Leonovich 报告的另一个问题表明，从单租户虚拟机迁移到多租户容器会增加内核的 posix\_fadvise() 调用率，从而产生瓶颈 \[Leonovich 18\]。

列表中的最后一项由资源控制管理。虽然其中一些因素存在于传统的多用户环境中，但它们在多租户容器系统中更为普遍。

##### 中央处理器

可以使用 cpusets cgroup 跨容器分配 CPU，并使用 CFS 调度程序的共享和带宽。

###### 中央处理器组

cpusets cgroup 允许将整个 CPU 分配给特定容器。好处是这些容器可以在 CPU 上运行而不会被其他容器中断，并且它们可用的 CPU 容量是一致的。缺点是空闲的 CPU 容量无法供其他容器使用。

###### 份额和带宽

CPU 份额由 CFS 调度程序提供，是一种不同的 CPU 分配方法，允许容器共享其空闲 CPU 容量。共享支持爆发的概念，其中容器可以通过使用其他容器的空闲 CPU 来有效地运行得更快。当没有空闲容量时，包括当主机被过度配置时，共享可以在需要它们的容器之间尽力分配 CPU 资源。

CPU 份额通过将称为份额的分配单元分配给容器来工作，这些分配单元用于计算繁忙容器在给定时间将获得的 CPU 量。该计算使用以下公式：

容器 CPU =（所有 CPU × 容器份额）/系统上总繁忙份额

考虑一个在多个容器之间分配 100 个共享的系统。某一时刻，只有容器 A 和 B 需要 CPU 资源。容器 A 有 10 股，容器 B 有 30 股。因此，容器 A 可以使用系统上总 CPU 资源的 25%：所有 CPU × 10/(10 + 30)。

现在考虑一个所有容器同时繁忙的系统。给定容器的 CPU 分配将是：

容器 CPU = 所有 CPU × 容器份额 / 系统上的总份额

对于所描述的场景，容器 A 将获得 10% 的 CPU 容量（CPU 数 × 10/100）。共享分配提供了 CPU 使用率的最低保证。爆裂可能会让容器使用更多。容器 A 可以使用 10% 到 100% 的 CPU 容量，具体取决于有多少其他容器处于繁忙状态。

共享的一个问题是突发可能会混淆容量规划，特别是因为许多监控系统不显示突发统计数据（它们应该显示）。测试容器的最终用户可能对其性能感到满意，但没有意识到这种性能只有通过突发才能实现。稍后，当其他租户搬进来时，他们的容器将不再爆裂，并且性能会降低。想象一下，容器 A 最初在空闲系统上进行测试并获得 100% 的 CPU，但后来由于添加了其他容器，因此仅获得 10%。我在现实生活中见过很多次这样的场景，最终用户认为一定是系统性能问题，并要求我帮忙调试。然后，他们失望地发现系统正在按预期工作，而且由于其他容器已移入，速度慢十倍已成为新常态。对于客户来说，这可能感觉像是诱饵和切换。

可以通过限制过度爆发来减少过度爆发的问题，以使性能下降不那么严重（尽管这也会限制性能）。在 Linux 上，这是使用 CFS 带宽控制来完成的，它可以设置 CPU 使用的上限。例如，容器 A 的带宽可以设置为系统范围 CPU 容量的 20%，这样，通过共享，它现在可以在 10% 到 20% 的范围内运行，具体取决于空闲可用性。从基于共享的最小 CPU 到最大带宽的范围如图 11.12 所示。它假设每个容器中有足够的繁忙线程来使用可用的 CPU（否则，在达到系统施加的限制之前，容器会因其自身的工作负载而受到 CPU 限制）。

管理爆破的另一种方法是，容器运营商在一段时间内（例如几天）爆破时通知其最终用户，以便他们不会对性能产生错误的期望。然后可以鼓励最终用户升级其容器大小，以便获得更多份额以及更高的 CPU 分配最低保证。

BPF 程序可以附加到 cgroup，以实现自定义编程资源控制和防火墙。一个例子是 Cilium 软件，它使用 XDP、cgroup 和 tc (qdiscs) 等各个层的 BPF 程序组合来支持容器之间的安全、负载平衡和防火墙功能 \[Cilium 20a\]。

#### 11.3.4 可观测性

可观察的内容取决于启动可观察工具的位置以及主机的安全设置。由于容器可以通过多种不同的方式进行配置，因此我将描述典型案例。一般来说：

*   从主机（最具特权的命名空间）：可以观察到一切，包括硬件资源、文件系统、来宾进程、来宾 TCP 会话等。无需登录来宾即可查看和分析来宾进程。还可以从主机（云提供商）轻松浏览来宾文件系统。
    
*   来自来宾：容器通常只能看到自己的进程、文件系统、网络接口和 TCP 会话。一个主要的例外是系统范围的统计信息，例如 CPU 和磁盘：这些通常显示主机而不仅仅是容器。这些统计信息的状态通常没有记录（我在下面的传统工具部分编写了自己的文档）。通常无法检查内核内部结构，因此使用内核跟踪框架（第 13 章到第 15 章）的性能工具通常不起作用。
    

最后一点已在前面描述过：容器更有可能遇到内核争用问题，同时它们使最终用户无法诊断这些问题。

容器性能分析的一个常见问题是可能存在“吵闹的邻居”，即其他容器租户，这些租户会大量消耗资源并导致其他人的访问争用。由于这些容器进程都在一个内核下，并且可以从主机同时进行分析，这与传统的在一个分时系统上运行的多个进程的性能分析没有什么不同。主要区别在于 cgroup 可能会在硬件限制之前施加额外的软件限制（资源控制）。

许多为独立系统编写的监控工具尚未开发出对操作系统虚拟化（容器）的支持，并且对 cgroup 和其他软件限制视而不见。尝试在容器中使用这些的客户可能会发现它们似乎可以工作，但实际上仅显示物理系统资源。如果不支持观察云资源控制，这些工具可能会错误地报告系统有空间，而实际上它们已达到软件限制。它们还可能显示出实际上由其他租户造成的高资源使用率。

在 Linux 上，由于内核中目前没有容器 ID，[^11] 也没有传统性能工具提供的太多容器支持，因此来自主机和来宾的容器可观察性变得更加复杂和耗时。

[^11]容器管理软件可能会在容器 ID 之后命名 cgroup，在这种情况下，内核内 cgroup 名称确实会显示用户级容器名称。默认 cgroup v2 ID 是内核内 ID 的另一个候选者，BPF 和 bpftrace 将其用于此目的。第 11.3.4 节，可观察性，在 BPF 跟踪部分下，展示了另一种可能的解决方案：来自 uts 命名空间的节点名，通常设置为容器名称。

以下各节描述了这些挑战，总结了传统性能工具的状态，探索了主机和容器的可观察性，并描述了分析性能的策略。

##### 11.3.4.2 主机

*   每个容器的统计数据
    
*   资源控制的效果

*   BPF 跟踪，可以从内核读取 cgroup 和命名空间信息。

###### 命名空间映射

容器通常对进程 ID 和挂载使用不同的命名空间。

对于进程命名空间，这意味着来宾中的 PID 不太可能与主机中的 PID 匹配。

在诊断性能问题时，我首先登录到容器，以便可以从最终用户的角度看到问题。稍后我可能会登录主机使用系统范围的工具继续排查，但PID可能不一样。该映射显示在 /proc/PID/status 文件中。例如，来自主机：

点击这里查看代码图片

host# **grep NSpid /proc/4915/status**
NSpid:   4915    753

这表明主机上的 PID 4915 是来宾中的 PID 753。不幸的是，我通常需要进行反向映射：给定容器 PID，我需要找到主机 PID。一种（有点低效）的方法是扫描所有状态文件：

点击这里查看代码图片

host# **awk '$1 == "NSpid:" && $3 == 753 { print $2 }' /proc/\*/status**
4915

在本例中，它显示来宾 PID 753 是主机 PID 4915。请注意，输出可能显示多个主机 PID，因为“753”可能出现在多个进程命名空间中。在这种情况下，您需要找出哪个 753 来自匹配的命名空间。 /proc/PID/ns 文件是包含命名空间 ID 的符号链接，可用于此目的。先从客人那里检查，然后从主人那里检查：

点击这里查看代码图片

guest# **ls -lh /proc/753/ns/pid**
lrwxrwxrwx 1 root root 0 Mar 15 20:47 /proc/753/ns/pid -> 'pid:\[4026532216\]'

host# **ls -lh /proc/4915/ns/pid**
lrwxrwxrwx 1 root root 0 Mar 15 20:46 /proc/4915/ns/pid -> 'pid:\[4026532216\]'

请注意匹配的命名空间 ID (4026532216)：这确认主机 PID 4915 与来宾 PID 753 相同。

挂载命名空间可能会带来类似的挑战。例如，从主机运行 perf(1) 命令会在 /tmp/perf-PID.map 中搜索补充符号文件，但容器应用程序将它们发送到容器中的 /tmp，这与容器中的 /tmp 不同主人。此外，由于进程命名空间的原因，PID 可能会有所不同。 Alice Goldfuss 首先发布了一个解决方法，其中涉及移动和重命名这些符号文件，以便它们在主机中可用 \[Goldfuss 17\]。 perf(1) 自此获得了命名空间支持以避免此问题，并且内核提供了 /proc/PID/root 挂载命名空间映射，用于直接访问容器的根目录（“/”）。例如：

点击这里查看代码图片

host# **ls -lh /proc/4915/root/tmp**
total 0
-rw-r--r-- 1 root root 0 Mar 15 20:54 I\_am\_in\_the\_container.txt

这列出了容器的 /tmp 中的一个文件。

除了 /proc 文件之外，nsenter(1) 命令还可以在选定的命名空间中执行其他命令。以下命令从主机在 PID 4915 (`-t 4915`) 的挂载 (`-m`) 和进程 (`-p`) 命名空间中运行 top(1) 命令：

点击这里查看代码图片

###### BPF追踪

一些 BPF 跟踪工具已经具有容器支持，但许多工具还没有。幸运的是，在需要时添加对 bpftrace 工具的支持通常并不困难；以下是一个例子。有关 bpftrace 编程的说明，请参阅第 15 章。

forks.bt 工具通过检测clone(2)、fork(2) 和vfork(2) 系统调用来计算跟踪时创建的新进程的数量。来源是：

点击这里查看代码图片

#!/usr/local/bin/bpftrace

tracepoint:syscalls:sys\_enter\_clone,
tracepoint:syscalls:sys\_enter\_fork,
tracepoint:syscalls:sys\_enter\_vfork
{
        @new\_processes = count();
}

输出示例：

\# **./forks.bt**
Attaching 3 probes...
^C

@new\_processes: 590

这显示了跟踪时在系统范围内创建的 590 个新进程。

要按容器对此进行分解，一种方法是从 uts 命名空间打印节点名（主机名）。这依赖于配置此命名空间的容器软件，通常就是这种情况。代码变为，突出显示添加内容：

点击这里查看代码图片

#!/usr/local/bin/bpftrace

**#include <linux/sched.h>**
**#include <linux/nsproxy.h>**
**#include <linux/utsname.h>**

tracepoint:syscalls:sys\_enter\_clone,
tracepoint:syscalls:sys\_enter\_fork,
tracepoint:syscalls:sys\_enter\_vfork
{
        **$task = (struct task\_struct \*)curtask;**
        **$nodename = $task->nsproxy->uts\_ns->name.nodename;**
        @new\_processes\[**$nodename**\] = count();
}

此额外代码从当前内核 task\_struct 走到 uts 命名空间节点名，并将其作为键包含在 @new\_processes 输出映射中。

输出示例：

点击这里查看代码图片

\# **./forks.bt**
Attaching 3 probes...
^C

@new\_processes\[ip-10-1-239-218\]: 171
@new\_processes\[efe9f9be6185\]: 743

输出现在按容器进行细分，显示节点名称 `6280172ea7b9`（容器）在跟踪时创建了 252 个进程。另一个节点名称 `ip-10-1-239-218` 是主机系统。

这只有效，因为系统调用在任务（进程）上下文中运行，因此 `curtask` 返回负责的 task\_struct，我们从中获取节点名。如果跟踪进程异步事件，例如磁盘 I/O 的完成中断，则原始进程可能不在 CPU 上，并且 `curtask` 将无法识别正确的节点名。

由于获取 uts 节点名可能会在 bpftrace 中常用，我想我们将添加一个内置变量 nodename，因此唯一的添加是：

点击这里查看代码图片

@new\_processes\[**nodename**\] = count();

检查 bpftrace 更新以查看是否已添加。

###### 资源控制

必须遵守第 11.3.3 节“资源控制”中列出的资源控制，以确定容器是否受其限制。传统的性能工具和文档侧重于物理资源，而忽视了这些软件施加的限制。

USE 方法中描述了检查资源控制（第 2 章，方法，第 2.5.9 节，USE 方法），该方法迭代资源并检查利用率、饱和度和错误。由于存在资源控制，因此也必须对每个资源进行检查。

前面的 Cgroup 统计部分显示了 /sys/fs/cgroup/.../cpu.stat 文件，该文件提供了有关 CPU 限制 (nr\_throttled) 和以纳秒为单位的限制时间 (throttled\_time) 的统计信息。这种限制指的是CPU带宽限制，并且很容易识别容器是否受到带宽限制：throttled\_time将会增加。如果改用 cpuset，则可以通过每个 CPU 的工具和指标（包括 mpstat(1)）检查其 CPU 利用率。

CPU 也可以通过共享进行管理，如前面的共享和带宽部分所述。股份有限的容器更难以识别，因为没有统计数据。我开发了图 11.13 中的流程图，用于确定容器 CPU 是否以及如何受到限制的过程 \[Gregg 17g\]：

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/11fig13.jpg)

图11.13 容器CPU限流分析

图 11.13 中的流程使用五个统计数据确定是否以及如何限制容器 CPU：

*   限制时间：cpu cgroupthrottled\_time
    
*   非自愿上下文切换：可以从 /proc/PID/status 中读取，作为 nonvolutical\_ctxt\_switches 的增加
    
*   主机有空闲 CPU：可以从 mpstat(1) `%idle`、/proc/stat 和其他工具读取
    
*   cpuset CPUs 100% busy：如果 cpusets 正在使用，可以从 mpstat(1)、/proc/stat 等读取它们的利用率。
    
*   所有其他租户都处于空闲状态：可以通过特定于容器的工具 (`docker stat`) 或显示 CPU 资源缺乏竞争的系统工具来确定（例如，如果 top(1) 仅显示消耗 `%CPU`)
    

可以为其他资源开发类似的流程，并且应在监控软件和工具中提供包括 cgroup 统计数据在内的支持统计数据。理想的监控产品或工具可以为您做出决定，并报告每个容器是否以及如何受到限制。

###### 容器意识

让工具具有容器感知能力并不一定意味着将其视图限制在自己的容器内：容器查看物理资源的状态是有好处的。这取决于使用容器的目标，可能是：

1.  容器是一个隔离的服务器：如果这是目标，就像云供应商的典型情况一样，那么使工具具有容器感知能力意味着它们只显示当前容器的活动。例如，iostat(1) 应该只显示容器调用的磁盘 I/O，而不显示其他调用的磁盘 I/O。这可以通过隔离所有统计源以仅显示当前容器（/proc、/sys、netlink 等）来实现。这种隔离既有助于也妨碍来宾的分析：诊断其他租户的资源争用问题将更加耗时，并且将依赖于无法解释的设备延迟增加的推断。
    
2.  容器是一种打包解决方案：对于运营自己的容器云的公司来说，可能不一定需要隔离容器的统计数据。允许容器查看主机统计信息（无论如何，目前经常出现这种情况，如前面使用 iostat(1) 看到的那样）意味着最终用户可以更好地了解硬件设备的状态以及由嘈杂的邻居引起的问题。在这种情况下，为了使 iostat(1) 更加了解容器，可能意味着提供显示当前容器与主机或其他容器使用情况的详细信息，而不是隐藏这些统计信息。
    

对于这两种情况，工具还应该支持在适当的情况下显示资源控制，作为容器感知的一部分。继续 iostat(1) 示例，除了设备 `%util`（在场景 A 中不可见）之外，它还可以根据 blkio 吞吐量和 IOPS 限制提供 `%cap`，以便容器知道磁盘 I/O 是否受资源控制限制。[^12]

[^12] iostat(1) -x 目前有太多字段，不适合我最宽的终端，我犹豫是否鼓励添加更多字段。我宁愿添加另一个开关，例如 -l，以显示软件限制列。

如果允许物理资源可观察性，客人将能够排除某些类型的问题，包括吵闹的邻居。这可以减轻集装箱运营商的支持负担：人们倾向于归咎于他们无法观察到的事情。这也是与硬件虚拟化的一个重要区别，硬件虚拟化对来宾隐藏了物理资源，并且无法共享这些统计数据（除非使用外部手段）。理想情况下，Linux 中未来会有一个设置来控制主机统计信息的共享，以便每个容器环境可以根据需要在 A 或 B 之间进行选择。

###### 追踪工具

基于内核的高级跟踪工具，例如 perf(1)、Ftrace 和 BPF，也有类似的问题，并且在它们之前就实现了容器感知。由于各种系统调用（perf\_event\_open(2)、bpf(2) 等）以及访问各种 /proc 和 /sys 文件所需的权限，它们目前无法在容器内工作。描述早期场景 A 和 B 的未来：

1.  所需隔离：允许容器使用跟踪工具可以通过以下方式实现：
    
    *   内核过滤：事件及其参数可以由内核过滤，因此，例如，跟踪 block:block\_rq\_issue 跟踪点仅显示当前容器的磁盘 I/O。
        
    *   主机 API：主机通过安全 API 或 GUI 公开对某些跟踪工具的访问。例如，容器可以请求执行常见的 BCC 工具，例如 execsnoop(8) 和 biolatency(8)，主机将验证该请求，执行该工具的过滤版本，并返回输出。
        
2.  不需要隔离：资源（系统调用、/proc 和 /sys）在容器中可用，以便跟踪工具可以工作。跟踪工具本身可以感知容器，以便于仅过滤当前容器的事件。
    

在 Linux 和其他内核中，使工具和统计数据能够感知容器是一个缓慢的过程，并且可能需要很多年才能全部完成。这对登录系统并在命令行使用性能工具的高级用户造成的伤害最大；许多用户使用具有容器感知（在某种程度上）的代理的监控产品。

##### 11.3.4.4 策略

前面的章节介绍了物理系统资源的分析技术并包括各种方法。主办方操作员可以遵循这些规定，客人在某种程度上也可以遵循这些规定，同时牢记前面提到的限制。对于来宾来说，高级资源使用情况通常是可观察的，但通常不可能深入到内核。

除了物理资源之外，主机运营商和来宾租户还应该检查资源控制所施加的云限制。由于这些限制（如果存在）是在物理限制之前很久就遇到的，因此它们更有可能生效，因此应首先检查。

由于许多传统的可观察性工具是在容器和资源控制存在之前创建的（例如top(1)、iostat(1)），因此它们默认不包含资源控制信息，用户可能会忘记检查它们。

以下是检查每个资源控制的一些注释和策略：

*   CPU：见图11.13流程图。 cpusets、带宽和共享的使用情况都需要检查。
    
*   内存：对于主内存，根据任何内存 cgroup 限制检查当前使用情况。
    
*   文件系统容量：对于任何其他文件系统（包括使用 df(1)），这应该是可观察到的。
    
*   磁盘 I/O：检查 blkio cgroup 节流阀的配置 (/sys/fs/cgroup/blkio) 以及 blkio.throttle.io\_serviced 和 blkio.throttle.io\_service\_bytes 文件中的统计信息：如果它们以与节流阀相同的速率递增，这证明磁盘 I/O 受到限制。如果 BPF 跟踪可用，则 blkthrot(8) 工具也可用于确认 blkio 节流 \[Gregg 19\]。
    
*   网络 I/O：根据任何已知的带宽限制检查当前网络吞吐量，该限制只能从主机观察到。遇到限制会导致网络 I/O 延迟增加，因为租户受到限制。
    

最后的评论：本节的大部分内容描述了 cgroup v1 和 Linux 容器的当前状态。内核功能正在迅速变化，使得供应商文档和工具容器意识等其他领域也迎头赶上。为了跟上 Linux 容器的最新动态，您需要检查新内核版本中的新增内容，并通读 Linux 源代码中的文档目录。我还建议阅读 cgroups v2 的首席开发人员 Tejun Heo 编写的任何文档，包括 Linux 源代码中的 Documentation/admin-guide/cgroup-v2.rst \[Heo 15\]。

### 11.4 轻量级虚拟化

轻量级硬件虚拟化被设计为两全其美：硬件虚拟化的安全性与容器的效率和快速启动时间。图 11.14 展示了它们基于 Firecracker 的情况，以及用于比较的容器。

轻量级硬件虚拟化使用基于处理器虚拟化的轻量级管理程序和最少数量的模拟设备。这与全机硬件管理程序（第 11.2 节，硬件虚拟化）不同，后者起源于桌面虚拟机，包括对视频、音频、BIOS、PCI 总线和其他设备的支持，以及不同级别的处理器支持。仅用于服务器计算的虚拟机管理程序不需要支持这些设备，今天编写的虚拟机管理程序可以假设现代处理器虚拟化功能可用。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/11fig14.jpg)

图11.14 轻量级虚拟化

描述一下这种差异：快速仿真器 (QEMU) 是用于 KVM 的全机硬件虚拟机管理程序，拥有超过 140 万行代码（QEMU 版本 4.2）。 Amazon Firecracker 是一个轻量级虚拟机管理程序，只有 5 万行代码 \[Agache 20\]。

轻量级 VM 的行为与第 11.2 节中描述的配置 B 硬件 VM 类似。与硬件虚拟机相比，轻量级虚拟机具有更快的启动时间、更低的内存开销和更高的安全性。轻量级管理程序可以通过将命名空间配置为另一层安全性来进一步提高安全性，如图 11.14 所示。

一些实现将轻量级虚拟机描述为容器，而其他实现则使用术语 MicroVM。我更喜欢 MicroVM 这个术语，因为容器这个术语通常与操作系统虚拟化相关。

#### 11.4.1 实施

有几个轻量级硬件虚拟化项目，包括：

*   Intel Clear Containers：于 2015 年推出，利用 Intel VT 功能提供轻量级虚拟机。该项目证明了轻量级容器的潜力，实现了 45 毫秒以下的启动时间 \[Kadera 16\]。 2017 年，英特尔 Clear Containers 加入了 Kata Containers 项目并继续开发。
    
*   Kata Containers：该项目于2017年启动，基于Intel Clear Containers和Hyper.sh RunV，并接受OpenStack基金会的管理。其网站口号是：“容器的速度，虚拟机的安全”\[Kata Containers 20\]。
    
*   Google gVisor：gVisor 于 2018 年以开源方式推出，使用用 Go 编写的专门用于访客的用户空间内核，提高了容器的安全性。
    
*   Amazon Firecracker：于 2019 年作为开源 \[Firecracker 20\] 推出，它使用 KVM 和新的轻量级 VMM 而不是 QEMU，并实现了约 100 毫秒的启动时间（系统启动）\[Agache 20\]。
    

以下部分描述了常用的实现：轻量级硬件管理程序（Intel Clear Containers、Kata Containers 和 Firecracker）。 gVisor 是一种不同的方法，它实现了自己的轻量级内核，并且具有更类似于容器的特性（第 11.3 节，操作系统虚拟化）。

#### 11.4.2 开销

开销与第 11.2.2 节“开销”中描述的 KVM 虚拟化的开销类似，但由于 VMM 更小，因此内存占用更低。 Intel Clear Containers 2.0 报告每个容器的内存开销为 48-50 MB \[Kadera 16\]； Amazon Firecracker 报告小于 5 MB \[Agache 20\]。

#### 11.4.3 资源控制

当 VMM 进程在主机上运行时，它们可以通过操作系统级别的资源控制进行管理：cgroup、qdisc 等，类似于第 11.2.3 节“资源控制”中描述的 KVM 虚拟化。第 11.3.3 节“资源控制”中还针对容器更详细地讨论了这些操作系统级资源控制。

#### 11.4.4 可观察性

可观察性类似于第 11.2.4 节“可观察性”中描述的 KVM 虚拟化。总之：

*   从主机：可以使用标准操作系统工具观察所有物理资源，如前面的章节所述。来宾虚拟机作为进程可见。无法直接观察来宾内部结构，包括虚拟机及其文件系统内的进程。对于要分析来宾内部结构的虚拟机管理程序操作员，必须授予他们访问权限（例如 SSH）。
    
*   从来宾：可以看到虚拟化资源及其对来宾的使用情况，并推断物理问题。内核跟踪工具（包括基于 BPF 的工具）都可以工作，因为 VM 有自己的专用内核。

### 11.5 其他类型

其他云计算原语和技术包括：

*   功能即服务（FaaS）：开发人员向云端提交应用程序功能，云端按需运行。尽管这简化了软件开发体验，但由于无需管理服务器（“无服务器”），因此会产生性能影响。该功能的启动时间可能很长，并且如果没有服务器，最终用户就无法运行传统的命令行可观察性工具。性能分析通常仅限于应用程序提供的时间戳。
    
*   软件即服务 (SaaS)：它提供高级软件，最终用户无需自行配置服务器或应用程序。性能分析仅限于操作员：如果无法访问服务器，最终用户除了基于客户端的计时之外几乎无能为力。
    
*   Unikernels：该技术将应用程序与最少的内核部分一起编译成单个软件二进制文件，可以直接由硬件管理程序执行，无需操作系统。虽然可以带来性能提升，例如最大限度地减少指令文本，从而减少 CPU 缓存污染，以及由于删除未使用的代码而带来的安全性收益，但 Unikernels 也带来了可观察性挑战，因为没有操作系统可以运行可观察性工具。内核统计信息（例如 /proc 中的统计信息）也可能不存在。幸运的是，实现通常允许 Unikernel 作为正常进程运行，从而提供一种分析路径（尽管在与虚拟机管理程序不同的环境中）。虚拟机管理程序还可以开发检查它们的方法，例如堆栈分析：我开发了一个原型，可以生成正在运行的 MirageOS Unikernel \[Gregg 16a\] 的火焰图。
    

对于所有这些，没有操作系统可供最终用户登录（或容器进入）并进行传统的性能分析。 FaaS和SaaS必须由运营商来分析。 Unikernels 需要自定义工具和统计数据，以及理想的虚拟机管理程序分析支持。

虚拟化仍然是一个不断发展的领域，轻量级硬件管理程序只是近年来才出现。鉴于它们的好处，特别是对于最终用户的可观察性，我预计它们的使用量会增长。

    *   什么是租户？

    *   什么是硬件虚拟化？
