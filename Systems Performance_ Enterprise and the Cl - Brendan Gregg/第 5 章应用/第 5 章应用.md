
性能最好在最接近执行工作的位置进行调整：在应用程序中。其中包括数据库、Web 服务器、应用程序服务器、负载平衡器、文件服务器等。后面的章节从应用程序消耗的资源的角度来介绍应用程序：CPU、内存、文件系统、磁盘和网络。本章讨论应用程序级别。

应用程序本身可能变得极其复杂，尤其是在涉及许多组件的分布式环境中。应用程序内部结构的研究通常是应用程序开发人员的领域，可以包括使用第三方工具进行内省。对于那些研究系统性能的人来说，应用程序性能分析包括配置应用程序以充分利用系统资源、描述应用程序如何使用系统以及分析常见问题。

本章的学习目标是：
*   描述性能调整目标。
*   熟悉性能改进技术，包括多线程编程、哈希表和非阻塞 I/O。
*   了解常见的锁定和同步原语。
*   了解不同编程语言带来的挑战。
*   遵循线程状态分析方法。
*   执行 CPU 和 CPU 外分析。
*   执行系统调用分析，包括跟踪进程执行。
*   了解堆栈跟踪陷阱：缺少符号和堆栈。

本章讨论应用程序基础知识、应用程序性能基础知识、编程语言和编译器、通用应用程序性能分析策略以及基于系统的应用程序可观察性工具。

### 5.1 应用基础知识

在深入了解应用程序性能之前，您应该熟悉应用程序的角色、基本特征及其在行业中的生态系统。这构成了您可以了解应用程序活动的上下文。它还为您提供了了解常见性能问题和调优的机会，并提供了进一步研究的途径。要了解此背景，请尝试回答以下问题：

*   功能：应用程序的作用是什么？它是数据库服务器、Web 服务器、负载均衡器、文件服务器、对象存储吗？
    
*   操作：应用程序服务什么请求，或者执行什么操作？数据库服务于查询（和命令），Web 服务器服务于 HTTP 请求，等等。这可以作为速率来测量，以衡量负载和容量规划。
    
*   性能要求：运行应用程序的公司是否有服务级别目标 (SLO)（例如，99.9% 的请求延迟时间小于 100 毫秒）？
    
*   CPU 模式：应用程序是作为用户级软件还是内核级软件实现的？大多数应用程序都是用户级的，作为一个或多个进程执行，但有些应用程序是作为内核服务（例如 NFS）实现的，BPF 程序也是内核级的。
    
*   配置：应用程序是如何配置的，为什么？该信息可以在配置文件中或通过管理工具找到。检查与性能相关的任何可调参数是否已更改，包括缓冲区大小、缓存大小、并行性（进程或线程）和其他选项。
    
*   主机：应用程序由什么主机托管？服务器还是云实例？ CPU、内存拓扑、存储设备等是什么？他们的极限是什么？
    
*   指标：是否提供应用程序指标，例如操作率？它们可能由捆绑工具或第三方工具、通过 API 请求或通过处理操作日志来提供。
    
*   日志：应用程序创建哪些操作日志？可以启用哪些日志？从日志中可以获得哪些性能指标（包括延迟）？例如，MySQL 支持慢查询日志，为每个慢于某个阈值的查询提供有价值的性能详细信息。
    
*   版本：该应用程序是最新版本吗？最新版本的发行说明中是否注明了性能修复或改进？
    
*   错误：应用程序是否有错误数据库？您的应用程序版本有哪些“性能”错误？如果您当前遇到性能问题，请搜索错误数据库以查看以前是否发生过类似情况、如何调查该问题以及还涉及哪些内容。
    
*   源代码：应用程序是开源的吗？如果是这样，则可以研究分析器和跟踪器识别的代码路径，这可能会带来性能提升。您可以自行修改应用程序代码以提高性能，并向上游提交改进以包含在官方应用程序中。
    
*   社区：该应用程序是否有一个可以共享性能调查结果的社区？社区可能包括论坛、博客、互联网中继聊天 (IRC) 频道、其他聊天频道（例如 Slack）、聚会和会议。聚会和会议经常在网上发布幻灯片和视频，这些都是多年后有用的资源。他们可能还有一位社区经理来分享社区更新和新闻。
    
*   书籍：有关于应用程序和/或其性能的书籍吗？它们是好书吗（例如，由专家撰写、实用/可操作、充分利用读者的时间、最新的等等）？
    
*   专家：谁是该应用程序公认的性能专家？了解他们的名字可以帮助您找到他们创作的材料。
    

无论来源如何，您的目标是从较高的层面了解应用程序 — 它的用途、运行方式以及执行方式。如果您能找到的话，一个非常有用的资源是说明应用程序内部结构的功能图。

接下来的部分将介绍其他应用程序基础知识：设定目标、优化常见情况、可观察性和大 O 表示法。

#### 5.1.1 目标

绩效目标为您的绩效分析工作提供方向，并帮助您选择要执行的活动。如果没有明确的目标，绩效分析就有可能变成随机的“钓鱼探险”。

对于应用程序性能，您可以从应用程序执行哪些操作（如前所述）以及性能目标是什么开始。目标可能是：

*   延迟：应用程序响应时间短或一致
    
*   吞吐量：较高的应用程序运行率或数据传输率
    
*   资源利用率：给定应用程序工作负载的效率
    
*   价格：提高性价比，降低计算成本
    

如果可以使用源自业务或服务质量要求的指标来量化这些，那就更好了。例子有：

*   平均应用程序请求延迟为 5 毫秒
    
*   95% 的请求在 100 毫秒或更短的延迟内完成
    
*   消除延迟异常值：超过 1,000 毫秒的零请求
    
*   每台给定大小的服务器每秒至少处理 10,000 个应用程序请求的最大吞吐量[^1]
    
    [^1]如果服务器大小可变（与云实例一样）。这可以用限制资源来更好地表达：例如，对于 CPU 限制的工作负载，每个 CPU 每秒最多 1,000 个应用程序请求。
    
*   每秒 10,000 个应用程序请求的平均磁盘利用率低于 50%
    

一旦选择了目标，您就可以研究该目标的限制因素。对于延迟，限制器可能是磁盘或网络 I/O；对于吞吐量，可能是CPU使用率。本章和其他章节中的策略将帮助您识别它们。

对于基于吞吐量的目标，请注意并非所有操作在性能或成本方面都是相同的。如果目标是一定的操作率，那么指定操作的类型可能也很重要。这可以是基于预期或测量的工作负载的分布。

第 5.2 节“应用程序性能技术”描述了提高应用程序性能的常用方法。其中一些可能对一个目标有意义，但对另一个目标则不然；例如，选择较大的 I/O 大小可能会以延迟为代价提高吞吐量。在确定哪些主题最适用时，请记住您所追求的目标。

##### Apdex

一些公司使用目标应用程序性能指数（ApDex 或 Apdex）作为目标和监控指标。它可以更好地传达客户体验，并首先对客户事件进行分类，以确定它们是“令人满意”、“可以忍受”还是“令人沮丧”。然后使用 \[Apdex 20\] 计算 Apdex：

Apdex =（满意 + 0.5 × 可容忍 + 0 × 令人沮丧）/总事件

生成的 Apdex 范围从 0（没有满意的客户）到 1（所有满意的客户）。

#### 5.1.2 优化常见案例

软件内部结构可能很复杂，具有许多不同的代码路径和行为。如果您浏览源代码，这一点可能尤其明显：应用程序通常有数万行代码，而操作系统内核则有数十万行以上。随机选择要优化的区域可能会涉及大量工作而收效甚微。

有效提高应用程序性能的一种方法是找到生产工作负载最常见的代码路径，并首先对其进行改进。如果应用程序受 CPU 限制，则可能意味着代码路径经常位于 CPU 上。如果应用程序受 I/O 限制，您应该查看经常导致 I/O 的代码路径。这些可以通过应用程序的分析和分析来确定，包括研究堆栈跟踪和火焰图，如后面的章节所述。应用程序可观察性工具还可以提供用于理解常见情况的更高级别的上下文。

#### 5.1.3 可观察性

正如我在本书的许多章节中重申的那样，最大的性能提升来自于消除不必要的工作。

当根据性能选择应用程序时，这一事实有时会被忽视。如果基准测试显示应用程序 A 比应用程序 B 快 10%，那么选择应用程序 A 可能很诱人。但是，如果应用程序 A 不透明并且应用程序 B 提供了一组丰富的可观察性工具，那么应用程序 B 很可能会成为最佳选择。从长远来看更好的选择。这些可观察性工具可以查看并消除不必要的工作，并更好地理解和调整主动工作。通过增强可观察性获得的性能胜利可能会使最初 10% 的性能差异相形见绌。对于语言和运行时的选择也是如此：比如选择成熟且有很多可观察性工具的Java或C，而不是选择新的语言。

### 5.2 应用性能技术

本节介绍一些可提高应用程序性能的常用技术：选择 I/O 大小、缓存、缓冲、轮询、并发性和并行性、非阻塞 I/O 以及处理器绑定。请参阅应用程序文档以了解使用了哪些功能，以及任何其他特定于应用程序的功能。

#### 5.2.1 选择 I/O 大小

与执行 I/O 相关的成本包括初始化缓冲区、进行系统调用、模式或上下文切换、分配内核元数据、检查进程权限和限制、将地址映射到设备、执行内核和驱动程序代码以交付 I/O，以及最后，释放元数据和缓冲区。小型和大型 I/O 都需要支付“初始化税”。就效率而言，每个 I/O 传输的数据越多越好。

增加 I/O 大小是应用程序用来提高吞吐量的常见策略。考虑到任何固定的每 I/O 成本，将 128 KB 作为单个 I/O 传输通常比 128 × 1 KB I/O 更高效。尤其是旋转磁盘 I/O，由于寻道时间，历史上每 I/O 成本较高。

当应用程序不需要更大的 I/O 大小时，就会有一个缺点。执行 8 KB 随机读取的数据库在使用 128 KB 磁盘 I/O 大小时可能会运行得更慢，因为浪费了 120 KB 的数据传输。这会引入 I/O 延迟，可以通过选择更接近应用程序请求的较小 I/O 大小来降低 I/O 延迟。不必要的较大 I/O 大小也会浪费缓存空间。

#### 5.2.2 缓存

操作系统使用缓存来提高文件系统读取性能和内存分配性能；应用程序经常出于类似的原因使用缓存。代替总是执行昂贵的操作，通常执行的操作的结果可以存储在本地高速缓存中以供将来使用。一个例子是数据库缓冲区高速缓存，它存储来自常用数据库查询的数据。

部署应用程序时的一项常见任务是确定提供或可以启用哪些缓存，然后配置其大小以适合系统。

虽然缓存可以提高读取性能，但它们的存储通常用作缓冲区来提高写入性能。

#### 5.2.3 缓冲

为了提高写入性能，数据可以在发送到下一级之前合并在缓冲区中。这增加了 I/O 大小和操作效率。根据写入的类型，它还可能会增加写入延迟，因为对缓冲区的第一次写入会在发送之前等待后续写入。

环形缓冲区（或循环缓冲区）是一种固定缓冲区，可用于异步作用于缓冲区的组件之间的连续传输。它可以使用开始和结束指针来实现，每个组件可以在添加或删除数据时移动开始指针和结束指针。

#### 5.2.4 轮询

轮询是一种技术，系统通过循环检查事件的状态来等待事件发生，检查之间有暂停。当几乎没有什么工作要做时，轮询存在一些潜在的性能问题：

*   重复检查会产生昂贵的 CPU 开销
    
*   事件发生与下一次轮询检查之间的高延迟
    

如果这是性能问题，应用程序可能能够更改其行为以侦听事件的发生，这会立即通知应用程序并执行所需的例程。

##### poll() 系统调用

有一个 poll(2) 系统调用来检查文件描述符的状态，它提供与轮询类似的功能，尽管它是基于事件的，因此不会遭受轮询的性能成本。

poll(2)接口支持多个文件描述符作为数组，这需要应用程序在事件发生时扫描数组以查找相关的文件描述符。此扫描的时间复杂度为 O(n)（请参阅第 5.1.4 节，大 O 表示法），其开销可能会成为大规模的性能问题。 Linux 上的替代方案是 epoll(2)，它可以避免扫描，因此时间复杂度为 O(1)。在 BSD 上，等效项是 kqueue(2)。

#### 5.2.5 并发和并行

分时系统（包括所有源自 Unix 的系统）提供程序并发性：加载并开始执行多个可运行程序的能力。虽然它们的运行时间可能重叠，但它们不一定同时在 CPU 上执行。这些程序中的每一个都可以是一个应用程序进程。

要利用多处理器系统，应用程序必须同时在多个 CPU 上执行。这就是并行性，应用程序可以通过使用多个进程（多进程）或多线程（多线程）来实现并行性，每个线程执行自己的任务。由于第 6 章 CPU、第 6.3.13 节“多进程、多线程”中解释的原因，多线程（或等效任务）效率更高，因此是首选方法。

除了增加 CPU 工作的吞吐量之外，多线程（或进程）是允许并发执行 I/O 的一种方法，因为当 I/O 阻塞的线程等待时，其他线程可以执行。 （另一种方式是异步 I/O。）

使用多进程或多线程架构意味着允许内核通过 CPU 调度程序来决定运行谁，并以上下文切换开销为代价。另一种方法是用户模式应用程序实现自己的调度机制和程序模型，以便它可以在同一操作系统线程中服务不同的应用程序请求（或程序）。机制包括：

*   纤程：也称为轻量级线程，它们是线程的用户模式版本，其中每个纤程代表一个可调度程序。应用程序可以使用自己的调度逻辑来选择运行哪个纤程。例如，它们可用于分配纤程来处理每个应用程序请求，其开销比使用操作系统线程执行相同操作要少。例如，Microsoft Windows 支持光纤。[^2]
    
    [^2]微软官方文档警告了纤程可能造成的问题：例如，线程本地存储在纤程之间共享，因此程序员必须切换到纤程本地存储，并且任何退出线程的例程都将退出该线程上的所有纤程那个线程。该文档指出：“一般来说，与精心设计的多线程应用程序相比，光纤没有优势”\[Microsoft 18\]。
    
*   协同例程：协同例程比纤程更轻量，是可以由用户模式应用程序调度的子例程，提供并发机制。
    
*   基于事件的并发：程序被分解为一系列事件处理程序，并且可以在队列上规划和执行可运行的事件。例如，可以通过为每个应用程序请求分配元数据（由事件处理程序引用）来使用这些元数据。例如，Node.js 运行时使用基于事件的并发，使用单个事件工作线程（这可能成为瓶颈，因为它只能在一个 CPU 上执行）。
    

使用所有这些机制，I/O 仍必须由内核处理，因此操作系统线程切换通常是不可避免的。[^3] 此外，为了实现并行性，必须使用多个操作系统线程，以便可以跨多个 CPU 调度它们。

[^3]有一些例外，例如使用 sendfile(2) 来避免 I/O 系统调用，以及 Linux io\_uring，它允许用户空间通过写入和读取 io\_uring 队列来调度 I/O（这些在第 1 节中进行了总结） 5.2.6，非阻塞 I/O）。

一些运行时使用协同例程来实现轻量级并发，并使用多个操作系统线程来实现并行性。一个例子是 Golang 运行时，它在操作系统线程池上使用 goroutine（协同例程）。为了提高性能，当 goroutine 进行阻塞调用时，Golang 的调度程序会自动将阻塞线程上的其他 goroutine 移动到其他线程来运行 \[Golang 20\]。

多线程编程的三种常见模型是：

*   服务线程池：服务网络请求的线程池，其中每个线程一次服务一个客户端连接。
    
*   CPU线程池：每个CPU创建一个线程。这通常用于长时间的批处理，例如视频编码。
    
*   分阶段事件驱动架构 (SEDA)：应用程序请求被分解为可由一个或多个线程池处理的阶段。
    

由于多线程编程与进程共享相同的地址空间，因此线程可以直接读写相同的内存，而不需要更昂贵的接口（例如用于多进程编程的进程间通信\[IPC\]）。为了完整性，使用同步原语，以便数据不会因多个线程同时读写而损坏。

##### 同步原语

同步原语管理对内存的访问以确保完整性，并且可以类似于交通灯调节对交叉路口的访问进行操作。而且，就像交通信号灯一样，它们可以阻止交通流动，从而导致等待时间（延迟）。应用程序常用的三种类型是：

*   互斥（MUTually EXclusive）锁：只有锁的持有者才能操作。其他的则阻塞并等待 CPU 关闭。
    
*   自旋锁：自旋锁允许持有者进行操作，而其他自旋锁则在紧密循环中在 CPU 上旋转，检查是否要释放锁。虽然它们可以提供低延迟访问（被阻塞的线程永远不会离开 CPU，并且一旦锁可用，就可以在几个周期内运行），但它们也会在线程旋转、等待时浪费 CPU 资源。
    
*   RW 锁：读取器/写入器锁通过允许多个读取器或仅允许一个写入器但不允许读取器来确保完整性。
    
*   信号量：这是一种变量类型，可以通过计数来允许给定数量的并行操作，或者通过二进制来仅允许一个（实际上是互斥锁）。
    

互斥锁可以由库或内核实现为自旋锁和互斥锁的混合体，如果持有者当前正在另一个 CPU 上运行，则互斥锁会自旋，否则会阻塞（或者达到自旋阈值）。它们最初于 2009 年在 Linux 上实现 \[Zijlstra 09\]，现在根据锁的状态具有三个路径（如 Documentation/locking/mutex-design.rst \[Molnar 20\] 中所述）：

1.  fastpath：尝试使用 cmpxchg 指令获取锁以设置所有者。仅当未持有锁时此操作才会成功。
    
2.  midpath：也称为乐观自旋，它在CPU上自旋，同时锁持有者也在运行，希望它很快被释放并且可以在不阻塞的情况下获取它。
    
3.  Slowpath：这会阻塞并取消调度线程，以便稍后在锁可用时唤醒。
    

Linux 读取-复制-更新 (RCU) 机制是另一种在内核代码中大量使用的同步机制。它允许读取操作而无需获取锁，从而比其他锁类型提高了性能。使用 RCU，写入操作会创建受保护数据的副本并更新该副本，而进行中的读取操作仍可以访问原始数据。它可以检测何时不再有任何读取器（基于每个 CPU 的各种条件），然后用更新的副本替换原始副本 \[Linux 20e\]。

研究涉及锁的性能问题可能非常耗时，并且通常需要熟悉应用程序源代码。这通常是开发人员的一项活动。

##### 哈希表

锁哈希表可用于为大量数据结构采用最佳数量的锁。虽然此处总结了哈希表，但这是一个高级主题，需要具备编程背景。

想象一下以下两种方法：

*   所有数据结构的单个全局互斥锁。虽然这种解决方案很简单，但是并发访问会遇到锁争用和等待时的延迟。需要锁的多个线程将串行化——按顺序执行，而不是同时执行。
    
*   每个数据结构的互斥锁。虽然这将争用减少到仅真正需要的次数（对同一数据结构的并发访问），但存在锁的存储开销以及为每个数据结构创建和销毁锁的 CPU 开销。
    

锁哈希表是一种中间解决方案，适用于预期锁争用较少的情况。创建固定数量的锁，并使用哈希算法来选择哪个锁用于哪个数据结构。这避免了数据结构的创建和销毁成本，也避免了只有单个锁的问题。

此示例还展示了一种解决哈希冲突的方法，其中两个或多个输入数据结构哈希到同一存储桶。在这里，创建了一个数据结构链，将它们全部存储在同一个存储桶中，哈希函数将在其中再次找到它们。如果这些哈希链变得太长并且串行行走，则可能会出现性能问题，因为它们仅受一个锁的保护，而该锁的保持时间可能会很长。可以选择哈希函数和表大小，以将数据结构均匀分布在多个存储桶上，以使哈希链长度保持最小。应检查生产工作负载的哈希链长度，以防哈希算法未按预期工作，而是创建性能不佳的长哈希链。

理想情况下，哈希表桶的数量应等于或大于 CPU 数量，以实现最大并行性的潜力。散列算法可能很简单，只需获取数据结构地址的低位位 [^4] 并将其用作两倍大小的锁数组的索引即可。这种简单的算法速度也很快，可以快速定位数据结构。

[^4]或中间位。结构数组地址的最低位可能有太多冲突。

对于内存中的一系列相邻锁，当锁落入同一高速缓存行时，可能会出现性能问题。两个 CPU 更新同一缓存行中的不同锁将遇到缓存一致性开销，每个 CPU 都会使对方缓存中的缓存行无效。这种情况称为错误共享，通常通过用未使用的字节填充锁来解决，以便内存中的每个缓存行中仅存在一个锁。

#### 5.2.6 非阻塞 I/O

Unix 进程生命周期，如第 3 章操作系统所示，显示进程在 I/O 期间阻塞并进入睡眠状态。该模型存在一些性能问题：

*   每个 I/O 操作在阻塞时都会消耗一个线程（或进程）。为了支持许多并发 I/O，应用程序必须创建许多线程（通常每个客户端一个），这会产生与线程创建和销毁相关的成本，以及保留它们所需的堆栈空间。
    
*   对于频繁的短期 I/O，频繁上下文切换的开销会消耗 CPU 资源并增加应用程序延迟。
    

非阻塞 I/O 模型异步发出 I/O，不会阻塞当前线程，然后该线程可以执行其他工作。这是 Node.js \[Node.js 20\] 的一个关键特性，Node.js 是一个服务器端 JavaScript 应用程序环境，指导代码以非阻塞方式开发。

有多种机制可以执行非阻塞或异步 I/O，包括：

*   open(2)：通过 O\_ASYNC 标志。当文件描述符上的 I/O 变为可能时，会使用信号通知进程。
    
*   io\_submit(2)：Linux 异步 I/O (AIO)。
    
*   sendfile(2)：这会将数据从一个文件描述符复制到另一个文件描述符，将 I/O 推迟到内核而不是用户级 I/O。[^5]
    
    [^5]Netflix CDN 使用它向客户发送视频资产，而无需用户级 I/O 开销。
    
*   io\_uring\_enter(2)：Linux io\_uring 允许使用在用户和内核空间之间共享的环形缓冲区提交异步 I/O \[Axboe 19\]。
    

检查操作系统文档以了解其他方法。

#### 5.2.7 处理器绑定

对于 NUMA 环境，进程或线程在单个 CPU 上保持运行并在执行 I/O 后在与之前相同的 CPU 上运行可能是有利的。这可以提高应用程序的内存局部性，减少内存 I/O 周期并提高整体应用程序性能。操作系统非常清楚这一点，并且旨在将应用程序线程保持在相同的 CPU 上（CPU 亲和性）。这些主题在第 7 章“内存”中介绍。

某些应用程序通过将自身绑定到 CPU 来强制执行此行为。这可以显着提高某些系统的性能。当绑定与其他 CPU 绑定（例如设备中断映射到 CPU）冲突时，它还会降低性能。

当同一系统上运行其他租户或应用程序时，请特别注意 CPU 绑定的风险。这是我在操作系统虚拟化（容器）环境中遇到的一个问题，其中应用程序可以看到所有 CPU，然后绑定到某些 CPU，假设它是服务器上唯一的应用程序。当服务器被其他也绑定的租户应用程序共享时，多个租户可能会在不知不觉中绑定到相同的 CPU，从而导致 CPU 争用和调度程序延迟，即使其他 CPU 处于空闲状态。

在应用程序的生命周期中，主机系统也可能会发生变化，未更新的绑定可能会损害而不是帮助性能，例如，当它们不必要地跨多个套接字绑定到 CPU 时。

#### 5.2.8 性能咒语

有关提高应用程序性能的更多技术，请参阅第 2 章中的 Performance Mantras 方法。 总之：

1.  不要这样做。
    
2.  做一次，但不要再做一次。
    
3.  少做一点吧
    
4.  稍后再做。
    
5.  当他们不注意的时候就做。
    
6.  同时进行。
    
7.  做起来更便宜。
    

第一项“不要这样做”是消除不必要的工作。有关此方法的更多详细信息，请参阅第 2 章“方法”、第 2.5.20 节“绩效准则”。

### 5.3 编程语言

编程语言可以被编译或解释，也可以通过虚拟机执行。许多语言将“性能优化”列为功能，但严格来说，这些通常是执行该语言的软件的功能，而不是语言本身。例如，Java HotSpot 虚拟机软件包含一个即时 (JIT) 编译器，可动态提高性能。

解释器和语言虚拟机还通过自己的特定工具提供不同级别的性能可观察性支持。对于系统性能分析师来说，使用这些工具进行基本分析可以快速取得成果。例如，高 CPU 使用率可能会被识别为垃圾收集 (GC) 的结果，然后通过一些常用的可调参数进行修复。或者它可能是由代码路径引起的，该代码路径可以在错误数据库中作为已知错误找到并通过升级软件版本来修复（这种情况经常发生）。

以下部分描述了每种编程语言类型的基本性能特征。有关个人语言表现的更多信息，请查找有关该语言的文本。

#### 5.3.1 编译语言

编译采用程序并在运行时之前生成机器指令，这些指令存储在称为二进制文件的二进制可执行文件中，通常在 Linux 和其他 Unix 衍生版本上使用可执行和链接格式 (ELF)，在 Windows 上使用可移植可执行 (PE) 格式。这些可以随时运行，无需再次编译。编译语言包括 C、C++ 和汇编语言。有些语言可能同时具有解释器和编译器。

编译后的代码通常具有高性能，因为在 CPU 执行之前不需要进一步翻译。编译代码的一个常见示例是 Linux 内核，它主要是用 C 语言编写的，其中一些关键路径是用汇编语言编写的。

编译语言的性能分析通常很简单，因为执行的机器代码通常与原始程序紧密映射（尽管这取决于编译优化）。在编译期间，可以生成将地址映射到程序函数和对象名称的符号表。稍后对 CPU 执行的分析和跟踪可以直接映射到这些程序名称，从而允许分析人员研究程序执行。堆栈跟踪及其包含的数字地址也可以映射并转换为函数名称以提供代码路径祖先。

编译器可以通过使用编译器优化（优化 CPU 指令的选择和放置的例程）来提高性能。

#### 5.3.2 解释语言

解释型语言通过在运行时将程序翻译成动作来执行程序，这个过程会增加执行开销。解释型语言预计不会表现出高性能，并且用于其他因素更重要的情况，例如易于编程和调试。 Shell 脚本是解释性语言的一个示例。

除非提供可观察性工具，否则解释型语言的性能分析可能会很困难。 CPU 分析可以显示解释器的操作（包括解析、翻译和执行操作），但它可能无法显示原始程序函数名称，从而使基本的程序上下文成为一个谜。这种解释器分析可能并非完全没有结果，因为解释器本身可能存在性能问题，即使它正在执行的代码看起来设计得很好。

根据解释器的不同，程序上下文可以作为解释器函数的参数使用，这可以使用动态检测来查看。另一种方法是在了解程序布局的情况下检查进程的内存（例如，使用 Linux process\_vm\_readv(2) 系统调用）。

通常，通过简单地添加打印语句和时间戳来研究这些程序。更严格的性能分析并不常见，因为解释语言通常不会首先被选择用于高性能应用程序。

#### 5.3.3 虚拟机

语言虚拟机（也称为进程虚拟机）是模拟计算机的软件。一些编程语言（包括 Java 和 Erlang）通常使用虚拟机 (VM) 执行，虚拟机为它们提供了独立于平台的编程环境。应用程序被编译为虚拟机指令集（字节码），然后由虚拟机执行。这允许编译对象的可移植性，前提是虚拟机可用于在目标平台上运行它们。

字节码可以由语言虚拟机以不同的方式执行。 Java HotSpot 虚拟机支持通过解释执行以及 JIT 编译，后者将字节码编译为机器代码以供处理器直接执行。这提供了编译代码的性能优势以及虚拟机的可移植性。

虚拟机通常是最难观察的语言类型。当程序在 CPU 上执行时，编译或解释的多个阶段可能已经过去，并且有关原始程序的信息可能不容易获得。性能分析通常侧重于语言虚拟机提供的工具集（其中许多提供 USDT 探针）和第三方工具。

#### 5.3.4 垃圾收集

某些语言使用自动内存管理，其中分配的内存不需要显式释放，将其留给异步垃圾收集过程。虽然这使得程序更容易编写，但也存在缺点：

*   内存增长：对应用程序内存使用的控制较少，当对象没有自动识别为有资格释放时，内存使用可能会增长。如果应用程序变得太大，它可能会达到自身的限制或遇到系统分页（Linux 交换），从而严重损害性能。
    
*   CPU 成本：GC 通常会间歇性运行，并涉及搜索或扫描内存中的对象。这会消耗 CPU 资源，从而减少应用程序短期内可用的资源。随着应用程序内存的增加，GC 的 CPU 消耗也可能会增加。在某些情况下，这可能会达到 GC 持续消耗整个 CPU 的程度。
    
*   延迟异常值：GC 执行时应用程序执行可能会暂停，导致偶尔出现高延迟的应用程序响应，并被 GC 中断。[^8] 这取决于 GC 类型：stop-the-world、增量或并发。
    

[^8]人们在减少 GC 时间或 GC 造成的应用程序中断的技术方面开展了大量工作。一个示例使用系统和应用程序指标来确定何时最好调用 GC \[Schwartz 18\]。

GC 是性能调整的常见目标，以降低 CPU 成本和延迟异常值的出现。例如，Java VM 提供了许多可调参数来设置 GC 类型、GC 线程数、最大堆大小、目标堆空闲比率等。

如果调整无效，问题可能是应用程序创建了太多垃圾或泄漏引用。这些都是应用程序开发人员需要解决的问题。一种方法是尽可能分配更少的对象，以减少 GC 负载。显示对象分配及其代码路径的可观察性工具可用于查找潜在的消除目标。

#### 5.4.1 CPU 分析

CPU 分析是应用程序性能分析的一项重要活动，在第 6 章“CPU”中从第 6.5.4 节“分析”开始进行了解释。本节总结了 CPU 分析和 CPU 火焰图，并描述了如何将 CPU 分析用于某些脱离 CPU 的分析。

Linux 有许多 CPU 分析器，包括 perf(1) 和 profile(8)，在第 5.5 节“可观察性工具”中进行了总结，它们都使用定时采样。这些分析器在内核模式下运行，可以捕获内核堆栈和用户堆栈，从而生成混合模式配置文件。这提供了（几乎）完整的 CPU 使用情况可见性。

应用程序和运行时有时会提供自己的在用户模式下运行的分析器，该分析器无法显示内核 CPU 使用情况。这些基于用户的分析器可能对 CPU 时间有一个扭曲的概念，因为它们可能不知道内核何时取消调度应用程序，并且不考虑它。我总是从基于内核的分析器（perf(1) 和 profile(8)）开始，并使用基于用户的分析器作为最后的手段。

基于样本的分析器会生成许多样本：Netflix 的典型 CPU 配置文件会以 49 赫兹的速度在（大约）32 个 CPU 上收集堆栈跟踪，持续 30 秒：这总共会生成 47,040 个样本。为了理解这些，分析器通常提供不同的方法来总结或可视化它们。采样堆栈跟踪的常用可视化称为火焰图，这是我发明的。

我不知道 crc32\_z() 或 MYSQL\_BIN\_LOG::commit() 做什么（尽管我可能猜得到）。 CPU 配置文件公开了应用程序的内部工作原理，除非您是应用程序开发人员，否则您不需要知道这些功能是什么。您需要研究它们以开发可行的性能改进。

例如，我在互联网上搜索 MYSQL\_BIN\_LOG::commit() 并很快找到了描述 MySQL 二进制日志记录（用于数据库恢复和复制）以及如何完全调整或禁用它的文章。快速搜索 crc32\_z() 显示它是 zlib 中的校验和函数。也许有更新更快的 zlib 版本？处理器是否具有优化的 CRC 指令，zlib 是否使用它？ MySQL是否需要计算CRC，或者可以将其关闭吗？有关这种思维方式的更多信息，请参阅第 2 章“方法论”、第 2.5.20 节“绩效咒语”。

##### CPU 外占用空间

CPU 配置文件不仅仅可以显示 CPU 使用情况。您可以查找其他 CPU 外问题类型的证据。例如，磁盘 I/O 在某种程度上可以通过文件系统访问和块 I/O 初始化的 CPU 使用情况来看出。这就像寻找熊的脚印：你没有看到熊，但你发现了一只熊的存在。

通过浏览 CPU 火焰图，您可能会发现文件系统 I/O、磁盘 I/O、网络 I/O、锁争用等的证据。图 5.3 突出显示了 ext4 文件系统 I/O 作为示例。如果您浏览了足够多的火焰图，您将熟悉要查找的函数名称：“tcp\_\*”表示内核 TCP 函数，“blk\_\*”表示内核块 I/O 函数等。以下是一些建议的搜索术语Linux系统：

*   “ext4”（或“btrfs”、“xfs”、“zfs”）：查找文件系统操作。
    
*   “blk”：查找块 I/O。
    
*   “tcp”：查找网络 I/O。
    
*   “utex”：显示锁争用（“mutex”或“futex”）。
    
*   “alloc”或“object”：显示执行内存分配的代码路径。
    

此方法仅识别这些活动的存在，而不是其强度。 CPU 火焰图显示了 CPU 使用率的大小，而不是 CPU 被阻塞所花费的时间。要直接测量脱离 CPU 的时间，您可以使用脱离 CPU 的分析（接下来将介绍），尽管测量的开销通常更大。

#### 5.4.2 脱离CPU分析

Off-CPU 分析是对当前不在 CPU 上运行的线程的研究：这种状态称为 off-CPU。它包括线程阻塞的所有原因：磁盘 I/O、网络 I/O、锁争用、显式睡眠、调度程序抢占等。对这些原因及其引起的性能问题的分析通常涉及多种工具。脱离 CPU 分析是一种分析所有这些方法的方法，并且可由单个脱离 CPU 分析工具支持。

脱离 CPU 分析可以通过不同的方式执行，包括：

*   采样：收集脱离 CPU 的线程的基于计时器的样本，或者只是收集所有线程（称为挂钟采样）。
    
*   调度程序跟踪：检测内核 CPU 调度程序以计算线程脱离 CPU 的持续时间，并使用脱离 CPU 堆栈跟踪记录这些时间。当线程脱离 CPU 时，堆栈跟踪不会更改（因为它不会运行来更改它），因此只需为每个阻塞事件读取一次堆栈跟踪。
    
*   应用程序检测：某些应用程序具有用于常见阻塞代码路径的内置检测，例如磁盘 I/O。此类检测可以包括特定于应用程序的上下文。虽然方便且有用，但这种方法通常对 CPU 外事件（调度程序抢占、页面错误等）视而不见。
    

前两种方法更可取，因为它们适用于所有应用程序并且可以查看所有脱离 CPU 的事件；然而，它们会带来很大的开销。在 8 CPU 系统上，以 49 赫兹进行采样的开销可以忽略不计，但 CPU 外采样必须对线程池而不是 CPU 池进行采样。同一个系统可能有 10,000 个线程，其中大部分是空闲的，因此对它们进行采样会增加 1,000x[^9] 的开销（想象一下对 10,000 个 CPU 的系统进行 CPU 分析）。调度程序跟踪也会消耗大量开销，因为同一系统每秒可能有 100,000 个或更多调度程序事件。

[^9]可能更高，因为它需要对脱离 CPU 的线程进行堆栈跟踪采样，并且它们的堆栈不太可能被 CPU 缓存（与 CPU 分析不同）。将其限制为单个应用程序应该有助于减少线程数，尽管配置文件将不完整。

调度程序跟踪是现在常用的技术，基于我自己的工具，例如 offcputime(8)（第 5.5.3 节，offcputime）。我使用的优化是仅记录超过极短持续时间的脱离 CPU 事件，这会减少样本数量。[^10] 我还使用 BPF 在内核上下文中聚合堆栈，而不是将所有样本发送到用户空间，进一步减少开销。尽管这些技术很有帮助，但您应该小心生产中的 CPU 外分析，并在使用前评估测试环境中的开销。

[^10]在您说“如果此优化排除了大量微小的睡眠持续时间怎么办？”之前，您应该在 CPU 配置文件中看到这一点的证据，因为如此频繁地调用调度程序，促使您转向关闭此优化。

此火焰图中的大部分时间现在都处于类似的 pthread\_cond\_wait() 和 futex() 代码路径中：这些是等待工作的线程。火焰图中可以看到线程函数：从右到左依次是srv\_worker\_thread()、srv\_purge\_coordinator\_thread()、srv\_monitor\_thread()等。

有几种方法可以找到重要的脱离 CPU 时间：

*   放大（或过滤）应用程序请求处理函数，因为我们最关心处理应用程序请求期间的 CPU 外时间。对于 MySQL 服务器，这是 do\_command() 函数。搜索 do\_command() 然后放大会生成与图 5.4 类似的火焰图。虽然这种方法很有效，但您需要知道在您的特定应用程序中要搜索什么功能。
    
*   在收集期间使用内核过滤器来排除不感兴趣的线程状态。有效性取决于内核；在 Linux 上，TASK\_UNINTERRUPTIBLE 上的匹配侧重于许多有趣的 CPU 外事件，但也排除了一些事件。
    

有时您会发现阻塞应用程序的代码路径正在等待其他东西，例如锁。为了进一步深入，您需要知道为什么锁的持有者花了这么长时间才释放它。除了第 5.4.7 节“静态性能调优”中描述的锁定分析之外，一种通用技术是检测唤醒事件。这是一项高级活动：请参阅 BPF 性能工具 \[Gregg 19\] 第 14 章，以及来自 BCC 的工具wakeuptime(8) 和 offwaketime(8)。

第 5.5.3 节，offcputime，显示了使用 BCC 中的 offcputime(8) 生成脱离 CPU 火焰图的指令。除了调度程序事件之外，系统调用事件是研究应用程序的另一个有用目标。

#### 5.4.3 系统调用分析

系统调用（syscalls）可以用于研究基于资源的性能问题。目的是找出系统调用时间花在哪里，包括系统调用的类型及其调用的原因。

系统调用分析的目标包括：

*   新进程跟踪：通过跟踪 execve(2) 系统调用，您可以记录新进程的执行情况，并分析短期进程的问题。请参阅第 5.5.5 节 execsnoop 中的 execsnoop(8) 工具。
    
*   I/O 分析：跟踪 read(2)/write(2)/send(2)/recv(2) 及其变体，并研究它们的 I/O 大小、标志和代码路径，将帮助您识别次优问题I/O，如大量小I/O。请参阅第 5.5.7 节 bpftrace 中的 bpftrace 工具。
    
*   内核时间分析：当系统显示大量内核 CPU 时间（通常报告为“%sys”）时，检测系统调用可以找到原因。请参阅第 5.5.6 节 syscount 中的 syscount(8) 工具。系统调用解释了大部分但不是全部的内核 CPU 时间；异常包括页面错误、异步内核线程和中断。
    

系统调用是一个记录良好的 API（手册页），使它们成为易于研究的事件源。它们还与应用程序同步调用，这意味着从系统调用收集堆栈跟踪将显示负责的应用程序代码路径。此类堆栈跟踪可以可视化为火焰图。

#### 5.4.4 使用方法

正如第 2 章“方法论”中介绍并在后面的章节中应用的那样，USE 方法检查所有硬件资源的利用率、饱和度和错误。通过显示资源已成为瓶颈，许多应用程序性能问题都可以通过这种方式解决。

USE 方法也可以应用于软件资源。如果您可以找到显示应用程序内部组件的功能图，请考虑每个软件资源的利用率、饱和度和错误指标，并看看什么是有意义的。

例如，应用程序可以使用工作线程池来处理请求，并使用等待轮流的请求队列。将其视为资源，然后可以通过以下方式定义三个指标：

*   利用率：在一个时间间隔内忙于处理请求的线程的平均数量，以总线程的百分比表示。例如，50% 意味着平均有一半线程忙于处理请求。
    
*   饱和度：一个时间间隔内请求队列的平均长度。这显示有多少请求已备份等待工作线程。
    
*   错误：请求因任何原因被拒绝或失败。
    

然后，您的任务是找到如何衡量这些指标。它们可能已经由应用程序在某处提供，或者可能需要使用其他工具（例如动态跟踪）添加或测量。

像这个例子一样，排队系统也可以使用排队论来研究（参见第 2 章，方法论）。

对于不同的示例，请考虑文件描述符。系统可能会施加限制，使得这些资源成为有限的。这三个指标可能如下：

*   利用率：正在使用的文件描述符的数量，以限制的百分比表示
    
*   饱和度：取决于操作系统行为：如果线程阻塞等待文件描述符，则这可能是等待该资源的阻塞线程数
    
*   错误：分配错误，例如EFILE、“打开文件太多”
    

对应用程序的组件重复此练习，并跳过任何没有意义的指标。此过程可以帮助您制定一个简短的清单，用于在继续使用其他方法之前检查应用程序的运行状况。

#### 5.4.5 线程状态分析

这是我针对每个性能问题使用的第一个方法，但它也是 Linux 上的高级活动。目标是在较高级别上识别应用程序线程将时间花在哪里，从而立即解决一些问题，并指导其他问题的调查。您可以通过将每个应用程序的线程时间划分为多个有意义的状态来实现此目的。

至少有两种线程状态：CPU 上和 CPU 外。您可以使用标准指标和工具（例如 top(1)）来识别线程是否处于 CPU 上状态，并根据情况进行 CPU 分析或 CPU 外分析（请参阅第 5.4.1 节“CPU 分析”和第 5.4 节） .2，CPU 外分析）。这种方法对于更多的状态更有效。

##### 九个州

这是我选择的九个线程状态的列表，以便为分析提供比两个早期状态（CPU 上和 CPU 外）更好的起点：

*   用户：用户模式下的 CPU 上
    
*   内核：内核模式下的 On-CPU
    
*   可运行：并且关闭 CPU 等待打开 CPU
    
*   交换（匿名分页）：可运行，但阻止匿名分页
    
*   磁盘 I/O：等待块设备 I/O：读/写、数据/文本换页
    
*   Net I/O：等待网络设备I/O：套接字读/写
    
*   睡眠：自愿睡眠
    
*   Lock：等待获取同步锁（等待别人）
    
*   空闲：等待工作
    

这个九状态模型如图 5.6 所示。

通过减少除空闲之外的每个状态的时间，可以提高应用程序请求的性能。在其他条件相同的情况下，这意味着应用程序请求具有较低的延迟，并且应用程序可以处理更多的负载。

一旦确定了线程花费时间的状态，您就可以进一步调查它们：

*   用户或内核：分析可以确定哪些代码路径正在消耗 CPU，包括旋转锁所花费的时间。请参见第 5.4.1 节，CPU 分析。
    
*   可运行：处于此状态的时间意味着应用程序需要更多的 CPU 资源。检查整个系统的 CPU 负载以及应用程序存在的任何 CPU 限制（例如资源控制）。
    
*   交换（匿名分页）：应用程序缺乏可用主内存可能会导致交换延迟。检查整个系统的内存使用情况以及存在的任何内存限制。有关详细信息，请参阅第 7 章“内存”。
    
*   磁盘：此状态包括直接磁盘 I/O 和页面错误。要进行分析，请参阅第 5.4.3 节“系统调用分析”、第 8 章“文件系统”和第 9 章“磁盘”。工作负载表征可以帮助解决许多磁盘 I/O 问题；检查文件名、I/O 大小和 I/O 类型。
    
*   网络：此状态用于网络 I/O（发送/接收）期间阻塞的时间，但不侦听新连接（即空闲时间）。分析请参见第 5.4.3 节“系统调用分析”；第 5.5.7 节，bpftrace 和 I/O 分析标题；以及第 10 章，网络。工作负载特征对于网络 I/O 问题也很有用；检查主机名、协议和吞吐量。
    
*   睡眠：分析睡眠的原因（代码路径）和持续时间。
    
*   锁：识别锁、固定锁的螺纹以及锁持有者长时间持有锁的原因。原因可能是支架被另一把锁卡住了，需要进一步解开。这是一项高级活动，通常由对应用程序及其锁定层次结构有深入了解的软件开发人员执行。我开发了一个 BCC 工具来帮助进行此类分析：offwaketime(8)（包含在 BCC 中），它显示阻塞堆栈跟踪以及唤醒器。
    

由于应用程序通常等待工作的方式，您经常会发现处于网络 I/O 和锁定状态的时间实际上是空闲时间。应用程序工作线程可以通过等待网络 I/O 等待下一个请求（例如，HTTP 保持活动）或通过等待条件变量（锁定状态）被唤醒来处理工作来实现空闲。

下面总结了如何在 Linux 上测量这些线程状态。

图5.7 Linux线程状态

内核线程状态基于内核task\_struct状态成员：Runnable为TASK\_RUNNING，Disk为TASK\_UNINTERRUPTIBLE，Sleep为TASK\_INTERRUPTIBLE。这些状态由 ps(1) 和 top(1) 等工具分别使用单字母代码 R、D 和 S 来显示。 （还有更多状态，例如被调试器停止，我没有在此处包含。）

虽然这为进一步分析提供了一些线索，但它还远远不能将时间划分为前面描述的九种状态。需要更多信息：例如，可以使用 /proc 或 getrusage(2) 统计信息将 Runnable 分为用户时间和内核时间。

其他内核通常提供更多状态，使这种方法更容易应用。我最初在 Solaris 内核上开发并使用了这种方法，受到其微状态记账功能的启发，该功能记录了八种不同状态下的线程时间：用户、系统、陷阱、文本错误、数据错误、锁定、睡眠和运行队列（调度程序延迟） ）。这些与我的理想状态不符，但却是一个更好的起点。

我将讨论我在 Linux 上使用的三种方法：基于线索、脱离 CPU 的分析和直接测量。

###### 基于线索

您可以首先使用常见的操作系统工具（例如 pidstat(1) 和 vmstat(8)）来建议线程状态时间可能花费在何处。感兴趣的工具和列是：

*   用户：pidstat(1)“%usr”（这个状态是直接测量的）
    
*   内核：pidstat(1)“%system”（这个状态是直接测量的）
    
*   可运行：vmstat(8)“r”（系统范围）
    
*   交换：vmstat(8)“si”和“so”（系统范围）
    
*   磁盘 I/O：pidstat(1) `-d`“iodelay”（包括交换状态）
    
*   网络 I/O：sar(1) `-n DEV`“rxkB/s”和“txkB/s”（系统范围）
    
*   睡觉：不容易
    
*   Lock: perf(1) `top` （可直接识别自旋锁时间）
    
*   空闲：不容易获得
    

其中一些统计数据是全系统范围的。如果您通过 vmstat(8) 发现存在系统范围的交换率，您可以使用更深入的工具调查该状态，以确认应用程序是否受到影响。以下各节和章节将介绍这些工具。

###### CPU 外分析

由于许多状态都是脱离 CPU 的（除了用户和内核之外的所有状态），因此您可以应用脱离 CPU 分析来确定线程状态。请参见第 5.4.2 节，脱离 CPU 分析。

###### 直接测量

通过线程状态精确测量线程时间如下：

用户：用户模式 CPU 可通过许多工具以及 /proc/PID/stat 和 getrusage(2) 中获得。 pidstat(1) 将其报告为 `%usr`。

内核：内核模式CPU也在/proc/PID/stat和getrusage(2)中。 pidstat(1) 将其报告为 `%system`。

可运行：由内核 schedstats 功能以纳秒为单位进行跟踪，并通过 /proc/PID/schedstat 公开。也可以使用跟踪工具（包括 perf(1) `sched` 子命令和 BCC runqlat(8)）来测量，但需要一些开销，这两个工具都在第 6 章“CPU”中介绍。

交换：以纳秒为单位的时间交换（匿名寻呼）可以通过延迟核算来测量，延迟核算在第 4 章“可观测性工具”、第 4.3.3 节“延迟核算”中介绍，其中包括一个示例工具：getdelays.c。跟踪工具也可用于检测交换延迟。

磁盘：pidstat(1) `-d` 将“iodelay”显示为时钟周期数，在此期间进程因块 I/O 和交换而延迟；如果没有系统范围的交换（如 vmstat(8) 所报告），您可以得出结论，任何 iodelay 都是 I/O 状态。延迟记账和其他记账功能（如果启用）还提供块 I/O 时间，如 iotop(8) 使用的那样。您还可以使用跟踪工具，例如 BCC 的 biotop(8)。

网络：可以使用 BCC 和 bpftrace 等跟踪工具来调查网络 I/O，包括用于 TCP 网络 I/O 的 tcptop(8) 工具。应用程序还可能具有跟踪 I/O（网络和磁盘）时间的工具。

睡眠：可以使用跟踪器和事件（包括 syscalls:sys\_enter\_nanosleep 跟踪点）检查进入自愿睡眠的时间。我的naptime.bt 工具跟踪这些睡眠并打印PID 和持续时间\[Gregg 19\]\[Gregg 20b\]。

锁定：可以使用跟踪工具来调查锁定时间，包括来自 BCC 的 klockstat(8)，以及来自 bpf-perf-tools-book 存储库的用于 pthread 互斥锁的 pmlock.bt 和 pmheld.bt，以及 mlock.bt 和 mheld。 bt 用于内核互斥体。

空闲：跟踪工具可用于检测处理等待工作的应用程序代码路径。

有时，应用程序可能看起来完全处于睡眠状态：它们在 CPU 之外保持阻塞状态，没有 I/O 或其他事件的速率。要确定应用程序线程处于什么状态，可能需要使用调试器（例如 pstack(1) 或 gdb(1)）来检查线程堆栈跟踪，或从 /proc/PID/stack 文件中读取它们。请注意，此类调试器可能会暂停目标应用程序并导致其自身的性能问题：在生产中尝试它们之前，请先了解如何使用它们及其风险。

#### 5.4.6 锁分析

对于多线程应用程序，锁可能成为瓶颈，抑制并行性和可扩展性。单线程应用程序可以被内核锁（例如文件系统锁）抑制。锁可以通过以下方式进行分析：

*   检查争用
    
*   检查保持时间是否过长
    

首先确定现在是否存在问题。过多的保持时间不一定是一个迫在眉睫的问题，但将来随着更多的并行负载，它们可能会变得如此。对于每个锁，尝试识别锁的名称（如果存在）以及导致使用它的代码路径。

虽然有用于锁定分析的专用工具，但有时您可以仅通过 CPU 分析来解决问题。对于自旋锁，争用显示为 CPU 使用情况，并且可以使用堆栈跟踪的 CPU 分析轻松识别。对于自适应互斥锁，争用通常涉及一些旋转，这也可以通过堆栈跟踪的 CPU 分析来识别。在这种情况下，请注意 CPU 配置文件仅提供了部分情况，因为线程在等待锁时可能已阻塞并休眠。请参见第 5.4.1 节，CPU 分析。

有关 Linux 上的特定锁分析工具，请参阅第 5.5.7 节 bpftrace。

#### 5.4.7 静态性能调优

静态性能调优重点关注配置环境的问题。对于应用程序性能，请检查静态配置的以下方面：

*   应用程序正在运行什么版本，它的依赖项是什么？有更新的版本吗？他们的发行说明是否提到了性能改进？
    
*   是否存在已知的性能问题？是否有列出它们的错误数据库？
    
*   应用程序是如何配置的？
    
*   如果配置或调整与默认值不同，原因是什么？ （是基于测量和分析，还是猜测？）
    
*   应用程序是否使用对象缓存？它的大小如何？
    
*   应用程序是否同时运行？如何配置（例如，线程池大小）？
    
*   应用程序是否在特殊模式下运行？ （例如，调试模式可能已启用并降低了性能，或者应用程序可能是调试版本而不是发布版本。）
    
*   该应用程序使用哪些系统库？它们是什么版本？
    
*   应用程序使用什么内存分配器？
    
*   应用程序是否配置为在其堆中使用大页面？
    
*   应用程序是否已编译？什么版本的编译器？有哪些编译器选项和优化？ 64 位？
    
*   本机代码是否包含高级指令？ （应该吗？）（例如，SIMD/向量指令，包括英特尔 SSE。）
    
*   应用程序是否遇到错误，现在是否处于降级模式？或者它是否配置错误并且始终以降级模式运行？
    
*   是否存在系统对 CPU、内存、文件系统、磁盘或网络使用施加的限制或资源控制？ （这些在云计算中很常见。）
    

回答这些问题可能会揭示被忽视的配置选择。

#### 5.4.8 分布式追踪

在分布式环境中，应用程序可能由在单独系统上运行的服务组成。虽然每个服务都可以像它自己的小型应用程序一样进行研究，但也有必要将分布式应用程序作为一个整体进行研究。这需要新的方法和工具，并且通常使用分布式跟踪来执行。

分布式跟踪涉及记录每个服务请求的信息，然后组合这些信息进行研究。然后，跨多个服务的每个应用程序请求都可以分解为其依赖项请求，并且可以识别导致高应用程序延迟或错误的服务。

收集的信息可包括：

*   应用程序请求的唯一标识符（外部请求ID）
    
*   有关其在依赖关系层次结构中的位置的信息
    
*   开始和结束时间
    
*   错误状态
    

分布式跟踪的一个挑战是生成的日志数据量：每个应用程序请求有多个条目。一种解决方案是执行基于头部的采样，在请求开始（“头部”）时，决定是否对其进行采样（“跟踪”）：例如，跟踪每万个请求中的一个。这足以分析大量请求的性能，但由于数据有限，可能会导致间歇性错误或异常值的分析变得困难。一些分布式跟踪器是基于尾部的，首先捕获所有事件，然后可能根据延迟和错误决定保留哪些事件。

一旦发现有问题的服务，就可以使用其他方法和工具对其进行更详细的分析。

###### strace 开销

警告：当前版本的 strace(1) 通过 Linux ptrace(2) 接口采用基于断点的跟踪。这会为所有系统调用的入口和返回设置断点（即使 `-e` 选项仅用于选择某些系统调用）。这是侵入性的，系统调用率较高的应用程序可能会发现其性能下降了一个数量级。为了说明这一点，下面是没有 strace(1) 的相同 dd(1) 命令：

点击这里查看代码图片

$ **dd if=/dev/zero of=/dev/null bs=1k count=5000k**
5120000+0 records in
5120000+0 records out
5242880000 bytes (5.2 GB) copied, 1.91247 s, 2.7 GB/s

dd(1) 包括最后一行的吞吐量统计信息：通过比较它们，我们可以得出结论，strace(1) 将 dd(1) 的速度减慢了 73 倍。这是一个特别严重的情况，因为 dd(1) 的性能很高系统调用率。

根据应用程序的要求，这种跟踪方式可以在短时间内使用，以确定正在调用的系统调用类型。如果开销不是一个问题的话，strace(1) 会有更大的用处。其他跟踪器（包括 perf(1)、Ftrace、BCC 和 bpftrace）通过使用缓冲跟踪大大减少了跟踪开销，其中事件被写入共享内核环形缓冲区，并且用户级跟踪器定期读取缓冲区。这减少了用户和内核上下文之间的上下文切换，从而降低了开销。

strace(1) 的未来版本可以通过成为 perf(1) `trace` 子命令（前面在第 5.5.1 节 perf 中描述）的别名来解决其开销问题。其他基于 BPF 的高性能 Linux 系统调用跟踪器包括：Intel 的 vltrace \[Intel 18\] 和 Microsoft 的 Windows ProcMon 工具的 Linux 版本 \[Microsoft 20\]。

#### 5.5.7 bpftrace

bpftrace 是一种基于 BPF 的跟踪器，它提供了一种高级编程语言，允许创建强大的单行代码和短脚本。它非常适合根据其他工具的线索进行自定义应用程序分析。

##### 应用程序内部结构

如果需要，您可以开发自定义工具来总结应用程序内部结构。首先检查 USDT 探针是否可用，或者是否可以使用（通常通过使用选项重新编译）。如果这些无法提供或不足，请考虑使用 uprobes。有关 bpftrace、uprobes 和 USDT 的示例，请参阅第 4 章“可观测性工具”、第 4.3.7 节“uprobes”和第 4.3.8 节“USDT”。第 4.3.8 节还描述了动态 USDT，这对于深入了解 JIT 编译的软件可能是必要的，而 uprobes 可能无法检测。

一个复杂的例子是 Java：uprobes 可以检测 JVM 运行时（C++ 代码）和操作系统库，USDT 可以检测高级 JVM 事件，动态 USDT 可以放置在 Java 代码中以提供对方法执行的洞察。

### 5.6 陷阱

以下部分描述了应用程序性能分析的常见问题，特别是缺少符号和堆栈跟踪。您可能首先在检查 CPU 配置文件（例如火焰图）时遇到这些问题，并发现它缺少函数名称和堆栈跟踪。

这些问题是一个高级主题，我在 BPF 性能工具的第 2、12 和 18 章中更详细地介绍了这些问题，并在此进行了总结。

#### 5.6.1 缺失符号

当探查器或跟踪器无法将应用程序指令地址解析为其函数名称（符号）时，它可能会将其打印为十六进制数字或字符串“\[unknown\]”。此问题的修复取决于应用程序的编译器、运行时、调优以及分析器本身。

##### ELF 二进制文件（C、C++、...）

已编译的二进制文件中可能会缺少符号，尤其是那些打包和分发的二进制文件，因为它们已使用 strip(1) 进行处理以减小文件大小。一种解决方法是调整构建过程以避免剥离符号；另一种是使用不同的符号信息源，例如 debuginfo 或 BPF 类型格式 (BTF)。通过 perf(1)、BCC 和 bpftrace 进行的 Linux 分析支持 debuginfo 符号。

##### JIT 运行时（Java、Node.js...）

缺少符号通常发生在 Java 和 Node.js 等即时 (JIT) 编译器运行时。在这些情况下，JIT 编译器有自己的符号表，该符号表在运行时会发生变化，并且不属于二进制文件中预编译符号表的一部分。常见的修复方法是使用运行时生成的补充符号表，这些符号表放置在 perf(1) 和 BCC 读取的 /tmp/perf-<PID>.map 文件中。

例如，Netflix 使用 perf-map-agent \[Rudolph 18\]，它可以附加到实时 Java 进程并转储补充符号文件。我已经使用另一个名为 jmaps \[Gregg 20c\] 的工具自动化了它的使用，该工具应该在配置文件之后和符号转换之前立即运行。例如，使用 perf(1)（第 13 章）：

点击这里查看代码图片

\# **perf record -F 49 -a -g -- sleep 10; jmaps**
# **perf script --header > out.stacks**
# **\[...\]**

并使用 bpftrace（第 15 章）：

点击这里查看代码图片

\# **bpftrace --unsafe -e 'profile:hz:49 { @\[ustack\] = count(); }**
    **interval:s:10 { exit(); } END { system("jmaps"); }'**

配置文件示例和符号表转储之间的符号映射可能会发生变化，从而在配置文件中产生无效的函数名称。这称为符号搅动，在 `perf record` 之后立即运行 jmap 可以减少符号搅动。到目前为止，这还不是一个严重的问题；如果是，则可以在配置文件之前和之后进行符号转储以查找更改。

还有其他方法可以解析 JIT 符号。一种是使用符号时间戳日志记录，它由 perf(1) 支持并解决符号改动问题，尽管启用时开销较高。另一个是让 perf(1) 调用运行时自己的堆栈遍历器（通常存在于异常堆栈中）。这种方法有时使用堆栈助手来调用，对于 Java，它已由 async-profiler \[Pangin 20\] 实现。

请注意，JIT 运行时也有预编译组件：JVM 还使用 libjvm 和 libc。请参阅前面的 ELF 二进制文件部分来了解这些组件的地址。

#### 5.6.2 丢失堆栈

另一个常见问题是堆栈跟踪丢失或不完整，可能短至一两帧。例如，来自 MySQL 服务器的 CPU 外配置文件：

点击这里查看代码图片

    finish\_task\_switch
    schedule
    futex\_wait\_queue\_me
    futex\_wait
    do\_futex
    \_\_x64\_sys\_futex
    do\_syscall\_64
    entry\_SYSCALL\_64\_after\_hwframe
    pthread\_cond\_timedwait@@GLIBC\_2.3.2
    \[unknown\]

该堆栈不完整：pthread\_cond\_timedwait() 之后是单个“\[unknown\]”帧。它缺少下面的 MySQL 函数，而正是这些 MySQL 函数是我们真正需要了解应用程序上下文的。

有时堆栈是单个框架：

    send

在火焰图中，这可以显示为“草”：轮廓底部有许多薄的单框架。

不幸的是，不完整的堆栈跟踪很常见，通常是由两个因素共同造成的：1) 可观察性工具使用基于帧指针的方法来读取堆栈跟踪，2) 目标二进制文件未保留寄存器（x86\_64 上的 RBP） ）对于帧指针，而是将其重新用作通用寄存器以优化编译器性能。可观察性工具读取此寄存器，期望它是帧指针，但实际上它现在可以包含任何内容：数字、对象地址、指向字符串的指针等。可观察性工具尝试在符号表中解析此数字，如果它是幸运的是，它没有找到它并且可以打印“\[unknown\]”。如果不幸的话，该随机数将解析为不相关的符号，现在打印的堆栈跟踪有一个错误的函数名称，让最终用户感到困惑。

由于 libc 库通常是在没有帧指针的情况下编译的，因此在通过 libc 的任何路径中，损坏的堆栈都很常见，包括前面的两个示例：pthread\_cond\_timedwait() 和 send()。[^17]

[^17]我已向软件包维护者发送了构建指令，要求打包带有帧指针的 libc。

最简单的解决方案通常是修复帧指针寄存器：

*   对于 C/C++ 软件以及使用 gcc(1) 或 LLVM 编译的其他软件：使用 `-fno-omit-frame-pointer` 重新编译软件。
    
*   对于 Java：使用 `-XX:+PreserveFramePointer` 运行 java(1)。
    

这可能会带来性能成本，但通常测得不到 1%；能够使用堆栈跟踪来发现性能优势的好处通常远远超过此成本。

另一种方法是切换到不基于帧指针的堆栈遍历技术。 perf(1) 支持基于 DWARF 的堆栈遍历、ORC 和最后一个分支记录 (LBR)。其他堆栈遍历方法在第 13 章，perf，第 13.9 节，perf record 中提到。

截至撰写本文时，BPF 尚不支持基于 DWARF 和 LBR 堆栈遍历，并且 ORC 尚未可用于用户级软件。
