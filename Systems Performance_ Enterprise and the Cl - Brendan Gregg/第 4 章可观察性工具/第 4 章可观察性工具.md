
操作系统历史上提供了许多用于观察系统软件和硬件组件的工具。对于新手来说，广泛的可用工具和指标表明一切（或者至少是重要的一切）都可以被观察到。事实上，存在许多差距，系统性能专家变得熟练于推理和解释的艺术：从间接工具和统计数据中找出活动。例如，网络数据包可以单独检查（嗅探），但磁盘 I/O 不能（至少不容易）。

由于动态跟踪工具的兴起，包括基于 BPF 的 BCC 和 bpftrace，Linux 的可观察性得到了极大的提高。现在，黑暗的角落都被照亮了，包括使用 `biosnoop(8)` 的单个磁盘 I/O。然而，许多公司和商业监控产品尚未采用系统跟踪，并且错过了它带来的洞察力。作者通过开发、发布和解释新的跟踪工具来带头，这些工具已经被 Netflix 和 Facebook 等公司使用。

本章的学习目标是：

- 识别静态绩效工具和危机工具。
- 了解工具类型及其开销：计数器、分析和跟踪。
- 了解可观测性来源，包括：`/proc`、`/sys`、tracepoints、kprobes、uprobes、USDT 和 PMC。
- 了解如何配置 `sar(1)` 以归档统计数据。

在第一章中，作者介绍了不同类型的可观察性：计数器、分析和跟踪，以及静态和动态检测。本章详细介绍了可观测性工具及其数据源，包括系统活动报告器 `sar(1)` 的摘要以及跟踪工具的介绍。这为您提供了理解 Linux 可观察性的基础知识；后面的章节（6 到 11）使用这些工具和资源来解决特定问题。第 13 章到第 15 章深入介绍了示踪剂。

本章以 Ubuntu Linux 发行版为例；这些工具中的大多数在其他 Linux 发行版中都是相同的，并且对于这些工具起源的其他内核和操作系统也存在一些类似的工具。

有些工具适用于多个象限；例如，`top(1)` 还具有系统范围的摘要，系统范围的事件工具通常可以过滤特定进程（`-p PID`）。

基于事件的工具包括分析器和跟踪器。分析器通过拍摄一系列事件快照来观察活动，绘制目标的粗略图像。跟踪器检测每个感兴趣的事件，并可以对它们执行处理，例如生成定制计数器。第 1 章介绍了计数器、跟踪和分析。

以下各节介绍使用固定计数器、跟踪和分析，以及执行监视（指标）的 Linux 工具。

#### 4.2.1 固定计数器

内核维护各种计数器以提供系统统计信息。它们通常被实现为无符号整数，当事件发生时这些整数会递增。例如，有接收的网络数据包数量、发出的磁盘 I/O 以及发生的中断数量的计数器。这些由监控软件作为指标公开（参见第 4.2.4 节，监控）。

一种常见的内核方法是维护一对累积计数器：一个用于对事件进行计数，另一个用于记录事件的总时间。这些通过将总时间除以计数来直接提供事件的计数以及事件的平均时间（或延迟）。由于它们是累积的，因此通过以时间间隔（例如一秒）读取该对，可以计算增量，并从中计算出每秒计数和平均延迟。这是计算系统统计信息的常见方法。

在性能方面，计数器被认为是“免费”使用的，因为它们默认启用并由内核持续维护。使用它们时唯一的额外成本是从用户空间读取它们的值（这通常可以忽略不计）。以下示例工具可以在系统范围内或按进程读取这些计数器。

##### 全系统

这些工具使用内核计数器检查系统软件或硬件资源上下文中的系统范围活动。常见的 Linux 工具包括：

*   `vmstat(8)`：系统范围的虚拟和物理内存统计信息
*   `mpstat(1)`：每个 CPU 使用情况
*   `iostat(1)`：每个磁盘 I/O 使用情况，从块设备接口报告
*   `nstat(8)`：TCP/IP 堆栈统计信息
*   `sar(1)`：各种统计数据；还可以将它们存档以用于历史报告

这些工具通常可供系统上的所有用户（非 root 用户）查看。它们的统计数据通常也通过监控软件绘制成图表。

许多工具遵循相同的使用约定，它们接受可选的间隔和计数。例如，`vmstat(8)` 的间隔为一秒，输出计数为三：

##### 每个进程

这些工具是面向进程的，并使用内核为每个进程维护的计数器。 Linux 工具包括：

*   `ps(1)`：显示进程状态，显示各种进程统计信息，包括内存和 CPU 使用情况。
*   `top(1)`：显示排名靠前的进程，按 CPU 使用率或其他统计数据排序。
*   `pmap(1)`：列出进程内存段及其使用统计信息。

这些工具通常从 `/proc` 文件系统读取统计信息。

#### 4.2.2 分析

分析通过收集目标行为的一组样本或快照来表征目标。 CPU 使用情况是分析的常见目标，其中从指令指针或堆栈跟踪中获取基于计时器的样本来表征 CPU 消耗代码路径。这些样本通常以固定速率（例如 100 Hz（每秒周期））在所有 CPU 上收集，并且持续时间很短（例如一分钟）。分析工具或分析器通常使用 99 Hz 而不是 100 Hz，以避免与目标活动同步采样，这可能导致计数过高或过低。

分析还可以基于不定时的硬件事件，例如 CPU 硬件缓存未命中或总线活动。它可以显示哪些代码路径负责，这些信息尤其可以帮助开发人员优化其代码的内存使用情况。

与固定计数器不同，分析（和跟踪）通常仅根据需要启用，因为它们可能会花费一些 CPU 开销来收集，并消耗一些存储开销来存储。这些开销的大小取决于工具及其检测的事件发生率。基于计时器的分析器通常更安全：事件速率已知，因此可以预测其开销，并且可以选择事件速率以使其开销可忽略不计。

##### 全系统

系统范围的 Linux 分析器包括：

*   `perf(1)`：标准 Linux 分析器，包括分析子命令。
*   `profile(8)`：来自 BCC 存储库的基于 BPF 的 CPU 分析器（在第 15 章 BPF 中介绍），它对内核上下文中的堆栈跟踪进行频率计数。
*   英特尔 VTune Amplifier XE：Linux 和 Windows 分析，具有图形界面，包括源浏览。

这些也可用于针对单个进程。

##### 每个进程

面向流程的分析器包括：

*   `gprof(1)`：GNU 分析工具，用于分析编译器添加的分析信息（例如，`gcc -pg`）。
*   `cachegrind`：valgrind 工具包中的一个工具，可以分析硬件缓存使用情况（以及更多）并使用 kcachegrind 可视化配置文件。
*   Java Flight Recorder (JFR)：编程语言通常有自己的专用分析器，可以检查语言上下文。例如，用于 Java 的 JFR。

有关分析工具的更多信息，请参阅第 6 章“CPU”和第 13 章“性能”。

#### 4.2.3 追踪

跟踪仪器每次发生事件，并且可以存储基于事件的详细信息以供以后分析或生成摘要。这与分析类似，但目的是收集或检查所有事件，而不仅仅是样本。与分析相比，跟踪可能会产生更高的 CPU 和存储开销，这可能会减慢跟踪目标的速度。应该考虑到这一点，因为它可能会对生产工作负载产生负面影响，并且测量的时间戳也可能被跟踪器扭曲。与分析一样，跟踪通常仅在需要时使用。

日志记录（将错误和警告等不常见事件写入日志文件以供以后读取）可以被视为默认启用的低频跟踪。日志包括系统日志。

以下是系统范围和每个进程的跟踪工具的示例。

##### 全系统

这些跟踪工具使用内核跟踪工具在系统软件或硬件资源的上下文中检查系统范围的活动。 Linux 工具包括：

*   `tcpdump(8)`：网络数据包跟踪（使用 libpcap）
*   `biosnoop(8)`：块 I/O 跟踪（使用 BCC 或 bpftrace）
*   `execsnoop(8)`：新进程跟踪（使用 BCC 或 bpftrace）
*   `perf(1)`：标准 Linux 分析器，还可以跟踪事件
*   `perf trace`：一个特殊的 perf 子命令，用于跟踪系统范围内的系统调用
*   Ftrace：Linux 内置跟踪器
*   BCC：基于 BPF 的跟踪库和工具包
*   bpftrace：基于 BPF 的跟踪器 (bpftrace(8)) 和工具包

`perf(1)`、Ftrace、BCC 和 bpftrace 在第 4.5 节“跟踪工具”中介绍，并在第 13 章到第 15 章中详细介绍。使用 BCC 和 bpftrace 构建的跟踪工具有一百多个，包括 `biosnoop(8)` 和 `execsnoop (8)` 从此列表中。本书中提供了更多示例。

##### 每个进程

这些跟踪工具是面向进程的，它们所基于的操作系统框架也是如此。 Linux 工具包括：

*   `strace(1)`：系统调用跟踪
*   `gdb(1)`：源代码级调试器

调试器可以检查每个事件的数据，但它们必须通过停止和启动目标的执行来完成此操作。这可能会带来巨大的管理成本，使它们不适合生产使用。

系统范围的跟踪工具（例如 `perf(1)` 和 `bpftrace`）支持用于检查单个进程的过滤器，并且可以以低得多的开销运行，这使得它们在可用时成为首选。

#### 4.2.4 监控

第 2 章“方法论”中介绍了监控。与前面介绍的工具类型不同，监控会连续记录统计数据，以备以后需要。

##### SAR(1)

用于监视单个操作系统主机的传统工具是系统活动报告器 `sar(1)`，源自 AT&T Unix。 `sar(1)` 是基于计数器的，并且有一个代理在预定时间执行（通过 cron）来记录系统范围计数器的状态。 `sar(1)` 工具允许在命令行查看这些内容，例如：

```bash
$ **sar -u 1 3**
Linux 4.9.0-6-amd64 (hostname)  10/05/2024  _x86_64_    (2 CPU)

11:20:00 AM     CPU     %user     %nice   %system   %iowait    %steal     %idle
11:20:01 AM     all      5.40      0.00      3.00      0.50      0.00      91.10
11:20:02 AM     all      5.30      0.00      2.90      0.60      0.00      91.20
11:20:03 AM     all      5.25      0.00      3.05      0.45      0.00      91.25
```



默认情况下，`sar(1)` 读取其统计档案（如果启用）以打印最近的历史统计信息。您可以指定一个可选的时间间隔并对其进行计数，以按指定的速率检查当前活动。

` sar(1)` 可以记录数十种不同的统计数据，以深入了解 CPU、内存、磁盘、网络、中断、电源使用情况等。第 4.4 节 `sar` 对此进行了更详细的介绍。

第三方监控产品通常基于 `sar(1)` 或其使用的相同可观测性统计数据构建，并通过网络公开这些指标。

##### SNMP 简单网络管理协议

传统的网络监控技术是简单网络管理协议（SNMP）。设备和操作系统可以支持 SNMP，并且在某些情况下默认提供它，从而无需安装第三方代理或导出程序。 SNMP 包括许多基本的操作系统指标，尽管它尚未扩展到涵盖现代应用程序。大多数环境已改用基于自定义代理的监控。

##### Agents 代理商

现代监控软件在每个系统上运行代理（也称为导出器或插件）来记录内核和应用程序指标。其中可以包括针对特定应用程序和目标的代理，例如 MySQL 数据库服务器、Apache Web 服务器和 Memcached 缓存系统。此类代理可以提供单独从系统计数器无法获得的详细应用程序请求指标。

适用于 Linux 的监控软件和代理包括：

*   性能副驾驶 (PCP)：PCP 支持数十种不同的代理（称为性能度量域代理：PMDA），包括基于 BPF 的度量 \[PCP 20\]。
*   Prometheus：Prometheus 监控软件支持数十种不同的导出器，包括数据库、硬件、消息传递、存储、HTTP、API 和日志记录 \[Prometheus 20\]。
*   collectd：支持数十种不同的插件。

图 4.4 显示了一个示例监控架构，其中包括用于归档指标的监控数据库服务器和用于提供客户端 UI 的监控 Web 服务器。指标由代理发送（或提供）到数据库服务器，然后提供给客户端 UI 以折线图和仪表板形式显示。例如，Graphite Carbon是一个监控数据库服务器，Grafana是一个监控 Web 服务器/仪表板。

#### 4.3.1 `/proc`

这是用于内核统计的文件系统接口。`/proc` 包含许多目录，其中每个目录都以其代表的进程的进程 ID 命名。每个目录中都有许多文件，其中包含从内核数据结构映射的每个进程的信息和统计信息。`/proc` 中还有用于系统范围统计信息的附加文件。

`/proc` 由内核动态创建，不受存储设备支持（它在内存中运行）。它大部分是只读的，为可观察性工具提供统计数据。有些文件是可写的，用于控制进程和内核行为。

文件系统接口非常方便：它是一个直观的框架，通过目录树向用户态公开内核统计信息，并且通过 POSIX 文件系统调用具有众所周知的编程接口：`open()`、`read()`、`close()`。您还可以在命令行中使用 `cd`、`cat`、`grep` 和 `awk` 来探索它。文件系统还通过使用文件访问权限来提供用户级安全性。在无法执行典型进程可观察性工具（如 `ps`、`top` 等）的极少数情况下，仍然可以通过 `/proc` 目录中的 shell 内置命令执行某些进程调试。

读取大多数 `/proc` 文件的开销可以忽略不计；例外情况包括一些遍历页表的内存映射相关文件。

- `**fd**`：文件描述符符号链接的目录（另请参阅 `fdinfo`）
- `**cgroup**`：Cgroup 成员信息
- `**task**`：每个任务（线程）统计信息的目录

以下是 `top` 如何读取每个进程的统计信息，并使用 `strace` 进行跟踪：

bash

Copy code

`stat("/proc/14704", {st_mode=S_IFDIR|0555, st_size=0, ...}) = 0 open("/proc/14704/stat", O_RDONLY) = 4 read(4, "14704 (sshd) S 1 14704 14704 0 -"..., 1023) = 232 close(4)`

这会在以进程 ID（14704）命名的目录中打开一个名为 `stat` 的文件，然后读取文件内容。

`top` 会对系统上的所有活动进程重复此操作。在某些系统上，尤其是那些具有许多进程的系统上，执行这些操作的开销可能会变得非常明显，尤其是对于每次屏幕更新时都重复此序列的 `top` 版本。这可能会导致 `top` 报告 `top` 本身是 CPU 消耗最高的进程！

###### CPU 统计准确度

`/proc/stat` 文件提供系统范围的 CPU 利用率统计信息，并由许多工具（如 `vmstat`、`mpstat`、`sar`、监视代理）使用。这些统计信息的准确性取决于内核配置。默认配置 (`CONFIG_TICK_CPU_ACCOUNTING`) 以时钟周期的粒度 Weisbecker13Weisbecker 13Weisbecker13 测量 CPU 利用率，该粒度可能是 4 毫秒（取决于 `CONFIG_HZ`）。这通常足够了。有一些选项可以通过使用更高分辨率的计数器来提高准确性，尽管性能成本很小（`VIRT_CPU_ACCOUNTING_NATIVE` 和 `VIRT_CPU_ACCOUNTING_GEN`），也可以选择更准确的 IRQ 时间（`IRQ_TIME_ACCOUNTING`）。获得准确的 CPU 利用率测量值的另一种方法是使用 MSR 或 PMC。

##### 文件内容

`/proc` 文件通常是文本格式的，允许它们轻松地从命令行读取并由 shell 脚本工具处理。例如：

`$ cat /proc/meminfo MemTotal:       15923672 kB MemFree:        10919912 kB MemAvailable:   15407564 kB Buffers:           94536 kB Cached:          2512040 kB SwapCached:            0 kB Active:          1671088 kB [...] $ grep Mem /proc/meminfo MemTotal:       15923672 kB MemFree:        10918292 kB MemAvailable:   15405968 kB`

虽然这很方便，但它确实为内核将统计信息编码为文本以及随后解析文本的任何用户态工具增加了少量开销。`netlink`（第 4.3.4 节 `netlink` 中介绍）是一个更高效的二进制接口。

`/proc` 的内容记录在 `proc(5)` 手册页和 Linux 内核文档中：`Documentation/filesystems/proc.txt` Bowden20Bowden 20Bowden20。有些部分有扩展文档，例如 `Documentation/iostats.txt` 中的 `diskstats` 和 `Documentation/scheduler/sched-stats.txt` 中的调度程序统计信息。除了文档之外，您还可以研究内核源代码以了解 `/proc` 中所有项目的确切来源。阅读使用它们的工具的源代码也很有帮助。

某些 `/proc` 条目取决于 `CONFIG` 选项：`schedstat` 使用 `CONFIG_SCHEDSTATS` 启用，`sched` 使用 `CONFIG_SCHED_DEBUG` 启用，`pressure` 使用 `CONFIG_PSI` 启用。

#### 4.3.3 延迟核算

启用了 `CONFIG_TASK_DELAY_ACCT` 选项的 Linux 系统可以跟踪每个任务在以下状态下的时间：

- **调度程序延迟**：等待 CPU 开启
- **块 I/O**：等待块 I/O 完成
- **交换**：等待分页（内存压力）
- **内存回收**：等待内存回收例程

从技术上讲，调度程序延迟的统计数据源自 `schedstats`（前面提到的 `/proc` 中的内容），但它与其他延迟状态一起公开。这些数据位于 `struct sched_info` 中，而不是 `struct task_delay_info` 中。

这些统计信息可以通过用户级工具 `taskstats` 读取，`taskstats` 是基于 netlink 的接口，用于获取每个任务和进程的统计数据。内核源码中包含了相关文档和示例：

- `Documentation/accounting/delay-accounting.txt`：文档
- `tools/accounting/getdelays.c`：消费者示例

以下是 `getdelays.c` 输出的示例：

#### 4.3.4 网络链接

`netlink` 是一种特殊的套接字地址族（`AF_NETLINK`），用于获取内核信息。使用 `netlink` 需要通过 `AF_NETLINK` 地址族打开网络套接字，然后使用一系列 `send(2)` 和 `receive(2)` 调用来传递请求并接收二进制结构中的信息。虽然这是一个比 `/proc` 更复杂的接口，但它更高效，并且还支持通知。`libnetlink` 库有助于简化使用。

与早期的工具一样，`strace(1)` 可用于显示内核信息的来源。检查套接字统计工具 `ss(8)`：

```bash
# strace ss
[...]
socket(AF_NETLINK, SOCK_RAW|SOCK_CLOEXEC, NETLINK_SOCK_DIAG) = 3
[...]
```

这将为 `NETLINK_SOCK_DIAG` 组打开一个 `AF_NETLINK` 套接字，该套接字返回有关套接字的信息。它记录在 `sock_diag(7)` 手册页中。`netlink` 组包括：

- **NETLINK_ROUTE**：路由信息（还有 `/proc/net/route`）
- **NETLINK_SOCK_DIAG**：套接字信息
- **NETLINK_SELINUX**：SELinux 事件通知
- **NETLINK_AUDIT**：审核（安全）
- **NETLINK_SCSITRANSPORT**：SCSI 传输
- **NETLINK_CRYPTO**：内核加密信息

使用 `netlink` 的命令包括 `ip(8)`、`ss(8)`、`routel(8)` 以及较旧的 `ifconfig(8)` 和 `netstat(8)`。
#### 4.3.5 跟踪点

跟踪点是基于静态检测的 Linux 内核事件源，该术语在第 1 章“简介”、第 1.7.3 节“跟踪”中介绍。跟踪点是放置在内核代码中逻辑位置的硬编码检测点。例如，系统调用、调度程序事件、文件系统操作和磁盘 I/O 的开始和结束处都有跟踪点[^4]。跟踪点基础设施由 Mathieu Desnoyers 开发，首次在 Linux 2.6 中提供。2009 年发布 32 版。跟踪点是一个稳定的 API[^5]，数量有限。

[^4]: 有些跟踪点是由 Kconfig 选项控制的，如果内核在没有它们的情况下编译，则可能不可用；例如，rcu 跟踪点和 `CONFIG_RCPU_TRACE`。
[^5]: 我称之为“尽力稳定”。这种情况很少见，但我见过跟踪点发生变化。

跟踪点是性能分析的重要资源，因为它们为高级跟踪工具提供了超越汇总统计的功能，可以更深入地了解内核行为。虽然基于函数的跟踪可以提供类似的功能（例如，第 4.3.6 节，kprobes），但只有跟踪点提供稳定的接口，从而允许开发强大的工具。

本节介绍跟踪点。这些可以被第 4.5 节“跟踪工具”中介绍的跟踪器使用，并在第 13 章到第 15 章中进行了深入介绍。

```
前三个字段是时间戳（秒）、进程详细信息（名称/线程 ID）和事件描述（后跟冒号分隔符而不是空格）。其余字段是跟踪点的参数，由接下来解释的格式字符串生成；对于特定的 `block:block_rq_issue` 格式字符串，请参见第 9 章，磁盘，第 9.6.5 节，perf。

关于术语的说明：从技术上讲，跟踪点（或跟踪点）是放置在内核源代码中的跟踪函数（也称为跟踪挂钩）。例如，`trace_sched_wakeup()` 是一个跟踪点，您会发现它是从 `kernel/sched/core.c` 调用的。该跟踪点可以通过使用名称 `sched:sched_wakeup` 的跟踪器进行检测；然而，从技术上讲，这是一个跟踪事件，由 `TRACE_EVENT` 宏定义。 `TRACE_EVENT` 还定义并格式化其参数，自动生成 `trace_sched_wakeup()` 代码，并将跟踪事件放置在 `tracefs` 和 `perf_event_open(2)` 接口中[^Ts'o 20]。跟踪工具主要检测跟踪事件，尽管它们可能将其称为“跟踪点”。 `perf(1)` 将跟踪事件称为“Tracepoint 事件”，这很令人困惑，因为基于 `kprobe` 和 `uprobe` 的跟踪事件也被标记为“Tracepoint 事件”。

##### 跟踪点接口

跟踪工具可以通过 Tracefs 中的跟踪事件文件（通常安装在 `/sys/kernel/debug/tracing`）或 `perf_event_open(2)` 系统调用来使用跟踪点。例如，我的基于 Ftrace 的 `iosnoop(8)` 工具使用 Tracefs 文件：
```shell
# strace -e openat ~/Git/perf-tools/bin/iosnoop

```
`perf_event_open(2)` 是内核 `perf_events` 子系统的接口，它提供各种分析和跟踪功能。有关更多详细信息，请参阅其手册页，以及第 13 章中的 `perf(1)` 前端。

##### 跟踪点开销

当跟踪点被激活时，它们会给每个事件增加少量的 CPU 开销。跟踪工具还可能会增加后处理事件的 CPU 开销，以及记录事件的文件系统开销。开销是否高到足以干扰生产应用程序取决于事件发生率和 CPU 数量，这是使用跟踪点时需要考虑的因素。

在当今的典型系统（4 到 128 个 CPU）上，我发现每秒低于 10,000 个的事件速率所花费的开销可以忽略不计，只有超过 100,000 个事件速率才开始变得可测量。作为事件示例，您可能会发现磁盘事件通常少于每秒 10,000 个，但调度程序事件每秒可能超过 100,000 个，因此跟踪成本可能很高。

我之前分析过特定系统的开销，发现最小跟踪点开销为 96 纳秒的 CPU 时间Gregg19Gregg 19Gregg19。2018 年 Linux 4.7 中添加了一种称为原始跟踪点的新型跟踪点，它避免了创建稳定跟踪点参数的成本，从而减少了这种开销。

除了使用跟踪点时启用的开销之外，还存在使它们可用的禁用开销。禁用的跟踪点变成少量指令：对于 x86_64，它是 5 字节无操作 (`nop`) 指令。函数末尾还添加了一个跟踪点处理程序，这会稍微增加其文本大小。虽然这些开销非常小，但在向内核添加跟踪点时您应该分析和理解它们。

##### 跟踪点文档

跟踪点技术记录在内核源代码中的 `Documentation/trace/tracepoints.rst` 下。跟踪点本身（有时）记录在定义它们的头文件中，可以在 Linux 源代码的 `include/trace/events` 下找到。我在 BPF 性能工具第 2 章Gregg19Gregg 19Gregg19 中总结了高级跟踪点主题：如何将它们添加到内核代码中，以及它们如何在指令级别工作。

#### 4.3.6 k探针

kprobes（内核探针的缩写）是基于动态检测的跟踪器的 Linux 内核事件源，该术语在第 1 章“简介”第 1.7.3 节“跟踪”中介绍。 kprobes 可以跟踪任何内核函数或指令，并在 2004 年发布的 Linux 2.6.9 中提供。它们被认为是不稳定的 API，因为它们公开了可能在内核版本之间发生变化的原始内核函数和参数。

kprobes 可以在内部以不同的方式工作。标准方法是修改运行内核代码的指令文本以在需要的地方插入检测。在检测函数条目时，可以在 kprobes 利用现有 Ftrace 函数跟踪的情况下进行优化，因为它的开销较低。[^7]

[^7]: 它也可以通过 debug.kprobes-optimization sysctl(8) 启用/禁用。

kprobes 很重要，因为它们是有关生产中内核行为的几乎无限信息的最后手段[^8]，这对于观察其他工具看不见的性能问题至关重要。它们可以被第 4.5 节“跟踪工具”中介绍的跟踪器使用，并在第 13 章到第 15 章中进行了深入介绍。

[^8]: 如果没有 kprobes，最后的选择是修改内核代码以在需要时添加检测、重新编译和重新部署。

kprobes 和 tracepoint 的比较见表 4.3。

表 4.3 kprobes 与跟踪点的比较

| 细节                | k探针           | 跟踪点       |
| ------------------- | --------------- | ------------ |
| 类型                | 动态的          | 静止的       |
| 事件的大致数量      | 50,000+         | 1,000+       |
| 内核维护            | 没有任何        | 必需的       |
| 禁用开销            | 没有任何        | 微小（NOP + 元数据） |
| 稳定的API           | 不              | 是的         |

kprobes 可以跟踪函数的入口以及函数内的指令偏移量。使用 kprobes 创建 kprobe 事件（基于 kprobe 的跟踪事件）。这些 kprobe 事件仅在跟踪器创建它们时才存在：默认情况下，内核代码运行时未修改。

##### kprobes 示例

作为使用 kprobes 的示例，以下 bpftrace 命令检测 `do_nanosleep()` 内核函数并打印 CPU 上的进程：

```bash
# bpftrace -e 'kprobe:do_nanosleep { printf("sleep by: %s\\n", comm); }'
Attaching 1 probe...
sleep by: mysqld
sleep by: mysqld
sleep by: sleep
^C
#

输出显示名为“mysqld”的进程进行了几次睡眠，以及“sleep”（可能是 `/bin/sleep`）进行了一次睡眠。 `do_nanosleep()` 的 kprobe 事件在 bpftrace 程序开始运行时创建，并在 bpftrace 终止 (Ctrl-C) 时删除。

##### kprobes 参数

由于 kprobes 可以跟踪内核函数调用，因此通常需要检查函数的参数以获取更多上下文。每个跟踪工具都以自己的方式公开它们，并将在后面的部分中介绍。例如，使用 bpftrace 打印 `do_nanosleep()` 的第二个参数，即 `hrtimer_mode`：

```bash
# bpftrace -e 'kprobe:do_nanosleep { printf("mode: %d\\n", arg1); }'
Attaching 1 probe...
mode: 1
mode: 1
mode: 1
[...]
```

使用 `arg0..argN` 内置变量可以在 bpftrace 中使用函数参数。

##### 电路探针

可以使用 kretprobes（内核返回探针的缩写）来跟踪内核函数的返回及其返回值，它与 kprobes 类似。 kretprobes 是使用 kprobe 作为函数入口来实现的，它插入一个 Trampoline 函数来检测返回。

当与 kprobes 和记录时间戳的跟踪器配合使用时，可以测量内核函数的持续时间。例如，使用 bpftrace 测量 `do_nanosleep()` 的持续时间：

```bash
# bpftrace -e 'kprobe:do_nanosleep { @ts[tid] = nsecs; }
    kretprobe:do_nanosleep /@ts[tid]/ { 
        @sleep_ms = hist((nsecs - @ts[tid]) / 1000000); delete(@ts[tid]); }
    END { clear(@ts); }'
Attaching 3 probes...
^C
```

输出显示 `do_nanosleep()` 通常是一个快速函数，在零毫秒内返回（向下舍入）1,280 次。有两次出现时间达到 512 到 1024 毫秒范围。

`bpftrace` 语法在第 15 章 BPF 中进行了解释，其中包括一个用于计时 `vfs_read()` 的类似示例。

##### kprobes 接口和开销

kprobes 接口类似于 tracepoints。有一种方法可以通过 `/sys` 文件、`perf_event_open(2)` 系统调用（这是首选）以及 `register_kprobe()` 内核 API 来检测它们。当跟踪函数的条目（Ftrace 方法，如果可用）时，开销与跟踪点的开销类似，而当跟踪函数偏移量（断点方法）或使用 kretprobe（蹦床方法）时，开销会更高。对于特定系统，我测量的最小 kprobe CPU 成本为 76 纳秒，最小 kretprobe CPU 成本为 212 纳秒 [Gregg 19]。

##### kprobes 文档

kprobes 记录在 Linux 源文件的 `Documentation/kprobes.txt` 下。他们所检测的内核函数通常不会在内核源代码之外记录（因为大多数不是 API，所以不需要）。我在 BPF 性能工具第 2 章 [Gregg 19] 中总结了 kprobe 的高级主题：它们如何在指令级别工作。

#### 4.3.7 上升脉冲

uprobes（用户空间探针）与 kprobes 类似，但用于用户空间。它们可以动态检测应用程序和库中的功能，并提供不稳定的 API，用于深入了解其他工具范围之外的软件内部结构。 uprobes 在 2012 年发布的 Linux 3.5 中可用。

uprobes 可由第 4.5 节“跟踪工具”中介绍的跟踪器使用，并在第 13 章至第 15 章中进行了深入介绍。

##### 示例

作为 uprobes 的示例，以下 bpftrace 命令列出了 bash(1) shell 中可能的 uprobe 函数入口位置：

```
# bpftrace -l 'uprobe:/bin/bash:*'
uprobe:/bin/bash:rl_old_menu_complete
uprobe:/bin/bash:maybe_make_export_env
uprobe:/bin/bash:initialize_shell_builtins
uprobe:/bin/bash:extglob_pattern_p
uprobe:/bin/bash:dispose_cond_node
uprobe:/bin/bash:decode_prompt_string
[..]
```

完整输出显示 1,507 种可能的 uprobe。 uprobes 仪器代码并在需要时创建 uprobe 事件（基于 uprobe 的跟踪事件）：默认情况下，用户空间代码运行时不进行修改。这类似于使用调试器向函数添加断点：在添加断点之前，函数未经修改地运行。

##### uprobes 参数

用户函数的参数由 uprobes 提供。作为示例，以下使用 bpftrace 来检测 `decode_prompt_string()` bash 函数并将第一个参数打印为字符串：

```
# bpftrace -e 'uprobe:/bin/bash:decode_prompt_string { printf("%s\\n", str(arg0)); }'
Attaching 1 probe...
\[\\e\[31;1m\\\]\\u@\\h:\\w>\\[\\e\[0m\\\]
\[\\e\[31;1m\\\]\\u@\\h:\\w>\\[\\e\[0m\\\]
^C
```

输出显示该系统上的 bash(1) 提示字符串。 `decode_prompt_string()` 的 uprobe 在 bpftrace 程序开始运行时创建，并在 bpftrace 终止 (Ctrl-C) 时删除。

##### 输尿管探针

用户函数的返回及其返回值可以使用 uretprobes（用户级返回探针的缩写）来跟踪，这与 uprobes 类似。当与 uprobes 和记录时间戳的跟踪器配合使用时，可以测量用户级函数的持续时间。请注意，输尿管探针的开销可能会严重影响快速函数的此类测量。

##### uprobes 接口和开销

uprobes 接口与 kprobes 类似。有一种方法可以通过 `/sys` 文件以及（最好）通过 `perf_event_open(2)` 系统调用来检测它们。

uprobes 目前通过捕获内核来工作。这比 kprobes 或跟踪点花费更高的 CPU 开销。对于我测量的特定系统，最小 uprobe 成本为 1,287 纳秒，最小 uretprobe 成本为 1,931 纳秒 [Gregg 19]。 uretprobe 的开销较高，因为它是 uprobe 加上 Trampoline 功能。

##### uprobe 文档

uprobes 记录在 Linux 源文件的 `Documentation/trace/uprobetracer.rst` 下。我在 BPF 性能工具第 2 章 [Gregg 19] 中总结了高级 uprobe 主题：它们如何在指令级别工作。他们所检测的用户功能通常不会记录在应用程序源之外（因为大多数不太可能是 API，所以也不需要）。对于记录的用户空间跟踪，请使用 USDT。

#### 4.3.8 泰达币

用户级静态定义跟踪 (USDT) 是跟踪点的用户空间版本。 USDT 之于 uprobes 就像跟踪点之于 kprobes 一样。一些应用程序和库已将 USDT 探针添加到其代码中，为跟踪应用程序级事件提供稳定（且有记录）的 API。例如，Java JDK、PostgreSQL 数据库、libc 中都有 USDT 探针。下面列出了使用 bpftrace 的 OpenJDK USDT 探针：

```
# bpftrace -lv 'usdt:/usr/lib/jvm/openjdk/libjvm.so:*'
usdt:/usr/lib/jvm/openjdk/libjvm.so:hotspot:class__loaded
usdt:/usr/lib/jvm/openjdk/libjvm.so:hotspot:class__unloaded
usdt:/usr/lib/jvm/openjdk/libjvm.so:hotspot:method__compile__begin
usdt:/usr/lib/jvm/openjdk/libjvm.so:hotspot:method__compile__end
usdt:/usr/lib/jvm/openjdk/libjvm.so:hotspot:gc__begin
usdt:/usr/lib/jvm/openjdk/libjvm.so:hotspot:gc__end
[..]
```

这里列出了用于 Java 类加载和卸载、方法编译和垃圾收集的 USDT 探针。还有更多被截断：完整列表显示了该 JDK 版本的 524 个 USDT 探针。

许多应用程序已经具有可以启用和配置的自定义事件日志，并且对于性能分析非常有用。 USDT 探针的不同之处在于，它们可以从各种跟踪器中使用，这些跟踪器可以将应用程序上下文与内核事件（例如磁盘和网络 I/O）结合起来。应用程序级记录器可能会告诉您，由于文件系统 I/O，数据库查询速度很慢，但跟踪器可以揭示更多信息：例如，由于文件系统中的锁争用，而不是磁盘 I/O，查询速度很慢。哦，正如你可能已经假设的那样。

某些应用程序包含 USDT 探针，但目前在应用程序的打包版本中尚未启用（OpenJDK 就是这种情况）。使用它们需要使用适当的配置选项从源代码重建应用程序。该选项可能在 DTrace 跟踪器之后被称为 --enable-dtrace-probes，它推动了 USDT 在应用程序中的采用。

USDT 探针必须编译成它们检测的可执行文件。对于 Java 等 JIT 编译语言来说这是不可能的，因为 Java 通常是动态编译的。对此的解决方案是动态 USDT，它将探针预编译为共享库，并提供从 JIT 编译语言调用它们的接口。适用于 Java、Node.js 和其他语言的动态 USDT 库。解释型语言也有类似的问题，需要动态 USDT。

USDT 探针是在 Linux 中使用 uprobes 实现的：有关 uprobes 及其开销的描述，请参阅上一节。除了启用的开销之外，USDT 探针还会在代码中放置 nop 指令，跟踪点也是如此。

USDT 探针可由第 4.5 节“跟踪工具”中介绍的跟踪器使用，第 13 章至第 15 章将深入介绍（尽管将 USDT 与 Ftrace 一起使用需要一些额外的工作）。

##### USDT文档

如果应用程序提供 USDT 探针，则应将其记录在应用程序的文档中。我在 BPF 性能工具第 2 章 [Gregg 19] 中总结了高级 USDT 主题：如何将 USDT 探针添加到应用程序代码中、它们内部如何工作以及动态 USDT。

#### 4.3.9 硬件计数器 (PMC)

处理器和其他设备通常支持用于观察活动的硬件计数器。主要来源是处理器，通常称为性能监控计数器 (PMC)。它们还有其他名称：CPU 性能计数器 (CPC)、性能仪表计数器 (PIC) 和性能监视单元事件（PMU 事件）。这些都指的是同一件事：处理器上的可编程硬件寄存器，可在 CPU 周期级别提供低级性能信息。

PMC 是绩效分析的重要资源。只有通过PMC才能测量CPU指令的效率、CPU缓存的命中率、内存和设备总线的利用率、互连利用率、停顿周期等等。使用这些来分析性能可以导致各种性能优化。

##### PMC 示例

虽然 PMC 有很多，但英特尔选择了 7 个作为“架构集”，它们提供了一些核心功能的高级概述 [Intel 16]。可以使用 cpuid 指令检查这些架构集 PMC 的存在。表 4.4 显示了该集合，它作为有用 PMC 的示例集。

**表 4.4 Intel 架构 PMC**

|活动名称|口罩|事件选择|事件掩码助记符示例|
|---|---|---|---|
|不间断的核心周期|00H|3通道|CPU_CLK_UNHALTED.THREAD_P|
|指令已停用|00H|碳氢化合物|INST_RETIRED.ANY_P|
|不间断的参考周期|01H|3通道|CPU_CLK_THREAD_UNHALTED.REF_XCLK|
|有限责任公司参考资料|4FH|2EH|LONGEST_LAT_CACHE.REFERENCE|
|有限责任公司失误|41小时|2EH|LONGEST_LAT_CACHE.MISS|

|分行错过退休|00H|C5H|BR_MISP_RETIRED.ALL_BRANCHES|

作为 PMC 的示例，如果您运行 `perf stat` 命令而不指定事件（无 `-e`），则它默认检测架构 PMC。例如，以下命令在 gzip(1) 命令上运行 `perf stat`：

```
# perf stat gzip words

        156.927428      task-clock (msec)         #    0.987 CPUs utilized
                 1      context-switches          #    0.006 K/sec
                 0      cpu-migrations            #    0.000 K/sec
               131      page-faults               #    0.835 K/sec
       209,911,358      cycles                    #    1.338 GHz
       288,321,441      instructions              #    1.37  insn per cycle
        66,240,624      branches                  #  422.110 M/sec
         1,382,627      branch-misses             #    2.09% of all branches

       0.159065542 seconds time elapsed
```

原始计数是第一列；哈希之后是一些统计信息，包括一个重要的性能指标：每个周期的指令数 (`insn per cycle`)。这显示了 CPU 执行指令的效率——越高越好。该指标在第 6 章“CPU”、第 6.3.7 节“IPC、CPI”中进行了解释。

##### PMC接口

在 Linux 上，PMC 通过 `perf_event_open(2)` 系统调用进行访问，并由包括 perf(1) 在内的工具使用。

虽然有数百个可用的 PMC，但 CPU 中只有固定数量的寄存器可用于同时测量它们，可能只有 6 个。您需要选择要在这六个寄存器上测量哪些 PMC，或者循环不同的 PMC 集作为对它们进行采样的方式（Linux perf(1) 自动支持此操作）。其他软件计数器不受这些限制。

PMC 可以用于不同的模式： 计数，它们以几乎为零的开销对事件进行计数；溢出采样，其中每可配置数量的事件中就有一个产生中断，以便可以捕获状态。计数可以用来量化问题；溢出采样可用于显示负责的代码路径。

`perf(1)` 可以使用 `stat` 子命令进行计数，并使用 `record` 子命令进行采样；参见第 13 章，性能。

##### PMC 挑战

使用 PMC 时的两个常见挑战是溢出采样的准确性及其在云环境中的可用性。

由于中断延迟（通常称为“滑动”）或乱序指令执行，溢出采样可能无法记录触发事件的正确指令指针。对于 CPU 周期分析，这种打滑可能不是问题，并且一些分析器故意引入抖动以避免锁步采样（或使用偏移采样率，例如 99 赫兹）。但为了测量其他事件，例如 LLC 未命

中，采样的指令指针需要准确。

解决方案是处理器对所谓的精确事件的支持。在英特尔上，精确事件使用一种称为精确基于事件采样 (PEBS) 的技术，[1](https://chatgpt.com/#user-content-fn-9) 该技术使用硬件缓冲区来记录 PMC 事件发生时更准确（“精确”）的指令指针。在 AMD 上，精确事件使用基于指令的采样 (IBS) [Drongowski 07]。 Linux `perf(1)` 命令支持精确事件（请参阅第 13 章，perf，第 13.9.2 节，CPU 分析）。

另一个挑战是云计算，因为许多云环境禁用其来宾的 PMC 访问。从技术上讲，启用它是可行的：例如，Xen 虚拟机管理程序具有 `vpmu` 命令行选项，该选项允许将不同的 PMC 集暴露给 guest[2](https://chatgpt.com/#user-content-fn-10) [Xenbits 20]。 Amazon 为其 Nitro 虚拟机管理程序来宾启用了许多 PMC。[3](https://chatgpt.com/#user-content-fn-11)此外，一些云提供商还提供“裸机实例”，其中来宾具有完全的处理器访问权限，因此也具有完全的 PMC 访问权限。

##### PMC 文档

PMC 是特定于处理器的，并记录在相应的处理器软件开发人员手册中。处理器制造商的示例：

- 英特尔：英特尔® 64 和 IA-32 架构软件开发人员手册第 3 卷 [英特尔 16] 第 19 章“性能监控事件”。
- AMD：AMD 系列 17h 处理器型号 00h-2Fh 的开源寄存器参考第 2.1.1 节“性能监视器计数器”[AMD 18]
- ARM：Arm® 架构参考手册 Armv8 的 D7.10 节“PMU 事件和事件编号”，适用于 Armv8-A 架构配置文件 [ARM 19]

人们一直在努力开发一种可在所有处理器上支持的 PMC 标准命名方案，称为性能应用程序编程接口 (PAPI) [UTK 20]。操作系统对 PAPI 的支持参差不齐：它需要频繁更新才能将 PAPI 名称映射到供应商 PMC 代码。

第 6 章“CPU”、第 6.4.1 节“硬件”、“硬件计数器 (PMC)”小节更详细地描述了它们的实现，并提供了其他 PMC 示例。

#### 4.3.10 其他可观测性来源

其他可观测性来源包括：

- MSR：PMC 使用模型特定寄存器 (MSR) 来实现。还有其他 MSR 用于显示系统的配置和运行状况，包括 CPU 时钟频率、使用情况、温度和功耗。可用的 MSR 取决于处理器类型（特定于型号）、BIOS 版本和设置以及虚拟机管理程序设置。一种用途是基于周期的 CPU 利用率的精确测量。
    
- ptrace(2)：此系统调用控制进程跟踪，gdb(1) 使用它进行进程调试，strace(1) 使用它来跟踪系统调用。它是基于断点的，可以使目标速度减慢一百倍以上。 Linux 还具有跟踪点（在第 4.3.5 节“跟踪点”中介绍），以实现更高效的系统调用跟踪。
    
- 函数分析：分析函数调用（mcount() 或 `__fentry__()`）被添加到 x86 上所有非内联内核函数的开头，以实现高效的 Ftrace 函数跟踪。在需要之前它们会被转换为 nop 指令。请参阅第 14 章，Ftrace。
    
- 网络嗅探 (libpcap)：这些接口提供了一种从网络设备捕获数据包的方法，以便详细调查数据包和协议性能。在 Linux 上，嗅探是通过 libpcap 库和 `/proc/net/dev` 提供的，并由 tcpdump(8) 工具使用。捕获和检查所有数据包会产生 CPU 和存储开销。有关网络嗅探的更多信息，请参阅第 10 章。
    
- netfilter conntrack：Linux netfilter 技术允许对事件执行自定义处理程序，不仅用于防火墙，还用于连接跟踪 (conntrack)。这允许创建网络流日志 [Ayuso 12]。
    
- 流程记账：这可以追溯到大型机，以及根据流程的执行和运行时间向部门和用户的计算机使用情况计费的需要。它以某种形式存在于 Linux 和其他系统中，有时有助于进程级别的性能分析。例如，Linux atop(1) 工具使用进程记帐来捕获并显示来自短期进程的信息，否则在拍摄 `/proc` 快照时会丢失这些信息 [Atoptool 20]。
    
- 软件事件：这些事件与硬件事件相关，但通过软件进行检测。页面错误就是一个例子。软件事件通过 `perf_event_open(2)` 接口提供，并由 `perf(1)` 和 bpftrace 使用。它们如图 4.5 所示。
    
- 系统调用：某些系统或库调用可能可用于提供某些性能指标。其中包括 `getrusage(2)`，这是一个系统调用，用于让进程获取自己的资源使用统计信息，包括用户和系统时间、故障、消息和上下文切换。
    

如果您对其中每个接口的工作原理感兴趣，您会发现通常可以找到文档，供在这些接口上构建工具的开发人员使用。

##### 还有更多

根据您的内核版本和启用的选项，甚至可能有更多的可观察性来源可用。本书后面的章节中会提到一些内容。对于 Linux，这些包括 I/O 统计、blktrace、timer_stats、lockstat 和 debugfs。

找到此类来源的一种方法是阅读您有兴趣观察的内核代码，并查看其中放置了哪些统计信息或跟踪点。

在某些情况下，可能没有您所追求的内核统计信息。除了动态检测（Linux kprobes 和 uprobes）之外，您可能会发现 gdb(1) 和 lldb(1) 等调试器可以获取内核和应用程序变量以阐明调查。

##### Solaris Kstat

作为提供系统统计信息的不同方式的示例，基于 Solaris 的系统使用内核统计 (Kstat) 框架，该框架提供一致的内核统计分层结构，每个结构都使用以下四元组命名：

```
_module:instance:name:statistic_
```

这些都是

- `module`：这通常是指创建统计信息的内核模块，例如 SCSI 磁盘驱动程序的 `sd` 或 ZFS 文件系统的 `zfs`。
- `实例`：某些模块作为多个实例存在，例如每个 SCSI 磁盘都有一个 `sd` 模块。实例是一个枚举。
- `name`：这是统计组的名称。
- `统计`：这是单独的统计名称。

Kstats 使用二进制内核接口进行访问，并且存在各种库。

作为 Kstat 示例，下面使用 `kstat(1M)` 并指定完整的四元组读取“nproc”统计信息：

```
$ kstat -p unix:0:system_misc:nproc
unix:0:system_misc:nproc        94
```

该统计信息显示当前正在运行的进程数。

相比之下，Linux 上的 `/proc/stat` 风格的源格式不一致，通常需要文本解析来处理，从而消耗一些 CPU 周期。
### 4.4 SAR（系统活动报告）

`**sar(1)**` 是一个强大的系统监控工具，通常用于收集和报告 Linux 系统的性能数据，尤其是 CPU、内存、磁盘 I/O 和网络活动。它是 `sysstat` 包的一部分，并为系统管理员提供了定期和实时性能数据。尽管近年来 BPF（伯克利包过滤器）追踪工具（如 `bpftrace`）得到广泛关注，但 `sar(1)` 依然是一个不可忽视的实用工具，能够帮助解决许多性能问题，且具有易于理解的列标题、丰富的文档支持。

#### 4.4.1 **sar(1) 的覆盖范围**

`sar(1)` 提供了广泛的系统和设备统计数据，包括 CPU 使用情况、磁盘 I/O、网络流量、内存使用等，甚至包括硬件温度和电源管理等方面。图 4.6 说明了 `sar(1)` 通过不同命令行选项提供的观测内容。

例如，使用 `-m`（电源管理）选项，`sar` 可报告诸如电压（`IN`）、温度（`TEMP`）和 USB 电源等电源相关的统计信息。

#### 4.4.2 **sar(1) 监测**

要检查系统是否启用了 `sar(1)` 数据收集，您可以直接运行 `sar` 命令。如果数据收集尚未启用，您会看到类似以下的错误信息：

```
$ sar
Cannot open /var/log/sysstat/sa19: No such file or directory
Please check if data collecting is enabled
```

启用数据收集的步骤可能因不同的 Linux 发行版而异。在 Ubuntu 系统上，可以通过修改 `/etc/default/sysstat` 文件，将 `ENABLED` 设置为 `true` 来启用数据收集：

```bash
# vi /etc/default/sysstat
ENABLED="true"
```

然后，重新启动 `sysstat` 服务：

```bash
$ sudo service sysstat restart
```

您还可以编辑 `sysstat` 的 crontab 文件，以调整数据收集的频率。例如，默认设置为每 10 分钟收集一次数据，您可以将其更改为每 5 分钟收集一次：

```bash
# vi /etc/cron.d/sysstat
*/5 * * * * root command -v debian-sa1 > /dev/null && debian-sa1 1 1 -S ALL
```

#####  **报告**

`SAR` 可以通过不同的选项来生成各种性能报告。例如，您可以使用以下命令来报告 CPU 使用情况（`-u`）、TCP 连接统计（`-n TCP`）以及 TCP 错误（`-n ETCP`）：

```bash
$ sar -u -n TCP,ETCP

10:45:01 AM     all      6.87      0.00      2.84      0.18      0.00     90.12
10:50:01 AM     all      6.87      0.00      2.49      0.06      0.00     90.58
...
10:40:01 AM  active/s passive/s    iseg/s    oseg/s
10:45:01 AM      0.16      0.00     10.98      9.27
10:50:01 AM      0.20      0.00     10.40      8.93

在上述示例中，`sar` 会提供系统的各类 CPU 使用统计和 TCP 状态统计数据。第一行输出是系统的摘要，显示操作系统内核版本、主机名、日期、CPU 数量等信息。

要查看所有统计信息，可以使用 `sar -A` 命令，它会输出系统上所有可用的性能数据。

#### 4.4.4 **sar(1) 文档**

`sar(1)` 的手册页提供了对所有统计项的详细描述。例如，以下是 TCP 统计的部分描述：

- **active/s**：每秒直接从 CLOSED 状态转到 SYN-SENT 状态的 TCP 连接数（SNMP 名称：`tcpActiveOpens`）。
- **iseg/s**：每秒收到的 TCP 数据段总数，包括错误的段（SNMP 名称：`tcpInSegs`）。

可以通过 `man sar` 查看完整的手册页，了解每个统计项的详细信息。
### 4.5 追踪工具

Linux 提供了多个强大的工具来进行高级性能分析，这些工具利用了前面描述的事件接口（如 tracepoints、kprobes、uprobes 和 USDT）来实现精确的跟踪。主要的追踪工具有：

1. **perf(1)**：官方的 Linux 性能分析工具。它非常适合进行 CPU 分析（如堆栈跟踪采样）和性能监控计数器（PMC）分析，并且可以检测到系统中的其他重要事件，通常将结果记录到输出文件中以便后处理。它的灵活性和强大功能使其成为最常用的性能分析工具之一。
    
2. **Ftrace**：Linux 内核的官方跟踪工具，包含多个实用程序，适用于内核代码路径分析。Ftrace 适合资源受限的系统，因为它可以在没有复杂依赖关系的情况下运行，且能够高效地获取内核级的行为信息。
    
3. **BPF（BCC、bpftrace）**：扩展 BPF 功能，允许开发者进行强大的系统性能监控和分析。BCC（BPF Compiler Collection）提供了一系列高级工具，而 bpftrace 则是一个为短小的自定义脚本而设计的工具。BPF 使得开发者能够以非常低的开销获取详细的系统行为信息。
    
4. **SystemTap**：一种高级的脚本语言和跟踪工具，拥有丰富的 Tapsets（用于跟踪各种目标的库）。它的灵活性和强大功能使其适合于高度定制化的跟踪任务。SystemTap 也正在开发 BPF 后端，这使其与 BPF 工具的结合更加紧密（参见 `stapbpf(8)` 手册页）。
    
5. **LTTng**：一个专门优化黑盒记录的跟踪工具，旨在以最佳的方式记录各种事件并为后期分析提供数据。它非常适合高性能、高吞吐量的记录需求。
    

在接下来的章节中（第 13 章到第 15 章），将详细介绍 **perf**、**Ftrace** 和 **BPF** 的使用方法和最佳实践。第 13 章专注于 perf 工具，第 14 章介绍 Ftrace，第 15 章讲解 BPF。这些工具在实际应用中的使用方式将通过各种示例展示，帮助您理解如何在生产环境中进行性能分析。

在我个人的工作中，使用 **perf** 进行 CPU 分析，使用 **Ftrace** 挖掘内核代码，使用 **BCC** 和 **bpftrace** 进行其他领域的性能分析，包括内存、文件系统、磁盘、网络和应用程序的跟踪。

### 4.6 观察可观察性

可观察性工具和构建它们的统计数据都是在软件中实现的，所有软件都可能存在错误。描述软件的文档也是如此。以健康的怀疑态度看待任何对你来说新的统计数据，质疑它们的真正含义以及它们是否真的正确。

指标可能会遇到以下任何问题：

*   工具和测量有时是错误的。
*   手册页并不总是正确的。
*   可用的指标可能不完整。
*   可用的指标可能设计不当且令人困惑。
*   指标收集器（例如解析工具输出）可能存在错误。[^13]
    [^13]在这种情况下，工具和测量结果是正确的，但自动收集器引入了错误。在 Surge 2013 上，我就一个令人震惊的案例 \[Gregg 13c\] 进行了一次闪电演讲：一家基准测试公司报告了我所支持的产品的不良指标，我深入研究了这一点。事实证明，他们用于自动化基准测试的 shell 脚本有两个错误。首先，在处理 fio(1) 的输出时，它将获取诸如“100KB/s”之类的结果，并使用正则表达式来删除非数字字符，包括“KB/s”以将其转换为“100”。由于 fio(1) 使用不同的单位（字节、千字节、兆字节）报告结果，因此引入了大量 (1024x) 错误。其次，他们还省略了小数位，因此“1.6”的结果变成了“16”。
*   度量处理（算法/电子表格）也会引入错误。

当多个可观测性工具的覆盖范围重叠时，您可以使用它们相互交叉检查。理想情况下，他们将采购不同的检测框架来检查框架中的错误。动态检测对于此目的特别有用，因为可以创建自定义工具来双重检查指标。

另一种验证技术是应用已知的工作负载，然后检查可观察性工具是否与您期望的结果一致。这可能涉及使用微基准测试工具来报告自己的统计数据以进行比较。

有时，错误的不是工具或统计数据，而是描述它的文档，包括手册页。软件可能已经发展但文档没有更新。

实际上，您可能没有时间仔细检查您使用的每个绩效衡量标准，只有在遇到异常结果或对公司至关重要的结果时才会这样做。即使您没有仔细检查，意识到您没有仔细检查并且您认为工具是正确的也是很有价值的。

指标也可能不完整。当面对大量的工具和指标时，人们可能很容易认为它们提供了完整且有效的覆盖范围。通常情况并非如此：程序员可能会添加度量来调试自己的代码，然后将其内置到可观察性工具中，而无需对实际客户需求进行太多研究。有些程序员可能根本没有向新子系统添加任何内容。

缺乏指标可能比存在不良指标更难识别。第 2 章“方法论”可以通过研究性能分析所需回答的问题来帮助您找到这些缺失的指标。
