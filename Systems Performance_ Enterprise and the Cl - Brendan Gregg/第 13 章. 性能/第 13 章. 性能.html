第 13 章. 性能

第13章

性能

perf(1) 是官方的 Linux 分析器，位于 tools/perf 下的 Linux 内核源代码中。1 它是一个多功能工具，具有分析、跟踪和脚本功能，并且是内核 perf_events 可观察性子系统。 perf_events 也称为 Linux 性能计数器 (PCL) 或 Linux 性能事件 (LPE)。 perf_events 和 perf(1) 前端最初具有性能监控计数器 (PMC) 功能，但后来也发展到支持基于事件的跟踪源：tracepoints、kprobes、uprobes 和 USDT。

1perf(1) 的不同寻常之处在于它是 Linux 内核源代码树中的一个大型、复杂的用户级程序。维护者 Arnaldo Carvalho de Melo 向我描述了这种情况，称其为“实验”。虽然这对 perf(1) 和 Linux 是有利的，因为它们是同步开发的，但有些人对它的包含感到不安，并且它可能仍然是 Linux 源代码中包含的唯一复杂的用户软件。

对于那些希望更详细地学习一个或多个系统跟踪器的人来说，本章以及第 14 章“Ftrace”和第 15 章“BPF”是可选读物。

与其他跟踪器相比，perf(1) 特别适合 CPU 分析：分析（采样）CPU 堆栈跟踪、跟踪 CPU 调度程序行为以及检查 PMC 以了解微架构级 CPU 性能（包括周期行为）。其跟踪功能还使其能够分析其他目标，包括磁盘 I/O 和软件功能。

perf(1) 可用于回答以下问题：

  • 哪些代码路径正在消耗 CPU 资源？

  • CPU 是否因内存加载/存储而停滞？

  • 线程离开 CPU 的原因是什么？

  • 磁盘 I/O 的模式是什么？

13.2 单行线

以下行话通过示例展示了各种 perf(1) 功能。这些来自我在线发布的一个更大的列表 [Gregg 20h]，它已被证明是解释 perf(1) 功能的有效方法。这些的语法将在后面的章节和 perf(1) 的手册页中介绍。

请注意，许多单行代码使用 -a 来指定所有 CPU，但这成为 Linux 4.11 中的默认设置，并且可以在该内核和更高版本中省略。

上市活动

列出所有当前已知的事件：

perf list

列出计划跟踪点：

perf list 'sched:*'

列出名称中包含字符串“block”的事件：

perf list block

列出当前可用的动态探针：

perf probe -l

计数事件

显示指定命令的 PMC 统计信息：

perf stat command

显示指定 PID 的 PMC 统计信息，直到 Ctrl-C：

perf stat -p PID

显示整个系统的 PMC 统计信息，持续 5 秒：

perf stat -a sleep 5

显示命令的 CPU 最后一级缓存 (LLC) 统计信息：

点击这里查看代码图片

perf stat -e LLC-loads,LLC-load-misses,LLC-stores,LLC-prefetches command

使用原始 PMC 规范（Intel）计算未停止的核心周期：

perf stat -e r003c -a sleep 5

使用详细的 PMC 原始规范（Intel）计算前端停顿：

点击这里查看代码图片

perf stat -e cpu/event=0x0e,umask=0x01,inv,cmask=0x01/ -a sleep 5

计算系统范围内每秒的系统调用数：

点击这里查看代码图片

perf stat -e raw_syscalls:sys_enter -I 1000 -a

按类型对指定 PID 的系统调用进行计数：

点击这里查看代码图片

perf stat -e 'syscalls:sys_enter_*' -p PID

计算整个系统的块设备 I/O 事件，持续 10 秒：

点击这里查看代码图片

perf stat -e 'block:*' -a sleep 10

分析

指定命令的 CPU 上函数示例（频率为 99 赫兹）：

perf record -F 99 command

系统范围内 10 秒的 CPU 堆栈跟踪示例（通过帧指针）：

点击这里查看代码图片

perf record -F 99 -a -g sleep 10

PID 的 CPU 堆栈跟踪示例，使用 dwarf (debuginfo) 展开堆栈：

点击这里查看代码图片

perf record -F 99 -p PID --call-graph dwarf sleep 10

容器的 /sys/fs/cgroup/perf_event cgroup 的示例 CPU 堆栈跟踪：

点击这里查看代码图片

perf record -F 99 -e cpu-clock --cgroup=docker/1d567f439319...etc... -a sleep 10

使用最后一个分支记录（LBR；Intel）的整个系统的 CPU 堆栈跟踪示例：

点击这里查看代码图片

perf record -F 99 -a --call-graph lbr sleep 10

CPU 堆栈跟踪示例，每 100 次最后一级缓存未命中一次，持续 5 秒：

点击这里查看代码图片

perf record -e LLC-load-misses -c 100 -ag sleep 5

精确采样 CPU 上的用户指令（例如，使用 Intel PEBS），持续 5 秒：

点击这里查看代码图片

perf record -e cycles:up -a sleep 5

49 赫兹的 CPU 示例，并实时显示顶级进程名称和段：

perf top -F 49 -ns comm,dso

静态追踪

跟踪新进程，直到 Ctrl-C：

点击这里查看代码图片

perf record -e sched:sched_process_exec -a

对具有堆栈跟踪的上下文切换子集进行 1 秒采样：

点击这里查看代码图片

perf record -e context-switches -a -g sleep 1

使用堆栈跟踪跟踪所有上下文切换 1 秒：

点击这里查看代码图片

perf record -e sched:sched_switch -a -g sleep 1

使用 5 级深度堆栈跟踪跟踪所有上下文切换 1 秒：

点击这里查看代码图片

perf record -e sched:sched_switch/max-stack=5/ -a sleep 1

使用堆栈跟踪跟踪 connect(2) 调用（出站连接），直到 Ctrl-C：

点击这里查看代码图片

perf record -e syscalls:sys_enter_connect -a -g

每秒最多采样 100 个块设备请求，直到 Ctrl-C：

点击这里查看代码图片

perf record -F 100 -e block:block_rq_issue -a

跟踪所有块设备问题和完成情况（有时间戳），直到 Ctrl-C：

点击这里查看代码图片

perf record -e block:block_rq_issue,block:block_rq_complete -a

跟踪大小至少为 64 KB 的所有块请求，直到 Ctrl-C：

点击这里查看代码图片

perf record -e block:block_rq_issue --filter 'bytes >= 65536'

跟踪所有 ext4 调用，并写入非 ext4 位置，直到 Ctrl-C：

点击这里查看代码图片

perf record -e 'ext4:*' -o /tmp/perf.data -a

跟踪 http__server__request USDT 事件（来自 Node.js；Linux 4.10+）：

点击这里查看代码图片

perf record -e sdt_node:http__server__request -a

使用实时输出（无 perf.data）跟踪块设备请求，直到 Ctrl-C：

点击这里查看代码图片

perf trace -e block:block_rq_issue

通过实时输出跟踪块设备请求和完成情况：

点击这里查看代码图片

perf trace -e block:block_rq_issue,block:block_rq_complete

通过实时输出（详细）跟踪系统范围内的系统调用：

perf trace

动态追踪

为内核 tcp_sendmsg() 函数条目添加探针（--add 可选）：

perf probe --add tcp_sendmsg

删除 tcp_sendmsg() 跟踪点（或 -d）：

perf probe --del tcp_sendmsg

列出 tcp_sendmsg() 的可用变量以及外部变量（需要内核调试信息）：

点击这里查看代码图片

perf probe -V tcp_sendmsg --externs

列出 tcp_sendmsg() 的可用线路探针（需要 debuginfo）：

perf probe -L tcp_sendmsg

列出第 81 行 tcp_sendmsg() 的可用变量（需要 debuginfo）：

perf probe -V tcp_sendmsg:81

添加带有条目参数寄存器的 tcp_sendmsg() 探测器（特定于处理器）：

点击这里查看代码图片

perf probe 'tcp_sendmsg %ax %dx %cx'

添加 tcp_sendmsg() 的探测器，并为 %cx 寄存器添加别名（“bytes”）：

点击这里查看代码图片

perf probe 'tcp_sendmsg bytes=%cx'

当字节（别名）大于 100 时，跟踪先前创建的探测：

点击这里查看代码图片

perf record -e probe:tcp_sendmsg --filter 'bytes > 100'

为 tcp_sendmsg() 返回添加跟踪点，并捕获返回值：

点击这里查看代码图片

perf probe 'tcp_sendmsg%return $retval'

为 tcp_sendmsg() 添加跟踪点，包含大小和套接字状态（需要 debuginfo）：

点击这里查看代码图片

perf probe 'tcp_sendmsg size sk->__sk_common.skc_state'

添加 do_sys_open() 的跟踪点，文件名作为字符串（需要 debuginfo）：

点击这里查看代码图片

perf probe 'do_sys_open filename:string'

为 libc 中的用户级 fopen(3) 函数添加跟踪点：

点击这里查看代码图片

perf probe -x /lib/x86_64-linux-gnu/libc.so.6 --add fopen

报告

如果可能的话，在 ncurses 浏览器 (TUI) 中显示 perf.data：

perf report

将 perf.data 显示为文本报告，其中包含数据合并、计数和百分比：

perf report -n --stdio

列出所有 perf.data 事件，带有数据头（推荐）：

perf script --header

列出所有 perf.data 事件，以及我推荐的字段（需要 record -a；Linux < 4.1 使用 -f 而不是 -F）：

点击这里查看代码图片

perf script --header -F comm,pid,tid,cpu,time,event,ip,sym,dso

生成火焰图可视化（Linux 5.8+）：

perf script report flamegraph

用百分比反汇编和注释指令（需要一些调试信息）：

perf annotate --stdio

这是我精选的俏皮话；还有更多此处未介绍的功能。有关更多 perf(1) 命令，请参阅上一节中的子命令以及本章和其他章节中的后续部分。

13.3 性能事件

可以使用 perf list 列出事件。我在此处包含了 Linux 5.8 中的一个选择，以显示不同类型的事件，并突出显示：

点击这里查看代码图片

13.4 硬件事件

硬件事件在第 4 章“可观测性工具”、第 4.3.9 节“硬件计数器 (PMC)”中介绍。它们通常使用 PMC 来实现，PMC 使用特定于处理器的代码进行配置；例如，Intel 处理器上的分支指令通常可以通过使用原始硬件事件描述符“r00c4”（寄存器代码的缩写：umask 0x0 和事件选择 0xc4）来使用 perf(1) 进行检测。这些代码发布在处理器手册中[Intel 16][AMD 18][ARM 19]； Intel 还以 JSON 文件的形式提供它们 [Intel 20c]。

您不需要记住这些代码，并且仅在需要时在处理器手册中参考它们。为了便于使用，perf(1) 提供了可供替代使用的人类可读映射。例如，事件“branch-instructions”有望映射到系统上的分支指令 PMC。2 其中一些人类可读的名称在前面的列表中可见（硬件和 PMU 事件）。

2我过去遇到过映射问题，人类可读的名称未映射到正确的 PMC。仅从 perf(1) 输出中很难识别这一点：您需要先前的 PMC 经验并对正常情况有预期，才能发现异常情况。请注意这种可能性。随着处理器更新的速度，我预计未来的映射也会出现错误。

处理器类型很多，并且定期发布新版本。您的处理器的人类可读映射可能在 perf(1) 中尚不可用，或者在较新的内核版本中可用。某些 PMC 可能永远不会通过人类可读的名称公开。一旦我转向缺乏映射的更深层次的 PMC，我经常必须从人类可读的名称切换到原始事件描述符。映射中也可能存在错误，如果您遇到可疑的 PMC 结果，您可能希望尝试使用原始事件描述符进行仔细检查。

13.4.1 频率采样

将 perf record 与 PMC 一起使用时，将使用默认采样频率，以便不会记录每个事件。例如，记录循环事件：

点击这里查看代码图片

# perf record -vve cycles -a sleep 1 Using CPUID GenuineIntel-6-8E

intel_pt default config: tsc,mtc,mtc_period=3,psb_period=3,pt,branch

perf_event_attr: size 112 { sample_period, sample_freq } 4000 sample_type IP|TID|TIME|CPU|PERIOD disabled 1 inherit 1 mmap 1 comm 1 freq 1 [...] [ perf record: Captured and wrote 3.360 MB perf.data (3538 samples) ]

输出显示频率采样已启用 (freq 1)，采样频率为 4000。这告诉内核调整采样率，以便每个 CPU 每秒捕获大约 4,000 个事件。这是可取的，因为某些 PMC 仪器事件每秒可能发生数十亿次（例如 CPU 周期），并且记录每个事件的开销将令人望而却步。3 但这也是一个陷阱：默认输出perf(1)（没有非常详细的选项：-vv）并不表示正在使用频率采样，并且您可能希望记录所有事件。此事件频率仅影响 record 子命令； stat 统计所有事件。

3尽管内核会限制采样率并丢弃事件以保护自身。请始终检查丢失的事件以查看是否发生这种情况（例如，检查来自 perf report -D | tail -20 的摘要计数器）。

可以使用 -F 选项修改事件频率，或使用 -c 更改为周期，该周期捕获每个周期一个事件（也称为溢出采样）。作为使用 -F 的示例：

点击这里查看代码图片

perf record -F 99 -e cycles -a sleep 1

此采样的目标速率为 99 赫兹（每秒事件数）。它类似于第 13.2 节 One-Liners 中的分析单行代码：它们不指定事件（无 -e cycles），这会导致 perf(1) 默认为循环（如果 PMC 可用），或者默认为循环。 cpu 时钟软件事件。有关更多详细信息，请参阅第 13.9.2 节“CPU 分析”。

请注意，频率有限制，perf(1) 也有 CPU 利用率百分比限制，可以使用 sysctl(8) 查看和设置：

点击这里查看代码图片

# sysctl kernel.perf_event_max_sample_rate kernel.perf_event_max_sample_rate = 15500

sysctl kernel.perf_cpu_time_max_percent

kernel.perf_cpu_time_max_percent = 25

这表明该系统的最大采样率为 15,500 赫兹，perf(1)（特别是 PMU 中断）允许的最大 CPU 利用率为 25%。

13.5 软件事件

这些事件通常映射到硬件事件，但在软件中进行检测。与硬件事件一样，它们可能具有默认采样频率（通常为 4000），因此在使用 record 子命令时仅捕获子集。

请注意上下文切换软件事件和等效跟踪点之间的以下区别。从软件事件开始：

点击这里查看代码图片

# perf record -vve context-switches -a -- sleep 1

[...]

perf_event_attr: type 1 size 112 config 0x3 { sample_period, sample_freq } 4000 sample_type IP|TID|TIME|CPU|PERIOD [...] freq 1 [...] [ perf record: Captured and wrote 3.227 MB perf.data (660 samples) ]

此输出显示软件事件默认以 4000 赫兹的速率进行频率采样。现在等效的跟踪点：

点击这里查看代码图片

# perf record -vve sched:sched_switch -a sleep 1

[...]

perf_event_attr: type 2 size 112 config 0x131 { sample_period, sample_freq } 1 sample_type IP|TID|TIME|CPU|PERIOD|RAW [...] [ perf record: Captured and wrote 3.360 MB perf.data (3538 samples) ]

这次使用周期采样（没有freq 1），采样周期为1（相当于-c 1）。这捕获了每个事件。您可以通过指定 -c 1 对软件事件执行相同的操作，例如：

点击这里查看代码图片

perf record -vve context-switches -a -c 1 -- sleep 1

请注意记录每个事件的量和所涉及的开销，尤其是频繁的上下文切换。您可以使用 perf stat 检查它们的频率：请参阅第 13.8 节，性能统计。

13.6 跟踪点事件

跟踪点在第 4 章“可观察性工具”第 4.3.5 节“跟踪点”中进行了介绍，其中包括使用 perf(1) 对其进行检测的示例。回顾一下，我使用了 block:block_rq_issue 跟踪点和以下示例。

跟踪系统范围 10 秒并打印事件：

点击这里查看代码图片

打印此跟踪点的参数及其格式字符串（元数据摘要）：

点击这里查看代码图片

仅过滤大于 65536 字节的块 I/O：

第 13.2 节“One-Liners”以及本书的其他章节中还有 perf(1) 和跟踪点的其他示例。

请注意，perf list 将显示初始化的探测事件，包括 kprobes（动态内核检测）作为“Tracepoint 事件”；请参见第 13.7 节，探测事件。

13.7 探测事件

perf(1) 使用术语探测事件来指代 kprobes、uprobes 和 USDT 探测。它们是“动态的”，必须首先初始化才能被追踪：默认情况下，它们不存在于 perf list 的输出中（某些 USDT 探针可能存在，因为它们已自动初始化）。一旦初始化，它们就会被列为“跟踪点事件”。

13.7.1 k探针

kprobes 在第 4 章“可观测性工具”第 4.3.6 节“kprobes”中介绍。以下是创建和使用 kprobe 的典型工作流程，在此示例中用于检测 do_nanosleep() 内核函数：

首先，如果内核 debuginfo 可用，则有关函数变量的信息（包括参数）可用于 perf(1)。使用 --vars 选项列出 do_nanosleep() kprobe 的变量：

点击这里查看代码图片

点击这里查看代码图片

      svscan  1470 \[012\] 4731125.216396: probe:do\_nanosleep: (ffffffffa8e4e440)

uprobes 在第 4 章“可观测性工具”第 4.3.7 节“uprobes”中介绍。使用 perf(1) 时，uprobe 的创建方式与 kprobes 类似。例如，要为 libc 文件打开函数 fopen(3) 创建 uprobe：

点击这里查看代码图片

# perf probe -x /lib/x86_64-linux-gnu/libc.so.6 --add fopen Added new event: probe_libc:fopen (on fopen in /lib/x86_64-linux-gnu/libc-2.27.so) You can now use it in all perf tools, such as:

    perf record -e probe\_libc:fopen -aR sleep 1

二进制路径使用 -x 指定。名为probe_libc:fopen的uprobe现在可以与perf record一起使用来记录事件。

当您完成 uprobe 后，您可以使用 --del 将其删除：

点击这里查看代码图片

# perf probe --del probe_libc:fopen Removed event: probe_libc:fopen

可以通过添加 %return 来检测函数的返回：

点击这里查看代码图片

perf probe -x /lib/x86_64-linux-gnu/libc.so.6 --add fopen%return

这使用了输尿管探针。

uprobe 参数

  • +0(...)：取消引用偏移量为零处的内容。如果没有这个，我们会意外地将地址打印为字符串，而不是将地址的内容打印为字符串。

  • :string：将其打印为字符串。

  • :u8：将其打印为无符号 8 位整数。

13.8 性能统计

perf stat 子命令对事件进行计数。这可用于测量事件发生率，或检查事件是否正在发生。 perf stat 非常高效：它使用 PMC 寄存器对内核上下文中的软件事件和硬件事件进行计数。这使得它适合通过首先使用 perf stat 检查事件率来衡量更昂贵的 perf record 子命令的开销。

例如，计算跟踪点 sched:sched_switch （使用 -e 作为事件）、系统范围 (-a) 和一秒（sleep 1：虚拟命令）：

点击这里查看代码图片

# perf stat -e sched:sched_switch -a -- sleep 1 Performance counter stats for 'system wide':

         5,705      sched:sched\_switch

   1.001892925 seconds time elapsed

这表明 sched:sched_switch 跟踪点在一秒钟内触发了 5,705 次。

我经常在 perf(1) 命令选项和它运行的虚拟命令之间使用“--” shell 分隔符，尽管在这种情况下并不是绝对必要的。

以下部分解释了选项和使用示例。

13.8.1 选项

stat 子命令支持许多选项，包括：

  • -a：跨所有 CPU 进行记录（这成为 Linux 4.11 中的默认设置）

  • -e _event_：记录此事件

  • --filter _filter_：为事件设置布尔过滤表达式

  • -p _PID_：仅记录此PID

  • -t _TID_：仅记录该线程ID

  • -G _cgroup_：仅记录此cgroup（用于容器）

  • -A：显示每个 CPU 的计数

  • -I _interval_ms_：每隔间隔（毫秒）打印输出

  • -v：显示详细消息； -vv获取更多消息

这些事件可以是跟踪点、软件事件、硬件事件、kprobes、uprobes 和 USDT 探针（请参见第 13.3 节到第 13.7 节）。通配符可用于匹配文件通配样式的多个事件（“*”匹配任何内容，“?”匹配任何一个字符）。例如，以下内容匹配 sched 类型的所有跟踪点：

# perf stat -e 'sched:*' -a

多个 -e 选项可用于匹配多个事件描述。例如，要计算计划跟踪点和块跟踪点，可以使用以下任一方法：

点击这里查看代码图片

# perf stat -e 'sched:*' -e 'block:*' -a

perf stat -e 'sched:*,block:*' -a

如果未指定事件，perf stat 将默认为架构 PMC：您可以在第 4 章“可观测性工具”、第 4.3.9 节“硬件计数器 (PMC)”中看到相关示例。

13.8.2 区间统计

13.8.5 影子统计

perf(1) 具有各种影子统计信息，当检测某些事件组合时将打印这些统计信息。例如，当检测 PMC 的周期和指令时，会打印每周期指令 (IPC) 统计数据：

要更详细地检查事件，可以使用 perf record 来捕获它们。

13.9 性能记录

perf record 子命令将事件记录到文件中以供以后分析。事件在 -e 之后指定，可以同时记录多个事件（可以使用多个 -e 或用逗号分隔）。

默认情况下，输出文件名为 perf.data。例如：

点击这里查看代码图片

# perf record -e sched:sched_switch -a ^C[ perf record: Woken up 9 times to write data ] [ perf record: Captured and wrote 6.060 MB perf.data (23526 samples) ]

请注意，输出包括 perf.data 文件的大小（6.060 MB）、它包含的样本数量（23,526）以及 perf(1) 唤醒记录数据的次数（9 次）。数据通过每个 CPU 的环形缓冲区从内核传递到用户空间，并且为了将上下文切换开销保持在最低限度，perf(1) 会以不频繁且动态的次数被唤醒来读取数据。

记录上一个命令，直到键入 Ctrl-C。虚拟 sleep(1) 命令（或任何命令）可用于设置持续时间（与之前的 perf stat 相同）。例如：

点击这里查看代码图片

perf record -e tracepoint -a -- sleep 1

此命令仅记录系统范围内的跟踪点 (-a) 1 秒。

13.9.1 选项

record 子命令支持许多选项，包括：

  • -a：跨所有 CPU 进行记录（这成为 Linux 4.11 中的默认设置）

  • -e _event_：记录此事件

  • --filter _filter_：为事件设置布尔过滤表达式

  • -p _PID_：仅记录此PID

  • -t _TID_：仅记录该线程ID

  • -G _cgroup_：仅记录此cgroup（用于容器）

  • -g：记录堆栈跟踪

  • --call-graph mode：使用给定方法（fp、dwarf 或 lbr）记录堆栈跟踪

  • -o _file_：设置输出文件

  • -v：显示详细消息； -vv获取更多消息

可以使用 perf stat 记录相同的事件，并使用 perf trace 实时打印（当事件发生时）。

13.9.2 CPU 分析

perf(1) 的常见用途是用作 CPU 分析器。以下分析示例以 99 赫兹的频率对所有 CPU 的堆栈跟踪进行采样，持续 30 秒：

点击这里查看代码图片

perf record -F 99 -a -g -- sleep 30

未指定事件（没有 -e），因此 perf(1) 将默认为可用的第一个（许多使用精确事件，在第 4 章“可观察性工具”第 4.3.9 节“硬件计数器”中介绍） （PMC））：

  1. 周期：ppp：基于 CPU 周期的频率采样，精确设置为零滑差

  2. Cycles:pp：基于 CPU 周期的频率采样，精确设置为请求零滑移（实际上可能不为零）

  3. 周期：p：基于 CPU 周期的频率采样，精确设置以请求恒定滑动

  4. Cycles：基于CPU周期的频率采样（不精确）

  5. cpu-clock：基于软件的CPU频率采样

订购时会选择可用的最准确的 CPU 分析机制。 :ppp、:pp 和 :p 语法激活精确事件采样模式，并且可以应用于支持它们的其他事件（周期除外）。事件还可以支持不同级别的精确度。在Intel上，精确事件使用PEBS；在 AMD 上，他们使用 IBS。这些在第 4.3.9 节“PMC 挑战”标题下定义。

13.9.3 堆栈遍历

可以使用 max-stack 配置选项，而不是使用 -g 来指定记录堆栈跟踪。它有两个好处：可以指定堆栈的最大深度，并且可以针对不同的事件使用不同的设置。例如：

点击这里查看代码图片

# perf record -e sched:sched_switch/max-stack=5/,sched:sched_wakeup/max-stack=1/ \ -a -- sleep 1

这会记录具有 5 帧堆栈的 sched_switch 事件，以及仅具有 1 帧堆栈的 sched_wakeup 事件。

请注意，如果堆栈跟踪出现损坏，可能是由于软件不支持帧指针寄存器。这已在第 5 章“应用程序”第 5.6.2 节“丢失堆栈”中讨论过。除了使用帧指针（例如 gcc(1) -fno-omit-frame-pointer）重新编译软件之外，还可以使用使用 --call-graph 选择的不同堆栈遍历方法。选项包括：

  • --call-graph dwarf：选择基于debuginfo的堆栈遍历，这需要可执行文件的debuginfo可用（对于某些软件，它是通过安装名称以“-dbgsym”或“-dbg”结尾的包来提供的） 。

  • --call-graph lbr：选择 Intel 最后分支记录 (LBR) 堆栈遍历，这是一种处理器提供的方法（尽管它通常仅限于 16 帧的堆栈深度，7 因此其用处也受到限制） 。

    7自 Haswell 以来堆栈深度为 16，自 Skylake 以来堆栈深度为 32。

  • --call-graph fp：选择基于帧指针的堆栈遍历（默认）。

基于帧指针的堆栈遍历在第 3 章“操作系统”第 3.2.7 节“堆栈”中进行了描述。其他类型（dwarf、LBR 和 ORC）在 BPF 性能工具 [Gregg 19] 的第 2 章“技术”第 2.4 节“堆栈跟踪遍历”中进行了描述。

记录事件后，可以使用 perf report 或 perf script 检查事件。

  • --tui：使用 TUI 界面（默认）

13.11.1 火焰图

火焰图可视化堆栈跟踪。虽然通常用于 CPU 配置文件，但它们可以可视化 perf(1) 收集的任何堆栈跟踪集合，包括在上下文切换事件上查看线程离开 CPU 的原因，以及在块 I/O 创建上查看哪些代码路径正在创建磁盘输入/输出。

两个常用的火焰图实现（我自己的和 d3 版本）可视化 perf script 的输出。 Linux 5.8 中添加了 perf(1) 中的火焰图支持。使用 perf(1) 创建火焰图的步骤包含在第 6 章“CPU”、第 6.6.13 节“perf”中的“CPU 火焰图”标题下。可视化本身在第 6.7.3 节“火焰图”中进行了解释。

FlameScope 是另一个用于可视化 perf script 输出的工具，它结合亚秒偏移热图来研究火焰图基于时间的变化。它还包含在第 6 章“CPU”、第 6.7.4 节“FlameScope”中。

13.12 性能跟踪

默认情况下，perf trace 子命令将跟踪系统调用并实时打印输出（无 perf.data 文件）。它是在第 5 章“应用程序”第 5.5.1 节“perf”中介绍的，作为 strace(1) 的低开销版本，可以在系统范围内进行跟踪。 perf trace 还可以使用与 perf record 类似的语法检查任何事件。

例如，跟踪磁盘 I/O 问题和完成情况：

点击这里查看代码图片

# perf trace -e block:block_rq_issue,block:block_rq_complete 0.000 auditd/391 block:block_rq_issue:259,0 WS 8192 () 16046032 + 16 [auditd] 0.566 systemd-journa/28651 block:block_rq_complete:259,0 WS () 16046032 + 16 [0] 0.748 jbd2/nvme0n1p1/174 block:block_rq_issue:259,0 WS 61440 () 2100744 + 120 [jbd2/nvme0n1p1-] 1.436 systemd-journa/28651 block:block_rq_complete:259,0 WS () 2100744 + 120 [0] 1.515 kworker/0:1H-k/365 block:block_rq_issue:259,0 FF 0 () 0 + 0 [kworker/0:1H] 1.543 kworker/0:1H-k/365 block:block_rq_issue:259,0 WFS 4096 () 2100864 + 8 [kworker/0:1H] 2.074 sshd/6463 block:block_rq_complete:259,0 WFS () 2100864 + 8 [0] 2.077 sshd/6463 block:block_rq_complete:259,0 WFS () 2100864 + 0 [0] 1087.562 kworker/0:1H-k/365 block:block_rq_issue:259,0 W 4096 () 16046040 + 8 [kworker/0:1H] [...]

与 perf record 一样，过滤器也可以用于事件。这些过滤器可以包含一些从内核头生成的字符串常量。例如，使用字符串“SHARED”跟踪标志为 MAP_SHARED 的 mmap(2) 系统调用：

点击这里查看代码图片

# perf trace -e syscalls:*enter_mmap --filter='flags==SHARED' 0.000 env/14780 syscalls:sys_enter_mmap(len: 27002, prot: READ, flags: SHARED, fd: 3) 16.145 grep/14787 syscalls:sys_enter_mmap(len: 27002, prot: READ, flags: SHARED, fd: 3) 18.704 cut/14791 syscalls:sys_enter_mmap(len: 27002, prot: READ, flags: SHARED, fd: 3) [...]

请注意，perf(1) 还使用字符串来提高格式字符串的可读性：它打印了“prot: READ”，而不是“prot: 1”。 perf(1) 将此功能称为“美化”。

13.12.1 内核版本

在 Linux 4.19 之前，除了指定的事件 (-e) 之外，perf trace 还会默认检测所有系统调用（--syscalls 选项）。要禁用跟踪其他系统调用，请指定 --no-syscalls （现在是默认值）。例如：

点击这里查看代码图片

# perf trace -e block:block_rq_issue,block:block_rq_complete --no-syscalls

请注意，自 Linux 3.8 起，跨所有 CPU (-a) 的跟踪已成为默认设置。 Linux 5.5 中添加了过滤器 (--filter)。

13.13 其他命令

还有更多 perf(1) 子命令和功能，其中一些在其他章节中使用。回顾一下其他子命令（完整列表请参见表 13.1）：

  • perf c2c (Linux 4.10+)：缓存到缓存和缓存行错误共享分析

  • perf kmem：内核内存分配分析

  • perf kvm：KVM 访客分析

  • perf lock：锁分析

  • perf mem：内存访问分析

  • perf sched：内核调度程序统计信息

  • perf 脚本：自定义 perf 工具

高级附加功能包括针对事件启动 BPF 程序，以及使用硬件跟踪（例如 Intel 处理器跟踪 (PT) 或 ARM CoreSight）进行每指令分析 [Hunter 20]。
