第 3 章操作系统

第三章

操作系统

了解操作系统及其内核对于系统性能分析至关重要。您经常需要开发并测试有关系统行为的假设，例如系统调用如何执行、内核如何调度 CPU 上的线程、有限的内存如何影响性能，或者文件系统如何处理 I/O。这些活动将要求您运用操作系统和内核的知识。

本章的学习目标是：

  • 学习内核术语：上下文切换、交换、分页、抢占等。

  • 了解内核和系统调用的作用。

  • 获得内核内部的实用知识，包括：中断、调度程序、虚拟内存和 I/O 堆栈。

  • 了解如何将内核性能功能从 Unix 添加到 Linux。

  • 对扩展 BPF 有基本的了解。

本章概述了操作系统和内核，并为本书的其余部分提供了假定的知识。如果您错过了操作系统课程，您可以将其视为速成课程。请留意您的知识差距，因为最后会有考试（我开玩笑的；这只是一个测验）。有关内核内部结构的更多信息，请参阅本章末尾的参考资料。

本章分为三个部分：

  • 术语列出了基本术语。

  • 背景总结了关键操作系统和内核概念。

  • 内核总结了 Linux 和其他内核的实现细节。

与性能相关的领域，包括 CPU 调度、内存、磁盘、文件系统、网络和许多特定的性能工具，将在后续章节中进行更详细的介绍。

3.1 术语

作为参考，这里是本书中使用的核心操作系统术语。其中许多也是本章和后续章节中更详细解释的概念。

  • 操作系统：指安装在系统上以便引导和执行程序的软件和文件。它包括内核、管理工具和系统库。

  • 内核：内核是管理系统的程序，包括（取决于内核型号）硬件设备、内存和CPU调度。它运行在允许直接访问硬件的特权CPU模式下，称为内核模式。

  • 进程：用于执行程序的操作系统抽象和环境。该程序在用户模式下运行，可以通过系统调用或陷阱访问内核模式（例如，用于执行设备 I/O）。

  • 线程：可以安排在 CPU 上运行的可执行上下文。内核有多个线程，一个进程包含一个或多个线程。

  • 任务：Linux 可运行实体，可以指进程（具有单个线程）、多线程进程中的线程或内核线程。

  • BPF程序：运行在BPF1执行环境中的内核模式程序。

    1BPF 最初代表伯克利数据包过滤器，但如今的技术与伯克利、数据包或过滤关系不大，以至于 BPF 本身已成为一个名称，而不是缩写词。

  • 主内存：系统的物理内存（例如RAM）。

  • 虚拟内存：支持多任务和超额订阅的主内存的抽象。实际上，它是一种无限的资源。

  • 内核空间：内核的虚拟内存地址空间。

  • 用户空间：进程的虚拟内存地址空间。

  • 用户域：用户级程序和库（/usr/bin、/usr/lib...）。

  • 上下文切换：从运行一个线程或进程到另一个线程或进程的切换。这是内核 CPU 调度程序的正常功能，涉及将正在运行的 CPU 寄存器组（线程上下文）切换到新组。

  • 模式切换：内核模式和用户模式之间的切换。

  • 系统调用（syscall）：一种定义明确的协议，供用户程序请求内核执行特权操作，包括设备 I/O。

  • 处理器：不要与进程混淆，处理器是包含一个或多个 CPU 的物理芯片。

  • 陷阱：发送到内核以请求系统例程（特权操作）的信号。陷阱类型包括系统调用、处理器异常和中断。

  • 硬件中断：物理设备向内核发送的信号，通常用于请求 I/O 服务。中断是一种陷阱。

如果本章需要，术语表包含更多术语供参考，包括地址空间、缓冲区、CPU、文件描述符、POSIX 和寄存器。

3.2 背景

以下部分描述通用操作系统和内核概念，并将帮助您了解任何操作系统。为了帮助您理解，本节包含一些 Linux 实现细节。接下来的部分，3.3 内核和 3.4 Linux，重点关注 Unix、BSD 和 Linux 内核实现细节。

3.2.1 内核

内核是操作系统的核心软件。它的作用取决于内核模型：包括 Linux 和 BSD 在内的类 Unix 操作系统具有管理 CPU 调度、内存、文件系统、网络协议和系统设备（磁盘、网络接口等）的整体内核。该内核模型如图 3.1 所示。

图 3.1 单片操作系统内核的作用

还显示了系统库，它们通常用于提供比单独的系统调用更丰富、更简单的编程接口。应用程序包括所有正在运行的用户级软件，包括数据库、Web 服务器、管理工具和操作系统 shell。

内核执行

内核是一个大型程序，通常有数百万行代码。它主要在用户级程序进行系统调用或设备发送中断时按需执行。一些内核线程异步操作以进行内务管理，其中可能包括内核时钟例程和内存管理任务，但这些任务尽量轻量级并且消耗很少的CPU资源。

执行频繁 I/O 的工作负载（例如 Web 服务器）主要在内核上下文中执行。计算密集型工作负载通常在用户模式下运行，不受内核中断。人们可能会认为内核不会影响这些计算密集型工作负载的性能，但在很多情况下确实会影响这些计算密集型工作负载的性能。最明显的是 CPU 争用，当其他线程竞争 CPU 资源时，内核调度程序需要决定哪些线程将运行，哪些线程将等待。内核还选择线程将在哪个 CPU 上运行，并且可以为进程选择具有更温暖的硬件缓存或更好的内存局部性的 CPU，以显着提高性能。

3.2.2 内核模式和用户模式

内核运行在一种称为内核模式的特殊 CPU 模式下，允许对设备的完全访问和特权指令的执行。内核仲裁设备访问以支持多任务处理，防止进程和用户访问彼此的数据，除非明确允许。

用户程序（进程）在用户模式下运行，它们通过系统调用向内核请求特权操作，例如 I/O。

内核和用户模式是在使用特权环（或保护环）的处理器上实现的，遵循图 3.1 中的模型。例如，x86 处理器支持四个特权环，编号为 0 到 3。通常仅使用两个或三个：用于用户模式、内核模式和管理程序（如果存在）。访问设备的特权指令只允许在内核模式下使用；在用户模式下执行它们会导致异常，然后由内核处理（例如，生成权限被拒绝错误）。

在传统的内核中，系统调用是通过切换到内核模式，然后执行系统调用代码来执行的。如图 3.3 所示。

图3.3 系统调用执行模式

用户模式和内核模式之间的切换就是模式切换。

由于模式和上下文切换会消耗少量开销（CPU 周期），3 有各种优化可以避免它们，包括：

  • 用户模式系统调用：可以单独在用户模式库中实现一些系统调用。 Linux 内核通过导出映射到进程地址空间的虚拟动态共享对象 (vDSO) 来实现此目的，其中包含 gettimeofday(2) 和 getcpu(2) 等系统调用 [Drysdale 14]。

  • 内核旁路：这允许用户模式程序直接访问设备，绕过系统调用和典型的内核代码路径。例如，用于网络的 DPDK：数据平面开发套件。

  • 内核模式应用程序：其中包括在内核中实现的 TUX Web 服务器 [Lever 00]，以及最近的扩展 BPF 技术，如图 3.2 所示。

3.2.3 系统调用

系统调用请求内核执行特权系统例程。有数百个可用的系统调用，但内核维护者做出了一些努力，使该数量尽可能小，以保持内核简单（Unix 哲学；[Thompson 78]）。可以在用户空间中将更复杂的接口构建为系统库，从而更容易开发和维护。操作系统通常包含一个 C 标准库，为许多常见的系统调用提供更易于使用的接口（例如 libc 或 glibc 库）。

ioctl(2) 系统调用可能是最难学习的，因为它的性质不明确。作为其用法的一个示例，Linux perf(1) 工具（在第 6 章 CPU 中介绍）执行特权操作来协调性能检测。不是为每个操作添加系统调用，而是添加一个系统调用：perf_event_open(2)，它返回一个与 ioctl(2) 一起使用的文件描述符。然后可以使用不同的参数调用此 ioctl(2) 以执行不同的所需操作。例如，ioctl(fd, PERF_EVENT_IOC_ENABLE) 启用检测。开发人员可以更轻松地添加和更改本示例中的参数 PERF_EVENT_IOC_ENABLE。

3.2.4 中断

同步中断

同步中断由软件指令产生。下面使用术语“陷阱”、“异常”和“故障”描述不同类型的软件中断；然而，这些术语经常可以互换使用。

  • 陷阱：故意调用内核，例如通过 int（中断）指令。系统调用的一种实现涉及使用系统调用处理程序的向量调用 int 指令（例如，Linux x86 上的 int 0x80）。 int 引发软件中断。

  • 异常：异常情况，例如执行除以零的指令。

  • 错误：通常用于内存事件的术语，例如在没有 MMU 映射的情况下访问内存位置而触发的页面错误。请参阅第 7 章“内存”。

中断服务例程 (ISR) 旨在尽可能快地运行，以减少中断活动线程的影响。如果中断需要执行大量工作，特别是如果它可能因锁而阻塞，则可以由可由内核调度的中断线程来处理。如图 3.6 所示。

图3.6 中断处理

如何实现取决于内核版本。在 Linux 上，设备驱动程序可以建模为两半，上半部分快速处理中断，并将工作安排到下半部分稍后处理 [Corbet 05]。快速处理中断非常重要，因为上半部分在中断禁用模式下运行，以推迟新中断的传递，如果运行时间过长，可能会导致其他线程出现延迟问题。下半部分可以是微线程或工作队列；后者是可以由内核调度的线程，并且可以在必要时休眠。

从中断到达到被服务的时间称为中断延迟，这取决于硬件和实现。这是实时或低延迟系统的研究主题。

中断屏蔽

内核中的某些代码路径无法安全中断。一个例子是在系统调用期间获取自旋锁的内核代码，中断也可能需要自旋锁。在持有此类锁的情况下进行中断可能会导致死锁。为了防止这种情况，内核可以通过设置CPU的中断屏蔽寄存器来暂时屏蔽中断。中断禁用时间应尽可能短，因为它会干扰由其他中断唤醒的应用程序的及时执行。对于具有严格响应时间要求的实时系统来说，这是一个重要因素。中断禁用时间也是性能分析的一个目标（此类分析由 Ftrace irqsoff 跟踪器直接支持，在第 14 章 Ftrace 中提到）。

一些高优先级事件不应被忽略，因此被实现为不可屏蔽中断（NMI）。例如，Linux 可以使用智能平台管理接口 (IPMI) 看门狗定时器，根据一段时间内没有中断来检查内核是否已锁定。如果是这样，看门狗可以发出 NMI 中断来重新启动系统。5

5Linux 还有一个用于检测锁定的软件 NMI 看门狗 [Linux 20d]。

3.2.5 时钟和空闲

原始 Unix 内核的核心组件是由定时器中断执行的 Clock() 例程。历史上它的执行速度为每秒 60、100 或 1,000 次6（通常以赫兹表示：每秒周期数），每次执行称为一个周期。7其功能包括更新系统时间、用于线程调度的到期计时器和时间片、维护 CPU 统计数据以及执行调度的内核例程。

6其他费率包括 Linux 2.6.13 为 250、Ultrix 为 256、OSF/1 为 1,024 [Mills 94]。

7Linux 还跟踪 jiffies，这是一种类似于时钟周期的时间单位。

时钟存在性能问题，在后来的内核中得到了改进，包括：

  • 滴答延迟：对于 100 赫兹时钟，定时器在等待下一个滴答时处理时可能会遇到最多 10 毫秒的额外延迟。已使用高分辨率实时中断修复了此问题，以便立即执行。

  • 滴答开销：滴答会消耗 CPU 周期并稍微扰乱应用程序，并且是操作系统抖动的原因之一。现代处理器还具有动态电源功能，可以在空闲期间关闭部件。时钟例程会中断此空闲时间，这会不必要地消耗功率。

现代内核已将许多功能从时钟例程转移到按需中断，以创建无滴答内核。这允许处理器更长时间地保持睡眠状态，从而减少开销并提高电源效率。

Linux 时钟例程是scheduler_tick()，Linux 有方法在没有任何 CPU 负载时省略调用时钟。时钟本身通常以 250 赫兹运行（由 CONFIG_HZ Kconfig 选项和变体配置），并且它的调用通过 NO_HZ 功能（由 CONFIG_NO_HZ 和变体配置）减少，该功能现在已普遍启用 [Linux 20a]。

空闲线程

当 CPU 没有工作需要执行时，内核会调度一个等待工作的占位线程，称为空闲线程。一个简单的实现将检查循环中新工作的可用性。在现代Linux中，空闲任务可以调用hlt（halt）指令来关闭CPU，直到收到下一个中断，从而节省电量。

3.2.6 流程

进程是执行用户级程序的环境。它由内存地址空间、文件描述符、线程堆栈和寄存器组成。在某些方面，进程就像一台虚拟的早期计算机，其中只有一个程序使用自己的寄存器和堆栈执行。

进程由内核执行多任务处理，通常支持在单个系统上执行数千个进程。它们通过进程 ID (PID) 进行单独标识，PID 是唯一的数字标识符。

一个进程包含一个或多个线程，这些线程在进程地址空间中运行并共享相同的文件描述符。线程是由堆栈、寄存器和指令指针（也称为程序计数器）组成的可执行上下文。多个线程允许单个进程在多个 CPU 上并行执行。在Linux上，线程和进程都是任务。

内核启动的第一个进程称为“init”，来自 /sbin/init（默认情况下），PID 为 1，用于启动用户空间服务。在 Unix 中，这涉及从 /etc 运行启动脚本，这种方法现在称为 SysV（以 Unix System V 命名）。 Linux 发行版现在通常使用 systemd 软件来启动服务并跟踪它们的依赖关系。

流程创建

3.2.7 堆栈

堆栈是临时数据的内存存储区域，以后进先出 (LIFO) 列表的形式组织。它用于存储比适合 CPU 寄存器组的数据不太重要的数据。当函数被调用时，返回地址被保存到堆栈中。如果调用后需要某些寄存器的值，则某些寄存器也可以保存到堆栈中。10 当被调用函数完成时，它会恢复任何所需的寄存器，并通过从堆栈中获取返回地址，将执行传递给调用函数。堆栈还可用于将参数传递给函数。堆栈上与函数执行相关的数据集称为堆栈帧。

10处理器 ABI 的调用约定指定哪些寄存器应在函数调用后保留其值（它们是非易失性的），并由被调用函数保存到堆栈（“被调用者保存”）。其他寄存器是易失性的，可能会被调用的函数破坏；如果调用者希望保留它们的值，则必须将它们保存到堆栈中（“调用者保存”）。

通过检查线程堆栈中所有堆栈帧中保存的返回地址，可以看到当前正在执行的函数的调用路径（这个过程称为堆栈遍历）。11此调用路径称为堆栈回溯或堆栈跟踪。在性能工程中，它通常简称为“堆栈”。这些堆栈可以回答某些内容正在执行的原因，并且是调试和性能分析的宝贵工具。

11有关堆栈遍历和不同可能技术（包括：基于帧指针、debuginfo、最后分支记录和 ORC）的更多详细信息，请参阅 BPF 性能的第 2 章“技术”、第 2.4 节“堆栈跟踪遍历”工具 [格雷格 19]。

3.2.8 虚拟内存

虚拟内存是主内存的抽象，为进程和内核提供了自己的、几乎无限的、12主内存的私有视图。它支持多任务处理，允许进程和内核在自己的私有地址空间上运行，而不必担心争用。它还支持主内存的超额订阅，允许操作系统根据需要在主内存和辅助存储（磁盘）之间透明地映射虚拟内存。

内存管理

虽然虚拟内存允许使用辅助存储来扩展主内存，但内核努力将最活跃的数据保留在主内存中。为此有两种内核方案：

  • 进程交换在主内存和辅助存储之间移动整个进程。

  • 分页移动称为页（例如 4 KB）的小内存单元。

进程交换是原始的 Unix 方法，会导致严重的性能损失。分页效率更高，并通过引入分页虚拟内存而添加到 BSD 中。在这两种情况下，最近最少使用（或最近未使用）的内存都会被移至辅助存储，并仅在再次需要时移回主内存。

在 Linux 中，术语交换用于指代分页。 Linux 内核不支持整个线程和进程的（较旧的）Unix 风格的进程交换。

有关分页和交换的更多信息，请参阅第 7 章“内存”。

3.2.9 调度程序

Unix 及其衍生系统是分时系统，通过在多个进程之间分配执行时间来允许多个进程同时运行。处理器和各个 CPU 上的进程调度由调度程序执行，调度程序是操作系统内核的关键组件。调度程序的作用如图 3.12 所示，它显示调度程序对线程（在 Linux 中为任务）进行操作，将它们映射到 CPU。

图3.12 内核调度器

基本目的是在活动进程和线程之间分配 CPU 时间，并维护优先级概念，以便更重要的工作可以更快执行。调度程序跟踪所有处于准备运行状态的线程，传统上在称为运行队列的按优先级队列上[Bach 86]。现代内核可以为每个 CPU 实现这些队列，并且还可以使用除队列之外的其他数据结构来跟踪线程。当想要运行的线程多于可用 CPU 时，优先级较低的线程将等待轮到它们。大多数内核线程的运行优先级高于用户级进程。

调度程序可以动态修改进程优先级，以提高某些工作负载的性能。工作负载可以分为以下几类：

  • CPU 密集型：执行繁重计算的应用程序，例如科学和数学分析，预计运行时间较长（秒、分钟、小时、天甚至更长）。这些受到 CPU 资源的限制。

  • I/O 密集型：执行 I/O 且计算量很少的应用程序，例如 Web 服务器、文件服务器和交互式 shell，需要低延迟响应。当负载增加时，它们会受到存储或网络资源 I/O 的限制。

一种可追溯到 UNIX 的常用调度策略可识别 CPU 密集型工作负载并降低其优先级，从而允许 I/O 密集型工作负载（更需要低延迟响应）更快地运行。这可以通过计算最近的计算时间（在 CPU 上执行的时间）与实时时间（经过的时间）的比率并降低具有高（计算）比率的进程的优先级来实现 [Thompson 78]。此机制优先考虑运行时间较短的进程，这些进程通常是执行 I/O 的进程，包括人工交互进程。

现代内核支持多个调度类或调度策略 (Linux)，它们应用不同的算法来管理优先级和可运行线程。这些可能包括实时调度，它使用高于所有非关键工作（包括内核线程）的优先级。除了抢占支持（稍后描述）之外，实时调度还为需要它的系统提供可预测的低延迟调度。

有关内核调度程序和其他调度算法的更多信息，请参阅第 6 章“CPU”。

3.2.10 文件系统

文件系统是将数据组织为文件和目录。它们有一个基于文件的接口来访问它们，通常基于 POSIX 标准。内核支持多种文件系统类型和实例。提供文件系统是操作系统最重要的角色之一，曾经被描述为最重要的角色[Ritchie 74]。

操作系统提供了一个全局文件命名空间，组织为从根级别（“/”）开始的自上而下的树形拓扑。文件系统通过安装将其自己的树附加到目录（安装点）来加入树。这允许最终用户透明地导航文件命名空间，而不管底层文件系统类型如何。

3.2.12 联网

现代内核提供了一系列内置网络协议，允许系统通过网络进行通信并参与分布式系统环境。这在常用的 TCP 和 IP 协议之后被称为网络堆栈或 TCP/IP 堆栈。用户级应用程序通过称为套接字的可编程端点访问网络。

连接到网络的物理设备是网络接口，通常由网络接口卡 (NIC) 提供。系统管理员的历史职责是将IP地址与网络接口相关联，以便它可以与网络通信；这些映射现在通常通过动态主机配置协议 (DHCP) 自动完成。

网络协议不会经常改变，但有一种新的传输协议得到了越来越多的采用：QUIC（在第 10 章“网络”中进行了总结）。协议增强功能和选项更频繁地更改，例如更新的 TCP 选项和 TCP 拥塞控制算法。较新的协议和增强功能通常需要内核支持（用户空间协议实现除外）。另一个变化是对不同网络接口卡的支持，这需要内核的新设备驱动程序。

有关网络和网络性能的更多信息，请参阅第 10 章，网络。

3.2.13 设备驱动程序

内核必须与各种物理设备进行通信。这种通信是使用设备驱动程序实现的：用于设备管理和 I/O 的内核软件。设备驱动程序通常由开发硬件设备的供应商提供。某些内核支持可插拔设备驱动程序，无需重新启动系统即可加载和卸载这些驱动程序。

设备驱动程序可以为其设备提供字符和/或块接口。字符设备，也称为原始设备，提供对任何 I/O 大小（小至单个字符）的无缓冲顺序访问，具体取决于设备。此类设备包括键盘和串行端口（以及原始 Unix 中的纸带和行式打印机设备）。

块设备以块为单位执行 I/O，历史上每个块为 512 字节。可以根据它们的块偏移量随机访问它们，块偏移量从块设备开头的 0 开始。在最初的 Unix 中，块设备接口还提供了块设备缓冲区的缓存以提高性能，该缓存位于主内存的一个称为缓冲区高速缓存的区域中。在 Linux 中，该缓冲区高速缓存现在是页面高速缓存的一部分。

3.2.14 多处理器

多处理器支持允许操作系统使用多个 CPU 实例并行执行工作。它通常以对称多处理 (SMP) 的形式实现，其中所有 CPU 都受到同等对待。这在技术上很难实现，导致在并行运行的线程之间访问和共享内存和 CPU 时出现问题。在多处理器系统上，还可能存在连接到非均匀内存访问 (NUMA) 架构中的不同插槽（物理处理器）的主内存组，这也带来了性能挑战。有关详细信息（包括调度和线程同步），请参阅第 6 章“CPU”；有关内存访问和体系结构的详细信息，请参阅第 7 章“内存”。

IPI

对于多处理器系统，有时 CPU 需要进行协调，例如内存转换条目的缓存一致性（通知其他 CPU 某个条目（如果已缓存）现在已过时）。 CPU 可以使用处理器间中断 (IPI)（也称为 SMP 调用或 CPU 交叉调用）请求其他 CPU 或所有 CPU 立即执行此类工作。 IPI 是旨在快速执行的处理器中断，以最大限度地减少其他线程的中断。

IPI 也可以通过抢占的方式使用。

3.2.15 抢占

内核抢占支持允许高优先级用户级线程中断内核并执行。这使得实时系统能够在给定的时间限制内执行工作，包括飞机和医疗设备使用的系统。支持抢占的内核被认为是完全可抢占的，尽管实际上它仍然会有一些不能被中断的小的关键代码路径。

Linux 支持的另一种方法是自愿内核抢占，其中内核代码中的逻辑停止点可以检查并执行抢占。这避免了支持完全抢占式内核的一些复杂性，并为常见工作负载提供低延迟抢占。 Linux 中通常通过 CONFIG_PREEMPT_VOLUNTARY Kconfig 选项启用自愿内核抢占；还有 CONFIG_PREEMPT 允许所有内核代码（关键部分除外）可抢占，以及 CONFIG_PREEMPT_NONE 禁用抢占，以更高的延迟为代价提高吞吐量。

3.2.16 资源管理

操作系统可以提供各种可配置的控制来微调对系统资源（例如CPU、内存、磁盘和网络）的访问。这些是资源控制，可用于管理运行不同应用程序或托管多个租户（云计算）的系统的性能。此类控制可能会对每个进程（或进程组）的资源使用施加固定限制，或更灵活的方法——允许在它们之间共享备用资源。

Unix 和 BSD 的早期版本具有基本的每进程资源控制，包括使用 Nice(1) 的 CPU 优先级，以及使用 ulimit(1) 的一些资源限制。

对于 Linux，控制组 (cgroup) 已在 Linux 2.6.24 (2008) 中开发并集成，此后还添加了各种附加控件。这些记录在内核源代码的 Documentation/cgroups 下。还有一种改进的统一分层方案，称为 cgroup v2，在 Linux 4.5 (2016) 中提供，并记录在 Documentation/admin-guide/cgroup-v2.rst 中。

具体的资源控制将在后面的章节中酌情提及。第 11 章“云计算”中描述了一个示例用例，用于管理操作系统虚拟化租户的性能。

3.2.17 可观察性

操作系统由内核、库和程序组成。这些程序包括观察系统活动和分析性能的工具，通常安装在 /usr/bin 和 /usr/sbin 中。系统上还可以安装第三方工具以提供额外的可观察性。

3.3.1 Unix

Unix 是由 Ken Thompson、Dennis Ritchie 和 AT&T 贝尔实验室的其他人在 1969 年及随后的几年中开发的。它的确切起源在 The UNIX Time-Sharing System [Ritchie 74] 中有描述：

当我们中的一个人（汤普森）对可用的计算机设施不满意时，发现了很少使用的 PDP-7 并开始创建一个更友好的环境，从而编写了第一个版本。

UNIX 的开发人员之前曾开发过多路信息和计算机服务 (Multics) 操作系统。 UNIX 是作为一个轻量级多任务操作系统和内核而开发的，最初名为UNiplexed 信息和计算服务(UNICS)，是Multics 的双关语。来自 UNIX 实现 [Thompson 78]：

内核是唯一不能由用户根据自己的喜好进行替换的 UNIX 代码。因此，内核应该做出尽可能少的实际决策。这并不意味着允许用户有一百万个选项来做同样的事情。相反，它意味着只允许一种方法来做一件事，但让这种方法成为可能提供的所有选项的最小公约数。

虽然内核很小，但它确实提供了一些高性能功能。进程具有调度程序优先级，可以减少高优先级工作的运行队列延迟。为了提高效率，磁盘 I/O 在大块（512 字节）中执行，并缓存在内存中的每设备缓冲区高速缓存中。空闲进程可以交换到存储中，从而允许繁忙的进程在主内存中运行。当然，该系统是多任务处理的——允许多个进程同时运行，从而提高作业吞吐量。

为了支持网络、多文件系统、分页和我们现在认为标准的其他功能，内核必须增长。随着多种衍生产品的出现，包括 BSD、SunOS (Solaris) 和后来的 Linux，内核性能变得具有竞争力，从而推动了更多功能和代码的添加。

3.3.2 BSD

Berkeley Software Distribution (BSD) 操作系统最初是加州大学伯克利分校对 Unix 第 6 版的增强，并于 1978 年首次发布。由于原始 Unix 代码需要 AT&T 软件许可证，到 20 世纪 90 年代初，该 Unix 代码已被在新的 BSD 许可证下用 BSD 重写，允许免费分发，包括 FreeBSD。

主要的 BSD 内核开发，尤其是与性能相关的开发，包括：

  • 分页虚拟内存：BSD 将分页虚拟内存引入 Unix：无需交换整个进程来释放主内存，而是可以移动（分页）较小的最近最少使用的内存块。请参见第 7 章“内存”、第 7.2.2 节“分页”。

  • 按需分页：这将物理内存到虚拟内存的映射推迟到首次写入时，避免了可能永远不会使用的页面的早期且有时不必要的性能和内存成本。 BSD 将请求分页引入 Unix。请参见第 7 章“内存”、第 7.2.2 节“分页”。

  • FFS：伯克利快速文件系统 (FFS) 将磁盘分配分组为柱面组，大大减少了碎片并提高了旋转磁盘的性能，并支持更大的磁盘和其他增强功能。 FFS 成为许多其他文件系统（包括 UFS）的基础。请参见第 8 章“文件系统”、第 8.4.5 节“文件系统类型”。

  • TCP/IP 网络堆栈：BSD 为 Unix 开发了第一个高性能 TCP/IP 网络堆栈，包含在 4.2BSD (1983) 中。 BSD 仍然以其高性能的网络堆栈而闻名。

  • 套接字：Berkeley 套接字是用于连接端点的 API。它们包含在 4.2BSD 中，已成为网络标准。请参阅第 10 章，网络。

  • Jails：轻量级操作系统级虚拟化，允许多个来宾共享一个内核。 Jails 首次在 FreeBSD 4.0 中发布。

  • 内核 TLS：由于传输层安全性 (TLS) 现在在 Internet 上广泛使用，因此内核 TLS 将大部分 TLS 处理移至内核，从而提高了性能14 [Stewart 15]。

14旨在提高 Netflix FreeBSD 开放连接设备 (OCA)（即 Netflix CDN）的性能。

虽然 BSD 不像 Linux 那样流行，但它也用于一些性能关键型环境，包括 Netflix 内容分发网络 (CDN) 以及 NetApp、Isilon 等公司的文件服务器。 Netflix 将 2019 年 FreeBSD 在其 CDN 上的表现总结为 [Looney 19]：

“使用 FreeBSD 和商用部件，我们在 16 核 2.6 GHz CPU 上以约 55% 的 CPU 实现了 90 Gb/s 的 TLS 加密连接服务。”

有一本关于 FreeBSD 内部结构的优秀参考资料，来自为您提供本书的同一出版商：《FreeBSD 操作系统的设计和实现》，第二版 [McKusick 15]。

3.3.3 Solaris

Solaris 是由 Sun Microsystems 于 1982 年创建的源自 Unix 和 BSD 的内核和操作系统。它最初被命名为 SunOS，并针对 Sun 工作站进行了优化。到 20 世纪 80 年代末，AT&T 基于 SVR3、SunOS、BSD 和 Xenix 的技术开发了新的 Unix 标准，即 Unix System V Release 4 (SVR4)。 Sun 基于 SVR4 创建了新内核，并将操作系统重新命名为 Solaris。

主要的 Solaris 内核开发，尤其是与性能相关的开发，包括：

  • VFS：虚拟文件系统（VFS）是一个抽象和接口，允许多个文件系统轻松共存。 Sun 最初创建它是为了让 NFS 和 UFS 能够共存。第 8 章“文件系统”中介绍了 VFS。

  • 完全可抢占的内核：这为高优先级工作（包括实时工作）提供了低延迟。

  • 多处理器支持：在 20 世纪 90 年代初期，Sun 在多处理器操作系统支持方面投入了大量资金，开发了对非对称和对称多处理（ASMP 和 SMP）的内核支持 [Mauro 01]。

  • Slab 分配器：取代 SVR4 伙伴分配器，内核 Slab 内存分配器通过可快速重用的预分配缓冲区的每 CPU 缓存提供了更好的性能。这种分配器类型及其派生物已成为包括 Linux 在内的内核的标准。

  • DTrace：一种静态和动态跟踪框架和工具，可在实时和生产中提供对整个软件堆栈几乎无限的可观察性。 Linux 有 BPF 和 bpftrace 用于这种类型的可观察性。

  • 区域：一种基于操作系统的虚拟化技术，用于创建共享一个内核的操作系统实例，类似于早期的 FreeBSD 监狱技术。操作系统虚拟化现在作为 Linux 容器被广泛使用。请参阅第 11 章，云计算。

  • ZFS：具有企业级功能和性能的文件系统。它现在可用于其他操作系统，包括 Linux。请参阅第 8 章，文件系统。

Oracle 于 2010 年收购了 Sun Microsystems，Solaris 现在更名为 Oracle Solaris。本书第一版更详细地介绍了 Solaris。

3.4 Linux

Linux 是由 Linus Torvalds 于 1991 年创建的，作为 Intel 个人计算机的免费操作系统。他在新闻组帖子中宣布了该项目：

我正在为 386(486) AT 克隆做一个（免费）操作系统（只是一个爱好，不会像 gnu 那样大和专业）。这个计划从四月份就开始酝酿，现在已经开始准备了。我希望得到关于人们喜欢/不喜欢 minix 的任何反馈，因为我的操作系统有点类似于它（文件系统的物理布局相同（由于实际原因）等）。

这是指 MINIX 操作系统，它是为小型计算机开发的免费小型（迷你）版本 Unix。 BSD 还旨在提供免费的 Unix 版本，尽管当时它遇到了法律问题。

Linux 内核的开发借鉴了许多祖先的一般思想，包括：

  • Unix（和 Multics）：操作系统层、系统调用、多任务、进程、进程优先级、虚拟内存、全局文件系统、文件系统权限、设备节点、缓冲区高速缓存

  • BSD：分页虚拟内存、请求分页、快速文件系统 (FFS)、TCP/IP 网络堆栈、套接字

  • Solaris：VFS、NFS、页面缓存、统一页面缓存、slab 分配器

  • 方案9：资源分叉（rfork），用于在进程和线程（任务）之间创建不同级别的共享

Linux 现在广泛用于服务器、云实例和包括移动电话在内的嵌入式设备。

3.4.1 Linux 内核发展

Linux 内核开发，特别是与性能相关的开发，包括以下内容（其中许多描述包括首次引入它们的 Linux 内核版本）：

  • CPU 调度类：已经开发了各种高级 CPU 调度算法，包括调度域 (2.6.7)，以便在非均匀内存访问 (NUMA) 方面做出更好的决策。请参阅第 6 章，CPU。

  • I/O 调度类：已经开发了不同的块 I/O 调度算法，包括截止时间 (2.5.39)、预期 (2.5.75) 和完全公平排队 (CFQ) (2.6.6)。这些在 Linux 5.0 之前的内核中可用，Linux 5.0 删除了它们以仅支持较新的多队列 I/O 调度程序。请参阅第 9 章“磁盘”。

  • TCP 拥塞算法：Linux 允许配置不同的 TCP 拥塞控制算法，并支持 Reno、Cubic 以及此列表中提到的后续内核中的更多算法。另请参见第 10 章，网络。

  • 过度使用：与内存不足 (OOM) 杀手一起，这是一种用更少的主内存做更多事情的策略。请参阅第 7 章“内存”。

  • Futex (2.5.7)：快速用户空间互斥体的缩写，用于提供高性能的用户级同步原语。

  • 大页面 (2.5.36)：这为内核和内存管理单元 (MMU) 预分配的大内存页面提供支持。请参阅第 7 章“内存”。

  • OProfile (2.5.43)：一个系统分析器，用于研究内核和应用程序的 CPU 使用情况和其他事件。

  • RCU (2.5.43)：内核提供了读-副本更新同步机制，允许多个读取与更新同时发生，从而提高了大部分读取数据的性能和可扩展性。

  • epoll (2.5.46)：一个系统调用，用于跨多个打开的文件描述符有效等待 I/O，从而提高服务器应用程序的性能。

  • 模块化I/O调度（2.6.10）：Linux提供了可插拔的调度算法来调度块设备I/O。请参阅第 9 章“磁盘”。

  • DebugFS (2.6.11)：内核向用户层公开数据的简单非结构化接口，被一些性能工具使用。

  • Cpusets (2.6.12)：进程的专用 CPU 分组。

  • 自愿内核抢占 (2.6.13)：该过程提供低延迟调度，而没有完全抢占的复杂性。

  • inotify (2.6.13)：用于监视文件系统事件的框架。

  • blktrace (2.6.17)：用于跟踪块 I/O 事件的框架和工具（后来迁移到跟踪点）。

  • splice (2.6.17)：一个系统调用，用于在文件描述符和管道之间快速移动数据，而无需穿过用户空间。 （sendfile(2) 系统调用可以有效地在文件描述符之间移动数据，现在是 splice(2) 的包装器。）

  • 延迟统计 (2.6.18)：跟踪每个任务的延迟状态。请参阅第 4 章，可观测性工具。

  • IO 统计 (2.6.20)：测量每个进程的各种存储 I/O 统计数据。

  • DynTicks (2.6.21)：动态滴答允许内核定时器中断（时钟）在空闲期间不触发，从而节省 CPU 资源和电量。

  • SLUB (2.6.22)：slab 内存分配器的新简化版本。

  • CFS (2.6.23)：完全公平的调度程序。请参阅第 6 章，CPU。

  • cgroups (2.6.24)：控制组允许测量和限制进程组的资源使用情况。

  • TCP LRO (2.6.24)：TCP 大型接收卸载 (LRO) 允许网络驱动程序和硬件在将数据包发送到网络堆栈之前将其聚合为更大的大小。 Linux 还支持发送路径的大型发送卸载 (LSO)。

  • Latencytop (2.6.25)：用于观察操作系统中延迟来源的仪器和工具。

  • 跟踪点 (2.6.28)：静态内核跟踪点（又名静态探针），用于检测内核中的逻辑执行点，供跟踪工具（以前称为内核标记）使用。第 4 章“可观察性工具”中介绍了跟踪工具。

  • perf (2.6.31)：Linux 性能事件 (perf) 是一组用于性能可观察性的工具，包括 CPU 性能计数器分析以及静态和动态跟踪。有关介绍，请参阅第 6 章“CPU”。

  • 无 BKL (2.6.37)：最终消除大内核锁 (BKL) 性能瓶颈。

  • 透明大页面 (2.6.38)：这是一个允许轻松使用大内存页面的框架。请参阅第 7 章“内存”。

  • KVM：基于内核的虚拟机 (KVM) 技术是由 Qumranet 为 Linux 开发的，该公司于 2008 年被 Red Hat 收购。KVM 允许创建虚拟操作系统实例，运行它们自己的内核。请参阅第 11 章，云计算。

  • BPF JIT (3.0)：伯克利数据包过滤器 (BPF) 的即时 (JIT) 编译器，通过将 BPF 字节码编译为本机指令来提高数据包过滤性能。

  • CFS带宽控制（3.2）：一种CPU调度算法，支持CPU配额和限制。

  • TCP 防缓冲区膨胀 (3.3+)：从 Linux 3.3 开始，针对缓冲区膨胀问题进行了各种增强，包括用于传输数据包数据的字节队列限制 (BQL) (3.3)、CoDel 队列管理 (3.5)、TCP 小队列(3.6) 和增强型比例积分控制器 (PIE) 数据包调度程序 (3.14)。

  • uprobes (3.5)：用于动态跟踪用户级软件的基础设施，由其他工具（perf、SystemTap 等）使用。

  • TCP 早期重传 (3.5)：RFC 5827，用于减少触发快速重传所需的重复确认。

  • TFO（3.6、3.7、3.13）：TCP Fast Open (TFO) 可以将 TCP 三向握手减少为带有 TFO cookie 的单个 SYN 数据包，从而提高性能。在 3.13 中它被设为默认值。

  • NUMA 平衡（3.8+）：这为内核添加了自动平衡多 NUMA 系统上的内存位置的方法，减少了 CPU 互连流量并提高了性能。

  • SO_REUSEPORT (3.9)：一个套接字选项，允许多个侦听器套接字绑定到同一端口，从而提高多线程可扩展性。

  • SSD 缓存设备 (3.9)：设备映射器支持 SSD 设备用作速度较慢的旋转磁盘的缓存。

  • bcache (3.10)：一种用于块接口的 SSD 缓存技术。

  • TCP TLP (3.10)：TCP 尾部丢失探测 (TLP) 是一种通过在较短的探测超时后发送新数据或最后一个未确认的数据段来避免基于计时器的昂贵重传的方案，以触发更快的恢复。

  • NO_HZ_FULL（3.10、3.12）：也称为无定时器多任务处理或无滴答内核，这允许非空闲线程在没有时钟滴答的情况下运行，从而避免工作负载扰动[Corbet 13a]。

  • 多队列块 I/O (3.13)：这提供了每个 CPU I/O 提交队列而不是单个请求队列，从而提高了可扩展性，特别是对于高 IOPS SSD 设备 [Corbet 13b]。

  • SCHED_DEADLINE (3.14)：一种可选的调度策略，实现最早截止时间优先 (EDF) 调度 [Linux 20b]。

  • TCP 自动塞缩 (3.14)：这允许内核合并小写入，减少发送的数据包。 TCP_CORK setsockopt(2) 的自动版本。

  • MCS 锁和 qspinlocks (3.15)：高效的内核锁，使用诸如每 CPU 结构之类的技术。 MCS 以最初的锁发明者 (Mellor-Crummey 和 Scott) 的名字命名 [Mellor-Crummey 91][Corbet 14]。

  • 扩展 BPF (3.18+)：用于运行安全内核模式程序的内核执行环境。大部分扩展 BPF 是在 4.x 系列中添加的。 3.19 中添加了对附加到 kprobes 的支持，4.7 中添加了对跟踪点的支持，4.9 中添加了对软件和硬件事件的支持，4.10 中添加了对 cgroup 的支持。 5.3 中添加了有界循环，这也增加了指令限制以允许复杂的应用程序。请参见第 3.4.4 节，扩展 BPF。

  • Overlayfs (3.18)：Linux 中包含的联合挂载文件系统。它在其他文件系统之上创建虚拟文件系统，也可以在不更改第一个文件系统的情况下对其进行修改。常用于容器。

  • DCTCP (3.18)：数据中心 TCP (DCTCP) 拥塞控制算法，旨在提供高突发容忍度、低延迟和高吞吐量 [Borkmann 14a]。

  • DAX (4.0)：直接访问 (DAX) 允许用户空间直接从持久内存存储设备读取数据，而无需缓冲区开销。 ext4 可以使用 DAX。

  • 排队自旋锁 (4.2)：在争用情况下提供更好的性能，这些成为 4.2 中默认的自旋锁内核实现。

  • TCP 无锁侦听器 (4.4)：TCP 侦听器快速路径变得无锁，从而提高了性能。

  • cgroup v2（4.5、4.15）：cgroup 的统一层次结构存在于早期内核中，并且在 4.5 中被认为稳定并公开，名为 cgroup v2 [Heo 15]。 cgroup v2 CPU 控制器是在 4.15 中添加的。

  • epoll 可扩展性 (4.5)：对于多线程可扩展性，epoll(7) 避免唤醒在每个事件的相同文件描述符上等待的所有线程，这会导致雷鸣般的性能问题 [Corbet 15]。

  • KCM (4.6)：内核连接多路复用器 (KCM) 通过 TCP 提供基于消息的高效接口。

  • TCP NV (4.8)：New Vegas (NV) 是一种新的 TCP 拥塞控制算法，适用于高带宽网络（以 10+ Gbps 运行的网络）。

  • XDP（4.8、4.18）：eXpress 数据路径（XDP）是一种基于 BPF 的可编程快速路径，用于高性能网络 [Herbert 16]。 4.18 中添加了可以绕过大部分网络堆栈的 AF_XDP 套接字地址系列。

  • TCP BBR (4.9)：瓶颈带宽和 RTT (BBR) 是一种 TCP 拥塞控制算法，可在遭受数据包丢失和缓冲区膨胀的网络上提供改进的延迟和吞吐量 [Cardwell 16]。

  • 硬件延迟跟踪器 (4.9)：Ftrace 跟踪器，可以检测由硬件和固件引起的系统延迟，包括系统管理中断 (SMI)。

  • perf c2c (4.10)：缓存到缓存 (c2c) perf 子命令可以帮助识别 CPU 缓存性能问题，包括错误共享。

  • Intel CAT (4.10)：支持 Intel 高速缓存分配技术 (CAT)，允许任务拥有专用的 CPU 高速缓存空间。容器可以使用它来帮助解决嘈杂的邻居问题。

  • 多队列 I/O 调度程序：BPQ、Kyber (4.12)：预算公平队列 (BFQ) 多队列 I/O 调度程序为交互式应用程序（尤其是速度较慢的存储设备）提供低延迟 I/O。 BFQ在5.2中得到显着改善。 Kyber I/O 调度程序适用于快速多队列设备 [Corbet 17]。

  • 内核 TLS（4.13、4.17）：Linux 版本的内核 TLS [Edge 15]。

  • MSG_ZEROCOPY (4.14)：一个 send(2) 标志，用于避免应用程序和网络接口之间的数据包字节的额外副本 [Linux 20c]。

  • PCID (4.14)：Linux 添加了对进程上下文 ID (PCID) 的支持，这是一种处理器 MMU 功能，有助于避免上下文切换时的 TLB 刷新。这降低了缓解崩溃漏洞所需的内核页表隔离（KPTI）补丁的性能成本。请参见第 3.4.3 节，KPTI（崩溃）。

  • PSI（4.20、5.2）：压力停顿信息 (PSI) 是一组新指标，用于显示 CPU、内存或 I/O 上停顿所花费的时间。 5.2 中添加了 PSI 阈值通知以支持 PSI 监控。

  • TCP EDT (4.20)：TCP 堆栈切换到早期出发时间 (EDT)：这使用定时轮调度程序来发送数据包，提供更好的 CPU 效率和更小的队列 [Jacobson 18]。

  • 多队列 I/O (5.0)：多队列块 I/O 调度程序在 5.0 中成为默认设置，并删除了经典调度程序。

  • UDP GRO (5.0)：UDP 通用接收卸载 (GRO) 通过允许驱动程序和卡聚合数据包并向上传递堆栈来提高性能。

  • io_uring (5.1)：一个通用异步接口，用于应用程序和内核之间的快速通信，利用共享环形缓冲区。主要用途包括快速磁盘和网络 I/O。

  • MADV_COLD、MADV_PAGEOUT (5.4)：这些 madvise(2) 标志向内核暗示需要内存，但不会很快。 MADV_PAGEOUT也是一个可以立即回收内存的提示。这些对于内存受限的嵌入式 Linux 设备特别有用。

  • 多路径 TCP (5.6)：可以使用多个网络链路（例如 3G 和 WiFi）来提高单个 TCP 连接的性能和可靠性。

  • 启动时间跟踪 (5.6)：允许 Ftrace 跟踪早期启动过程。 （systemd 可以提供有关后期启动过程的计时信息：请参阅第 3.4.2 节，systemd。）

  • 热压力 (5.7)：调度程序考虑热节流以做出更好的布局决策。

  • perf 火焰图 (5.8)：perf(1) 支持火焰图可视化。

这里没有列出许多小的性能改进，包括锁定、驱动程序、VFS、文件系统、异步 I/O、内存分配器、NUMA、新处理器指令支持、GPU 以及性能工具 perf(1) 和 Ftrace。采用 systemd 也改善了系统启动时间。

以下各节更详细地描述了对性能很重要的三个 Linux 主题：systemd、KPTI 和扩展 BPF。

3.4.2 系统

3.4.3 KPTI（崩溃）

2018 年 Linux 4.14 中添加的内核页表隔离 (KPTI) 补丁是针对名为“meltdown”的英特尔处理器漏洞的缓解措施。较旧的 Linux 内核版本具有用于类似目的的 KAISER 补丁，其他内核也采用了缓解措施。虽然这些解决了安全问题，但由于额外的 CPU 周期以及上下文切换和系统调用上的额外 TLB 刷新，它们也会降低处理器性能。 Linux 在同一版本中添加了进程上下文 ID (PCID) 支持，只要处理器支持 PCID，就可以避免一些 TLB 刷新。

我评估了 KPTI 对 Netflix 云生产工作负载的性能影响在 0.1% 到 6% 之间，具体取决于工作负载的系统调用率（越高成本越高）[Gregg 18a]。额外的调整将进一步降低成本：使用大页面以便刷新的 TLB 更快地预热，并使用跟踪工具检查系统调用以确定降低其速率的方法。许多此类跟踪工具是使用扩展 BPF 实现的。

3.4.4 扩展BPF

BPF 代表伯克利数据包过滤器，这是一项于 1992 年首次开发的不起眼的技术，它提高了数据包捕获工具的性能 [McCanne 92]。 2013年，Alexei Starovoitov提出了BPF的重大重写[Starovoitov 13]，由他和Daniel Borkmann进一步开发，并于2014年包含在Linux内核中[Borkmann 14b]。这将 BPF 变成了一个通用执行引擎，可用于多种用途，包括网络、可观察性和安全性。

BPF 本身是一种灵活高效的技术，由指令集、存储对象（映射）和辅助函数组成。由于其虚拟指令集规范，它可以被视为虚拟机。 BPF 程序在内核模式下运行（如图 3.2 所示），并配置为在事件上运行：套接字事件、跟踪点、USDT 探针、kprobes、uprobes 和 perf_events。如图 3.16 所示。

图 3.16 BPF 组件

BPF 字节码必须首先通过检查安全性的验证器，确保 BPF 程序不会崩溃或损坏内核。它还可以使用 BPF 类型格式 (BTF) 系统来理解数据类型和结构。 BPF 程序可以通过 perf 环形缓冲区（一种发出每个事件数据的有效方式）或通过适合统计的映射来输出数据。

由于 BPF 为新一代高效、安全和先进的跟踪工具提供动力，因此它对于系统性能分析非常重要。它为现有内核事件源提供了可编程性：tracepoints、kprobes、uprobes 和 perf_events。例如，BPF 程序可以在 I/O 的开始和结束处记录时间戳以计算其持续时间，并将其记录在自定义直方图中。本书包含许多使用 BCC 和 bpftrace 前端的基于 BPF 的程序。这些前端将在第 15 章中介绍。

3.5 其他主题

其他一些值得总结的内核和操作系统主题包括 PGO 内核、Unikernels、微内核、混合内核和分布式操作系统。

3.5.1 PGO 内核

配置文件引导优化 (PGO)，也称为反馈导向优化 (FDO)，使用 CPU 配置文件信息来改进编译器决策 [Yuan 14a]。这可以应用于内核构建，其过程是：

  1. 在生产过程中，获取 CPU 配置文件。

  2. 根据该 CPU 配置文件重新编译内核。

  3. 在生产中部署新内核。

这将创建一个针对特定工作负载具有改进性能的内核。 JVM 等运行时会自动执行此操作，根据运行时性能重新编译 Java 方法，并结合即时 (JIT) 编译。创建 PGO 内核的过程涉及手动步骤。

相关的编译优化是链接时优化 (LTO)，其中一次编译整个二进制文件以允许对整个程序进行优化。 Microsoft Windows 内核大量使用了 LTO 和 PGO，与 PGO 相比，性能提升了 5% 到 20% [Bearman 20]。 Google 还使用 LTO 和 PGO 内核来提高性能 [Tolvanen 20]。

gcc 和 clang 编译器以及 Linux 内核都支持 PGO。内核 PGO 通常涉及运行专门检测的内核来收集配置文件数据。 Google 发布了一个 AutoFDO 工具，可以绕过对此类特殊内核的需求：AutoFDO 允许使用 perf(1) 从普通内核收集配置文件，然后将其转换为编译器使用的正确格式 [Google 20a]。

关于使用 PGO 或 AutoFDO 构建 Linux 内核的唯一最新文档是 Microsoft [Bearman 20] 和 Google [Tolvanen 20] 在 2020 年 Linux Plumber’s Conference 上的两场演讲。15

15有一段时间，最新的文档来自 2014 年的 Linux 3.13 [Yuan 14b]，这阻碍了新内核的采用。

3.5.2 单内核

unikernel 是一个单一应用程序机器映像，它将内核、库和应用程序软件组合在一起，通常可以在硬件虚拟机或裸机上的单个地址空间中运行。这可能具有性能和安全优势：更少的指令文本意味着更高的 CPU 缓存命中率和更少的安全漏洞。这也产生了一个问题：可能没有 SSH、shell 或性能工具可供您登录和调试系统，也没有任何方法添加它们。

为了在生产中对 Unikernel 进行性能调整，必须构建新的性能工具和指标来支持它们。作为概念证明，我构建了一个基本的 CPU 分析器，它从 Xen dom0 运行来分析 domU unikernel guest 虚拟机，然后构建了一个 CPU 火焰图，只是为了表明这是可能的 [Gregg 16a]。

Unikernel 的示例包括 MirageOS [MirageOS 20]。

3.5.3 微内核和混合内核

本章的大部分内容讨论类 Unix 内核，也称为单片内核，其中管理设备的所有代码作为单个大型内核程序一起运行。对于微内核模型，内核软件保持在最低限度。微内核支持内存管理、线程管理和进程间通信 (IPC) 等基本功能。文件系统、网络堆栈和驱动程序作为用户模式软件实现，这使得这些用户模式组件可以更轻松地修改和替换。想象一下，不仅要选择要安装的数据库或 Web 服务器，还要选择要安装的网络堆栈。微内核还具有更高的容错能力：驱动程序崩溃不会导致整个内核崩溃。微内核的示例包括 QNX 和 Minix 3。

微内核的缺点是需要额外的 IPC 步骤来执行 I/O 和其他功能，从而降低性能。一种解决方案是混合内核，它结合了微内核和整体内核的优点。混合内核将性能关键型服务移回到内核空间（使用直接函数调用而不是 IPC），就像使用单片内核一样，但保留了微内核的模块化设计和容错能力。混合内核的示例包括 Windows NT 内核和 Plan 9 内核。

3.5.4 分布式操作系统

分布式操作系统跨一组联网的独立计算机节点运行单个操作系统实例。每个节点上通常使用微内核。分布式操作系统的示例包括贝尔实验室的 Plan 9 和 Inferno 操作系统。

虽然是一种创新设计，但该模型尚未得到广泛应用。 Plan 9 和 Inferno 的联合创始人 Rob Pike 描述了造成这种情况的各种原因，包括 [Pike 00]：

“20 世纪 70 年代末和 80 年代初有人声称 Unix 扼杀了操作系统研究，因为没有人愿意尝试其他任何东西。当时我不相信。今天，我勉强接受这一说法可能是真的（尽管是微软）。”

在云中，当今扩展计算节点的常见模型是在一组相同的操作系统实例之间进行负载平衡，这些实例可以根据负载进行扩展（请参阅第 11 章“云计算”，第 11.1.3 节“容量规划”）。

3.6 内核比较

哪个内核最快？这部分取决于操作系统配置和工作负载以及内核的参与程度。总的来说，我预计 Linux 将优于其他内核，因为它在性能改进、应用程序和驱动程序支持方面所做的大量工作、广泛的使用以及发现和报告性能问题的大型社区。自 1993 年以来 TOP500 列表追踪的前 500 名超级计算机在 2017 年已实现 100% Linux [TOP500 17]。会有一些例外；例如，Netflix 使用云端 Linux 和 FreeBSD 作为其 CDN。16

16FreeBSD 为 Netflix CDN 工作负载提供了更高的性能，特别是由于 Netflix OCA 团队对内核进行了改进。这是定期测试的，最近一次是在 2019 年，我帮助分析了 Linux 5.0 与 FreeBSD 的生产比较。

通常使用微基准来比较内核性能，这很容易出错。此类基准测试可能会发现一个内核在特定系统调用上要快得多，但该系统调用并未在生产工作负载中使用。 （或者使用了它，但某些标志未经过微基准测试，这极大地影响了性能。）准确比较内核性能是高级性能工程师的一项任务，这项任务可能需要数周时间。请参阅第 12 章“基准测试”、第 12.3.2 节“主动基准测试”作为要遵循的方法。

在本书的第一版中，我在结束本节时指出，Linux 没有成熟的动态跟踪器，如果没有它，您可能会错失巨大的性能优势。自第一版发布以来，我转而担任全职 Linux 性能角色，并帮助开发了 Linux 所缺少的动态跟踪器：基于扩展 BPF 的 BCC 和 bpftrace。这些内容在第 15 章和我之前的书 [Gregg 19] 中进行了介绍。

第 3.4.1 节“Linux 内核开发”列出了第一版和本版本之间发生的许多其他 Linux 性能开发，涵盖内核版本 3.1 和 5.8。前面未列出的一项重大发展是 OpenZFS 现在支持 Linux 作为其主要内核，在 Linux 上提供高性能且成熟的文件系统选项。

然而，随着 Linux 的发展，复杂性也随之而来。 Linux 上有如此多的性能功能和可调参数，因此为每个工作负载配置和调整它们变得很费力。我见过许多未调整的部署。在比较内核性能时请记住这一点：每个内核是否都经过调整？本书后面的章节及其调整部分可以帮助您解决这个问题。
