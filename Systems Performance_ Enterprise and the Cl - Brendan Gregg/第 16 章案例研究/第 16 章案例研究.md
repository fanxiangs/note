## 第16章

案例研究

本章是一个系统性能案例研究：现实世界性能问题的故事，从最初的报告到最终的解决方案。这个特殊问题发生在生产云计算环境中；我选择它作为系统性能分析的常规示例。

我在本章中的目的不是介绍新的技术内容，而是通过讲故事来展示如何在实际工作环境中应用工具和方法。这对于尚未处理现实世界系统性能问题的初学者特别有用，提供了专家如何处理这些问题的全面视图，对专家在分析过程中可能想到的内容以及原因的评论。这不一定是记录可能的最佳方法，而是记录为什么采用一种方法。

### 16.1 无法解释的胜利

Netflix 的微服务在基于容器的新平台上进行了测试，结果发现可以将请求延迟减少三到四倍。虽然容器平台有很多好处，但如此大的收益却出乎意料！这听起来好得令人难以置信，我被要求调查并解释这是如何发生的。

为了进行分析，我使用了各种工具，包括基于计数器、静态配置、PMC、软件事件和跟踪的工具。所有这些工具类型都发挥了作用并提供了相互配合的线索。由于这对系统性能分析进行了广泛的介绍，因此我将其用作 USENIX LISA 2019 系统性能演讲的开场故事 \[Gregg 19h\]，并将其作为案例研究纳入此处。

#### 16.1.1 问题陈述

通过与服务团队的交谈，我了解了该微服务的详细信息：它是一个用于计算客户推荐的 Java 应用程序，目前正在 AWS EC2 云中的虚拟机实例上运行。该微服务由两个组件组成，其中一个组件正在名为 Titus 的新 Netflix 容器平台上进行测试，该平台也在 AWS EC2 上运行。该组件在虚拟机实例上的请求延迟为三到四秒，在容器上变为一秒：速度快了三到四倍！

问题是解释这种性能差异。如果仅仅是由于容器移动，微服务可以通过移动获得永久的 3-4 倍的胜利。如果这是由于其他一些因素造成的，那么就值得了解它是什么以及它是否会是永久性的。也许它也可以更大程度地应用于其他地方。

我立即想到的是独立运行工作负载的一个组件的好处：它将能够使用整个 CPU 缓存，而不会与其他组件争用，从而提高缓存命中率，从而提高性能。另一个猜测是容器平台上的爆发，容器可以使用其他容器的空闲 CPU 资源。

#### 16.1.2 分析策略

由于流量由负载均衡器 (AWS ELB) 处理，因此可以在虚拟机和容器之间拆分流量，以便我可以同时登录两者。这是比较分析的理想情况：我可以在一天中的同一时间（相同的流量组合和负载）运行相同的分析命令并立即比较输出。

在这种情况下，我可以访问容器主机，而不仅仅是容器，这允许我使用任何分析工具，并为这些工具提供进行任何系统调用的权限。如果我只有容器访问权限，由于可观察性来源和内核权限有限，分析会更加耗时，需要从有限的指标中进行更多的推断，而不是直接测量。目前，仅从容器中分析某些性能问题是不切实际的（请参阅第 11 章，云计算）。

对于方法论，我计划从 60 秒清单（第 1 章，简介，第 1.10.1 节，60 年代的 Linux 性能分析）和 USE 方法（第 2 章，方法论，第 2.5.9 节，USE 方法）开始，并根据他们的线索进行深入分析（第 2.5.12 节，深入分析）和其他方法。

我在以下部分中包含了我运行的命令及其输出，对虚拟机实例使用“serverA#”提示符，对容器主机使用“serverB#”提示符。

mpstat(1) 将 CPU 数量打印为第一行。输出显示虚拟机有 48 个 CPU，容器主机有 64 个。这帮助我进一步解释负载平均值：如果它们是基于 CPU 的，则表明虚拟机实例运行良好，达到了 CPU 饱和状态，因为平均负载大约是 CPU 数量的两倍，而容器主机利用率不足。 mpstat(1) 指标支持这一假设：虚拟机上的空闲时间约为 2%，而容器主机上的空闲时间约为 78%。

通过检查其他 mpstat(1) 统计数据，我发现了其他线索：

*   CPU 利用率 (`%usr` + `%sys` + ...) 显示虚拟机的利用率为 98%，而容器的利用率为 22%。这些处理器的每个 CPU 核心有两个超线程，因此超过 50% 的利用率通常意味着超线程核心争用，从而降低性能。虚拟机已经很好地进入了这一领域，而容器主机可能仍然受益于每个核心只有一个繁忙的超线程。
    
*   VM 上的系统时间 (`%sys`) 要长得多：大约 8% 对 0.38%。如果虚拟机在 CPU 饱和状态下运行，则额外的 `%sys` 时间可能包括内核上下文切换代码路径。内核跟踪或分析可以确认。
    

我继续执行 60 秒清单上的其他命令。 vmstat(8) 显示的运行队列长度与平均负载相似，确认平均负载是基于 CPU 的。 iostat(1) 显示很少的磁盘 I/O，sar(1) 显示很少的网络 I/O。 （这些输出未包含在此处。）这证实了虚拟机在 CPU 饱和状态下运行，导致可运行线程等待轮到它们，而容器主机则不然。容器主机上的 top(1) 还显示只有一个容器正在运行。

这些命令提供了 USE 方法的统计信息，这也确定了 CPU 负载问题。

我解决了这个问题吗？我发现虚拟机在 48 个 CPU 的系统上的平均负载为 85，并且该平均负载是基于 CPU 的。这意味着线程大约有 77% 的时间在等待 (85/48 – 1)，消除等待时间将产生大约 4 倍的加速 (1 / (1 – 0.77))。虽然这个幅度与问题相对应，但我还无法解释为什么平均负载较高：需要进行更多分析。

结果显示，虚拟机的每周期指令数 (IPC) 约为 0.57，而容器的每周期指令数 (IPC) 约为 1.52：相差 2.6 倍。

IPC 较低的原因之一可能是超线程争用，因为 VM 主机的 CPU 利用率超过 50%。最后一列显示了另一个原因：虚拟机的末级缓存 (LLC) 命中率仅为 30%，而容器的命中率约为 90%。这会导致虚拟机上的指令在访问主内存时频繁停滞，从而降低 IPC 和指令吞吐量（性能）。

虚拟机上较低的 LLC 命中率可能至少归因于三个因素：

*   LLC 大小较小（33 MB 与 45 MB）。
    
*   运行完整的工作负载而不是子组件（如问题陈述中所述）；子组件可能会更好地缓存：更少的指令和数据。
    
*   CPU 饱和会导致更多的上下文切换以及代码路径（包括用户和内核）之间的跳转，从而增加缓存压力。
    

最后一个因素可以使用跟踪工具进行调查。

现在，应用程序通常在 CPU 上花费 2 到 4 毫秒。其他工具表明，它并没有因为非自愿的上下文切换而受到太大干扰。

虚拟机上的非自愿上下文切换以及随后出现的高频率上下文切换导致了性能问题。导致应用程序在通常不到 10 微秒后离开 CPU 也不会给 CPU 缓存太多时间来预热当前的代码路径。

#### 16.1.8 结论

我得出的结论是，性能提升的原因是：

*   无容器邻居：容器主机除了一个容器外处于空闲状态。这使得容器可以拥有全部 CPU 缓存，并且可以在没有 CPU 争用的情况下运行。虽然这在测试期间产生了对容器有利的结果，但这并不是长期生产使用的预期情况，相邻容器将成为常态。当其他租户搬入时，微服务可能会发现 3-4 倍的性能优势消失了。
    
*   LLC 大小和工作负载差异：VM 上的 IPC 降低了 2.6 倍，这可以解释 2.6 倍的速度下降。原因之一可能是超线程争用，因为虚拟机主机的利用率超过 50%（并且每个核心有两个超线程）。然而，主要原因可能是 LLC 命中率较低：虚拟机上为 30%，而容器上为 90%。 LLC 命中率低可能有以下三个原因：
    
    *   VM 上的 LLC 大小较小：33 MB 与 45 MB。
        
    *   VM 上的工作负载更加复杂：完整的应用程序需要更多的指令文本和数据，而不是在容器上运行的组件。
        
    *   VM 上的上下文切换率很高：每秒大约 200 万次。这些可以防止线程在 CPU 上长时间运行，从而干扰缓存预热。 VM 上的 CPU 持续时间通常小于 10 微秒，而容器主机上的持续时间为 2-4 毫秒。
        
*   CPU 负载差异：更高的负载被定向到 VM，导致 CPU 饱和：在 48-CPU 系统上基于 CPU 的负载平均值为 85。这导致每秒大约 200 万次上下文切换，并在线程等待轮到时产生运行队列延迟。负载平均值暗示的运行队列延迟表明虚拟机的运行速度大约慢了 4 倍。
    

这些问题解释了观察到的性能差异。
