## 6.1 术语

作为参考，本章中使用的 CPU 相关术语包括以下内容：

*   处理器：插入系统或处理器板上插槽的物理芯片，包含一个或多个作为内核或硬件线程实现的 CPU。
    
*   核心：多核处理器上的独立CPU实例。使用内核是一种扩展处理器的方法，称为芯片级多处理（CMP）。
    
*   硬件线程：支持在单核上并行执行多个线程的CPU架构（包括Intel的超线程技术），其中每个线程都是一个独立的CPU实例。这种扩展方法称为同时多线程 (SMT)。
    
*   CPU指令：单个CPU操作，来自其指令集。有算术运算、存储器 I/O 和控制逻辑的指令。
    
*   逻辑CPU：也称为虚拟处理器，[^1]操作系统CPU实例（可调度的CPU实体）。这可以由处理器作为硬件线程（在这种情况下也可以称为虚拟核心）、核心或单核处理器来实现。
    
    [^1]有时也称为虚拟 CPU；但是，该术语更常用于指虚拟化技术提供的虚拟 CPU 实例。请参阅第 11 章，云计算。
    
*   调度程序：分配线程在 CPU 上运行的内核子系统。
    
*   运行队列：等待 CPU 服务的可运行线程的队列。现代内核可能使用一些其他数据结构（例如红黑树）来存储可运行线程，但我们仍然经常使用术语运行队列。
    

本章将介绍其他术语。术语表包含供参考的基本术语，包括 CPU、CPU 周期和堆栈。另请参阅第 2 章和第 3 章中的术语部分。

## 6.3 概念

以下是有关 CPU 性能的一些重要概念，首先是处理器内部结构的概述：CPU 时钟速率以及指令的执行方式。这是后续性能分析的背景，特别是了解每周期指令 (IPC) 指标。

### 6.3.1 时钟频率

时钟是驱动所有处理器逻辑的数字信号。每条 CPU 指令可能需要一个或多个时钟周期（称为 CPU 周期）来执行。 CPU 以特定的时钟频率执行；例如，4 GHz CPU 每秒执行 40 亿个时钟周期。

某些处理器能够改变时钟速率，提高时钟速率以提高性能，或降低时钟速率以降低功耗。该速率可以根据操作系统的请求而改变，或者由处理器本身动态地改变。例如，内核空闲线程可以请求 CPU 降低速度以节省电量。

时钟频率通常被宣传为处理器的主要特性，但这可能有点误导。即使系统中的 CPU 看起来已得到充分利用（瓶颈），更快的时钟速率也可能不会提高性能 — 这取决于那些快速的 CPU 周期实际上在做什么。如果它们在等待内存访问时大部分都是停顿周期，那么更快地执行它们实际上并不会增加 CPU 指令速率或工作负载吞吐量。

### 6.3.2 使用说明

CPU 执行从其指令集中选择的指令。一条指令包括以下步骤，每个步骤都由称为功能单元的 CPU 组件处理：

1.  取指令
    
2.  指令译码
    
3.  执行
    
4.  内存访问
    
5.  寄存器回写
    

最后两个步骤是可选的，具体取决于说明。许多指令仅对寄存器进行操作，不需要内存访问步骤。

其中每个步骤至少需要一个时钟周期来执行。内存访问通常是最慢的，因为可能需要数十个时钟周期来读取或写入主内存，在此期间指令执行已停止（这些停止的周期称为停止周期）。这就是为什么 CPU 缓存很重要，如第 6.4.1 节“硬件”中所述：它可以显着减少内存访问所需的周期数。

### 6.3.3 指令流水线

指令流水线是一种可以并行执行多条指令的CPU架构，通过同时执行不同指令的不同部分来实现。它类似于工厂装配线，生产阶段可以并行执行，从而提高吞吐量。

请考虑前面列出的说明步骤。如果每个指令需要一个时钟周期，则需要五个周期才能完成该指令。在该指令的每一步，只有一个功能单元处于活动状态，四个处于空闲状态。通过使用流水线，多个功能单元可以同时活动，处理流水线中的不同指令。理想情况下，处理器可以在每个时钟周期完成一条指令。

指令流水线可能涉及将指令分解为多个简单步骤以并行执行。 （根据处理器的不同，这些步骤可能会变成称为微操作（uOps）的简单操作，由称为后端的处理器区域执行。此类处理器的前端负责获取指令和分支预测。）

#### 分支预测

现代处理器可以执行流水线的无序执行，其中较晚的指令可以完成，而较早的指令则停止，从而提高指令吞吐量。然而，条件分支指令会带来问题。分支指令将执行跳转到不同的指令，条件分支基于测试来执行此操作。对于条件分支，处理器不知道后面的指令是什么。作为一种优化，处理器通常会实现分支预测，其中它们将猜测测试的结果并开始处理结果指令。如果后来证明猜测是错误的，则必须丢弃指令管道中的进度，从而损害性能。为了提高正确猜测的机会，程序员可以在代码中放置提示（例如，Linux 内核源代码中的likely()和unlikely()宏）。

### 6.3.4 指令宽度

但我们还可以走得更快。可以包括相同类型的多个功能单元，以便每个时钟周期可以向前推进更多指令。这种 CPU 架构称为超标量，通常与流水线一起使用以实现高指令吞吐量。

指令宽度描述了要并行处理的指令的目标数量。现代处理器是 3 宽或 4 宽，这意味着它们每个周期最多可以完成三个或四个指令。它的工作原理取决于处理器，因为每个阶段可能有不同数量的功能单元。

### 6.3.5 指令大小

另一个指令特征是指令大小：对于某些处理器架构来说，它是可变的：例如，被归类为复杂指令集计算机（CISC）的 x86 最多允许 15 字节指令。 ARM 是精简指令集计算机 (RISC)，对于 AArch32/A32 具有 4 字节指令（4 字节对齐），对于 ARM Thumb 则具有 2 或 4 字节指令。

### 6.3.6 贴片

同步多线程利用超标量架构和硬件多线程支持（由处理器）来提高并行性。它允许 CPU 内核运行多个线程，在指令期间有效地在它们之间进行调度，例如，当一条指令在内存 I/O 上停滞时。内核将这些硬件线程呈现为虚拟 CPU，并像往常一样在它们上调度线程和进程。第 6.2.1 节“CPU 架构”对此进行了介绍和描述。

一个示例实现是英特尔的超线程技术，其中每个内核通常有两个硬件线程。另一个例子是 POWER8，它的每个核心有 8 个硬件线程。

每个硬件线程的性能与单独的 CPU 核心不同，并且取决于工作负载。为了避免性能问题，内核可以将 CPU 负载分散到多个核心上，以便每个核心上只有一个硬件线程忙碌，从而避免硬件线程争用。停顿周期较多（低 IPC）的工作负载也可能比指令较多（高 IPC）的工作负载具有更好的性能，因为停顿周期减少了核心争用。

### 6.3.7 IPC、CPI

每周期指令数 (IPC) 是一个重要的高级指标，用于描述 CPU 如何使用其时钟周期并了解 CPU 利用率的性质。该指标也可以表示为每条指令的周期数 (CPI)，即 IPC 的倒数。 IPC 更常被 Linux 社区和 Linux perf(1) 分析器使用，而 CPI 更常被 Intel 和其他地方使用。[^3]

[^3]在本书的第一版中我使用了 CPI；从那时起，我开始更多地在 Linux 上工作，包括切换到 IPC。

IPC 较低表示 CPU 经常停滞，通常是在内存访问时。高 IPC 表明 CPU 通常不会停滞并且具有高指令吞吐量。这些指标表明性能调优工作最好花在哪里。

例如，可以通过安装更快的内存 (DRAM)、改进内存局部性（软件配置）或减少内存 I/O 量来改善内存密集型工作负载。安装时钟频率较高的 CPU 可能无法将性能提高到预期的程度，因为 CPU 可能需要等待相同的时间才能完成内存 I/O。换句话说，更快的 CPU 可能意味着更多的停顿周期，但每秒完成指令的速率相同。

高或低 IPC 的实际值取决于处理器和处理器功能，并且可以通过运行已知工作负载来通过实验确定。例如，您可能会发现低 IPC 工作负载以 0.2 或更低的 IPC 运行，而高 IPC 工作负载以超过 1.0 的 IPC 运行（这可能是由于指令流水线和宽度所致，如前所述）。在 Netflix，云工作负载的 IPC 范围从 0.2（被认为是慢）到 1.5（被认为是好）。以 CPI 表示，该范围为 5.0 至 0.66。

需要注意的是，IPC 显示的是指令处理的效率，而不是指令本身的效率。考虑一个软件更改，它添加了一个低效的软件循环，该循环主要在 CPU 寄存器上运行（无停顿周期）：这样的更改可能会导致更高的总体 IPC，但也会导致更高的 CPU 使用率和利用率。

### 6.3.8 使用

CPU 利用率通过 CPU 实例在某个时间间隔内忙于执行工作的时间来衡量，以百分比表示。它可以通过 CPU 不运行内核空闲线程而是运行用户级应用程序线程或其他内核线程或处理中断的时间来测量。

CPU 利用率高不一定是问题，而是系统正在工作的标志。有些人还认为这是一个投资回报（ROI）指标：利用率高的系统被认为具有良好的投资回报率，而闲置的系统则被认为是浪费。与其他资源类型（磁盘）不同，在高利用率下性能不会急剧下降，因为内核支持优先级、抢占和分时。这些共同使内核能够了解什么具有更高的优先级，并确保它首先运行。

CPU 利用率的测量涵盖符合条件的活动的所有时钟周期，包括内存停顿周期。这可能会产生误导：CPU 可能会被高度利用，因为它经常停止等待内存 I/O，而不仅仅是执行指令，如上一节所述。 Netflix 云就是这种情况，其中 CPU 利用率主要是内存停顿周期 \[Gregg 17b\]。

CPU 利用率通常分为单独的内核时间和用户时间指标。

### 6.3.9 用户时间/内核时间

CPU执行用户级软件所花费的时间称为用户时间，而内核级软件则称为内核时间。内核时间包括系统调用、内核线程和中断期间的时间。当对整个系统进行测量时，用户时间/内核时间比率表明所执行的工作负载类型。

计算密集型应用程序可能几乎将所有时间都花在执行用户级代码上，并且用户/内核比率接近 99/1。示例包括图像处理、机器学习、基因组学和数据分析。

I/O 密集型应用程序的系统调用率很高，系统调用执行内核代码来执行 I/O。例如，执行网络 I/O 的 Web 服务器可能具有大约 70/30 的用户/内核比率。

这些数字取决于许多因素，并用于表达预期的比率类型。

### 6.3.10 饱和度

利用率为 100% 的 CPU 已饱和，线程在等待在 CPU 上运行时会遇到调度程序延迟，从而降低整体性能。此延迟是等待 CPU 运行队列或用于管理线程的其他结构所花费的时间。

CPU 饱和的另一种形式涉及 CPU 资源控制，这可能在多租户云计算环境中实施。虽然 CPU 可能没有 100% 被利用，但已经达到了强加的限制，并且可运行的线程必须等待轮到它们。这对系统用户的可见程度取决于所使用的虚拟化类型；请参阅第 11 章“云计算”。

与其他资源类型相比，以饱和状态运行的 CPU 问题较小，因为较高优先级的工作可以抢占当前线程。

### 6.3.11 抢占

第 3 章操作系统中介绍的抢占允许较高优先级的线程抢占当前正在运行的线程并开始自己执行。这消除了高优先级工作的运行队列延迟，从而提高了其性能。

### 6.3.12 优先级反转

当较低优先级线程持有资源并阻止较高优先级线程运行时，就会发生优先级反转。这会降低高优先级工作的性能，因为它会被阻塞等待。

这可以使用优先级继承方案来解决。以下是其工作原理的示例（基于真实案例）：

1.  线程A执行监控，优先级较低。它获取生产数据库的地址空间锁，以检查内存使用情况。
    
2.  线程 B（执行系统日志压缩的例行任务）开始运行。
    
3.  没有足够的 CPU 来运行两者。线程 B 抢占 A 并运行。
    
4.  线程C来自生产数据库，优先级高，一直休眠等待I/O。此 I/O 现在已完成，将线程 C 重新置于可运行状态。
    
5.  线程 C 抢占 B，运行，但随后阻塞线程 A 持有的地址空间锁。线程 C 离开 CPU。
    
6.  调度程序选择下一个最高优先级的线程来运行：B.
    
7.  当线程 B 运行时，高优先级线程 C 实际上被阻塞在较低优先级线程 B 上。这就是优先级反转。
    
8.  优先级继承赋予线程A线程C高优先级，抢占B，直到它释放锁。线程 C 现在可以运行。
    

Linux 自 2.6.18 起提供了支持优先级继承的用户级互斥体，旨在用于实时工作负载 \[Corbet 06a\]。

### 6.3.13 多进程、多线程

大多数处理器提供某种形式的多个 CPU。对于要使用它们的应用程序，它需要单独的执行线程，以便它可以并行运行。例如，对于 64-CPU 系统，这可能意味着如果应用程序可以并行使用所有 CPU，或者处理 64 倍的负载，则应用程序的执行速度可以提高 64 倍。应用程序随着 CPU 数量的增加而有效扩展的程度是可扩展性的衡量标准。

### 6.3.14 字长

处理器是围绕最大字大小（32 位或 64 位）设计的，即整数大小和寄存器大小。根据处理器的不同，字大小也常用来表示地址空间大小和数据路径宽度（有时称为位宽）。

更大的尺寸意味着更好的性能，尽管它并不像听起来那么简单。较大的大小可能会导致某些数据类型中未使用的位产生内存开销。当指针大小（字大小）增加时，数据占用空间也会增加，这可能需要更多内存 I/O。对于 x86 64 位架构，这些开销可以通过增加寄存器和更高效的寄存器调用约定来补偿，因此 64 位应用程序可能会比其 32 位版本更快。

处理器和操作系统可以支持多种字长，并且可以同时运行针对不同字长编译的应用程序。如果软件是针对较小的字长进行编译的，它可能会成功执行，但性能相对较差。

### 6.3.15 编译器优化

通过编译器选项（包括设置字大小）和优化，可以显着提高应用程序的CPU运行时间。编译器也经常更新，以利用最新的 CPU 指令集并实现其他优化。有时，只需使用较新的编译器即可显着提高应用程序性能。

#### P 状态和 C 状态

英特尔处理器使用的高级配置和电源接口 (ACPI) 标准定义了处理器性能状态（P 状态）和处理器电源状态（C 状态）\[ACPI 17\]。

P 状态通过改变 CPU 频率在正常执行期间提供不同级别的性能：P0 是最高频率（对于某些 Intel CPU，这是最高的“涡轮增压”级别），P1...N 是较低频率状态。这些状态可以由硬件（例如，基于处理器温度）或通过软件（例如，内核节能模式）控制。可以使用特定于模型的寄存器 (MSR) 来观察当前的工作频率和可用状态（例如，使用第 6.6.10 节 showboost 中的 showboost(8) 工具）。

##### 延迟

多级缓存用于提供大小和延迟的最佳配置。 1 级高速缓存的访问时间通常为几个 CPU 时钟周期，而较大的 2 级高速缓存的访问时间约为 12 个时钟周期。主内存访问可能需要大约 60 ns（4 GHz 处理器大约 240 个周期），并且 MMU 的地址转换也会增加延迟。

##### 关联性

关联性是一种高速缓存特性，描述在高速缓存中定位新条目的约束。类型有：

*   完全关联：缓存可以在任何地方找到新条目。例如，最近最少使用（LRU）算法可用于跨整个高速缓存的逐出。
    
*   直接映射：每个条目在缓存中只有一个有效位置，例如内存地址的哈希，使用地址位的子集形成缓存中的地址。
    
*   设置关联：缓存的子集通过映射（例如散列）来标识，在该子集内可以执行另一种算法（例如 LRU）。它是用子集大小来描述的；例如，四路组关联将地址映射到四个可能的位置，然后从这四个位置中选择最好的位置（例如，最近最少使用的位置）。
    

CPU 缓存通常使用设置关联性作为完全关联（执行成本昂贵）和直接映射（命中率较低）之间的平衡。

##### 缓存线

CPU 高速缓存的另一个特征是其高速缓存线大小。这是作为一个单元存储和传输的字节范围，可提高内存吞吐量。 x86 处理器的典型缓存线大小为 64 字节。编译器在优化性能时会考虑到这一点。程序员有时也会这样做；请参阅第 5 章“应用程序”第 5.2.5 节“并发性和并行性”中的哈希表。

##### 缓存一致性

内存可能同时缓存在不同处理器上的多个CPU缓存中。当一个CPU修改内存时，所有缓存都需要知道它们的缓存副本现在已经过时并且应该被丢弃，以便任何未来的读取都将检索新修改的副本。这个过程称为高速缓存一致性，可确保 CPU 始终访问正确的内存状态。

缓存一致性的影响之一是 LLC 访问惩罚。以下示例作为粗略指南提供（这些示例来自 \[Levinthal 09\]）：

*   LLC 命中，线路未共享：约 40 个 CPU 周期
    
*   LLC 命中，线路在另一个核心中共享：约 65 个 CPU 周期
    
*   LLC 命中，在另一个核心中修改线路：约 75 个 CPU 周期
    

缓存一致性是设计可扩展多处理器系统时面临的最大挑战之一，因为内存可以快速修改。

内存管理单元（MMU）负责虚拟地址到物理地址的转换。

互连通常是针对高带宽而设计的，因此它们不会成为系统瓶颈。如果这样做，性能将会下降，因为 CPU 指令在涉及互连的操作（例如远程内存 I/O）中遇到停顿周期。一个关键指标是 IPC 的下降。可以使用 CPU 性能计数器来分析 CPU 指令、周期、IPC、停顿周期和内存 I/O。

#### 硬件计数器 (PMC)

性能监控计数器 (PMC) 在第 4 章“可观测性工具”、第 4.3.9 节“硬件计数器 (PMC)”中被总结为可观测性统计数据的来源。本节更详细地描述了它们的 CPU 实现，并提供了其他示例。

PMC 是在硬件中实现的处理器寄存器，可对其进行编程以对低级 CPU 活动进行计数。它们通常包括以下计数器：

*   CPU周期：包括停顿周期和停顿周期类型
    
*   CPU指令：退休（执行）
    
*   1、2、3 级缓存访问：命中、未命中
    
*   浮点单元：运算
    
*   内存 I/O：读取、写入、停顿周期
    
*   资源 I/O：读取、写入、停顿周期
    

每个 CPU 都有少量寄存器，通常在 2 到 8 个之间，可以对它们进行编程来记录此类事件。可用的选项取决于处理器类型和型号，并记录在处理器手册中。

作为一个相对简单的示例，Intel P6 系列处理器通过四个特定于模型的寄存器 (MSR) 提供性能计数器。两个MSR是计数器并且是只读的。另外两个 MSR 称为事件选择 MSR，用于对计数器进行编程并可读写。性能计数器是 40 位寄存器，事件选择 MSR 是 32 位。事件选择MSR 的格式如图6.11 所示。

#### GPU

图形处理单元 (GPU) 旨在支持图形显示，现在可用于其他工作负载，包括人工智能、机器学习、分析、图像处理和加密货币挖掘。对于服务器和云实例，GPU 是一种类似处理器的资源，可以执行一部分工作负载（称为计算内核），适合高度并行的数据处理，例如矩阵转换。 Nvidia 使用统一计算设备架构 (CUDA) 的通用 GPU 已得到广泛采用。 CUDA 提供用于使用 Nvidia GPU 的 API 和软件库。

虽然处理器 (CPU) 可能包含十几个核心，但 GPU 可能包含数百或数千个称为流处理器 (SP) 的较小核心，[^5] 每个核心都可以执行一个线程。由于 GPU 工作负载是高度并行的，因此可以并行执行的线程被分组为线程块，它们可以在线程块中相互协作。这些线程块可以由称为流式多处理器 (SM) 的 SP 组执行，这些处理器还提供包括内存高速缓存在内的其他资源。表 6.6 进一步比较了处理器 (CPU) 和 GPU \[Ather 19\]。

#### 其他加速器

请注意，除了 GPU 之外，还可能存在其他加速器，用于将 CPU 工作卸载到更快的专用集成电路。其中包括现场可编程门阵列 (FPGA) 和张量处理单元 (TPU)。如果使用，它们的使用情况和性能应与 CPU 一起分析，尽管它们通常需要自定义工具。

GPU 和 FPGA 用于提高加密货币挖掘的性能。

### 6.4.2 软件

支持 CPU 的内核软件包括调度程序、调度类和空闲线程。

#### 调度程序

内核CPU调度器的主要功能如图6.12所示。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/06fig12.jpg)

图6.12 内核CPU调度器功能

这些功能是：

*   分时：可运行线程之间的多任务处理，首先执行优先级最高的线程。
    
*   抢占：对于已经以高优先级运行的线程，调度程序可以抢占当前正在运行的线程，以便立即开始执行更高优先级的线程。
    
*   负载平衡：将可运行的线程移至空闲或不太繁忙的 CPU 的运行队列。
    

图 6.12 显示了运行队列，这就是调度最初的实现方式。该术语和心理模型仍然用于描述等待任务。然而，Linux CFS 调度程序实际上使用未来任务执行的红/黑树。

在Linux中，时间共享是通过调用scheduler\_tick()由系统定时器中断驱动的，它调用调度器类函数来管理优先级和称为时间片的CPU时间单位的到期。当线程变得可运行并且调用调度程序类 check\_preempt\_curr() 函数时，会触发抢占。线程切换由\_\_schedule()管理，它通过pick\_next\_task()选择最高优先级的线程来运行。负载平衡由 load\_balance() 函数执行。

当预计成本超过收益时，Linux 调度程序还使用逻辑来避免迁移，宁愿让繁忙的线程在 CPU 缓存仍处于热状态的同一 CPU 上运行（CPU 亲和性）。在 Linux 源代码中，请参阅idle\_balance() 和task\_hot() 函数。

请注意，所有这些函数名称都可能会更改；有关更多详细信息，请参阅 Linux 源代码，包括 Documentation 目录中的文档。

#### 安排课程

调度类管理可运行线程的行为，特别是它们的优先级、它们的 CPU 时间是否是时间片的，以及这些时间片的持续时间（也称为时间量子）。还有通过调度策略的附加控制，可以在调度类中选择这些策略，并且可以控制相同优先级的线程之间的调度。图 6.13 描述了 Linux 的它们以及线程优先级范围。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/06fig13.jpg)

图6.13 Linux线程调度程序优先级

用户级线程的优先级受用户定义的nice值影响，可以设置该值来降低不重要工作的优先级（以便对其他系统用户友好）。在Linux中，nice值设置线程的静态优先级，它与调度程序计算的动态优先级是分开的。

对于 Linux 内核，调度类是：

*   RT：为实时工作负载提供固定的高优先级。内核支持用户级和内核级抢占，允许以低延迟调度 RT 任务。优先级范围为 0–99 (MAX\_RT\_PRIO-1)。
    
*   O(1)：O(1) 调度程序是在 Linux 2.6 中引入的，作为用户进程的默认分时调度程序。该名称源自 O(1) 的算法复杂度（有关大 O 表示法的摘要，请参阅第 5 章“应用”）。之前的调度程序包含迭代所有任务的例程，使其成为 O(n)，这成为了可扩展性问题。 O(1) 调度程序动态提高了 I/O 密集型工作负载相对于 CPU 密集型工作负载的优先级，以减少交互和 I/O 工作负载的延迟。
    
*   CFS：Linux 2.6.23 内核中添加了完全公平调度，作为用户进程的默认分时调度器。调度程序在以任务 CPU 时间为键的红黑树上管理任务，而不是传统的运行队列。这使得低 CPU 消耗者可以轻松找到并优先执行 CPU 密集型工作负载，从而提高交互式和 I/O 密集型工作负载的性能。
    
*   空闲：以尽可能低的优先级运行线程。
    
*   Deadline：添加到 Linux 3.14，使用三个参数应用最早截止时间优先 (EDF) 调度：运行时间、周期和截止时间。任务应该每周期微秒接收运行时微秒的 CPU 时间，并在截止日期内执行此操作。
    

要选择调度类，用户级进程使用 sched\_setscheduler(2) 系统调用或 chrt(1) 工具选择映射到类的调度策略。

调度程序策略是：

*   RR：SCHED\_RR 是循环调度。一旦线程用完其时间量，它就会被移动到该优先级的运行队列的末尾，从而允许相同优先级的其他线程运行。使用RT调度类。
    
*   FIFO：SCHED\_FIFO 是先进先出调度，它继续运行运行队列头部的线程，直到它自愿离开，或者直到更高优先级的线程到达。即使运行队列中存在相同优先级的其他线程，该线程仍会继续运行。使用 RT 类。
    
*   NORMAL：SCHED\_NORMAL（以前称为SCHED\_OTHER）是分时调度，是用户进程的默认调度。调度器根据调度类别动态调整优先级。对于 O(1)，时间片持续时间是根据静态优先级设置的：较高优先级工作的持续时间较长。对于CFS，时间片是动态的。使用CFS调度类。
    
*   BATCH：SCHED\_BATCH 与 SCHED\_NORMAL 类似，但预期线程将受 CPU 限制，并且不应被调度来中断其他 I/O 限制的交互工作。使用CFS调度类。
    
*   IDLE：SCHED\_IDLE 使用 Idle 调度类。
    
*   DEADLINE：SCHED\_DEADLINE 使用 Deadline 调度类。
    

随着时间的推移，可能会添加其他类别和策略。人们已经研究了超线程感知 \[Bulpin 05\] 和温度感知 \[Otto 06\] 的调度算法，这些算法通过考虑其他处理器因素来优化性能。

当没有线程可运行时，将执行一个特殊的空闲任务（也称为空闲线程）作为占位符，直到另一个线程可运行。

#### 空闲线程

在第 3 章中介绍，当没有其他可运行线程时，内核“空闲”线程（或空闲任务）在 CPU 上运行，并且具有最低的优先级。它通常被编程为通知处理器 CPU 执行可能会停止（停止指令）或节流以节省电量。 CPU 将在下一个硬件中断时唤醒。

#### NUMA 分组

通过使内核具有 NUMA 感知能力，可以显着提高 NUMA 系统的性能，从而可以做出更好的调度和内存布局决策。这可以自动检测和创建本地化 CPU 和内存资源组，并将它们组织在拓扑中以反映 NUMA 架构。这种拓扑允许估计任何内存访问的成本。

在 Linux 系统上，这些称为调度域，它们处于从根域开始的拓扑中。

系统管理员可以通过将进程绑定为仅在一个或多个 CPU 上运行，或者通过创建一组独占的 CPU 供进程运行来执行手动形式的分组。请参见第 6.5.10 节，CPU 绑定。

#### 处理器资源感知

CPU资源拓扑也可以被内核理解，以便它可以为电源管理、硬件缓存使用和负载平衡做出更好的调度决策。

### 6.5.1 工具方法

工具方法是迭代可用工具、检查它们提供的关键指标的过程。虽然这是一种简单的方法，但它可能会忽略工具提供的可见性较差或没有可见性的问题，并且执行起来可能非常耗时。

对于 CPU，工具方法可能涉及检查以下内容 (Linux)：

*   `uptime/top`：检查负载平均值，看看负载是否随着时间的推移而增加或减少。使用以下工具时请记住这一点，因为负载在分析过程中可能会发生变化。
    
*   `vmstat`：以一秒的间隔运行 vmstat(1) 并检查系统范围的 CPU 利用率（“us”+“sy”）。利用率接近 100% 会增加调度程序延迟的可能性。
    
*   `mpstat`：检查每个 CPU 的统计信息并检查各个热（繁忙）CPU，识别可能的线程可扩展性问题。
    
*   `top`：查看哪些进程和用户是 CPU 消耗最多的进程和用户。
    
*   `pidstat`：将最大的 CPU 消耗者分解为用户时间和系统时间。
    
*   `perf/profile`：分析用户或内核时间的 CPU 使用情况堆栈跟踪，以确定使用 CPU 的原因。
    
*   `perf`：衡量 IPC 作为基于周期的低效率指标。
    
*   `showboost/turboboost`：检查当前的 CPU 时钟频率，以防异常低。
    
*   `dmesg`：检查 CPU 温度停顿消息（“CPU 时钟节流”）。
    

如果发现问题，请检查可用工具中的所有字段以了解更多背景信息。有关每个工具的更多信息，请参阅第 6.6 节“可观测性工具”。

### 6.5.2 使用方法

USE 方法可用于在性能调查的早期阶段识别所有组件的瓶颈和错误，然后再尝试更深入、更耗时的策略。

对于每个 CPU，检查：

*   利用率：CPU 繁忙的时间（不在空闲线程中）
    
*   饱和度：可运行线程排队等待 CPU 启动的程度
    
*   错误：CPU 错误，包括可纠正的错误
    

您可以首先检查错误，因为它们通常可以快速检查并且最容易解释。某些处理器和操作系统会检测到可纠正错误（错误纠正代码、ECC）的增加，并在不可纠正的错误导致 CPU 故障之前将 CPU 脱机作为预防措施。检查这些错误可以是检查所有 CPU 是否仍然在线。

通常可以通过操作系统工具轻松获得利用率（以忙碌百分比表示）。应针对每个 CPU 检查此指标，以检查可扩展性问题。通过使用分析和周期分析可以了解高 CPU 和核心利用率。

对于实施 CPU 限制或配额（资源控制；例如 Linux 任务集和 cgroup）的环境（如云计算环境中常见的那样），除了物理限制之外，还应根据施加的限制来衡量 CPU 利用率。您的系统可能会在物理 CPU 达到 100% 利用率之前就耗尽其 CPU 配额，从而比预期更早地遇到饱和。

饱和度指标通常在系统范围内提供，包括作为平均负载的一部分。该指标量化了 CPU 过载的程度，或者 CPU 配额（如果存在）用完的程度。

您可以按照类似的流程来检查 GPU 和其他加速器（如果正在使用）的运行状况，具体取决于可用指标。

### 6.5.3 工作负载特征

表征所应用的负载对于容量规划、基准测试和模拟工作负载非常重要。通过识别可以消除的不必要的工作，它还可以带来一些最大的性能提升。

用于表征 CPU 工作负载的基本属性包括：

*   CPU 平均负载（利用率 + 饱和度）
    
*   用户时间与系统时间的比率
    
*   系统调用率
    
*   自愿上下文切换率
    
*   中断率
    

目的是表征所施加的负载，而不是所提供的性能。某些操作系统（例如 Solaris）上的平均负载仅显示 CPU 需求，这使其成为 CPU 工作负载特征的主要指标。然而，在 Linux 上，平均负载包括其他负载类型。请参阅第 6.6.1 节“正常运行时间”中的示例和进一步说明。

速率指标有点难以解释，因为它们既反映了应用的负载，又在某种程度上反映了交付的性能，这可能会限制其速率。[^6]

[^6]例如，假设发现给定的批量计算工作负载在更快的 CPU 上运行时具有更高的系统调用率，即使工作负载相同。它完成得更快！

用户时间与系统时间的比率显示了所应用的负载类型，如前面第 6.3.9 节“用户时间/内核时间”中介绍的那样。高用户时间率是由于应用程序花费时间执行自己的计算。高系统时间表明花费在内核中的时间，这可以通过系统调用和中断率进一步理解。由于线程会阻塞等待 I/O，因此与 CPU 密集型工作负载相比，I/O 密集型工作负载具有更高的系统时间、系统调用和更高的自愿上下文切换。

以下是一个工作负载描述示例，旨在展示如何一起表达这些属性：

#### 高级工作负载特征/清单

可能会包含其他详细信息来描述工作负载的特征。此处列出了这些供考虑的问题，也可以作为彻底研究 CPU 问题时的清单：

*   系统范围内的 CPU 利用率是多少？每个CPU？每个核心？
    
*   CPU 负载的并行程度如何？是单线程的吗？有多少线程？
    
*   哪些应用程序或用户正在使用 CPU？多少？
    
*   哪些内核线程正在使用 CPU？多少？
    
*   中断的CPU使用率是多少？
    
*   CPU 互连利用率是多少？
    
*   为什么要使用 CPU（用户级和内核级调用路径）？
    
*   遇到什么类型的失速周期？
    

请参阅第 2 章“方法论”，了解该方法论的高级摘要以及要衡量的特征（谁、为什么、什么、如何）。接下来的部分扩展了此列表中的最后两个问题：如何使用分析来分析调用路径，以及如何使用周期分析来停止周期。

### 6.5.4 分析

分析构建了研究目标的图片。 CPU 分析可以通过不同的方式执行，通常是：

*   基于计时器的采样：收集当前运行的函数或堆栈跟踪的基于计时器的样本。使用的典型速率是每个 CPU 99 赫兹（每秒采样）。这提供了 CPU 使用情况的粗略视图，并为大大小小的问题提供了足够的详细信息。 99 用于避免可能在 100 赫兹发生的锁步采样，这会产生倾斜的轮廓。如果需要，可以降低计时器速率并扩大时间跨度，直到开销可以忽略不计并适合生产使用。
    
*   函数跟踪：检测所有或部分函数调用以测量其持续时间。这提供了精细级别的视图，但对于生产使用而言，开销可能过高，通常为 10% 或更多，因为函数跟踪为每个函数调用添加了检测。
    

生产中使用的大多数分析器以及本书中的分析器都使用基于计时器的采样。如图 6.14 所示，其中应用程序调用函数 A()，函数 A() 调用函数 B()，依此类推，同时收集堆栈跟踪样本。有关堆栈跟踪以及如何读取它们的说明，请参阅第 3 章“操作系统”、第 3.2.7 节“堆栈”。

### 6.5.5 周期分析

您可以使用性能监控计数器 (PMC) 来了解周期级别的 CPU 利用率。这可能表明周期花费在 1、2 或 3 级高速缓存未命中、内存或资源 I/O 上，或者花费在浮点运算或其他活动上。此信息可能显示您可以通过调整编译器选项或更改代码来实现性能提升。

通过测量 IPC（CPI 的倒数）开始周期分析。如果 IPC 较低，请继续调查停顿周期的类型。如果 IPC 高，请在代码中寻找减少执行指令的方法。 “高”或“低”IPC 的值取决于您的处理器：低可能小于 0.2，高可能大于 1。您可以通过执行已知的工作负载（内存 I/O）来了解这些值-密集型或指令密集型，并测量每个的结果IPC。

除了测量计数器值之外，PMC 还可以配置为在给定值溢出时中断内核。例如，每发生 10,000 次 3 级缓存未命中，内核就会被中断以收集堆栈回溯。随着时间的推移，内核会构建导致 3 级缓存未命中的代码路径的配置文件，而不会产生测量每个未命中的过高开销。这通常由集成开发环境 (IDE) 软件使用，用导致内存 I/O 和停顿周期的位置来注释代码。

如第 4 章“可观测性工具”第 4.3.9 节“PMC 挑战”中所述，溢出采样可能会由于滑动和乱序执行而错过记录正确的指令。在 Intel 上，解决方案是 PEBS，它由 Linux perf(1) 工具支持。

周期分析是一项高级活动，使用命令行工具可能需要数天时间才能完成，如第 6.6 节“可观测性工具”中所示。您还应该花一些时间阅读 CPU 供应商的处理器手册。 Intel vTune \[Intel 20b\] 和 AMD uprof \[AMD 20\] 等性能分析器可以节省时间，因为它们经过编程可以找到您感兴趣的 PMC。

### 6.5.6 性能监控

性能监控可以随着时间的推移识别活动问题和行为模式。 CPU 的关键指标包括：

*   利用率：繁忙百分比
    
*   饱和度：运行队列长度或调度程序延迟
    

应以每个 CPU 为基础监控利用率，以识别线程可扩展性问题。对于实施 CPU 限制或配额（资源控制）的环境，例如云计算环境，还应记录与这些限制相比的 CPU 使用情况。

选择正确的测量和归档间隔是监控 CPU 使用情况的一个挑战。一些监控工具使用五分钟的间隔，这可以隐藏较短的 CPU 使用率突发的存在。每秒测量更可取，但您应该意识到，即使在一秒内也可能存在突发。这些可以从饱和度中识别出来，并使用 FlameScope（第 6.7.4 节）进行检查，该火焰示波器是为亚秒级分析而创建的。

### 6.5.7 静态性能调优

静态性能调优重点关注配置环境的问题。对于 CPU 性能，请检查静态配置的以下方面：

*   有多少个 CPU 可供使用？他们是核心吗？硬件线程？
    
*   GPU 或其他加速器是否可用并正在使用中？
    
*   CPU 架构是单处理器还是多处理器？
    
*   CPU 缓存的大小是多少？它们是共享的吗？
    
*   CPU 时钟速度是多少？它是动态的（例如 Intel Turbo Boost 和 SpeedStep）吗？ BIOS 中是否启用了这些动态功能？
    
*   BIOS 中启用或禁用了哪些其他与 CPU 相关的功能？例如，涡轮增压、总线设置、省电设置？
    
*   此处理器型号是否存在性能问题（错误）？它们是否列在处理器勘误表中？
    
*   微码版本是什么？它是否包括针对安全漏洞（例如 Spectre/Meltdown）的影响性能的缓解措施？
    
*   此 BIOS 固件版本是否存在性能问题（错误）？
    
*   是否存在软件施加的 CPU 使用限制（资源控制）？这些是什么？
    

这些问题的答案可能会揭示以前被忽视的配置选择。

最后一个问题对于云计算环境尤其如此，在云计算环境中，CPU 使用率通常受到限制。

### 6.5.8 优先级调整

Unix一直提供nice(2)系统调用来调整进程优先级，它设置一个nice-ness值。正的 Nice 值会导致较低的进程优先级（更好），负值（只能由超级用户 (root)[^7] 设置）会导致较高的优先级。一个nice(1)命令可以用来启动具有nice值的程序，并且后来添加了一个renice(1M)命令（在BSD中）来调整已经运行的进程的nice值。 Unix 第四版的手册页提供了这个示例 \[TUHS 73\]：

[^7]自 Linux 2.6.12 起，可以修改每个进程的“nice上限”，允许非根进程具有较低的nice值。例如，使用：prlimit `--nice=-19 -p PID.`

建议希望执行长时间运行的程序而不受管理人员批评的用户将值设置为 16。

今天，nice 值对于调整进程优先级仍然有用。当存在 CPU 争用、导致高优先级工作的调度程序延迟时，这是最有效的。您的任务是确定低优先级工作，其中可能包括监视代理和计划备份，可以将其修改为以良好的值开始。还可以执行分析来检查调整是否有效，以及调度程序延迟对于高优先级工作是否保持较低。

除了好之外，操作系统还可以为进程优先级提供更高级的控制，例如更改调度程序类和调度程序策略以及可调参数。 Linux 包含一个实时调度类，它可以允许进程抢占所有其他工作。虽然这可以消除调度程序延迟（其他实时进程和中断除外），但请确保您了解后果。如果实时应用程序遇到多个线程进入无限循环的错误，则可能会导致所有 CPU 无法用于所有其他工作，包括手动修复问题所需的管理 shell。[^8]

[^8]Linux 自 2.6.25 起针对此问题提供了一个解决方案：RLIMIT\_RTTIME，它设置了实时线程在进行阻塞系统调用之前可能消耗的 CPU 时间（以微秒为单位）的限制。

### 6.5.9 资源控制

操作系统可以提供细粒度的控制来将CPU周期分配给进程或进程组。这些可能包括对 CPU 利用率和份额的固定限制，以实现更灵活的方法 — 允许根据份额值消耗空闲 CPU 周期。这些工作如何工作是特定于实现的，并在第 6.9 节“调优”中讨论。

### 6.5.10 CPU 绑定

调整 CPU 性能的另一种方法是将进程和线程绑定到单个 CPU 或 CPU 集合。这可以增加进程的 CPU 缓存热度，从而提高其内存 I/O 性能。对于 NUMA 系统，它还改进了内存局部性，进一步提高了性能。

通常有两种方式执行此操作：

*   CPU 绑定：将进程配置为仅在单个 CPU 上运行，或仅在定义的集合中的一个 CPU 上运行。
    
*   独占 CPU 集：对一组 CPU 进行分区，使其只能由分配给它们的进程使用。这可以进一步提高 CPU 缓存的热度，因为当进程空闲时，其他进程无法使用这些 CPU。
    

在基于 Linux 的系统上，可以使用 cpuset 来实现独占 CPU 集方法。第 6.9 节“调整”中提供了配置示例。

### 6.5.11 微基准测试

CPU 微基准测试工具通常会测量多次执行简单操作所需的时间。该操作可以基于：

*   CPU指令：整数算术、浮点运算、内存加载和存储、分支和其他指令
    
*   内存访问：研究不同CPU缓存的延迟和主内存吞吐量
    
*   高级语言：与CPU指令测试类似，但用高级解释或编译语言编写
    
*   操作系统操作：测试受 CPU 限制的系统库和系统调用函数，例如 getpid(2) 和进程创建
    

CPU 基准测试的一个早期示例是国家物理实验室的 Whetstone，于 1972 年在 Algol 60 中编写，旨在模拟科学工作负载。 Dhrystone 基准测试于 1984 年开发，用于模拟当时的整数工作负载，并成为比较 CPU 性能的流行手段。这些以及各种 Unix 基准测试（包括进程创建和管道吞吐量）都包含在名为 UnixBench 的集合中，该集合最初来自莫纳什大学，由 BYTE 杂志 \[Hinnant 84\] 出版。最近创建了 CPU 基准测试来测试压缩速度、素数计算、加密和编码。

无论您使用哪种基准测试，在比较系统之间的结果时，了解真正测试的内容非常重要。前面列出的基准通常最终会测试编译器版本之间编译器优化的差异，而不是基准代码或 CPU 速度。许多基准测试执行单线程，其结果在具有多个 CPU 的系统中失去意义。 （四 CPU 系统的基准测试速度可能比八 CPU 系统稍快，但当提供足够的并行可运行线程时，后者可能会提供更大的吞吐量。）

有关基准测试的更多信息，请参阅第 12 章，基准测试。

### 6.6.1 正常运行时间

uptime(1) 是打印系统平均负载的几个命令之一：

点击这里查看代码图片

$ **uptime**
  9:04pm  up 268 day(s), 10:16,  2 users,  **load average: 7.76, 8.32, 8.60**

最后三个数字是 1 分钟、5 分钟和 15 分钟的平均负载。通过比较这三个数字，您可以确定负载在过去 15 分钟（左右）内是增加、减少还是稳定。了解这一点很有用：如果您正在响应生产性能问题并发现负载正在下降，则您可能错过了该问题；如果负载增加，问题可能会变得更糟！

以下部分更详细地解释了平均负载，但它们只是一个起点，因此在继续讨论其他指标之前，考虑它们的时间不应超过五分钟。

#### 平均负载

负载平均值表明对系统资源的需求：较高意味着需求更多。在某些操作系统（例如 Solaris）上，负载平均值显示 CPU 需求，早期版本的 Linux 也是如此。但在 1993 年，Linux 更改了负载平均值以显示系统范围的需求：CPU、磁盘和其他资源。[^9] 这是通过将线程包含在 TASK\_UNINTERRUPTIBLE 线程状态（某些工具将其显示为状态“D”）中来实现的（该状态在第 5 章“应用程序”第 5.4.5 节“线程状态分析”中提到过。

[^9]这种变化发生得太久了，以至于其原因已被遗忘并且没有记录在案，早于 git 和其他资源中的 Linux 历史；我最终在旧邮件系统存档的在线 tarball 中找到了原始补丁。 Matthias Urlichs 做出了这一改变，指出如果需求从 CPU 转移到磁盘，那么平均负载应该保持不变，因为需求没有改变 \[Gregg 17c\]。我给他发了一封电子邮件（有史以来第一次）关于他 24 岁的变化，并在一小时内得到了回复！

负载的测量方式为当前资源使用情况（利用率）加上排队请求（饱和度）。想象一个汽车收费广场：您可以通过计算正在服务的汽车数量（利用率）以及排队的汽车数量（饱和度）来测量一天中不同时间点的负载。

该平均值是指数阻尼移动平均值，反映了 1 分钟、5 分钟和 15 分钟时间之外的负载（这些时间实际上是指数移动和 \[Myer 73\] 中使用的常数）。图 6.17 显示了一个简单实验的结果，其中启动了单个 CPU 绑定线程并绘制了负载平均值。

到 1 分钟、5 分钟和 15 分钟时，平均负载已达到已知负载 1.0 的约 61%。

负载平均值在早期 BSD 中被引入到 Unix，并且基于早期操作系统（CTSS、Multics \[Saltzer 70\]、TENEX \[Bobrow 72\]）常用的调度程序平均队列长度和负载平均值。 RFC 546 \[Thomas 73\] 中对它们进行了描述：

\[1\] TENEX 平均负载是 CPU 需求的衡量标准。平均负载是给定时间段内可运行进程数的平均值。例如，每小时平均负载为 10 意味着（对于单个 CPU 系统）在那一小时内的任何时间，我们都可以看到 1 个进程正在运行，而其他 9 个进程已准备好运行（即，未因 I/O 而阻塞）等待对于CPU。

作为一个现代示例，考虑一个平均负载为 128 的 64-CPU 系统。如果负载仅是 CPU，则意味着平均每个 CPU 上始终有一个线程在运行，并且每个 CPU 上都有一个线程在等待。 CPU 平均负载为 20 的同一系统表明有很大的空间，因为它可以在所有 CPU 都忙之前运行另外 44 个受 CPU 限制的线程。 （一些公司监控标准化负载平均指标，该指标会自动除以 CPU 数量，从而无需知道 CPU 数量即可对其进行解释。）
