## 第2章

方法论

我作为一名初级系统管理员开始了我的技术职业生涯，我认为我可以通过单独研究命令行工具和指标来学习性能。我错了。我从上到下阅读了手册页，了解了页面错误、上下文切换和各种其他系统指标的定义，但我不知道如何处理它们：如何从信号转向解决方案。

我注意到，每当出现性能问题时，高级系统管理员都有自己的思维程序，可以快速通过工具和指标来找到根本原因。他们了解哪些指标很重要，何时指出问题，以及如何使用它们来缩小调查范围。手册页中缺少的正是这种专业知识——通常是通过高级管理员或工程师的观察来学习的。

从那时起，我收集、记录、分享和开发了自己的绩效方法。本章包括这些方法和系统性能的其他基本背景：概念、术语、统计数据和可视化。在后面的章节深入实施之前，这将涵盖理论。

本章的学习目标是：

*   了解关键性能指标：延迟、利用率和饱和度。
    
*   培养对测量时间尺度（小至纳秒）的感觉。
    
*   了解调整权衡、目标以及何时停止分析。
    
*   确定工作负载与架构的问题。
    
*   考虑资源与工作负载分析。
    
*   遵循不同的性能方法，包括：使用方法、工作负载特征、延迟分析、静态性能调整和性能准则。
    
*   了解统计学和排队论的基础知识。
    

在本书的所有章节中，这一章自第一版以来变化最小。在我的职业生涯中，软件、硬件、性能工具和性能可调参数都发生了变化。保持不变的是理论和方法：本章涵盖的持久技能。

本章分为三个部分：

*   背景介绍术语、基本模型、关键性能概念和观点。其中大部分内容将作为本书其余部分的假定知识。
    
*   方法论讨论了观察和实验的性能分析方法；造型;和容量规划。
    
*   Metrics 引入了性能统计、监控和可视化。
    

这里介绍的许多方法将在后面的章节中进行更详细的探讨，包括第 5 章到第 10 章的方法部分。

### 2.1 术语

以下是系统性能的关键术语。后面的章节提供了额外的术语，并在不同的上下文中描述了其中的一些术语。

*   IOPS：每秒输入/输出操作数是数据传输操作速率的度量。对于磁盘 I/O，IOPS 是指每秒的读取和写入次数。
    
*   吞吐量：完成工作的速率。特别是在通信中，该术语用于指数据速率（每秒字节数或每秒位数）。在某些情况下（例如数据库），吞吐量可以指操作速率（每秒操作数或每秒事务数）。
    
*   响应时间：操作完成的时间。这包括等待所花费的时间和服务所花费的时间（服务时间），包括传输结果的时间。
    
*   延迟：衡量操作等待服务所花费的时间。在某些上下文中，它可以指操作的整个时间，相当于响应时间。有关示例，请参见第 2.3 节“概念”。
    
*   利用率：对于服务请求的资源，利用率是衡量资源繁忙程度的指标，基于给定时间间隔内资源主动执行工作的时间。对于提供存储的资源，利用率可以指消耗的容量（例如，存储器利用率）。
    
*   饱和度：资源将其无法服务的工作排队的程度。
    
*   瓶颈：在系统性能中，瓶颈是限制系统性能的资源。识别和消除系统瓶颈是系统性能的关键活动。
    
*   工作负载：系统的输入或施加的负载就是工作负载。对于数据库，工作负载由客户端发送的数据库查询和命令组成。
    
*   缓存：快速存储区域，可以复制或缓冲有限数量的数据，以避免直接与较慢的存储层通信，从而提高性能。出于经济原因，缓存通常小于较慢的层。
    

如果需要，术语表包含更多术语供参考。

### 2.3 概念

以下是系统性能的重要概念，并且是本章其余部分和本书的假定知识。在后面章节的架构部分介绍特定于实现的细节之前，以通用方式描述这些主题。

#### 2.3.1 延迟

对于某些环境，延迟是性能的唯一关注点。对于其他人来说，它是分析的首要一两个关键指标以及吞吐量。

作为延迟的示例，图 2.3 显示了网络传输，例如 HTTP GET 请求，其中时间分为延迟和数据传输部分。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/02fig03.jpg)

图2.3 网络连接延迟

延迟是执行操作之前等待的时间。在此示例中，该操作是传输数据的网络服务请求。在执行此操作之前，系统必须等待网络连接建立，这是此操作的延迟。响应时间跨越该延迟和操作时间。

由于延迟可以从不同的位置测量，因此通常用测量的目标来表示。例如，网站的加载时间可能由从不同位置测量的三个不同时间组成：DNS 延迟、TCP 连接延迟和 TCP 数据传输时间。 DNS 延迟是指整个 DNS 操作。 TCP 连接延迟仅指初始化（TCP 握手）。

在更高的层面上，所有这些，包括 TCP 数据传输时间，都可以被视为其他事物的延迟。例如，从用户点击网站链接到结果页面完全加载的时间可以称为延迟，其中包括浏览器通过网络获取网页并呈现它的时间。由于“延迟”一词可能含糊不清，因此最好包含限定术语来解释其测量的内容：请求延迟、TCP 连接延迟等。

由于延迟是基于时间的度量，因此可以进行各种计算。性能问题可以使用延迟进行量化，然后进行排名，因为它们使用相同的单位（时间）来表达。通过考虑何时可以减少或消除延迟，还可以计算预测的加速比。例如，使用 IOPS 指标无法准确执行这些操作。

#### 2.3.3 权衡

您应该了解一些常见的性能权衡。图 2.4 显示了好/快/便宜的“二选一”权衡，以及针对 IT 项目调整的术语。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/02fig04.jpg)

图 2.4 权衡：选择两个

许多 IT 项目选择准时且便宜，而将性能留到以后修复。当早期决策阻碍性能提高时，例如选择和填充次优存储架构、使用实施效率低下的编程语言或操作系统，或者选择缺乏全面性能分析工具的组件，这种选择可能会出现问题。

性能调优中的一个常见权衡是 CPU 和内存之间的权衡，因为内存可用于缓存结果，从而减少 CPU 使用率。在具有大量 CPU 的现代系统上，这种交易可能会以另一种方式进行：CPU 时间可能会花在压缩数据以减少内存使用上。

可调参数通常需要权衡。这里有几个例子：

*   文件系统记录大小（或块大小）：较小的记录大小，接近应用程序 I/O 大小，对于随机 I/O 工作负载性能更好，并且在应用程序运行时更有效地利用文件系统缓存。大记录大小将改善流工作负载，包括文件系统备份。
    
*   网络缓冲区大小：较小的缓冲区大小将减少每个连接的内存开销，有助于系统扩展。大尺寸将提高网络吞吐量。
    

在对系统进行更改时，请考虑此类权衡。

#### 2.3.4 调优工作

当在最接近工作执行地点进行性能调整时，性能调整最为有效。对于应用程序驱动的工作负载，这意味着在应用程序本身内。表 2.3 显示了具有调整可能性的示例软件堆栈。

通过在应用程序级别进行调整，您也许能够消除或减少数据库查询并大幅提高性能（例如 20 倍）。调低到存储设备级别可能会消除或改善存储 I/O，但在执行更高级别的操作系统堆栈代码时已经缴纳了税款，因此这可能只会按百分比（例如 20%）提高最终的应用程序性能。

在应用程序级别获得巨大性能提升还有另一个原因。当今许多环境的目标是快速部署特性和功能，每周或每天将软件更改推送到生产中。[^2]因此，应用程序开发和测试往往关注正确性，在生产部署之前很少或根本没有时间进行性能测量或优化。当性能成为问题时，这些活动才会进行。

[^2]快速变化的环境示例包括 Netflix 云和 Shopify，它们每天都会推送多次更改。

虽然应用程序可能是最有效的调整级别，但它不一定是最有效的基础观察级别。慢速查询可以通过它们在 CPU 上花费的时间或它们执行的文件系统和磁盘 I/O 来最好地理解。这些可以通过操作系统工具观察到。

在许多环境（尤其是云计算）中，应用程序级别不断发展，每周或每天将软件更改推入生产。随着应用程序代码的更改，经常会发现巨大的性能提升，包括回归修复。在这些环境中，操作系统的调整和操作系统的可观察性很容易被忽视。请记住，操作系统性能分析还可以识别应用程序级别的问题，而不仅仅是操作系统级别的问题，在某些情况下比单独从应用程序中更容易。

#### 2.3.5 适当程度

不同的组织和环境对性能有不同的要求。您可能加入了一个组织，在该组织中，进行比您以前见过的更深入的分析是常态，甚至知道这是可能的。或者您可能会发现，在您的新工作场所，您认为的基本分析被认为是高级的，并且以前从未进行过（好消息：唾手可得的成果！）。

这并不一定意味着有些组织做得正确，有些组织做得错误。这取决于绩效专业知识的投资回报 (ROI)。拥有大型数据中心或大型云环境的组织可能会聘请性能工程师团队来分析所有内容，包括内核内部结构和 CPU 性能计数器，并经常使用各种跟踪工具。他们还可以正式对绩效进行建模并为未来的增长做出准确的预测。对于每年在计算上花费数百万美元的环境来说，很容易证明雇用这样一个性能团队是合理的，因为他们发现的胜利就是投资回报率。计算支出适中的小型初创公司可能只执行表面检查，信任第三方监控解决方案来检查其性能并提供警报。

然而，正如第 1 章所述，系统性能不仅仅与成本有关：它还与最终用户体验有关。初创公司可能会发现有必要投资性能工程来改善网站或应用程序延迟。这里的投资回报率不一定是成本的降低，而是让客户而不是以前的客户更满意。

最极端的环境包括证券交易所和高频交易者，其中性能和延迟至关重要，并且需要付出大量的努力和费用。举个例子，纽约和伦敦交易所之间的跨大西洋电缆计划耗资 3 亿美元，以将传输延迟减少 6 毫秒 \[Williams 11\]。

在进行性能分析时，适当程度也会在决定何时停止分析时发挥作用。

#### 2.3.6 何时停止分析

进行性能分析时面临的一个挑战是知道何时停止。有这么多的工具，有这么多的东西需要检查！

当我教授表演课时（正如我最近再次开始做的那样），我可以给我的学生一个表现问题，其中有三个原因，然后发现有些学生在找到一个原因后就停止了，另一些学生找到两个原因，还有一些学生找到了三个原因。一些学生继续努力，试图找到性能问题的更多原因。谁做得对？在找到所有三个原因后，你可能很容易说你应该停止，但对于现实生活中的问题，你不知道原因的数量。

以下是您可以考虑停止分析的三种情况，并提供一些个人示例：

*   当您解释了大部分性能问题时。 Java 应用程序消耗的 CPU 比以前多了三倍。我发现的第一个问题是异常堆栈消耗 CPU。然后，我量化了这些堆栈中的时间，发现它们仅占总体 CPU 占用空间的 12%。如果这个数字接近 66%，我就可以停止分析——3 倍的减速已经被考虑在内。但在这种情况下，在 12% 的情况下，我需要继续寻找。
    
*   当潜在的投资回报率小于分析成本时。我处理的一些性能问题每年可以带来数千万美元的收益。对于这些，我可以证明花费几个月的时间（工程成本）进行分析是合理的。其他性能提升（例如小型微服务）可能需要数百美元来衡量：甚至可能不值得花费一个小时的工程时间来分析它们。例外情况可能包括当我在公司时间没有更好的事情做时（这在实践中从未发生过），或者如果我怀疑这可能是以后出现更大问题的金丝雀，因此值得在问题扩大之前进行调试。
    
*   当其他地方有更大的投资回报率时。即使前两种情况没有得到满足，其他地方也可能有更大的投资回报率需要优先考虑。
    

如果您是一名全职性能工程师，那么根据潜在的投资回报率对不同问题进行优先分析可能是一项日常任务。

#### 2.3.7 时间点建议

由于更多用户、更新的硬件以及更新的软件或固件的添加，环境的性能特征随着时间的推移而变化。当前受 10 Gbit/s 网络基础设施限制的环境在升级到 100 Gbit/s 后可能会开始遇到磁盘或 CPU 性能瓶颈。

性能建议，尤其是可调参数的值，仅在特定时间点有效。性能专家一周提出的最佳建议可能会在软件或硬件升级或添加更多用户后一周后变得无效。

在某些情况下，通过在互联网上搜索找到的可调参数值可以快速获胜。如果它们不适合您的系统或工作负载，曾经适合但现在不适合，或者仅适合作为在以后的软件升级中正确修复的软件错误的临时解决方法，它们也可能会降低性能。这类似于搜查别人的药柜并服用可能不适合您、可能已经过期或应该只短期服用的药物。

浏览此类建议只是为了查看存在哪些可调参数以及过去需要更改哪些参数，这可能很有用。然后，您的任务就变成查看是否以及如何针对您的系统和工作负载进行调整。但是，如果其他人之前不需要调整该参数，或者已经调整了它但没有在任何地方分享他们的经验，那么您可能仍然会错过一个重要的参数。

更改可调参数时，将它们存储在具有详细历史记录的版本控制系统中会很有帮助。 （在使用 Puppet、Salt、Chef 等配置管理工具时，您可能已经做了类似的事情。）这样，可以稍后检查可调参数更改的时间和原因。

应用程序可能会由于软件配置和运行它的硬件（其架构和实现）的问题而表现不佳。然而，应用程序也可能仅仅由于施加过多的负载而表现不佳，从而导致排队和长时间的延迟。负载和架构如图 2.5 所示。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/02fig05.jpg)

图 2.5 负载与架构

如果架构分析显示工作排队，但工作执行方式没有问题，则问题可能是施加了过多的负载。在云计算环境中，此时可以按需引入更多服务器实例来处理工作。

例如，架构问题可能是 CPU 上繁忙的单线程应用程序，请求排队，而其他 CPU 可用且空闲。在这种情况下，性能受到应用程序的单线程架构的限制。架构的另一个问题可能是多线程程序争夺单个锁，这样只有一个线程可以向前推进，而其他线程则等待。

负载问题可能是多线程应用程序在所有可用 CPU 上都很忙，并且请求仍在排队。在这种情况下，性能受到可用 CPU 容量的限制，或者换句话说，性能受到超出 CPU 处理能力的负载的限制。

系统在负载增加的情况下的性能就是其可扩展性。图 2.6 显示了系统负载增加时的典型吞吐量曲线。

当然，较长的响应时间是不好的。当系统开始将内存页移动到磁盘以释放主内存时，内存负载可能会出现“快速”降级情况。 CPU 负载可能会出现“缓慢”降级情况。

另一个“快速”配置文件示例是磁盘 I/O。随着负载（以及由此产生的磁盘利用率）的增加，I/O 更有可能排在其他 I/O 之后。空闲的旋转（非固态）磁盘可以以大约 1 毫秒的响应时间提供 I/O 服务，但当负载增加时，这可能会接近 10 毫秒。这是在第 2.6.5 节“排队理论”中的 M/D/1 和 60% 利用率下建模的，磁盘性能在第 9 章“磁盘”中介绍。

#### 2.3.10 指标

性能指标是由系统、应用程序或测量感兴趣的活动的附加工具生成的选定统计数据。对它们进行研究以进行性能分析和监控，无论是在命令行以数字方式还是使用可视化以图形方式。

常见的系统性能指标类型包括：

*   吞吐量：每秒的操作数或数据量
    
*   IOPS：每秒 I/O 操作数
    
*   利用率：资源的繁忙程度（百分比）
    
*   延迟：操作时间，以平均值或百分位数表示
    

吞吐量的使用取决于其上下文。数据库吞吐量通常是每秒查询或请求（操作）的度量。网络吞吐量是每秒位数或字节（容量）的度量。

IOPS 是仅针对 I/O 操作（读取和写入）的吞吐量测量。同样，上下文很重要，并且定义可能会有所不同。

##### 开销

绩效指标不是免费的；在某些时候，必须花费 CPU 周期来收集和存储它们。这会产生开销，从而对测量目标的性能产生负面影响。这称为观察者效应。 （它经常与海森堡的不确定性原理相混淆，海森堡的不确定性原理描述了已知的物理属性对（例如位置和动量）的精确度极限。）

##### 问题

您可能会认为软件供应商提供的指标经过精心挑选，没有错误，并且提供完整的可见性。事实上，指标可能会令人困惑、复杂、不可靠、不准确，甚至完全错误（由于错误）。有时，某个指标在一个软件版本中是正确的，但没有进行更新以反映新代码和代码路径的添加。

有关指标问题的更多信息，请参阅第 4 章“可观测性工具”、第 4.6 节“观察可观测性”。

#### 2.3.11 使用

术语“利用率”[^3] 通常用于操作系统来描述设备使用情况，例如 CPU 和磁盘设备。利用率可以是基于时间的或基于容量的。

[^3]世界某些地区的拼写使用情况。

##### 基于时间的

基于时间的利用率在排队论中被正式定义。例如\[冈瑟97\]：

服务器或资源繁忙的平均时间

随着比例

U=B/T

其中 U = 利用率，B = T（观察期）期间系统繁忙的总时间。

这也是操作系统性能工具最容易获得的“利用率”。磁盘监控工具 iostat(1) 将该指标称为“繁忙百分比”`%b`，该术语可以更好地传达底层指标：B/T。

该利用率指标告诉我们组件的繁忙程度：当组件利用率接近 100% 时，如果存在资源争用，性能可能会严重下降。可以检查其他指标来确认并查看该组件是否因此成为系统瓶颈。

有些组件可以并行服务多个操作。对于他们来说，在 100% 利用率下性能可能不会下降太多，因为他们可以接受更多的工作。要理解这一点，请考虑建筑物电梯。当它在楼层之间移动时可以认为被利用，而当其空闲等待时则不被利用。然而，即使电梯 100% 忙于响应呼叫，即处于 100% 利用率，电梯也可能能够容纳更多乘客。

100% 繁忙的磁盘也可能能够接受和处理更多工作，例如，通过缓冲磁盘缓存中稍后完成的写入操作。存储阵列经常以 100% 的利用率运行，因为某些磁盘 100% 的时间都忙，但阵列有大量空闲磁盘，可以接受更多的工作。

##### 基于能力

IT 专业人员在容量规划中使用利用率的另一个定义 \[Wong 97\]：

系统或组件（例如磁盘驱动器）能够提供一定量的吞吐量。在任何性能级别，系统或组件都以其容量的一定比例工作。该比例称为利用率。

这定义了容量而不是时间的利用率。这意味着利用率为 100% 的磁盘无法再接受任何工作。根据基于时间的定义，100% 利用率仅意味着 100% 的时间处于繁忙状态。

100% 繁忙并不意味着 100% 容量。

以电梯为例，100% 容量可能意味着电梯处于其最大有效负载能力，无法容纳更多乘客。

在理想的情况下，我们将能够测量设备的两种类型的利用率，这样，例如，您就会知道磁盘何时 100% 繁忙并且性能因争用而开始下降，以及何时处于最佳状态。 100% 容量，无法接受更多工作。不幸的是，这通常是不可能的。对于磁盘，需要了解磁盘的板载控制器正在做什么并预测容量。磁盘当前不提供此信息。

在本书中，利用率通常指的是基于时间的版本，也可以称为非空闲时间。容量版本用于某些基于卷的指标，例如内存使用情况。

#### 2.3.12 饱和度

资源请求的工作量超出其处理能力的程度即为饱和。利用率在 100%（基于容量）时开始出现饱和，因为无法处理额外的工作并开始排队。如图 2.8 所示。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/02fig08.jpg)

图 2.8 利用率与饱和度

该图显示，随着负载持续增加，饱和度线性增加，超过 100% 基于容量的利用率标记。任何程度的饱和都是性能问题，因为时间都花在等待（延迟）上。对于基于时间的利用率（繁忙百分比），排队并因此饱和可能不会在 100% 利用率标记时开始，具体取决于资源可以并行运行工作的程度。

#### 2.3.13 分析

分析构建了可以研究和理解的目标图片。在计算性能领域，分析通常是通过按时间间隔对系统状态进行采样，然后研究样本集来执行的。

与之前介绍的指标（包括 IOPS 和吞吐量）不同，采样的使用提供了目标活动的粗略视图。粗略程度取决于采样率。

作为分析的一个示例，可以通过频繁采样 CPU 指令指针或堆栈跟踪来收集有关消耗 CPU 资源的代码路径的统计信息，从而合理详细地了解 CPU 使用情况。第 6 章“CPU”中介绍了该主题。

#### 2.3.14 缓存

缓存经常用于提高性能。缓存将较慢存储层的结果存储在较快存储层中，以供参考。一个示例是在主内存 (RAM) 中缓存磁盘块。

可以使用多层高速缓存。 CPU 通常对主内存采用多个硬件高速缓存（1 级、2 级和 3 级），从非常快但很小的高速缓存（1 级）开始，然后存储大小和访问延迟都会增加。这是密度和延迟之间的经济权衡；选择级别和尺寸是为了在可用的片上空间中获得最佳性能。这些缓存将在第 6 章“CPU”中介绍。

系统中还存在许多其他缓存，其中许多是使用主内存进行存储的软件实现的。有关缓存层的列表，请参阅第 3 章“操作系统”、第 3.2.11 节“缓存”。

了解缓存性能的一个指标是每个缓存的命中率，即在缓存中找到所需数据的次数（命中）与总访问次数（命中 + 未命中）：

命中率=命中数/（命中数+未命中数）

越高越好，因为比率越高反映从更快的介质成功访问的数据越多。图 2.9 显示了提高缓存命中率所带来的预期性能改进。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/02fig09.jpg)

图2.9 缓存命中率和性能

98%和99%之间的性能差异远大于10%和11%之间的性能差异。这是一个非线性曲线，因为缓存命中和未命中（两个存储层在起作用）之间的速度存在差异。差异越大，坡度就越陡。

了解缓存性能的另一个指标是缓存未命中率，以每秒未命中数为单位。这与每次未命中的性能损失成正比（线性），并且更容易解释。

例如，工作负载 A 和 B 使用不同的算法执行相同的任务，并使用主内存缓存来避免从磁盘读取。工作负载 A 的缓存命中率为 90%，工作负载 B 的缓存命中率为 80%。仅此信息就表明工作负载 A 的性能更好。如果工作负载 A 的缺失率为 200/s，工作负载 B 的缺失率为 20/s，会怎样？从这些角度来看，工作负载 B 执行的磁盘读取次数要少 10 倍，这可能比 A 更快地完成任务。可以肯定的是，每个工作负载的总运行时间可以计算为

运行时间 = (命中率 × 命中延迟) + (未命中率 × 未命中延迟)

此计算使用平均命中和未命中延迟，并假设工作是序列化的。

##### 算法

缓存管理算法和策略决定在缓存可用的有限空间中存储什么内容。

最近使用（MRU）是指缓存保留策略，它决定哪些内容有利于保留在缓存中：最近使用过的对象。最近最少使用（LRU）可以参考等效的缓存驱逐策略，决定当需要更多空间时从缓存中删除哪些对象。还有最常用 (MFU) 和最不常用 (LFU) 策略。

您可能会遇到不常用（NFU），这可能是 LRU 的廉价但不太彻底的版本。

##### 热缓存、冷缓存和温缓存

这些词通常用来描述缓存的状态：

*   冷：冷缓存为空，或填充了不需要的数据。冷缓存的命中率为零（或在开始预热时接近于零）。
    
*   热：热缓存是一种填充了有用数据但命中率不够高而无法被视为热的缓存。
    
*   热：热缓存填充有常用请求的数据，并且具有高命中率，例如超过99%。
    
*   温度：缓存温度描述缓存的热度或冷度。提高缓存热度的活动旨在提高缓存命中率。
    

当缓存首次初始化时，它们开始变冷，然后随着时间的推移而变暖。当缓存很大或下一级存储很慢（或两者兼而有之）时，缓存可能需要很长时间才能填充并预热。

例如，我开发的存储设备具有 128 GB 的 DRAM 作为文件系统缓存、600 GB 的闪存作为二级缓存以及用于存储的旋转磁盘。在随机读取工作负载下，磁盘的读取速度约为 2,000 次/秒。对于 8 KB I/O 大小，这意味着缓存只能以 16 Mbytes/s (2,000 × 8 KB) 的速率预热。当两个缓存都开始变冷时，DRAM缓存需要2个多小时才能预热，而闪存缓存则需要10多个小时才能预热。

#### 2.3.15 已知-未知

前言中介绍的已知已知、已知未知和未知未知的概念对于性能领域很重要。细分如下，并举例进行系统性能分析：

*   已知的知识：这些是你知道的事情。您知道您应该检查性能指标，并且您知道其当前值。例如，您知道您应该检查 CPU 利用率，并且您还知道该值为平均 10%。
    
*   已知-未知：这些是你知道但你不知道的事情。您知道可以检查指标或子系统是否存在，但您尚未观察到它。例如，您知道可以使用分析来检查 CPU 繁忙的原因，但尚未这样做。
    
*   Unknown-unknowns：这些是你不知道但你不知道的事情。例如，您可能不知道设备中断会成为 CPU 的重度消耗者，因此您没有检查它们。
    

表演是一个“你知道的越多，你不知道的就越多”的领域。你对系统了解得越多，你就会意识到更多的未知，然后你可以检查这些已知的未知。

#### 2.4.1 资源分析

资源分析从分析系统资源开始：CPU、内存、磁盘、网络接口、总线和互连。它很可能由系统管理员（负责物理资源的人员）执行。活动包括

*   性能问题调查：查看是否是特定类型的资源造成的
    
*   容量规划：获取有助于调整新系统规模以及了解现有系统资源何时可能耗尽的信息
    

此视角侧重于利用率，以确定资源何时达到或接近其极限。某些资源类型（例如 CPU）具有随时可用的利用率指标。可以根据可用指标来估计其他资源的利用率，例如，通过将每秒发送和接收的兆比特数（吞吐量）与已知或预期的最大带宽进行比较来估计网络接口利用率。

最适合资源分析的指标包括：

*   IOPS
    
*   吞吐量
    
*   利用率
    
*   饱和
    

这些衡量资源被要求做什么，以及给定负载的利用或饱和程度。其他类型的指标（包括延迟）也有助于了解资源对给定工作负载的响应情况。

资源分析是性能分析的常用方法，部分归功于该主题的广泛可用文档。此类文档重点关注操作系统“stat”工具：vmstat(8)、iostat(1)、mpstat(1)。当您阅读此类文档时，重要的是要了解这是一种观点，但不是唯一的观点。

#### 2.4.2 工作负载分析

工作负载分析（参见图 2.11）检查应用程序的性能：应用的工作负载以及应用程序的响应方式。它最常由应用程序开发人员和支持人员（负责应用程序软件和配置的人员）使用。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/02fig11.jpg)

图2.11 工作负载分析

工作负载分析的目标是：

*   请求：应用的工作负载
    
*   延迟：应用程序的响应时间
    
*   完成：寻找错误
    

研究工作负载请求通常涉及检查和总结其属性：这是工作负载特征化的过程（在第 2.5 节“方法论”中更详细地描述）。对于数据库，这些属性可能包括客户端主机、数据库名称、表和查询字符串。此数据可能有助于识别不必要的工作或不平衡的工作。即使系统很好地执行其当前工作负载（低延迟），检查这些属性也可以找到减少或消除所应用工作的方法。请记住，最快的查询是您根本不执行的查询。

延迟（响应时间）是表达应用程序性能的最重要指标。对于MySQL数据库来说，是查询延迟；对于 Apache 来说，是 HTTP 请求延迟；等等。在这些上下文中，术语“延迟”的含义与响应时间相同（有关上下文的更多信息，请参阅第 2.3.1 节“延迟”）。

工作负载分析的任务包括识别和确认问题（例如，通过查找超出可接受阈值的延迟），然后找到延迟的来源并确认应用修复后延迟是否得到改善。请注意，起点是应用程序。调查延迟通常涉及更深入地研究应用程序、库和操作系统（内核）。

可以通过研究与事件完成相关的特征（包括其错误状态）来识别系统问题。虽然请求可能会很快完成，但它可能会出现错误状态，从而导致重试请求，从而累积延迟。

最适合工作负载分析的指标包括：

*   吞吐量（每秒事务数）
    
*   延迟
    

这些衡量请求率和最终的性能。

#### 2.5.1 路灯反方法

这种方法实际上是缺乏刻意的方法论。用户通过选择熟悉的、在互联网上找到的或随机的可观察工具来分析性能，以查看是否出现任何明显的情况。这种方法时好时坏，可能会忽略许多类型的问题。

可以以类似的试错方式尝试调整性能，将已知且熟悉的任何可调参数设置为不同的值，看看是否有帮助。

即使这种方法揭示了一个问题，它也可能会很慢，因为仅仅因为它们很熟悉，就发现并尝试了与该问题无关的工具或调整。因此，这种方法以称为路灯效应的观察偏差命名，如以下寓言所示：

一天晚上，一名警察看到一名醉汉在路灯下搜寻地面，并询问他在找什么。醉汉说他丢了钥匙。警察也找不到，问道：“你确定是在路灯下丢的吗？”醉汉回答：“没有，但这里光线最好。”

性能等同于查看 top(1)，不是因为它有意义，而是因为用户不知道如何阅读其他工具。

这种方法确实发现的问题可能是问题，但不是问题。其他方法可以量化结果，以便更快地排除误报，并优先处理更大的问题。

#### 2.5.2 随机变化反法

这是一种实验性的反方法论。用户随机猜测问题可能出在哪里，然后进行更改，直到问题消失。为了确定每次更改的结果是否提高了性能，需要研究一个指标，例如应用程序运行时间、操作时间、延迟、操作速率（每秒操作数）或吞吐量（每秒字节数）。做法如下：

1.  选择一个随机项目进行更改（例如，可调参数）。
    
2.  朝一个方向改变它。
    
3.  衡量绩效。
    
4.  把它换到另一个方向。
    
5.  衡量绩效。
    
6.  步骤 3 或步骤 5 中的结果是否优于基线？如果是，请保留更改并返回步骤 1。
    

虽然此过程最终可能会发现适合测试工作负载的调整，但它非常耗时，而且还可能导致调整从长远来看没有意义。例如，应用程序更改可能会提高性能，因为它解决了稍后修复的数据库或操作系统错误。但应用程序仍然会进行不再有意义的调整，并且一开始就没有人正确理解。

另一个风险是，未正确理解的更改会在高峰生产负载期间导致更严重的问题，并且需要取消更改。

#### 2.5.3 责备他人反方法

这种反方法论遵循以下步骤：

1.  查找您不负责的系统或环境组件。
    
2.  假设问题出在该组件上。
    
3.  将问题重定向至负责该组件的团队。
    
4.  当证明错误时，返回步骤 1。
    

“也许是网络的原因。你能和网络团队核实一下他们是否有丢包之类的情况吗？”

这种方法的使用者没有调查性能问题，而是将其视为别人的问题，这可能会浪费其他团队的资源，而事实证明这根本不是他们的问题。这种反方法论可以通过缺乏导致该假设的数据来识别。

为了避免成为指责他人的受害者，请向原告询问屏幕截图，显示运行了哪些工具以及如何解释输出。您可以将这些屏幕截图和解释提供给其他人以获得第二意见。

#### 2.5.4 临时清单法

当支持专业人员被要求检查和调整系统时（通常是在很短的时间内），逐步完成预设的检查表是一种常见的方法。典型的场景涉及在生产中部署新服务器或应用程序，并且支持专业人员在系统处于实际负载下时花费半天时间检查常见问题。这些清单是临时的，是根据该系统类型的最新经验和问题构建的。

以下是清单条目示例：

运行 `iostat –x 1` 并检查 `r_await` 列。如果加载期间持续超过 10（毫秒），则磁盘读取速度缓慢或磁盘过载。

一个检查表可能由十几个这样的检查组成。

虽然这些清单可以在最短的时间内提供最大的价值，但它们是时间点建议（请参阅第 2.3 节，概念），需要经常刷新以保持最新状态。他们还倾向于关注那些可以轻松记录的已知修复的问题，例如可调参数的设置，而不是对源代码或环境的自定义修复。

如果您管理着一支由支持专业人员组成的团队，则临时检查表可能是确保每个人都知道如何检查常见问题的有效方法。可以编写清晰且规范的清单，显示如何识别每个问题以及解决方案。但请记住，该列表必须不断更新。

#### 2.5.5 问题陈述

定义问题陈述是支持人员首次响应问题时的一项例行任务。这是通过询问客户以下问题来完成的：

1.  是什么让您认为存在性能问题？
    
2.  这个系统曾经表现良好吗？
    
3.  最近发生了什么变化？软件？硬件？加载？
    
4.  问题可以用延迟或运行时间来表达吗？
    
5.  该问题是否影响其他人或应用程序（或者只是您）？
    
6.  环境如何？使用什么软件和硬件？版本？配置？
    

只要提出并回答这些问题，通常就能指出直接原因和解决方案。因此，问题陈述作为其自己的方法包含在此处，并且应该是解决新问题时使用的第一个方法。

我仅使用问题陈述方法通过电话解决了性能问题，无需登录任何服务器或查看任何指标。

#### 2.5.6 科学方法

科学方法通过提出假设然后进行测试来研究未知事物。可以概括为以下几个步骤：

1.  问题
    
2.  假设
    
3.  预言
    
4.  测试
    
5.  分析
    

问题是性能问题陈述。由此您可以推测性能不佳的原因可能是什么。然后，您构建一个测试，可以是观察性的或实验性的，用于测试基于假设的预测。您将完成对收集到的测试数据的分析。

例如，您可能会发现在迁移到主内存较少的系统后应用程序性能下降，并且您假设性能不佳的原因是文件系统缓存较小。您可以使用观察测试来测量两个系统上的缓存未命中率，预测较小系统上的缓存未命中率会更高。实验测试是增加缓存大小（添加 RAM），预测性能将会提高。另一种可能更简单的实验测试是人为地减小缓存大小（使用可调参数），预测性能会更差。

以下是更多示例。

##### 示例（观察）

1.  问题：什么原因导致数据库查询缓慢？
    
2.  假设：吵闹的邻居（其他云计算租户）正在执行磁盘 I/O，与数据库磁盘 I/O 竞争（通过文件系统）。
    
3.  预测：如果在查询期间测量文件系统 I/O 延迟，则表明文件系统是导致查询缓慢的原因。
    
4.  测试：将数据库文件系统延迟跟踪为查询延迟的比率，结果显示不到 5% 的时间用于等待文件系统。
    
5.  分析：文件系统和磁盘不是查询慢的原因。
    

尽管问题仍未解决，但环境的一些重要组成部分已被排除。进行此项调查的人员可以返回步骤 2 并提出新的假设。

##### 示例（实验）

1.  问题：为什么从主机 A 到主机 C 的 HTTP 请求比从主机 B 到主机 C 花费的时间更长？
    
2.  假设：主机A和主机B位于不同的数据中心。
    
3.  预测：将主机 A 移至与主机 B 相同的数据中心将解决该问题。
    
4.  测试：移动主机A并测量性能。
    
5.  分析：性能已修复——与假设一致。
    

如果问题没有得到解决，请在开始新的假设之前逆转实验更改（在本例中将主机 A 移回原处）——同时更改多个因素会使确定哪个因素重要变得更加困难！

##### 示例（实验）

1.  问题：为什么文件系统性能会随着文件系统缓存大小的增加而下降？
    
2.  假设：较大的缓存存储更多的记录，与较小的缓存相比，管理较大的缓存需要更多的计算。
    
3.  预测：使记录大小逐渐变小，从而导致使用更多记录来存储相同数量的数据，将使性能逐渐变差。
    
4.  测试：使用逐渐减小的记录大小来测试相同的工作负载。
    
5.  分析：结果被绘制成图表并且与预测一致。现在对缓存管理例程进行深入分析。
    

这是一个负面测试的例子——故意损害性能以了解有关目标系统的更多信息。

#### 2.5.7 诊断周期

与科学方法类似的是诊断周期：

假设→仪器→数据→假设

与科学方法一样，这种方法也通过收集数据来故意检验假设。该循环强调数据可以快速得出新的假设，并对其进行测试和完善，等等。这类似于医生进行一系列小测试来诊断患者，并根据每个测试的结果完善假设。

这两种方法都在理论和数据上取得了良好的平衡。尝试快速从假设转向数据，以便尽早识别并丢弃不良理论，并开发出更好的理论。

#### 2.5.8 工具方法

面向工具的方法如下：

1.  列出可用的性能工具（可以选择安装或购买更多工具）。
    
2.  对于每个工具，列出它提供的有用指标。
    
3.  对于每个指标，列出解释它的可能方法。
    

其结果是一个规范性清单，显示要运行哪个工具、要读取哪些指标以及如何解释它们。虽然这可能相当有效，但它完全依赖于可用（或已知）的工具，这些工具可以提供不完整的系统视图，类似于路灯反方法。更糟糕的是，用户没有意识到他们的视图不完整，并且可能一直没有意识到。需要自定义工具（例如动态跟踪）的问题可能永远不会被识别和解决。

在实践中，工具方法确实可以识别某些资源瓶颈、错误和其他类型的问题，尽管它可能无法有效地做到这一点。

当有大量工具和指标可用时，迭代它们可能会非常耗时。当多个工具似乎具有相同的功能并且您花费额外的时间试图了解每个工具的优缺点时，情况会变得更糟。在某些情况下，例如文件系统微基准测试工具，有十几种工具可供选择，而您可能只需要一种。[^4]

[^4]顺便说一句，我遇到的支持多个重叠工具的一个论点是“竞争是好事”。我对此持谨慎态度：虽然使用重叠工具来交叉检查结果可能会有所帮助（并且我经常使用 Ftrace 交叉检查 BPF 工具），但多个重叠工具可能会浪费开发人员的时间，而这些时间本可以更有效地使用对于必须评估每个选择的最终用户来说也是浪费时间。

#### 2.5.9 使用方法

应在性能调查的早期使用利用率、饱和度和错误 (USE) 方法来识别系统瓶颈 \[Gregg 13b\]。它是一种关注系统资源的方法，可以概括为：

对于每个资源，检查利用率、饱和度和错误。

这些术语定义如下：

*   资源：所有物理服务器功能组件（CPU、总线……）。如果指标有意义，也可以检查一些软件资源。
    
*   利用率：在设定的时间间隔内，资源忙于服务工作的时间百分比。当繁忙时，资源可能仍然能够接受更多工作；它不能做到这一点的程度由饱和度来确定。
    
*   饱和度：资源具有无法服务的额外工作的程度，通常在队列中等待。另一个术语是压力。
    
*   错误：错误事件的计数。
    

对于某些资源类型（包括主内存），利用率是指所使用资源的容量。这与基于时间的定义不同，并在前面第 2.3.11 节“利用率”中进行了解释。一旦容量资源达到 100% 利用率，就无法接受更多工作，资源要么将工作排队（饱和），要么返回错误，这也可以使用 USE 方法来识别。

应调查错误，因为它们会降低性能，但在故障模式可恢复时可能不会立即被注意到。这包括失败并重试的操作，以及冗余设备池中失败的设备。

与工具方法相比，USE 方法涉及迭代系统资源而不是工具。这可以帮助您创建完整的问题列表，然后您才可以搜索工具来回答这些问题。即使无法找到工具来回答某些问题，了解这些问题尚未得到解答对于性能分析师来说也非常有用：它们现在是“已知的未知数”。

USE 方法还将分析引导到有限数量的关键指标，以便尽快检查所有系统资源。此后，如果没有发现问题，则可以使用其他方法。

##### 表达指标

USE方法指标通常表示如下：

*   利用率：一段时间间隔内的百分比（例如，“一个 CPU 以 90% 的利用率运行”）
    
*   饱和度：作为等待队列长度（例如，“CPU 的平均运行队列长度为 4”）
    
*   错误：报告的错误数（例如，“此磁盘驱动器有 50 个错误”）
    

尽管这似乎有悖常理，但短暂的高利用率可能会导致饱和和性能问题，即使整体利用率在很长一段时间内较低。一些监控工具报告超过 5 分钟的平均利用率。例如，CPU 利用率每秒都会发生巨大变化，因此 5 分钟平均值可能会掩盖短期的 100% 利用率，从而掩盖饱和度。

考虑高速公路上的收费站。利用率可以定义为有多少个收费站正忙于为汽车提供服务。利用率为 100% 意味着您找不到空摊位，必须在某人后面排队（饱和）。如果我告诉你摊位全天利用率为 40%，你能告诉我当天是否有车在排队吗？他们可能在高峰时段这样做，当时利用率为 100%，但这在日平均值中不可见。

##### 资源列表

USE 方法的第一步是创建资源列表。尝试尽可能完整。以下是服务器硬件资源的通用列表以及具体示例：

*   CPU：套接字、内核、硬件线程（虚拟 CPU）
    
*   主存：DRAM
    
*   网络接口：以太网端口、Infiniband
    
*   存储设备：磁盘、存储适配器
    
*   加速器：GPU、TPU、FPGA 等（如果使用）
    
*   控制器：存储、网络
    
*   互连：CPU、内存、I/O
    

每个组件通常充当单一资源类型。例如，主内存是容量资源，网络接口是 I/O 资源（可以表示 IOPS 或吞吐量）。某些组件可以表现为多种资源类型：例如，存储设备既是 I/O 资源，又是容量资源。考虑所有可能导致性能瓶颈的类型。另请注意，I/O 资源可以作为排队系统进一步研究，该系统对这些请求进行排队，然后为这些请求提供服务。

某些物理组件，例如硬件缓存（例如 CPU 缓存），可以排除在清单之外。 USE 方法对于在高利用率或饱和情况下性能下降、导致瓶颈的资源最有效，而缓存则在高利用率下提高性能。这些可以使用其他方法进行检查。如果您不确定是否包含某个资源，请将其包含在内，然后查看指标在实践中的效果如何。

##### 软件资源

可以类似地检查某些软件资源。这通常适用于软件的较小组件（而不是整个应用程序），例如：

*   互斥锁：利用率可以定义为持有锁的时间、排队等待锁的线程的饱和度。
    
*   线程池：利用率可以定义为线程忙于处理工作的时间，以及等待线程池服务的请求数量的饱和度。
    
*   进程/线程容量：系统可能具有有限数量的进程或线程，其当前使用情况可定义为利用率；等待分配可能会饱和；错误是指分配失败（例如“无法分叉”）。
    
*   文件描述符容量：与进程/线程容量类似，但针对文件描述符。
    

如果这些指标适合您的情况，请使用它们；否则，可以应用延迟分析等替代方法。

##### 建议的解释

以下是解释指标类型的一些一般建议：

*   利用率：100% 的利用率通常是瓶颈的标志（检查饱和度及其效果以确认）。由于以下几个原因，利用率超过 60% 可能会成为问题：根据时间间隔，它可能会隐藏 100% 利用率的短暂爆发。此外，某些资源（例如硬盘（但不是 CPU））在操作过程中通常不能中断，即使对于更高优先级的工作也是如此。随着利用率的提高，排队延迟变得更加频繁和明显。有关 60% 利用率的更多信息，请参阅第 2.6.5 节“排队理论”。
    
*   饱和度：任何饱和度（非零）都可能是一个问题。它可以以等待队列的长度或在队列上等待所花费的时间来衡量。
    
*   错误：非零错误计数器值得研究，特别是当它们在性能较差时增加时。
    

负面情况很容易解释：利用率低、不饱和、没有错误。这比听起来更有用 - 缩小调查范围可以帮助您快速关注问题区域，并确定它可能不是资源问题。这就是消除的过程。

##### 资源控制

在云计算和容器环境中，软件资源控制可能会限制或限制共享一个系统的租户。这些可能会限制内存、CPU、磁盘 I/O 和网络 I/O。例如，Linux 容器使用 cgroup 来限制资源使用。这些资源限制中的每一个都可以使用 USE 方法进行检查，类似于检查物理资源。

例如，“内存容量利用率”可以是租户的内存使用量与其内存上限。即使主机系统没有遇到内存压力，也可以通过该租户的限制分配错误或交换来看到“内存容量饱和”。这些限制将在第 11 章“云计算”中讨论。

##### 微服务

微服务架构提出了与过多资源指标类似的问题：每个服务可能有太多指标，以至于很难检查所有指标，并且可能会忽略尚不存在指标的区域。 USE 方法也可以解决微服务的这些问题。例如，对于典型的 Netflix 微服务，USE 指标为：

*   利用率：整个实例集群的平均 CPU 利用率。
    
*   饱和度：近似值是第 99 个延迟百分位数与平均延迟之间的差异（假设第 99 个延迟百分位数是饱和驱动的）。
    
*   错误：请求错误。
    

Netflix 已使用 Atlas 云范围监控工具 \[Harrington 14\] 对每个微服务检查了这三个指标。

有一种专门为服务设计的类似方法：RED 方法。

#### 2.5.10 红色方法

这种方法的重点是服务，通常是微服务架构中的云服务。它确定了从用户角度监控健康状况的三个指标，可概括为 \[Wilkie 18\]：

对于每项服务，检查请求率、错误和持续时间。

指标是：

*   请求率：每秒的服务请求数
    
*   错误：失败的请求数
    
*   持续时间：请求完成的时间（除了平均值之外还考虑分布统计数据，例如百分位数：请参阅第 2.8 节，统计数据）
    

您的任务是绘制微服务架构图，并确保针对每个服务监控这三个指标。 （分布式追踪工具可能会为您提供这样的图表。）优点与 USE 方法类似：RED 方法快速且易于遵循，而且全面。

RED 方法由 Tom Wilkie 创建，他还使用 Grafana 通过仪表板为 Prometheus 开发了 USE 和 RED 方法指标的实现 \[Wilkie 18\]。这些方法是互补的：用于机器健康状况的 USE 方法和用于用户健康状况的 RED 方法。

请求率的包含为调查提供了重要的早期线索：性能问题是否是负载与架构之间的问题（请参阅第 2.3.8 节“负载与架构”）。如果请求率一直稳定，但请求持续时间增加，则表明架构存在问题：服务本身。如果请求率和持续时间都增加，则问题可能出在所施加的负载之一上。可以使用工作负载特征进一步研究这一点。

#### 2.5.11 工作负载特征

工作负载特征描述是一种简单而有效的方法，用于识别一类问题：由于所施加的负载而产生的问题。它关注的是系统的输入，而不是最终的性能。您的系统可能不存在架构、实现或配置问题，但正在经历超出其合理处理能力的负载。

工作负载可以通过回答以下问题来表征：

*   谁造成了负载？进程ID、用户ID、远程IP地址？
    
*   为什么要调用负载？代码路径、堆栈跟踪？
    
*   负载特性有哪些？ IOPS、吞吐量、方向（读/写）、类型？在适当的情况下包括方差（标准差）。
    
*   负载如何随时间变化？有日常模式吗？
    

即使您对答案抱有强烈期望，检查所有这些内容也很有用，因为您可能会感到惊讶。

考虑以下场景：您的数据库存在性能问题，该数据库的客户端是 Web 服务器池。您是否应该检查正在使用该数据库的人的 IP 地址？根据配置，您已经期望它们成为 Web 服务器。不管怎样，您检查后发现整个互联网似乎都在向数据库施加负载，从而破坏了它们的性能。您实际上受到了拒绝服务 (DoS) 攻击！

最佳性能的胜利是消除不必要的工作的结果。有时，不必要的工作是由应用程序故障引起的，例如，线程陷入循环中，从而产生不必要的 CPU 工作。它也可能是由错误的配置引起的，例如，在高峰时段运行的系统范围的备份，甚至是前面描述的 DoS 攻击。表征工作负载可以识别这些问题，并且通过维护或重新配置可以消除这些问题。

如果无法消除所识别的工作负载，另一种方法可能是使用系统资源控制来限制它。例如，系统备份任务可能会消耗 CPU 资源来压缩备份，然后消耗网络资源来传输备份，从而干扰生产数据库。可以使用资源控制（如果系统支持它们）来限制 CPU 和网络使用，以便备份运行得更慢，而不会损害数据库。

除了识别问题之外，还可以输入工作负载特征来设计模拟基准。如果工作负载测量是平均值，理想情况下您还将收集分布和变化的详细信息。这对于模拟各种预期的工作负载（而不是仅测试平均工作负载）非常重要。有关平均值和变异（标准差）的更多信息，请参阅第 2.8 节“统计”，以及第 12 章“基准测试”。

通过识别负载问题和架构问题，负载分析还有助于将负载问题与架构问题分开。第 2.3.8 节“负载与架构”中介绍了负载与架构。

用于执行工作负载表征的特定工具和指标取决于目标。一些应用程序记录客户端活动的详细日志，这可以作为统计分析的来源。他们还可能已经提供了客户使用情况的每日或每月报告，可以从中挖掘详细信息。

#### 2.5.12 深入分析

深入分析首先从高层次审视一个问题，然后根据之前的发现缩小焦点，丢弃看似无趣的领域，并深入挖掘有趣的领域。该过程可能涉及深入挖掘软件堆栈的更深层次，必要时深入挖掘硬件，以找到问题的根本原因。

以下是系统性能的三阶段深入分析方法\[McDougall 06a\]：

1.  监控：用于随着时间的推移持续记录高级统计数据，并识别可能存在的问题或发出警报。
    
2.  识别：考虑到可疑的问题，这会将调查范围缩小到特定资源或感兴趣的领域，从而识别可能的瓶颈。
    
3.  分析：对特定系统区域进行进一步检查，以尝试找出根本原因并量化问题。
    

可以在公司范围内执行监控，并汇总所有服务器或云实例的结果。实现此目的的一项历史技术是简单网络监控协议 (SNMP)，它可用于监控任何支持该协议的网络连接设备。现代监控系统使用导出器：在每个系统上运行的软件代理来收集和发布指标。生成的数据由监控系统记录并由前端 GUI 可视化。这可能会揭示在短时间内使用命令行工具时可能会错过的长期模式。如果怀疑有问题，许多监控解决方案会发出警报，促使分析进入下一阶段。

通过直接分析服务器并检查系统组件（CPU、磁盘、内存等）来进行识别。历史上，它是使用命令行工具（例如 vmstat(8)、iostat(1) 和 mpstat(1)）执行的。如今，有许多 GUI 仪表板公开相同的指标以实现更快的分析。

分析工具包括基于追踪或分析的工具，用于更深入地检查可疑区域。这种更深入的分析可能涉及创建自定义工具和检查源代码（如果可用）。这是大部分深入研究发生的地方，根据需要剥离软件堆栈的各个层以找到根本原因。在 Linux 上执行此操作的工具包括 strace(1)、perf(1)、BCC 工具、bpftrace 和 Ftrace。

作为此三阶段方法的实施示例，以下是 Netflix 云所使用的技术：

1.  监控：Netflix Atlas：开源全云监控平台 \[Harrington 14\]。
    
2.  识别：Netflix perfdash（以前称为 Netflix Vector）：一个用于使用仪表板分析单个实例的 GUI，包括 USE 方法指标。
    
3.  分析：Netflix FlameCommander，用于生成不同类型的火焰图；以及通过 SSH 会话的命令行工具，包括基于 Ftrace 的工具、BCC 工具和 bpftrace。
    

作为我们如何在 Netflix 使用此序列的示例：Atlas 可能会识别有问题的微服务，perfdash 然后可能会将问题缩小到某个资源，然后 FlameCommander 显示消耗该资源的代码路径，然后可以使用 BCC 工具和自定义工具来检测该资源bpftrace 工具。

##### 五个为什么

在深入分析阶段可以使用的另一种方法是“五个为什么”技术 \[维基百科 20\]：问自己“为什么？”然后回答问题，总共重复五次（或更多）。下面是一个示例过程：

1.  数据库对于许多查询来说已经开始表现不佳。为什么？
    
2.  由于内存分页，它会被磁盘 I/O 延迟。为什么？
    
3.  数据库内存使用量变得太大。为什么？
    
4.  分配器消耗的内存超出了应有的范围。为什么？
    
5.  分配器存在内存碎片问题。
    

这是一个真实的示例，意外地导致了系统内存分配库的修复。正是对核心问题的持续质疑和深入研究才导致了解决。

#### 2.5.13 时延分析

延迟分析检查完成操作所需的时间，然后将其分解为更小的组件，继续细分延迟最高的组件，以便可以识别和量化根本原因。与深入分析类似，延迟分析可以深入软件堆栈的各层以找到延迟问题的根源。

分析可以从应用的工作负载开始，检查应用程序中如何处理该工作负载，然后深入到操作系统库、系统调用、内核和设备驱动程序。

例如，MySQL 查询延迟的分析可能涉及回答以下问题（此处给出了示例答案）：

1.  是否存在查询延迟问题？ （是的）
    
2.  查询时间主要花在 CPU 上还是在 CPU 外等待？ （脱离CPU）
    
3.  等待的off-CPU时间是多少？ （文件系统 I/O）
    
4.  文件系统 I/O 时间是由于磁盘 I/O 还是锁争用造成的？ （磁盘输入/输出）
    
5.  磁盘 I/O 时间主要花在排队或维护 I/O 上吗？ （维修）
    
6.  磁盘服务时间主要是I/O初始化还是数据传输？ （数据传输）
    

对于此示例，该过程的每个步骤都会提出一个问题，将延迟分为两部分，然后继续分析较大的部分：如果愿意的话，可以对延迟进行二分搜索。该过程如图2.14所示。

当 A 或 B 中较慢的一个被识别出来后，它会被进一步分成 A 或 B，进行分析，等等。

#### 2.5.16 基线统计

环境通常使用监控解决方案来记录服务器性能指标，并将其可视化为折线图，时间在 x 轴上（请参阅第 2.9 节，监控）。这些折线图可以显示某个指标最近是否发生了变化，如果是，那么它现在有何不同，只需检查折线中的变化即可。有时会添加额外的行以包含更多历史数据，例如历史平均值或简单的历史时间范围以与当前范围进行比较。例如，许多 Netflix 仪表板会额外绘制一条线来显示前一周的相同时间范围，因此下午 3 点的行为会显示在同一时间范围内。周二可以直接与下午 3 点进行比较。上周二。

这些方法与已经监控的指标和可视化它们的 GUI 配合良好。然而，命令行上还有更多系统指标和详细信息可能无法监控。您可能会遇到不熟悉的系统统计数据，并想知道它们对于服务器来说是否“正常”，或者是否是存在问题的证据。

这不是一个新问题，并且早在使用折线图的监控解决方案广泛使用之前就有一种解决该问题的方法。它是基线统计数据的收集。这可能涉及在系统处于“正常”负载时收集所有系统指标，并将它们记录在文本文件或数据库中以供以后参考。基线软件可以是运行可观察性工具并收集其他源（/proc 文件的 cat(1)）的 shell 脚本。分析器和跟踪工具可以包含在基线中，提供比监控产品通常记录的更多详细信息（但要小心这些工具的开销，以免干扰生产）。可以定期（每天）以及系统或应用程序更改之前和之后收集这些基线，以便可以分析性能差异。

如果尚未收集基线并且无法进行监视，则某些可观察性工具（基于内核计数器的工具）可以显示自启动以来的平均值摘要，以便与当前活动进行比较。这很粗糙，但总比没有好。

#### 2.5.17 静态性能调优

静态性能调优重点关注已配置架构的问题。其他方法关注所施加负载的性能：动态性能 \[Elling 00\]。静态性能分析可以在系统静止且未施加负载时进行。

对于静态性能分析和调整，请逐步检查系统的所有组件并检查以下内容：

*   该组件有意义吗？ （过时、动力不足等）
    
*   该配置对于预期工作负载是否有意义？
    
*   组件是否已自动配置为适合预期工作负载的最佳状态？
    
*   组件是否遇到了错误，导致其现在处于降级状态？
    

以下是使用静态性能调整可能发现的一些问题示例：

*   网络接口协商：选择 1 Gbits/s 而不是 10 Gbit/s
    
*   RAID 池中出现故障的磁盘
    
*   使用旧版本的操作系统、应用程序或固件
    
*   文件系统几乎已满（可能导致性能问题）
    
*   文件系统记录大小与工作负载 I/O 大小不匹配
    
*   以昂贵的调试模式运行的应用程序意外地保持启用状态
    
*   服务器意外配置为网络路由器（启用 IP 转发）
    
*   服务器配置为使用来自远程数据中心而不是本地的资源，例如身份验证
    

幸运的是，这些类型的问题很容易检查；困难的部分是记住去做！

#### 2.5.18 缓存调优

应用程序和操作系统可以采用多个缓存来提高从应用程序到磁盘的 I/O 性能。有关完整列表，请参阅第 3 章“操作系统”、第 3.2.11 节“缓存”。以下是调整每个缓存级别的一般策略：

1.  目标是在堆栈中尽可能高地缓存，更靠近执行工作的位置，从而减少缓存命中的操作开销。该位置还应该有更多可用的元数据，可用于改进缓存保留策略。
    
2.  检查缓存是否已启用并且正在工作。
    
3.  检查缓存命中/未命中率和未命中率。
    
4.  如果缓存大小是动态的，请检查其当前大小。
    
5.  针对工作负载调整缓存。此任务取决于可用的缓存可调参数。
    
6.  调整缓存的工作负载。这样做包括减少不必要的缓存消耗，从而为目标工作负载释放更多空间。
    

留意双重缓存，例如，两个不同的缓存消耗主内存并将相同的数据缓存两次。

还要考虑每个级别的缓存调整的整体性能增益。调整 CPU 1 级缓存可能会节省纳秒，因为缓存未命中可能会由 2 级来处理。但改进 CPU 3 级缓存可能会避免更慢的 DRAM 访问，并带来更大的整体性能增益。 （这些 CPU 缓存在第 6 章 CPU 中进行了描述。）

#### 2.5.19 微基准测试

微基准测试测试简单和人工工作负载的性能。这与宏观基准测试（或行业基准测试）不同，后者通常旨在测试真实世界和自然的工作负载。宏观基准测试是通过运行工作负载模拟来执行的，并且执行和理解可能会变得复杂。

由于起作用的因素较少，微基准测试的执行和理解也更加简单。常用的微基准是 Linux iperf(1)，它执行 TCP 吞吐量测试：这可以通过在生产工作负载期间检查 TCP 计数器来识别外部网络瓶颈（否则很难发现）。

微基准测试可以通过应用工作负载并测量其性能的微基准测试工具来执行，或者您可以使用仅应用工作负载的负载生成器工具，将性能测量留给其他可观察性工具（示例负载生成器在第 1 章中） 12，基准测试，第 12.2.2 节，模拟）。两种方法都可以，但使用微基准测试工具并使用其他工具双重检查性能可能是最安全的。

微基准的一些示例目标（包括测试的第二个维度）是：

*   系统调用时间：fork(2)、execve(2)、open(2)、read(2)、close(2)
    
*   文件系统读取：从缓存文件中读取大小从 1 字节到 1 MB
    
*   网络吞吐量：在 TCP 端点之间传输数据，适用于不同的套接字缓冲区大小
    

微基准测试通常会尽快执行目标操作，并测量大量此类操作完成的时间。然后可以计算平均时间（平均时间=运行时间/操作次数）。

后面的章节包括具体的微基准测试方法，列出了要测试的目标和属性。第 12 章“基准测试”更详细地介绍了基准测试主题。

#### 2.5.20 性能咒语

这是一种调整方法，展示了如何最好地提高性能，并按从最有效到最不有效的顺序列出了可操作的项目。这是：

1.  不要这样做。
    
2.  做一次，但不要再做一次。
    
3.  少做一点吧
    
4.  稍后再做。
    
5.  当他们不注意的时候就做。
    
6.  同时进行。
    
7.  做起来更便宜。
    

以下是每项的一些示例：

1.  不要这样做：消除不必要的工作。
    
2.  做一次，但不要再做一次：缓存。
    
3.  少做：减少刷新、轮询或更新的频率。
    
4.  稍后再做：回写缓存。
    
5.  在他们不注意的时候做：安排工作在非高峰时段进行。
    
6.  并发执行：从单线程切换到多线程。
    
7.  更便宜地做到这一点：购买更快的硬件。
    

这是我最喜欢的方法之一，是我从 Netflix 的斯科特·埃蒙斯那里学到的。他将其归功于克雷格·汉森和帕特·克雷恩（尽管我还没有找到已发表的参考文献）。

### 2.6 建模

系统的分析建模可用于各种目的，特别是可扩展性分析：研究性能如何随着负载或资源的增加而扩展。资源可以是硬件（例如CPU核心）或软件（进程或线程）。

分析建模可以被认为是第三种类型的性能评估活动，另外还有生产系统的可观察性（“测量”）和实验测试（“模拟”）\[Jain 91\]。当至少执行其中两项活动时，才能最好地理解性能：分析建模和模拟，或模拟和测量。

如果分析是针对现有系统，您可以从测量开始：表征负载和产生的性能。如果系统尚未具有生产负载，或者为了测试超出生产中所见的工作负载，则可以通过测试工作负载模拟来进行实验分析。分析建模可用于预测性能，并且可以基于测量或模拟的结果。

可扩展性分析可能会揭示，由于资源限制，性能在特定点停止线性扩展。这被称为拐点：当一个功能切换到另一个功能时，在这种情况下，从线性缩放到争用。找出这些点是否存在以及在哪里，可以指导调查抑制可扩展性的性能问题，以便在生产中遇到这些问题之前将其修复。

有关这些步骤的更多信息，请参阅第 2.5.11 节“工作负载特征”和第 2.5.19 节“微基准测试”。

#### 2.6.1 企业与云

虽然建模使我们能够模拟大型企业系统，而无需花费拥有一个系统的费用，但大规模环境的性能通常很复杂，并且难以准确建模。

借助云计算，任何规模的环境都可以短期租用——基准测试的长度。无需创建数学模型来预测性能，而是可以对工作负载进行表征、模拟，然后在不同规模的云上进行测试。一些发现（例如拐点）可能是相同的，但现在将基于测量数据而不是理论模型，并且通过测试真实环境，您可能会发现模型中未包含的限制因素。

*   线性可扩展性：随着资源的扩展，性能成比例增加。这可能不会永远持续下去，而可能是另一种可扩展性模式的早期阶段。
    
*   争用：架构的某些组件是共享的，只能串行使用，对这些共享资源的争用开始降低扩展的有效性。
    
*   一致性：维持数据一致性（包括传播变更）的成本开始超过扩展带来的好处。
    
*   拐点：在可扩展性点遇到改变可扩展性配置文件的因素。
    
*   可扩展性上限：已达到硬限制。这可能是设备瓶颈，例如达到最大吞吐量的总线或互连，或软件施加的限制（系统资源控制）。
    

虽然视觉识别既简单又有效，但您可以通过使用数学模型了解有关系统可扩展性的更多信息。模型可能会以意想不到的方式偏离数据，这对于调查可能很有用：要么模型存在问题，从而影响您对系统的理解，要么问题出在系统的实际可扩展性上。接下来的部分介绍阿姆达尔可扩展性定律、通用可扩展性定律和排队论。

#### 2.6.3 阿姆达尔可扩展性定律

该定律以计算机架构师 Gene Amdahl \[Amdahl 67\] 命名，对系统可扩展性进行建模，考虑到不能并行扩展的工作负载的串行组件。它可用于研究 CPU、线程、工作负载等的扩展。

阿姆达尔可扩展性定律在早期的可扩展性配置文件中显示为争用，它描述了串行资源或工作负载组件的争用。它可以定义为\[Gunther 97\]：

C(N) = N/(1 + α(N – 1))

相对容量为C(N)，N是缩放维度，例如CPU数量或用户负载。 α 参数（其中 0 <= α <= 1）表示序列性程度，以及它如何偏离线性可扩展性。

可以通过以下步骤应用阿姆达尔可扩展性定律：

1.  通过观察现有系统或使用微基准测试或负载生成器进行实验，收集 N 范围内的数据。
    
2.  进行回归分析以确定Amdahl参数（α）；这可以使用 gnuplot 或 R 等统计软件来完成。
    
3.  呈现结果进行分析。收集的数据点可以与模型函数一起绘制，以预测缩放并揭示数据和模型之间的差异。这也可以使用 gnuplot 或 R 来完成。
    

以下是阿姆达尔可扩展性定律回归分析的示例 gnuplot 代码，旨在让您了解如何执行此步骤：

点击这里查看代码图片

inputN = 10                     # rows to include as model input
alpha = 0.1                     # starting point (seed)
amdahl(N) = N1 \* N/(1 + alpha \* (N - 1))
# regression analysis (non-linear least squares fitting)
fit amdahl(x) filename every ::1::inputN using 1:2 via alpha

在 R 中处理此问题需要类似数量的代码，其中涉及用于非线性最小二乘拟合的 nls() 函数来计算系数，然后在绘图期间使用该系数。有关 gnuplot 和 R 中的完整代码，请参阅本章末尾参考文献中的性能可扩展性模型工具包 \[Gregg 14a\]。

#### 2.6.4 通用可扩展性法则

通用可扩展性定律 (USL)，以前称为超级串行模型 \[Gunther 97\]，由 Neil Gunther 博士开发，其中包含相干延迟参数。这在前面被描述为一致性可伸缩性配置文件，其中包括争用的影响。

USL 可以定义为：

C(N) = N/(1 + α(N – 1) + βN(N – 1))

C(N)、N 和 α 与阿姆达尔可扩展性定律相同。 β 是相干参数。当 β == 0 时，这成为阿姆达尔可扩展性定律。

#### 2.6.5 排队理论

排队论是对队列系统的数学研究，提供了分析队列长度、等待时间（延迟）和利用率（基于时间）的方法。计算中的许多组件（无论是软件还是硬件）都可以建模为排队系统。多个排队系统的建模称为排队网络。

本节总结了排队论的作用并提供了一个示例来帮助您理解该作用。这是一个很大的研究领域，其他文本中有详细介绍\[Jain 91\]\[Gunther 97\]。

排队论建立在数学和统计学的各个领域之上，包括概率分布、随机过程、Erlang 的 C 公式（Agner Krarup Erlang 发明了排队论）和 Little 定律。利特尔定律可以表示为

L = λW

它确定系统中的平均请求数 L，即平均到达率 λ 乘以系统中的平均请求时间 W。这可以应用于队列，使得 L 是系统中的请求数队列，W 是队列的平均等待时间。

排队系统可用于回答各种问题，包括：

*   如果负载加倍，平均响应时间是多少？
    
*   添加额外处理器后对平均响应时间有何影响？
    
*   当负载加倍时，系统能否提供低于 100 毫秒的 90% 响应时间？
    

除了响应时间之外，还可以研究其他因素，包括利用率、队列长度和常驻作业数量。

一个简单的排队系统模型如图2.18所示。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/02fig18.jpg)

图2.18 排队模型

它有一个处理队列中的作业的服务中心。排队系统可以有多个并行处理工作的服务中心。在排队论中，服务中心通常称为服务器。

排队系统可以根据三个因素进行分类：

*   到达过程：这描述了对排队系统的请求的到达间隔时间，该时间可以是随机的、固定时间的或诸如泊松（其使用到达时间的指数分布）之类的过程。
    
*   服务时间分布：描述服务中心的服务时间。它们可以是固定的（确定性的）、指数的或其他分布类型。
    
*   服务中心数量：一个或多个。
    

这些因子可以用肯德尔符号表示。

##### 肯德尔的符号

该表示法为每个属性分配代码。它具有以下形式

A/S/米

它们是到达过程（A）、服务时间分布（S）和服务中心数量（m）。肯德尔符号还有一种扩展形式，其中包含更多因素：系统中缓冲区的数量、人口规模和服务纪律。

经常研究的排队系统的例子是

*   M/M/1：马尔可夫到达（指数分布到达时间）、马尔可夫服务时间（指数分布）、1个服务中心
    
*   M/M/c：与 M/M/1 相同，但多服务器
    
*   M/G/1：马尔可夫到达，服务时间的一般分布（任意），一个服务中心
    
*   M/D/1：马尔可夫到达，确定性服务时间（固定），一个服务中心
    

M/G/1 通常用于研究旋转硬盘的性能。

##### M/D/1 和 60% 利用率

作为排队论的一个简单示例，请考虑确定性地响应工作负载的磁盘（这是一种简化）。型号为M/D/1。

提出的问题是：随着磁盘利用率的增加，磁盘的响应时间如何变化？

排队论允许计算 M/D/1 的响应时间：

r = s(2 - ρ)/2(1 - ρ)

其中响应时间 r 根据服务时间 s 和利用率 ρ 定义。

对于 1 毫秒的服务时间和从 0 到 100% 的利用率，这种关系如图 2.19 所示。

利用率超过 60% 时，平均响应时间会加倍。 80%，增加了两倍。由于磁盘 I/O 延迟通常是应用程序的限制资源，因此平均延迟增加一倍或更高可能会对应用程序性能产生显着的负面影响。这就是为什么磁盘利用率在达到 100% 之前就可能成为问题的原因，因为它是一个排队系统，请求（通常）不能被中断并且必须等待轮到它们。例如，这与 CPU 不同，在 CPU 中，较高优先级的工作可以抢占。

### 2.7 容量规划

容量规划检查系统处理负载的能力以及系统如何随着负载的扩展而扩展。它可以通过多种方式执行，包括研究资源限制和因素分析（如本文所述）以及建模（如前所述）。本节还包括扩展解决方案，包括负载均衡器和分片。有关此主题的更多信息，请参阅容量规划的艺术 \[Allspaw 08\]。

对于特定应用程序的容量规划，有一个量化的性能目标来规划是有帮助的。确定这一点已在第 5 章“应用”中讨论。

#### 2.7.1 资源限制

该方法是搜索在负载下将成为瓶颈的资源。对于容器来说，资源可能会遇到软件施加的限制，从而成为瓶颈。该方法的步骤是：

1.  测量服务器请求的速率，并随时间监控该速率。
    
2.  测量硬件和软件资源的使用情况。随着时间的推移监控这个比率。
    
3.  根据使用的资源表达服务器请求。
    
4.  将服务器请求推断为每个资源的已知（或实验确定的）限制。
    

首先确定服务器的角色及其所服务的请求类型。例如，Web 服务器服务于 HTTP 请求，网络文件系统 (NFS) 服务器服务于 NFS 协议请求（操作），数据库服务器服务于查询请求（或命令请求，查询是其子集）。

下一步是确定每个请求的系统资源消耗。对于现有系统，可以测量当前的请求率以及资源利用率。然后可以使用外推法来查看哪个资源将首先达到 100% 利用率，以及请求率是多少。

对于未来的系统，可以使用微基准测试或负载生成工具来模拟测试环境中的预期请求，同时测量资源利用率。如果有足够的客户端负载，您也许能够通过实验找到限制。

要监控的资源包括：

*   硬件：CPU利用率、内存使用率、磁盘IOPS、磁盘吞吐量、磁盘容量（使用的卷）、网络吞吐量
    
*   软件：虚拟内存使用、进程/任务/线程、文件描述符
    

假设您正在查看当前每秒执行 1,000 个请求的现有系统。最繁忙的资源是 16 个 CPU，平均利用率为 40%；您预测，一旦 100% 得到利用，它们将成为该工作负载的瓶颈。问题是：此时每秒的请求率是多少？

每个请求的 CPU% = 总 CPU%/请求 = 16 × 40%/1,000 = 每个请求的 0.64% CPU

最大请求数/秒 = 100% × 16 个 CPU/每个请求的 CPU% = 1,600 / 0.64 = 2,500 个请求/秒

预测为 2,500 个请求/秒，此时 CPU 将 100% 繁忙。这是对容量的粗略最佳情况估计，因为在请求达到该速率之前可能会遇到一些其他限制因素。

此练习仅使用一个数据点：应用程序吞吐量（每秒请求数）为 1,000，设备利用率为 40%。如果启用了随时间的监控，则可以包括不同吞吐量和利用率的多个数据点，以提高估计的准确性。图 2.20 说明了处理这些数据并推断最大应用程序吞吐量的可视化方法。

2,500 个请求/秒够吗？回答这个问题需要了解峰值工作负载是什么，这会显示在日常访问模式中。对于您长期监控的现有系统，您可能已经了解峰值的样子。

考虑一个每天处理 100,000 次网站点击的 Web 服务器。这听起来可能很多，但平均而言只有一个请求——不多。然而，100,000 次网站点击中的大部分可能发生在新内容发布后的几秒钟内，因此峰值非常显着。

#### 2.7.2 因素分析

购买和部署新系统时，通常需要改变许多因素才能达到所需的性能。这些可能包括改变磁盘和 CPU 的数量、RAM 的数量、闪存设备的使用、RAID 配置、文件系统设置等。任务通常是以最小的成本实现所需的性能。

测试所有组合将确定哪种组合具有最佳性价比；然而，这很快就会失控：八个二元因素将需要 256 次测试。

解决方案是测试一组有限的组合。这是一种基于了解最大系统配置的方法：

1.  将所有因素配置为最大值来测试性能。
    
2.  一项一项地改变因素，测试性能（每个因素都应该下降）。
    
3.  根据测量结果以及成本节省，将性能下降的百分比归因于每个因素。
    
4.  从最大性能（和成本）开始，选择因素来节省成本，同时根据综合性能下降维持每秒所需的请求数。
    
5.  重新测试计算的配置以确认交付的性能。
    

对于八因素系统，这种方法可能只需要十次测试。

例如，考虑新存储系统的容量规划，要求 1 GB/s 读取吞吐量和 200 GB 工作集大小。最大配置达到 2 GB/s，包括四个处理器、256 GB DRAM、2 个双端口 10 GbE 网卡、巨型帧，并且不启用压缩或加密（激活成本高昂）。切换到两个处理器会导致性能降低 30%，一张网卡性能会降低 25%，非巨型性能会降低 35%，加密会降低 10%，压缩会降低 40%，并且 DRAM 会减少 90%，因为工作负载不再需要完全缓存。考虑到这些性能下降及其已知的节省，现在可以计算出满足要求的最佳性价比系统；它可能是带有一个网卡的双处理器系统，可满足所需的吞吐量：估计为 2 × (1 – 0.30) × (1 – 0.25) = 1.04 GB/s。然后，明智的做法是测试此配置，以防这些组件在一起使用时表现与其预期性能不同。

#### 2.7.3 扩容解决方案

满足更高的性能需求通常意味着更大的系统，这种策略称为垂直扩展。将负载分散到多个系统（通常由称为负载均衡器的系统负责，使它们全部显示为一个系统）称为水平扩展。

云计算通过构建更小的虚拟化系统而不是整个系统，进一步实现水平扩展。这在购买计算来处理所需负载时提供了更细的粒度，并允许以小而高效的增量进行扩展。由于不需要像企业大型机那样进行初始大量采购（包括支持合同承诺），因此在项目的早期阶段不需要进行严格的容量规划。

有一些技术可以根据性能指标自动进行云扩展。为此的 AWS 技术称为自动扩展组 (ASG)。可以创建自定义扩展策略，以根据使用指标增加和减少实例数量。如图 2.21 所示。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/02fig21.jpg)

图2.21 弹性伸缩组

Netflix 通常使用的 ASG 的目标是 CPU 利用率为 60%，并且会随着负载的增加和减少来维持该目标。

容器编排系统还可以提供对自动扩展的支持。例如，Kubernetes 提供水平 Pod 自动缩放器 (HPA)，可以根据 CPU 利用率或其他自定义指标来缩放 Pod（容器）数量 \[Kubernetes 20a\]。

对于数据库，常见的扩展策略是分片，其中数据被分成逻辑组件，每个组件由自己的数据库（或冗余数据库组）管理。例如，可以通过将客户名称分成按字母顺序排列的范围来将客户数据库分成多个部分。选择有效的分片键对于在数据库之间均匀分布负载至关重要。

### 2.8 统计

充分了解如何使用统计数据及其局限性非常重要。本节讨论使用统计数据（指标）和统计类型（包括平均值、标准差和百分位数）来量化性能问题。

#### 2.8.1 量化性能增益

量化问题以及解决这些问题的潜在性能改进可以对它们进行比较并确定优先级。该任务可以通过观察或实验来完成。

##### 基于观察

使用观察来量化性能问题：

1.  选择一个可靠的指标。
    
2.  估计解决问题后的性能增益。
    

例如：

*   观察到：应用程序请求需要 10 毫秒。
    
*   观察结果：其中，9 毫秒是磁盘 I/O。
    
*   建议：将应用程序配置为在内存中缓存 I/O，预计 DRAM 延迟约为 10 μs。
    
*   估计增益：10 ms → 1.01 ms (10 ms - 9 ms + 10 μs) = ~9x 增益。
    

正如第 2.3 节概念中所介绍的，延迟（时间）非常适合于此，因为它可以在组件之间直接进行比较，这使得这样的计算成为可能。

使用延迟时，请确保将其作为应用程序请求的同步组件进行测量。有些事件是异步发生的，例如后台磁盘 I/O（写入刷新到磁盘），并且不会直接影响应用程序性能。

##### 基于实验的

通过实验量化性能问题：

1.  应用修复。
    
2.  使用可靠的指标之前和之后进行量化。
    

例如：

*   观察结果：应用程序事务延迟平均为 10 毫秒。
    
*   实验：增加应用程序线程数以允许更多并发而不是排队。
    
*   观察结果：应用程序事务延迟平均为 2 毫秒。
    
*   增益：10 ms → 2 ms = 5x。
    

如果在生产环境中尝试修复成本高昂，则此方法可能不合适！对于这种情况，应使用基于观察的方法。

#### 2.8.2 平均值

平均值通过单个值表示数据集：集中趋势指数。最常用的平均值类型是算术平均值（或简称平均值），它是值的总和除以值的计数。其他类型包括几何平均数和调和平均数。

##### 几何平均数

几何平均值是相乘值的 n 次方根（其中 n 是值的计数）。 \[Jain 91\]对此进行了描述，其中包括一个使用它进行网络性能分析的示例：如果单独测量内核网络堆栈每一层的性能改进，那么平均性能改进是多少？由于各层在同一个数据包上协同工作，因此性能改进具有“乘法”效应，这可以通过几何平均值来最好地概括。

##### 调和平均值

调和平均值是值的计数除以它们的倒数之和。取速率平均值更为合适，例如，计算 800 MB 数据的平均传输速率，此时前 100 Mbytes 将以 50 Mbytes/s 的速度发送，其余 700 Mbytes 以 10 Mbytes 的节流速率发送/s。使用调和平均值得出的答案是 800/(100/50 + 700/10) = 11.1 Mbytes/s。

##### 一段时间内的平均值

对于性能，我们研究的许多指标都是一段时间内的平均值。 CPU 永远不会“利用率达到 50%”；它已在某个时间间隔（可以是一秒、一分钟或一小时）的 50% 内被使用。每当考虑平均值时，检查间隔非常重要。

例如，我遇到了一个问题，客户遇到了由 CPU 饱和（调度程序延迟）引起的性能问题，尽管他们的监控工具显示 CPU 利用率从未高于 80%。监控工具报告 5 分钟平均值，掩盖了 CPU 利用率一次达到 100% 达几秒的时间段。

##### 衰减平均值

有时在系统性能中使用衰减平均值。一个例子是由各种工具（包括正常运行时间）报告的系统“平均负载”(1)。

衰减平均值仍然是在一段时间间隔内测量的，但最近时间的权重比更远的过去时间的权重更大。这减少（抑制）平均值的短期波动。

有关详细信息，请参阅第 6 章“CPU”中的“负载平均值”、第 6.6 节“可观测性工具”。

##### 局限性

平均值是隐藏细节的汇总统计数据。我分析过许多磁盘 I/O 延迟异常值偶尔超过 100 毫秒的案例，而平均延迟接近 1 毫秒。为了更好地理解数据，您可以使用第 2.8.3 节“标准差、百分位数、中位数”（下一节）中介绍的其他统计数据以及第 2.10 节“可视化”中介绍的可视化效果。

#### 2.8.3 标准差、百分位数、中位数

标准差和百分位数（例如第 99 个百分位数）是提供数据分布信息的统计技术。标准差是方差的度量，值越大表示与平均值的方差越大。第 99 个百分位数显示分布中包含 99% 值的点。图 2.22 显示了这些正态分布，以及最小值和最大值。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/02fig22.jpg)

图2.22 统计值

第 90、95、99 和 99.9 等百分位数用于请求延迟的性能监控，以量化总体中最慢的。这些也可以在服务级别协议 (SLA) 中指定，作为衡量大多数用户可接受的性能的一种方式。

第 50 个百分位（称为中位数）可以通过检查来显示大部分数据的位置。

#### 2.8.4 变异系数

由于标准差是相对于平均值的，因此只有同时考虑标准差和平均值才能理解方差。仅 50 的标准差告诉我们的信息很少。再加上 200 的平均值告诉了我们很多信息。

有一种方法可以将变异表达为单一指标：标准差与平均值的比率，称为变异系数（CoV 或 CV）。对于本例，CV 为 25%。 CV 越低意味着方差越小。

作为单一指标的方差的另一种表达是 z 值，它是一个值与平均值的标准差。

#### 2.8.5 多峰分布

均值、标准差和百分位数存在问题，从上一张图表中可以明显看出：它们适用于类正态分布或单峰分布。系统性能通常是双峰的，快速代码路径返回低延迟，慢速代码路径返回高延迟，或者缓存命中返回低延迟，缓存未命中返回高延迟。也可以有两种以上的模式。

图 2.23 显示了读写混合工作负载（包括随机 I/O 和顺序 I/O）的磁盘 I/O 延迟分布。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/02fig23.jpg)

图2.23 延迟分布

它以直方图的形式呈现，显示两种模式。左侧的模式显示延迟小于 1 毫秒，适用于磁盘缓存命中。右侧峰值约为 7 毫秒，用于磁盘缓存未命中：随机读取。平均 I/O 延迟为 3.3 毫秒，绘制为垂直线。该平均值不是集中趋势指数（如前所述）；事实上，情况几乎相反。作为一种衡量标准，该分布的平均值具有严重的误导性。

#### 2.8.6 异常值

另一个统计问题是异常值的存在：极少数极高或极低的值似乎不符合预期分布（单模或多模）。

磁盘 I/O 延迟异常值就是一个例子，当大多数磁盘 I/O 介于 0 到 10 毫秒之间时，非常偶然的磁盘 I/O 可能会花费超过 1,000 毫秒。像这样的延迟异常值可能会导致严重的性能问题，但除了最大值之外，很难从大多数指标类型中识别它们的存在。另一个例子是基于 TCP 计时器的重传导致的网络 I/O 延迟异常值。

对于正态分布，异常值的存在可能会稍微改变均值，但不会改变中位数（考虑这一点可能很有用）。标准差和 99% 有更好的机会识别异常值，但这仍然取决于它们的频率。

为了更好地理解多模态分布、异常值和其他复杂但常见的行为，请检查完整的分布，例如使用直方图。有关执行此操作的更多方法，请参阅第 2.10 节“可视化”。

### 2.9 监控

系统性能监控记录一段时间内（时间序列）的性能统计数据，以便可以将过去与现在进行比较，并可以识别基于时间的使用模式。这对于容量规划、量化增长和显示峰值使用情况非常有用。历史值还可以通过显示过去的“正常”范围和平均值，为理解绩效指标的当前值提供背景。

基于时间的模式示例如图 2.24、2.25 和 2.26 所示，其中绘制了文件系统在不同时间间隔从云计算服务器读取数据的情况。

这些图表显示了每天的模式，上午 8 点左右开始上升，下午略有下降，然后在夜间衰减。长期图表显示周末的活动较少。 30 日图表中还可以看到一些短期峰值。

包括图中所示的各种行为周期通常可以在历史数据中看到，包括：

*   每小时：应用程序环境中的活动可能每小时发生一次，例如监视和报告任务。这些以 5 或 10 分钟为周期执行也很常见。
    
*   每日：可能存在与工作时间（上午 9 点至下午 5 点）一致的每日使用模式，如果服务器适用于多个时区，则可能会延长。对于互联网服务器，当全球用户活跃时，可能会遵循这种模式。其他日常活动可能包括夜间日志轮换和备份。
    
*   每周：除了每日模式外，还可能存在基于工作日和周末的每周模式。
    
*   季度：财务报告按季度完成。
    
*   每年：每年的负荷模式可能取决于学校时间表和假期。
    

其他活动可能会导致负载不规则增加，例如在网站上发布新内容和销售（美国的黑色星期五/网络星期一）。由于外部活动，例如大面积停电或互联网中断以及体育决赛（每个人都观看比赛而不是使用您的产品），可能会出现负载不规律的下降。[^6]

[^6]当我参加 Netflix SRE 值班轮换时，我学习了一些针对这些案例的非传统分析工具：检查社交媒体是否有可疑的停电情况，以及在球队聊天室询问是否有人知道体育决赛。

#### 2.9.2 监控产品

有许多用于系统性能监控的第三方产品。典型功能包括归档数据并将其呈现为基于浏览器的交互式图表，以及提供可配置的警报。

其中一些通过在系统上运行代理（也称为导出程序）来收集统计数据来操作。这些代理要么执行操作系统可观察性工具（例如 iostat(1) 或 sar(1)）并解析输出文本（这被认为效率低下），要么直接从操作系统库和内核接口读取。监控产品支持一组自定义代理，用于从特定目标导出统计信息：Web 服务器、数据库和语言运行时。

随着系统变得更加分布式以及云计算的使用不断增长，您将更频繁地需要监控大量系统：数百、数千甚至更多。这就是集中监控产品特别有用的地方，它允许从一个界面监控整个环境。

举一个具体的例子：Netflix 云由超过 200,000 个实例组成，并使用 Atlas 云范围监控工具进行监控，该工具是 Netflix 定制构建的，用于在这种规模下运行，并且是开源的 \[Harrington 14\]。其他监控产品将在第 4 章“可观测性工具”、第 4.2.4 节“监控”中讨论。

#### 2.9.3 启动后总结

如果尚未执行监控，请检查操作系统是否至少提供了summary-since-boot值，该值可用于与当前值进行比较。

### 2.10 可视化

可视化允许检查比容易理解（或有时甚至显示）为文本的数据更多的数据。它们还支持模式识别和模式匹配。这可能是识别不同指标源之间的相关性的有效方法，这可能很难以编程方式完成，但很容易以视觉方式完成。

#### 2.10.1 折线图

折线图（也称为折线图）是一种众所周知的基本可视化效果。它通常用于检查一段时间内的性能指标，在 x 轴上显示时间的流逝。

#### 2.10.3 热图

热图（更准确地称为列量化）可以通过将 x 和 y 范围量化为称为桶的组来解决散点图可扩展性问题。它们显示为大像素，并根据该 x 和 y 范围内的事件数量进行着色。这种量化还解决了散点图视觉密度限制，允许热图以相同的方式显示来自单个系统或数千个系统的数据。热图以前用于定位，例如磁盘偏移（例如，TazTool \[McDougall 06a\]）；我发明了它们在计算延迟、利用率和其他指标方面的用途。延迟热图首次包含在 2008 年发布的 Sun Microsystems ZFS 存储设备分析中 \[Gregg 10a\]\[Gregg 10b\]，现在在 Grafana \[Grafana 20\] 等性能监控产品中很常见。

#### 2.10.6 可视化工具

Unix 性能分析历来侧重于使用基于文本的工具，部分原因是图形支持有限。此类工具可以通过登录会话快速执行并实时报告数据。可视化的访问更加耗时，并且通常需要跟踪和报告周期。在处理紧急性能问题时，访问指标的速度可能至关重要。

现代可视化工具提供系统性能的实时视图，可通过浏览器和移动设备访问。有许多产品可以执行此操作，其中许多产品可以监控整个云。第 1 章“简介”、第 1.7.1 节“计数器、统计数据和指标”包括来自此类产品 Grafana 的示例屏幕截图，第 4 章“可观察性工具”第 4.2.4 节“监控”中讨论了其他监控产品。
