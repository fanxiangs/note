
系统主内存存储应用程序和内核指令、它们的工作数据以及文件系统缓存。该数据的辅助存储通常是存储设备（磁盘），其运行速度要慢几个数量级。一旦主存储器已满，系统就可以开始在主存储器和存储设备之间切换数据。这是一个缓慢的过程，通常会成为系统瓶颈，从而显着降低性能。系统还可能终止最大消耗内存的进程，导致应用程序中断。

其他需要考虑的性能因素包括分配和释放内存、复制内存以及管理内存地址空间映射的 CPU 开销。在多套接字架构上，内存局部性可能成为一个因素，因为连接到本地套接字的内存比远程套接字具有更低的访问延迟。

本章的学习目标是：

*   了解内存概念。
*   熟悉内存硬件内部结构。
*   熟悉内核和用户分配器的内部结构。
*   具备 MMU 和 TLB 的应用知识。
*   遵循不同的内存分析方法。
*   描述系统范围和每个进程的内存使用情况。
*   识别由可用内存不足引起的问题。
*   定位进程地址空间和内核板中的内存使用情况。
*   使用分析器、跟踪器和火焰图调查内存使用情况。
*   了解内存的可调参数。

本章分为五个部分，前三个部分提供内存分析的基础，后两个部分展示其在基于 Linux 的系统中的实际应用。各部分如下：

*   背景介绍内存相关术语和关键内存性能概念。

## 7.1 术语

作为参考，本章中使用的与内存相关的术语包括以下内容：

*   主存储器：也称为物理存储器，描述计算机的快速数据存储区域，通常称为 DRAM。
*   虚拟内存：主内存的抽象，（几乎）无限且无竞争。虚拟内存不是真实内存。
*   常驻内存：当前驻留在主存中的内存。
*   匿名内存：没有文件系统位置或路径名的内存。它包括进程地址空间（称为堆）的工作数据。
*   地址空间：内存上下文。每个进程和内核都有虚拟地址空间。
*   段：标记为特定用途的虚拟内存区域，例如用于存储可执行或可写页面。
*   指令文本：指内存中的CPU指令，通常在一个段中。
*   OOM：内存不足，当内核检测到可用内存不足时。
*   页：操作系统和 CPU 使用的内存单位。从历史上看，它是 4 或 8 KB。现代处理器具有对更大尺寸的多种页面尺寸支持。
*   页面错误：无效的内存访问。这些是使用按需虚拟内存时的正常现象。
*   分页：主存和存储设备之间的页面传输。
*   交换：Linux 使用术语交换来指代交换设备的匿名分页（交换页的传输）。在 Unix 和其他操作系统中，交换是主内存和交换设备之间整个进程的传输。本书使用该术语的 Linux 版本。
*   交换：用于分页匿名数据的磁盘区域。它可能是存储设备上的一个区域，也称为物理交换设备，也可能是文件系统文件，称为交换文件。有些工具使用术语“交换”来指代虚拟内存（这是令人困惑且不正确的）。

本章将介绍其他术语。术语表包括供需要时参考的基本术语，包括地址、缓冲区和 DRAM。另请参阅第 2 章和第 3 章中的术语部分。

### 7.2 概念

以下是有关内存和内存性能的一些重要概念。

### 7.2.1 Virtual Memory

虚拟内存是一种抽象，它为每个进程和内核提供自己的大型、线性和私有地址空间。它简化了软件开发，将物理内存放置留给操作系统来管理。它还支持多任务处理（虚拟地址空间通过设计分开）和超额订阅（使用中的内存可以扩展到主内存之外）。虚拟内存在第 3 章“操作系统”第 3.2.8 节“虚拟内存”中介绍。有关历史背景，请参阅\[Denning 70\]。

图 7.1 显示了在具有交换设备（辅助存储）的系统上虚拟内存对于进程的作用。显示了内存页面，因为大多数虚拟内存实现都是基于页面的。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/07fig01.jpg)

图7.1 进程虚拟内存

进程地址空间由虚拟内存子系统映射到主内存和物理交换设备。内核可以根据需要在它们之间移动内存页面，Linux 称为交换（其他操作系统称为匿名分页）。这允许内核超额订阅主内存。

内核可能会对超额订阅施加限制。常用的限制是主内存加上物理交换设备的大小。尝试超出此限制的内核分配可能会失败。这种“虚拟内存不足”错误乍一看可能会令人困惑，因为虚拟内存本身是一种抽象资源。

Linux 还允许其他行为，包括对内存分配不设限制。这称为过量使用，并在以下有关分页和请求调页的部分之后进行描述，这是过量使用工作所必需的。

### 7.2.2 Paging

分页是将页面移入和移出主存储器，分别称为页入和页出。它于 1962 年首次由 Atlas Computer 引入 \[Corbató 68\]，允许：

*   要执行的部分加载的程序
*   大于主存执行的程序
*   程序在主内存和存储设备之间高效移动

这些能力在今天仍然有效。与早期交换整个程序的技术不同，分页是一种管理和释放主内存的细粒度方法，因为页面大小单位相对较小（例如 4 KB）。

虚拟内存分页（分页虚拟内存）通过 BSD \[Babaoglu 79\] 引入 Unix 并成为标准。

随着后来添加用于共享文件系统页面的页面缓存（请参阅第 8 章，文件系统），两种不同类型的分页变得可用：文件系统分页和匿名分页。

##### 文件系统分页

文件系统分页是由内存映射文件中页面的读写引起的。对于使用文件内存映射 (mmap(2)) 的应用程序和使用页面缓存的文件系统（大多数这样做；请参阅第 8 章，文件系统），这是正常行为。它被称为“好的”寻呼 \[McDougall 06a\]。

当需要时，内核可以通过调出一些内存来释放内存。这就是术语有点棘手的地方：如果主内存中的文件系统页面被修改（称为脏页面），则页面调出将要求将其写入磁盘。相反，如果文件系统页面未被修改（称为清理），则页面调出仅释放内存以供立即重用，因为磁盘上已存在副本。因此，术语“页出”意味着页面被移出内存，这可能包括也可能不包括对存储设备的写入（您可能会在其他文本中看到术语“页出”的不同定义）。

##### Anonymous Paging (Swapping)

匿名分页涉及进程私有的数据：进程堆和堆栈。它被称为匿名是因为它在操作系统中没有命名位置（即没有文件系统路径名）。匿名页面调出需要将数据移动到物理交换设备或交换文件。 Linux 使用术语交换来指代这种类型的分页。

匿名寻呼会损害性能，因此被称为“坏”寻呼 \[McDougall 06a\]。当应用程序访问已调出的内存页时，它们会阻塞将它们读回主内存所需的磁盘 I/O。[^1]这是匿名页调入，会给应用程序带来同步延迟。匿名页面调出可能不会直接影响应用程序性能，因为它们可以由内核异步执行。

[^1]如果使用更快的存储设备作为交换设备，例如具有低于 10 μs 延迟的 3D XPoint，交换可能不会像以前那样“糟糕”分页，而是成为有意扩展主内存的简单方法，一款拥有成熟内核支持的。

当没有匿名分页（交换）时，性能最佳。这可以通过将应用程序配置为保留在可用主内存内并监视页面扫描、内存利用率和匿名分页来实现，以确保没有内存短缺的迹象。

### 7.2.3 请求分页

支持按需调页的操作系统（大多数都支持）按需将虚拟内存页面映射到物理内存，如图 7.2 所示。这会将创建映射的 CPU 开销推迟到实际需要和访问它们时，而不是在首次分配内存范围时。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/07fig02.jpg)

图 7.2 页面错误示例

图 7.2 中所示的序列首先提供分配的内存的 malloc()（步骤 1），然后是指向新分配的内存的存储指令（步骤 2）。为了让 MMU 确定存储的主内存位置，它对内存页面执行虚拟到物理的查找（步骤 3），但由于尚未存在映射而失败。这种失败称为页面错误（步骤 4），它会触发内核创建按需映射（步骤 5）。一段时间后，内存页面可以被调出到交换设备以释放内存（步骤 6）。

如果可以从内存中的另一个页面满足映射，则称为小错误。在进程的内存增长期间，从可用内存映射新页面时可能会发生这种情况（如图所示）。它也可能发生在映射到另一个现有页面时，例如从映射的共享库读取页面。

需要访问存储设备（图中未显示）的页面错误（例如访问未缓存的内存映射文件）称为主要错误。

虚拟内存模型和需求分配的结果是虚拟内存的任意页都可能处于以下状态之一：

1.  未分配
2.  已分配，但未映射（未填充且尚未出现故障）
3.  分配并映射到主内存 (RAM)
4.  分配并映射到物理交换设备（磁盘）

如果页面由于系统内存压力而被调出，则达到状态 (D)。从 (B) 到 (C) 的转换是页错误。如果需要磁盘 I/O，则为主要页面错误；否则，会出现轻微页面错误。

从这些状态中，还可以定义两个内存使用术语：

*   驻留集大小（RSS）：分配的主内存页的大小（C）
*   虚拟内存大小：所有分配区域的大小（B + C + D）

请求分页与分页虚拟内存一起通过 BSD 添加到 Unix。它已成为标准并被 Linux 使用。

### 7.2.4 过度使用

Linux 支持过量使用的概念，它允许分配比系统可能存储的内存更多的内存，比物理内存和交换设备的总和还要多。它依赖于需求分页以及应用程序不使用其分配的大量内存的趋势。

通过过量使用，应用程序对内存的请求（例如，malloc(3)）将会成功，否则它们会失败。应用程序程序员可以慷慨地分配内存，然后根据需要稀疏地使用它，而不是保守地分配内存以保持在虚拟内存限制内。

在 Linux 上，可以使用可调参数配置过量使用的行为。有关详细信息，请参阅第 7.6 节“调整”。过度使用的后果取决于内核如何管理内存压力；请参阅第 7.3 节“架构”中对 OOM 杀手的讨论。

### 7.2.5 进程交换

进程交换是整个进程在主内存和物理交换设备或交换文件之间的移动。这是管理主内存的原始 Unix 技术，也是术语“交换”的起源 \[Thompson 78\]。

要换出进程，必须将其所有私有数据写入交换设备，包括进程堆（匿名数据）、其打开的文件表以及仅在进程处于活动状态时才需要的其他元数据。源自文件系统且未经修改的数据可以在需要时删除并从原始位置再次读取。

进程交换会严重损害性能，因为已换出的进程需要大量磁盘 I/O 才能再次运行。对于当时的机器来说，它在早期的 Unix 上更有意义，例如 PDP-11，它的最大进程大小为 64 KB \[Bach 86\]。 （现代系统允许以 GB 为单位测量进程大小。）

此描述是为了提供历史背景。 Linux 系统根本不交换进程，仅依赖分页。

### 7.2.6 文件系统缓存使用

系统启动后内存使用量增加是正常现象，因为操作系统使用可用内存来缓存文件系统，从而提高性能。原则是：如果有空闲的主内存，就将其用于有用的事情。这可能会让天真的用户感到困扰，因为他们会在启动后发现可用内存缩减到接近于零。但它不会给应用程序带来问题，因为当应用程序需要时，内核应该能够快速从文件系统缓存中释放内存。

有关可以消耗主内存的各种文件系统缓存的更多信息，请参阅第 8 章，文件系统。

### 7.2.7 利用率和饱和度

主内存利用率可以计算为已用内存与总内存之比。文件系统缓存使用的内存可以视为未使用，因为它可供应用程序重用。

如果对内存的需求超过主内存的容量，主内存就会饱和。然后，操作系统可以通过使用分页、进程交换（如果支持）以及 Linux 上的 OOM Killer（稍后描述）来释放内存。任何这些活动都是主内存饱和的指标。

如果系统对其愿意分配的虚拟内存量施加限制（Linux 过量使用则没有），则还可以根据容量利用率来研究虚拟内存。如果是这样，一旦虚拟内存耗尽，内核将无法分配；例如，malloc(3) 失败，errno 设置为 ENOMEM。

请注意，系统上当前可用的虚拟内存有时（令人困惑地）称为可用交换区。

### 7.2.8 分配器

虽然虚拟内存处理物理内存的多任务处理，但虚拟地址空间内的实际分配和放置通常由分配器处理。这些是用户态库或基于内核的例程，它们为软件程序员提供了一个简单的内存使用界面（例如，malloc(3)、free(3)）。

分配器可以对性能产生重大影响，并且系统可以提供多个用户级分配器库以供选择。它们可以通过使用包括每线程对象缓存在内的技术来提高性能，但如果分配变得碎片化和浪费，它们也会损害性能。第 7.3 节“架构”中介绍了具体示例。

### 7.2.9 共享内存

内存可以在进程之间共享。这通常用于系统库，通过与使用它的所有进程共享一份只读指令文本副本来节省内存。

这给显示每个进程主内存使用情况的可观察性工具带来了困难。报告进程的总内存大小时是否应该包括共享内存？ Linux 使用的一项技术是提供额外的衡量标准，即比例集大小 (PSS)，其中包括私有内存（非共享）加上共享内存除以用户数量。请参阅第 7.5.9 节，pmap，了解可以显示 PSS 的工具。

### 7.2.10 工作集大小

工作集大小 (WSS) 是进程经常用于执行工作的主内存量。对于内存性能调整来说，这是一个有用的概念：如果 WSS 可以适应 CPU 缓存而不是主内存，性能应该会大大提高。此外，如果 WSS 超过主内存大小，性能将大大降低，并且应用程序必须交换才能执行工作。

虽然作为一个概念很有用，但在实践中很难测量：可观测性工具中没有 WSS 统计数据（它们通常报告 RSS，而不是 WSS）。第 7.4.10 节“内存收缩”描述了 WSS 估计的实验方法，第 7.5.12 节“wss”展示了一个实验性工作集大小估计工具 wss(8)。

### 7.2.11 字长

正如第 6 章中所介绍的，CPU、处理器可以支持多种字大小，例如 32 位和 64 位，从而允许任一软件运行。由于地址空间大小受到字大小可寻址范围的限制，因此需要 4 GB 以上内存的应用程序对于 32 位地址空间来说太大，需要针对 64 位或更高位进行编译。[^2]

[^2]x86 还具有物理地址扩展 (PAE) 功能（解决方法），允许 32 位处理器访问更大的内存范围（但不能在单个进程中访问）。

根据内核和处理器的不同，某些地址空间可能会保留给内核地址，并且不能供应用程序使用。一个极端的情况是具有 32 位字大小的 Windows，默认情况下为内核保留 2 GB，只为应用程序保留 2 GB \[Hall 09\]。在 Linux（或启用了 /3GB 选项的 Windows）上，内核预留空间为 1 GB。对于 64 位字大小（如果处理器支持的话），地址空间要大得多，因此内核预留不应该成为问题。

根据 CPU 架构，使用更大的位宽也可以提高内存性能，因为指令可以对更大的字长进行操作。如果数据类型具有较大位宽的未使用位，则可能会浪费少量内存。

## 7.3 架构

本节介绍内存架构，包括硬件和软件，包括处理器和操作系统的详细信息。

这些主题已被总结为性能分析和调优的背景。有关更多详细信息，请参阅本章末尾列出的供应商处理器手册和有关操作系统内部结构的文本。

### 7.3.1 硬件

内存硬件包括主存、总线、CPU 缓存和 MMU。

#### 主内存

当今使用的常见主存储器类型是动态随机存取存储器（DRAM）。这是一种易失性存储器，断电时其内容就会丢失。 DRAM 提供高密度存储，因为每个位仅使用两个逻辑组件来实现：电容器和晶体管。电容器需要定期刷新以保持电量。

企业服务器根据其用途配置不同数量的 DRAM，通常范围从 1 GB 到 1 TB 甚至更大。云计算实例通常较小，每个实例在 512 MB 到 256 GB 之间。[^3]但是，云计算旨在将负载分散到实例池上，因此它们可以共同为分布式应用程序带来更多的在线 DRAM，尽管一致性成本要高得多。

[^3]例外情况包括 AWS EC2 高内存实例，其内存达到 24 TB \[Amazon 20\]。

##### 延迟

主存储器的访问时间可以用列地址选通 (CAS) 延迟来衡量：向存储器模块发送所需地址（列）与数据可供读取之间的时间。这取决于内存类型（对于 DDR4，约为 10 至 20 纳秒 \[Crucial 18\]）。对于存储器 I/O 传输，对于存储器总线（例如，64 位宽）传输高速缓存行（例如，64 字节宽）而言，该延迟可能会发生多次。 CPU 和 MMU 读取新的可用数据时还存在其他延迟。读取指令从 CPU 缓存返回时可避免这些延迟；如果处理器支持回写缓存（例如英特尔处理器），写入指令也可以避免它们。

每个 CPU 通过共享系统总线对所有内存具有统一的访问延迟。当由跨所有处理器统一运行的单个操作系统内核实例管理时，这也是对称多处理 (SMP) 架构。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/07fig04.jpg)

图 7.4 示例 NUMA 主内存架构，双处理器

CPU 1 可以通过其内存总线直接对 DRAM A 执行 I/O。这称为本地存储器。 CPU 1 通过 CPU 2 和 CPU 互连（两跳）执行到 DRAM B 的 I/O。这被称为远程存储器并且具有较高的访问延迟。

连接到每个 CPU 的内存组称为内存节点，或简称为节点。操作系统可以基于处理器提供的信息了解存储器节点拓扑。然后，它允许它根据内存局部性分配内存和调度线程，尽可能支持本地内存以提高性能。

#### Buses

主内存如何物理连接到系统取决于主内存架构，如前图所示。实际的实现可能涉及 CPU 和内存之间的附加控制器和总线。主存储器可以通过以下方式之一访问：

*   共享系统总线：单处理器或多处理器，通过共享系统总线、内存桥控制器，最后是内存总线。这在第 6 章“CPU”中被描绘为 UMA 示例（图 7.3）和 Intel 前端总线示例（图 6.9）。该示例中的内存控制器是北桥。
*   直接：通过内存总线直接连接内存的单处理器。
*   互连：多处理器，每个处理器都通过内存总线直接连接内存，处理器通过 CPU 互连连接。这在之前的图 7.4 中被描述为 NUMA 示例； CPU 互连将在第 6 章“CPU”中讨论。

如果您怀疑您的系统不属于上述任何一种情况，请找到系统功能图并跟踪 CPU 和内存之间的数据路径，并记下沿途的所有组件。

##### DDR内存

系统架构可以支持并行使用多个存储器总线，以提高带宽。公倍数有双通道、三通道和四通道。例如，Intel Core i7 处理器最多支持四通道 DDR3-1600，最大内存带宽为 51.2 GB/s。

#### CPU缓存

处理器通常包括片上硬件缓存以提高内存访问性能。缓存可能包括以下级别，速度逐渐降低，大小逐渐增加：

*   Level 1：通常分为单独的指令缓存和数据缓存
    
*   2 级：指令和数据的缓存
    
*   第 3 级：另一个更大级别的缓存
    

根据处理器的不同，级别 1 通常由虚拟内存地址引用，级别 2 及以上由物理内存地址引用。

这些缓存在第 6 章“CPU”中进行了更深入的讨论。本章讨论另一种类型的硬件高速缓存，即 TLB。

#### 内存管理单元

MMU（内存管理单元）负责虚拟地址到物理地址的转换。这些是按页执行的，并且直接映射页内的偏移量。 MMU 在第 6 章“CPU”中在附近 CPU 缓存的上下文中介绍。

##### 多种页面尺寸

现代处理器支持多种页面大小，这允许操作系统和 MMU 使用不同的页面大小（例如 4 KB、2 MB、1 GB）。 Linux 大页面功能支持更大的页面大小，例如 2 MB 或 1 GB。

##### TLB

图 7.5 中的 MMU（内存管理单元）使用 **TLB（转换后备缓冲区）** 作为第一级地址转换缓存，后面则是主存中的页表。TLB 可以分为指令页和数据页的单独高速缓存。

由于 TLB 用于映射的条目数量有限，因此使用较大的页面大小可以增加可从 TLB 缓存转换的内存范围（即“范围”），从而减少 TLB 未命中并提高系统性能。TLB 还可以针对每个页面大小进一步划分为单独的高速缓存，从而提高在高速缓存中保留较大映射的可能性。

### 7.3.2 软件

内存管理软件包括虚拟内存系统、地址转换、交换、分页和分配。本节包含与性能最相关的主题：释放内存、空闲列表、页面扫描、交换、进程地址空间和内存分配器。

#### 释放内存

当系统上的可用内存变低时，内核可以使用多种方法来释放内存，将其添加到空闲页面列表中。 Linux 中的这些方法如图 7.6 所示，按照可用内存减少时的一般顺序排列。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/07fig06.jpg)

图7.6 Linux内存可用性管理

这些方法是：

*   空闲列表：未使用（也称为空闲内存）且可立即分配的页面列表。这通常被实现为多个空闲页面列表，每个位置组（NUMA）一个。
*   页面缓存：文件系统缓存。称为 swappiness 的可调参数设置系统应支持从页面缓存释放内存而不是交换的程度。
*   交换：这是由页面调出守护进程 kswapd 进行的分页，它会查找最近未使用的页面以添加到空闲列表中，包括应用程序内存。这些被调出，这可能涉及写入基于文件系统的交换文件或交换设备。当然，只有配置了交换文件或设备后，这才可用。
*   收获：当超过低内存阈值时，可以指示内核模块和内核平板分配器立即释放任何可以轻松释放的内存。这也称为收缩。
*   OOM 杀手：内存不足杀手将通过查找并杀死牺牲进程来释放内存，使用 select\_bad\_process() 找到该进程，然后通过调用 oom\_kill\_process() 杀死该进程。这可能会作为“内存不足：终止进程”消息记录在系统日志 (/var/log/messages) 中。

Linux swappiness 参数控制是否支持通过分页应用程序或通过从页面缓存回收内存来释放内存。它是 0 到 100 之间的数字（默认值为 60），其中较高的值有利于通过分页释放内存。控制这些内存释放技术之间的平衡可以通过保留热文件系统缓存同时调出冷应用程序内存来提高系统吞吐量\[Corbet 04\]。

询问如果没有配置交换设备或交换文件会发生什么也很有趣。这限制了虚拟内存大小，因此如果禁用了过度使用，内存分配将很快失败。在Linux上，这也可能意味着OOM杀手会更快被使用。

考虑一个存在内存无限增长问题的应用程序。对于交换，这可能首先由于分页而成为性能问题，这是实时调试问题的机会。如果没有交换，就没有分页宽限期，因此应用程序要么遇到“内存不足”错误，要么 OOM 杀手终止它。如果仅在使用数小时后才发现问题，这可能会延迟调试问题。

在 Netflix 云中，实例通常不使用交换，因此应用程序如果耗尽内存就会被 OOM 终止。应用程序分布在大量实例中，杀死一个 OOM 会导致流量立即重定向到其他运行状况良好的实例。这被认为比允许一个实例由于交换而缓慢运行更好。

当使用内存 cgroup 时，可以使用如图 7.6 所示的类似内存释放技术来管理 cgroup 内存。系统可能拥有大量可用内存，但正在交换或遇到 OOM 杀手，因为容器已耗尽其 cgroup 控制的限制 \[Evans 17\]。有关 cgroup 和容器的更多信息，请参阅第 11 章，云计算。

#### 空闲列表

最初的 Unix 内存分配器使用内存映射和首次适应扫描。随着 BSD 中分页虚拟内存的引入，添加了空闲列表和页出守护进程 \[Babaoglu 79\]。空闲列表（如图 7.7 所示）允许立即找到可用内存。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/07fig07.jpg)

图7.7 空闲列表操作

释放的内存将添加到列表的头部以供将来分配。由页面调出守护进程释放的内存（并且可能仍然包含有用的缓存文件系统页面）被添加到尾部。如果在重用有用页面之前发生对这些页面之一的未来请求，则可以回收该页面并将其从空闲列表中删除。

基于 Linux 的系统仍在使用一种形式的空闲列表，如图 7.6 所示。空闲列表通常通过分配器使用，例如内核的slab分配器和用户空间的libc malloc()（有自己的空闲列表）。这些依次消耗页面，然后通过其分配器 API 公开它们。

Linux 使用伙伴分配器来管理页面。这为不同大小的内存分配提供了多个空闲列表，遵循二次方方案。术语“伙伴”是指查找空闲内存的相邻页面，以便可以将它们一起分配。有关历史背景，请参阅\[Peterson 77\]。

伙伴空闲列表位于以下层次结构的底部，从每个内存节点 pg\_data\_t 开始：

*   节点：内存组、NUMA 感知
*   区域：用于特定目的的内存范围（直接内存访问 \[DMA\]、[^4] 正常、highmem）
    [^4]尽管 ZONE\_DMA 可能会被删除 \[Corbet 18a\]。
*   迁移类型：不可移动、可回收、可移动等。
*   尺寸：页数的二次方

在节点空闲列表内进行分配可以提高内存局部性和性能。对于最常见的分配（单页），伙伴分配器为每个 CPU 保留单页列表，以减少 CPU 锁争用。

#### Reaping

收获主要涉及从内核平板分配器缓存中释放内存。这些缓存包含平板大小块中未使用的内存，可供重用。收割将此内存返回给系统以进行页面分配。

在Linux上，内核模块还可以调用register\_shrinker()来注册特定函数以获取自己的内存。

#### 页面扫描

通过分页释放内存是由内核页面调出守护进程管理的。当空闲列表中的可用主内存低于阈值时，页面调出守护程序开始页面扫描。仅在需要时才进行页面扫描。正常平衡的系统可能不会经常进行页面扫描，并且可能只在短时间内进行页面扫描。

在 Linux 上，页调出守护进程称为 kswapd，它扫描非活动和活动内存的 LRU 页面列表以释放页面。它根据可用内存和两个阈值来唤醒以提供迟滞，如图 7.8 所示。

一旦可用内存达到最低阈值，kswapd 将在前台运行，根据请求同步释放内存页，这种方法有时称为直接回收 \[Gorman 04\]。这个最低阈值是可调的 (vm.min\_free\_kbytes)，其他阈值基于它进行缩放（低阈值为 2 倍，高阈值为 3 倍）。对于具有超过 kswap 回收速度的高分配突发的工作负载，Linux 提供了额外的可调参数以进行更积极的扫描，vm.watermark\_scale\_factor 和 vm.watermark\_boost\_factor：请参阅第 7.6.1 节, 可调参数。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/07fig10.jpg)

图 7.10 进程虚拟内存地址空间示例

在 SPARC 上，内核驻留在单独的完整地址空间中（图 7.10 中未显示）。请注意，在 SPARC 上，无法仅根据指针值来区分用户地址和内核地址； x86 采用不同的方案，其中用户地址和内核地址不重叠。[^5]

>[^5]请注意，对于 64 位地址，处理器可能不支持完整的 64 位范围：AMD 规范允许实现仅支持 48 位地址，其中未使用的高阶位设置为最后一位：这将创建两个可用的地址范围，称为规范地址，从 0 到 0x00007ffff，用于用户空间，以及0xffff80000 到 0xffff，用于内核空间。这就是 x86 内核地址以 0xffff 开头的原因。

程序可执行段包含单独的文本段和数据段。库还由单独的可执行文本和数据段组成。这些不同的段类型是：

*   可执行文本：包含进程的可执行CPU指令。这是从文件系统上二进制程序的文本段映射的。它是只读的，具有执行权限。
*   可执行数据：包含从二进制程序的数据段映射的初始化变量。它具有读/写权限，以便在程序运行时可以修改变量。它还有一个私有标志，以便修改不会刷新到磁盘。
*   堆：这是程序的工作内存，并且是匿名内存（无文件系统位置）。它根据需要增长并通过 malloc(3) 分配。
*   堆栈：正在运行的线程的堆栈，映射读/写。

库文本段可以由使用同一库的其他进程共享，每个进程都有库数据段的私有副本。

#### 堆增长

造成混乱的一个常见原因是堆的无休止增长。是内存泄漏吗？对于简单的分配器， free(3) 不会将内存返回给操作系统；相反，保留内存是为了将来的分配。这意味着进程驻留内存只能增长，这是正常的。进程减少系统内存使用的方法包括：

*   重新执行：调用 execve(2) 从空地址空间开始
    
*   内存映射：使用mmap(2)和munmap(2)，它将内存返回给系统
    

内存映射文件在第 8 章“文件系统”、第 8.3.10 节“内存映射文件”中进行了描述。

Glibc 通常在 Linux 上使用，是一种高级分配器，支持 mmap 操作模式，以及用于向系统释放可用内存的 malloc\_trim(3) 函数。当堆顶可用内存变大时，malloc\_trim(3) 由 free(3) 自动调用，[^6] 并使用 sbrk(2) 系统调用释放它。

[^6]大于 M\_TRIM\_THRESHOLD mallopt(3) 参数，默认情况下为 128 KB。

有多种用于内存分配的用户级和内核级分配器。图7.11展示了分配器的作用，包括一些常见的类型。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/07fig11.jpg)

内存分配器功能可包括：

*   简单API：例如malloc(3)、free(3)。
*   高效的内存使用：在处理各种大小的内存分配时，内存使用可能会变得碎片化，其中有许多未使用的区域浪费了内存。分配器可以努力合并未使用的区域，以便更大的分配可以利用它们，从而提高效率。
*   性能：内存分配可能很频繁，并且在多线程环境中，由于同步原语的争用，它们的性能可能会很差。分配器可以设计为谨慎使用锁，还可以利用每线程或每 CPU 缓存来提高内存局部性。
*   可观察性：分配器可以提供统计信息和调试模式来显示它是如何使用的，以及哪些代码路径负责分配。

平板分配器是为 Solaris 2.4 \[Bonwick 94\] 开发的，后来通过称为 magazines \[Bonwick 01\] 的每 CPU 缓存进行了增强：

我们的基本方法是为每个 CPU 提供一个称为杂志的 M 元素缓存，类似于自动武器。在 CPU 需要重新加载之前，每个 CPU 的磁带盒都可以满足 M 次分配，即将空磁带盒更换为满的磁带盒。

除了高性能之外，原始的slab分配器还具有调试和分析功能，包括审计跟踪分配细节和堆栈跟踪。

平板分配已被各种操作系统采用。 BSD 有一个称为通用内存分配器 (UMA) 的内核板分配器，它非常高效且支持 NUMA。在 Linux 2.2 版本中，slab 分配器也被引入，多年来它一直是默认选项。 Linux 此后已将 SLUB 作为选项或默认选项。

#### SLUB

Linux内核SLUB分配器基于slab分配器，旨在解决各种问题，特别是关于slab分配器的复杂性。改进包括删除对象队列和每 CPU 缓存，将 NUMA 优化留给页面分配器（请参阅前面的空闲列表部分）。

SLUB 分配器在 Linux 2.6.23 \[Lameter 07\] 中成为默认选项。

#### glibc

用户级 GNU libc 分配器基于 Doug Lea 的 dlmalloc。其行为取决于分配请求的大小。小分配由内存箱提供，其中包含类似大小的单元，可以使用类似伙伴的算法来合并这些单元。较大的分配可以使用树查找来有效地查找空间。非常大的分配切换到使用 mmap(2)。最终结果是一个高性能的分配器，受益于多种分配策略。

#### TCMalloc

TCMalloc 是用户级线程缓存 malloc，它使用每线程缓存进行小型分配，减少锁争用并提高性能 \[Ghemawat 07\]。定期垃圾收集将内存迁移回中央堆进行分配。

#### jemalloc

libjemalloc 最初是 FreeBSD 用户级 libc 分配器，也可用于 Linux。它使用多竞技场、每线程缓存和小对象板等技术来提高可扩展性并减少内存碎片。它可以使用mmap(2)和sbrk(2)来获取系统内存，优先使用mmap(2)。 Facebook 使用 jemalloc 并添加了分析和其他优化 \[Facebook 11\]。

### 7.4.1 工具方法

工具方法是迭代可用工具、检查它们提供的关键指标的过程。这是一种简单的方法，可能会忽略您所拥有的工具提供的可见性较差或没有可见性的问题，并且执行起来可能非常耗时。

对于内存，工具方法可能涉及针对 Linux 检查以下内容：

*   页面扫描：寻找连续的页面扫描（超过 10 秒）作为内存压力的迹象。这可以使用 `sar -B` 并检查 `pgscan` 列来完成。
*   压力失速信息 (PSI)：`cat /proc/pressure/memory` (Linux 4.20+)，用于检查内存压力（饱和度）统计信息及其随时间的变化情况。
*   交换：如果配置了交换，则内存页面的交换（Linux 对交换的定义）进一步表明系统内存不足。您可以使用 vmstat(8) 并检查 `si` 和 `so` 列。
*   `vmstat`：运行 `vmstat 1` 并检查 `free` 列的可用内存。
*   OOM 杀手：这些事件可以在系统日志 /var/log/messages 中或从 dmesg(1) 中看到。搜索“内存不足”。
*   `top`：查看哪些进程和用户是最大的物理内存消耗者（常驻）和虚拟内存消耗者（有关列名称的信息，请参阅手册页，根据版本的不同而有所不同）。 top(1) 还总结了可用内存。
*   perf(1)/BCC/bpftrace：使用堆栈跟踪跟踪内存分配，以识别内存使用的原因。请注意，这可能会花费相当大的开销。一种更便宜但粗糙的解决方案是执行 CPU 分析（定时堆栈采样）并搜索分配代码路径。

有关每个工具的更多信息，请参阅第 7.5 节“可观测性工具”。

### 7.4.2 USE Method

USE 方法用于在性能调查的早期识别所有组件的瓶颈和错误，然后再采取更深入、更耗时的策略。

检查系统范围内的：

*   利用率：有多少内存正在使用中，有多少内存可用。物理内存和虚拟内存都应该检查。
*   饱和度：作为缓解内存压力的措施而执行的页面扫描、分页、交换和 Linux OOM 杀手牺牲的程度。
*   错误：软件或硬件错误。

您可能需要首先检查饱和度，因为持续饱和是内存问题的迹象。这些指标通常可以从操作系统工具中轻松获得，包括用于交换统计信息的 vmstat(8) 和 sar(1)，以及用于 OOM 杀手牺牲的 dmesg(1)。对于配置有单独磁盘交换设备的系统，交换设备的任何活动都是内存压力的另一个迹象。 Linux 还提供内存饱和统计信息作为压力失速信息 (PSI) 的一部分。

不同的工具可以以不同的方式报告物理内存利用率，具体取决于它们是否考虑了未引用的文件系统缓存页面或非活动页面。系统可能会报告它只有 10 MB 的可用内存，而实际上它有 10 GB 的文件系统缓存，可以在需要时立即由应用程序回收。检查工具文档以了解其中包含的内容。

可能还需要检查虚拟内存利用率，具体取决于系统是否执行过量使用。对于不这样做的系统，一旦虚拟内存耗尽，内存分配就会失败——这是一种内存错误。

内存错误可能是由软件引起的，例如失败的内存分配或 Linux OOM Killer，也可能是由硬件引起的，例如 ECC 错误。从历史上看，内存分配错误一直由应用程序报告，尽管并非所有应用程序都这样做（并且，由于 Linux 过度使用，开发人员可能认为没有必要）。硬件错误也很难诊断。当使用 ECC 内存时，某些工具可能会报告 ECC 可纠正错误（例如，在 Linux 上，dmidecode(8)、edac-utils、`ipmitool sel`）。这些可纠正的错误可以用作 USE 方法错误度量，并且可能是不可纠正的错误可能很快发生的标志。对于实际的（不可纠正的）内存错误，您可能会遇到任意应用程序无法解释的、不可重现的崩溃（包括段错误和总线错误信号）。

对于实施内存限制或配额（资源控制）的环境（如在某些云计算环境中），可能需要以不同的方式测量内存利用率和饱和度。即使主机上有足够的物理内存，您的操作系统实例也可能处于软件内存限制和交换状态。请参阅第 11 章，云计算。

### 7.4.3 Characterizing Usage

在进行容量规划、基准测试和模拟工作负载时，表征内存使用情况是一项重要的练习。它还可以通过查找和纠正错误配置来带来一些最大的性能提升。例如，数据库缓存可能配置得太小，命中率低，或者太大，导致系统分页。

对于内存，表征使用情况涉及识别内存的使用位置和数量：

*   系统范围的物理和虚拟内存利用率
*   饱和度：交换和 OOM 杀死
*   内核和文件系统缓存内存使用情况
*   每个进程的物理和虚拟内存使用情况
*   内存资源控制的使用（如果存在）

此示例描述显示了如何一起表达这些属性：

系统有 256 GB 的主内存，其中 1% 被进程使用（已利用），30% 被文件系统缓存使用。最大的进程是数据库，消耗 2 GB 主内存 (RSS)，这是其迁移来源的先前系统的配置限制。

随着更多内存用于缓存工作数据，这些特征可能会随着时间的推移而变化。除了常规缓存增长之外，内核或应用程序内存也可能由于内存泄漏（软件错误）而随着时间的推移而不断增长。

#### Advanced Usage Analysis/Checklist

此处列出了其他特征作为需要考虑的问题，也可以作为彻底研究内存问题时的清单：

*   应用程序的工作集大小 (WSS) 是多少？
*   内核内存用在哪里？每块板？
*   文件系统缓存有多少是活动的，而不是非活动的？
*   进程内存在哪里使用（指令、缓存、缓冲区、对象等）？
*   为什么进程要分配内存（调用路径）？
*   为什么内核要分配内存（调用路径）？
*   进程库映射有什么奇怪的吗（例如，随着时间的推移而变化）？
*   哪些进程正在被主动换出？
*   哪些流程之前已被换出？
*   进程或内核是否存在内存泄漏？
*   在 NUMA 系统中，内存在内存节点之间的分布情况如何？
*   IPC 和内存停顿周期率是多少？
*   内存总线的平衡程度如何？
*   与远程内存 I/O 相比，执行了多少本地内存 I/O？

以下部分可以帮助回答其中一些问题。请参阅第 2 章“方法论”，了解该方法论的高级摘要以及要衡量的特征（谁、为什么、什么、如何）。

### 7.4.4 Cycle Analysis

内存总线负载可以通过检查 CPU 性能监控计数器 (PMC) 来确定，该计数器可以编程来计算内存停顿周期、内存总线使用情况等。首先的指标是每周期指令数 (IPC)，它反映了 CPU 负载对内存的依赖程度。请参阅第 6 章，CPU。

### 7.4.5 性能监控

性能监控可以随着时间的推移识别活动问题和行为模式。内存的关键指标是：

*   利用率：使用百分比，可以从可用内存中推断出来
*   饱和：交换、OOM 杀死

对于实施内存限制或配额（资源控制）的环境，可能还需要收集与所施加的限制相关的统计信息。

还可以监视错误（如果可用），如第 7.4.2 节“使用方法”中的利用率和饱和度所述。

监控一段时间内的内存使用情况，尤其是按进程监控内存使用情况，可以帮助识别内存泄漏的存在和发生率。

### 7.4.6 泄漏检测

当应用程序或内核模块无休止地增长，消耗来自空闲列表、文件系统缓存以及最终来自其他进程的内存时，就会出现此问题。这可能首先会被注意到，因为系统开始交换或应用程序被 OOM 终止，以应对无尽的内存压力。

此类问题由以下任一原因引起：
*   内存泄漏：一种软件错误，内存不再使用但从未释放。这可以通过修改软件代码或应用补丁或升级（修改代码）来解决。
*   内存增长：软件正常消耗内存，但消耗速度远高于系统所需的速度。通过更改软件配置或软件开发人员更改应用程序消耗内存的方式可以解决此问题。

内存增长问题经常被误认为是内存泄漏。首先要问的问题是：它应该这样做吗？检查内存使用情况、应用程序的配置及其分配器的行为。应用程序可以被配置为填充存储器高速缓存，并且观察到的增长可以是高速缓存预热。

如何分析内存泄漏取决于软件和语言类型。一些分配器提供调试模式来记录分配详细信息，然后可以在事后分析这些详细信息以识别负责的调用路径。某些运行时具有用于进行堆转储分析的方法，以及用于进行内存泄漏调查的其他工具。

Linux BCC 跟踪工具包括用于增长和泄漏分析的 memleak(8)：它跟踪分配并记录在一段时间内未释放的分配以及分配代码路径。它无法判断这些是泄漏还是正常增长，因此您的任务是分析代码路径以确定是哪种情况。 （请注意，该工具还会因高分配率而产生高开销。）BCC 在第 15 章 BPF、第 15.1 节 BCC 中介绍。

### 7.4.7 静态性能调优

静态性能调优重点关注配置环境的问题。对于内存性能，请检查静态配置的以下方面：

*   主存总共有多少？
*   应用程序配置使用多少内存（它们自己的配置）？
*   应用程序使用哪些内存分配器？
*   主存的速度是多少？它是可用的最快类型 (DDR5)吗？
*   主内存是否已经过全面测试（例如，使用 Linux memtester）？
*   系统架构是怎样的？努玛，乌玛？
*   操作系统是否支持 NUMA？它提供 NUMA 可调参数吗？
*   内存是连接到同一个插槽，还是跨插槽分开？
*   有多少条内存总线？
*   CPU 高速缓存的数量和大小是多少？ TLB？
*   BIOS 设置是什么？
*   是否配置和使用了大页面？
*   过量使用是否可用并已配置？
*   正在使用哪些其他系统内存可调参数？
*   是否存在软件施加的内存限制（资源控制）？

回答这些问题可能会揭示被忽视的配置选择。

### 7.4.8 资源控制

操作系统可以提供细粒度的控制以将存储器分配给进程或进程组。这些控制可能包括主存储器和虚拟存储器使用的固定限制。它们的工作方式是特定于实现的，并在第 7.6 节“调优”和第 11 章“云计算”中讨论。

### 7.4.9 微基准测试

微基准测试可用于确定主存储器的速度和特征，例如CPU高速缓存和高速缓存行大小。这在分析系统之间的差异时可能会有所帮助，因为内存访问速度对性能的影响可能比 CPU 时钟速度更大，具体取决于应用程序和工作负载。

在第 6 章“CPU”中，“CPU 缓存”下的“延迟”部分（第 6.4.1 节“硬件”）显示了微基准测试内存访问延迟的结果，以确定 CPU 缓存的特性。

### 7.4.10 内存收缩

这是一种使用负实验的工作集大小 (WSS) 估计方法，需要配置交换设备来执行实验。在测量性能和交换时，应用程序的可用主内存逐渐减少：性能急剧下降且交换大幅增加的点表明 WSS 不再适合可用内存。

虽然值得一提的是作为一个负面实验的例子，但不建议将其用于生产用途，因为它会故意损害性能。对于其他 WSS 估计技术，请参阅第 7.5.12 节中的实验性 wss(8) 工具 wss 和我关于 WSS 估计的网站 \[Gregg 18c\]。

- **bpftrace**：利用 BPF（伯克利包过滤）技术提供内存分析和跟踪功能，是一个灵活的内存分析工具。

请参阅每个工具的手册页或文档，以获取其完整的功能和使用说明。
### 7.5.1 vmstat

`vmstat`（虚拟内存统计）命令用于提供系统内存、分页、CPU 等的统计信息。它能够帮助我们获取关于内存使用的高级视图，包括当前可用内存、交换（swap）信息以及页面缓存的使用情况。在 1979 年，Bill Joy 和 Ozalp Babaoglu 为 BSD 操作系统引入了这个命令，原始的手册页中提到：

> 错误：打印出的数字太多，有时很难弄清楚该看什么。

以下是一个 Linux 版本 `vmstat` 输出的示例：

```bash
$ vmstat 1
procs -----------memory---------- ---swap-- -----io---- -system-- ----cpu----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa
 4  0      0 34454064 111516 13438596   0   0    0    5    2    0  0  0 100  0
 4  0      0 34455208 111516 13438596   0   0    0    0 2262 15303 16 12 73  0
 5  0      0 34455588 111516 13438596   0   0    0    0 1961 15221 15 11 74  0
 4  0      0 34456300 111516 13438596   0   0    0    0 2343 15294 15 11 73  0
 [ ... ]
```

输出解释

- `r` 和 `b`：分别表示运行队列中的进程数和阻塞的进程数。
- `swpd`：换出的内存量（已交换到磁盘的内存）。
- `free`：当前系统可用的内存。
- `buff`：缓冲区高速缓存中的内存。
- `cache`：页面缓存中的内存。
- `si`：内存交换的入（即交换到内存）量。
- `so`：内存换出的出（即从内存交换到磁盘）量。
- `bi` 和 `bo`：分别表示块设备（磁盘）的读取和写入量。
- `in` 和 `cs`：中断和上下文切换次数。
- `us`、`sy`、`id` 和 `wa`：CPU 的使用情况（用户时间、系统时间、空闲时间和等待I/O时间）。

例如，在以下行中：

```bash
 4  0      0 34454064 111516 13438596   0   0    0    5    2    0  0  0 100  0
```

- 系统处于运行状态，有 4 个进程处于运行队列（r=4），没有进程被阻塞（b=0）。
- 没有内存被交换到磁盘（`swpd=0`），且有 34,454,064 KB 的可用内存（`free=34454064`）。
- `buff` 和 `cache` 中分别有 111,516 KB 和 13,438,596 KB 的内存在使用中。
- 没有发生内存交换（`si=0` 和 `so=0`），I/O 操作较少（`bi=0` 和 `bo=5`）。
- CPU 处于 100% 空闲状态（`us=0`, `sy=0`, `id=100`, `wa=0`）。

默认情况下，`vmstat` 使用千字节（KB）作为单位。为了以更易读的格式显示数据，可以使用 `-S` 选项来将输出单位更改为兆字节（MB）。例如：

```bash
$ vmstat -Sm 1
procs -----------memory---------- ---swap-- -----io---- -system-- ----cpu----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa
 4  0      0  35280    114  13761    0    0     0     5    2    1  0  0 100  0
 4  0      0  35281    114  13761    0    0     0     0 2027 15146 16 13 70  0
 [ ... ]
```

这将以兆字节（MB）为单位显示内存和交换统计信息，使得输出更加易于理解。

 `-a` 选项

使用 `-a` 选项，`vmstat` 可以显示更详细的内存信息，特别是活动和非活动内存。例如：

```bash
$ vmstat -a 1
procs -----------memory---------- ---swap-- -----io---- -system-- ----cpu----
 r  b   swpd   free  inact active   si   so    bi    bo   in   cs us sy id wa
 5  0      0 34453536 10358040 3201540   0   0    0    5    2    0  0  0 100  0
 4  0      0 34453228 10358040 3200648   0   0    0    0 2464 15261 16 12 71  0
 [ ... ]
```

在这个输出中，`inact` 表示非活动内存，`active` 表示活动内存。可以用这些信息来更好地理解内存的使用情况，尤其是哪些内存被系统认为是不活跃的。

 `-s` 选项

如果你想以更简洁的格式查看内存统计信息，可以使用 `-s` 选项，这将以列表形式打印详细的统计数据：

```bash
$ vmstat -s
       4194304 K total memory
       3197848 K used memory
        495832 K active memory
        544640 K inactive memory
        105240 K free memory
        271060 K buffer memory
        639236 K swap cache
        104856 K total swap
        104856 K used swap
             0 K free swap
```


#### 7.5.2 PSI

Linux 4.20 中添加的 Linux 压力失速信息 (PSI) 包括内存饱和度的统计信息。这些不仅显示是否存在内存压力，还显示过去五分钟内存压力的变化情况。输出示例：

点击这里查看代码图片

\# **cat /proc/pressure/memory**
some avg10=2.84 avg60=1.23 avg300=0.32 total=1468344
full avg10=1.85 avg60=0.66 avg300=0.16 total=702578

此输出显示内存压力正在增加，10 秒平均值 (2.84) 高于 300 秒平均值 (0.32)。这些平均值是任务内存停滞的时间百分比。 `some` 行显示某些任务（线程）何时受到影响，`full` 行显示何时所有可运行任务受到影响。

PSI 统计数据也按 cgroup2 进行跟踪（第 11 章云计算中介绍了 cgroup）\[Facebook 19\]。
### 7.5.2 PSI

PSI（压力失速信息，Pressure Stall Information）是 **Linux 4.20** 中引入的一项功能，它为内存饱和度和其他资源的压力情况提供统计数据。PSI 不仅能显示系统当前是否存在内存压力，还能够显示过去不同时间段内的内存压力变化情况。以下是一个输出示例：

```bash
# cat /proc/pressure/memory
some avg10=2.84 avg60=1.23 avg300=0.32 total=1468344
full avg10=1.85 avg60=0.66 avg300=0.16 total=702578
```

输出解释
- **some**：表示某些任务（线程）受到内存压力影响的情况。
- **full**：表示所有可运行任务（线程）都受到内存压力影响的情况。
- **avg10**、**avg60**、**avg300**：分别表示过去 10 秒、60 秒和 300 秒内，任务因内存压力导致停滞的时间百分比。
  - 例如，`avg10=2.84` 表示过去 10 秒内，任务因内存压力而停滞的时间占比为 2.84%。
  - `avg300=0.32` 表示过去 300 秒内，任务因内存压力而停滞的时间占比为 0.32%。
- **total**：表示在指定时间段内任务所受到的内存压力的总时间。

在此示例中，内存压力正在增加，因为 `avg10`（过去10秒的平均值）为 2.84，明显高于 `avg300`（过去300秒的平均值）为 0.32。表示系统近期面临较高的内存压力。
PSI 统计数据也可以按 **cgroup2**（控制组）进行跟踪，这为容器化应用程序提供了更精确的内存压力统计。有关 cgroup 的详细信息，请参阅第 11 章《云计算》。

### 7.5.3 swapon

`swapon(1)` 命令用于显示当前系统的交换设备状态，显示交换设备是否已配置，并展示各个交换设备的使用情况。输出示例如下：

```bash
$ swapon
NAME      TYPE      SIZE   USED PRIO
/dev/dm-2 partition 980M 611.6M   -2
/swap1    file       30G  10.9M   -3
```

输出字段说明：

- **NAME**：交换设备的名称。
- **TYPE**：交换设备的类型，可以是分区（`partition`）或文件（`file`）。
- **SIZE**：交换设备的总大小。
- **USED**：当前已经使用的交换空间。
- **PRIO**：交换设备的优先级。优先级越高，交换设备被使用的优先级越高，通常值为负数。

在上面的示例中，系统配置了两个交换设备：
1. `/dev/dm-2`：一个大小为 980MB 的物理交换分区，当前已使用 611.6MB。
2. `/swap1`：一个大小为 30GB 的交换文件，当前已使用 10.9MB。

许多现代系统可能没有配置交换空间。在这种情况下，`swapon(1)` 命令将不会显示任何输出。

如果交换设备正在被使用（即有活动的 I/O 操作），可以通过其他工具查看相关的 I/O 信息：

- `vmstat(1)` 命令中的 `si`（交换输入）和 `so`（交换输出）列。
- `iostat(1)` 命令中的设备 I/O 统计信息（详见第 9 章）。

- **pgfree/s**：表示每秒将页面添加到空闲列表的次数。
- **pgscank/s / pgscand/s**：分别表示由后台页出守护进程（`kswapd`）扫描和直接扫描页面的速率。
- **%vmeff**：表示页面回收效率的比率。高值表示页面回收效率高，低值表示系统的页面回收效率差。

- **kbswpfree / kbswpused**：分别表示可用和已用的交换空间大小。
- **%swpused**：表示已用交换空间占总交换空间的百分比。

- **`%vmeff`** 是衡量内存回收效率的一个重要指标。高值表明系统能够有效地回收内存，而低值则表明系统可能存在内存管理问题。
- **`pgscand/s`** 指标可以显示应用程序阻塞内存分配的速率和进入直接回收的速率。这个值越高，表示系统越有效地进行内存回收。

要深入了解这些统计数据，您可以使用性能分析工具，如 `perf(1)` 和 `bpftrace`，来进一步追踪内存回收的过程和内核行为。

这些统计信息可以帮助系统管理员及时发现系统内存的潜在问题，并采取相应的优化措施。
### 7.5.5 slabtop

`slabtop(1)` 是一个用于监视和分析 Linux 内核 slab 缓存的命令。它以类似于 `top(1)` 命令的实时更新方式显示系统的 slab 分配器的使用情况。`slabtop` 可以帮助管理员了解内核如何管理和分配内存，以便识别潜在的内存问题或优化机会。

以下是运行 `slabtop -sc` 命令时的一个输出示例：

```bash

 Active / Total Size (% used)       : 157680.28K / 200462.06K (78.7%)
 Minimum / Average / Maximum Object : 0.01K / 0.23K / 12.00K

 45450  33712  74%    1.05K   3030       15     48480K ext4_inode_cache
161091  81681  50%    0.19K   7671       21     30684K dentry
222963 196779  88%    0.10K   5717       39     22868K buffer_head
 35763  35471  99%    0.58K   2751       13     22008K inode_cache
 26033  13859  53%    0.57K   1860       14     14880K radix_tree_node
 93330  80502  86%    0.13K   3111       30     12444K kernfs_node_cache
  2104   2081  98%    4.00K    263        8      8416K kmalloc-4k
   528    431  81%    7.50K    132        4      4224K task_struct
[...]
```

 输出说明
1. **摘要信息**
   - **Active / Total Objects**：显示当前活跃对象的数量和总对象的数量，以及活跃对象占总对象的百分比。例如，`686110 / 867574 (79.1%)` 表示有 79.1% 的对象正在被使用。
   - **Active / Total Slabs**：显示活跃 slabs 和总 slabs 的数量以及使用比例。
   - **Active / Total Caches**：显示当前活跃的缓存和总缓存的数量，以及使用比例。
   - **Active / Total Size**：显示活跃内存和总内存的大小以及其使用比例。
   - **Minimum / Average / Maximum Object**：显示对象的最小、平均和最大大小。
2. **详细的 slab 缓存信息**
   以下是每个 slab 缓存的一些关键字段：
   - **OBJS**：缓存中所有对象的总数量。
   - **ACTIVE**：当前活跃（正在使用）的对象数量。
   - **USE**：表示缓存的使用百分比（即活跃对象占总对象的比例）。
   - **OBJ SIZE**：每个对象的大小（以字节为单位）。
   - **SLABS**：该缓存使用的 slabs 数量。
   - **OBJ/SLAB**：每个 slab 中包含的对象数。
   - **CACHE SIZE**：缓存所占的总内存大小（以字节为单位）。
   - **NAME**：缓存的名称。

以第一个条目 `ext4_inode_cache` 为例：
- **OBJS**: 45450 表示该缓存有 45450 个对象。
- **ACTIVE**: 33712 表示其中 33712 个对象正在被使用。
- **USE**: 74% 表示该缓存的 74% 被活跃使用。
- **OBJ SIZE**: 1.05K 表示每个对象的大小为 1.05KB。
- **SLABS**: 3030 表示该缓存使用了 3030 个 slabs。
- **OBJ/SLAB**: 15 表示每个 slab 中有 15 个对象。
- **CACHE SIZE**: 48480K 表示该缓存占用了 48,480KB 的内存。

 使用 `slabtop` 的选项

- `-sc`：按缓存大小排序显示 slabs，最大缓存显示在最上面。
- `-s`：周期性刷新屏幕。可以通过指定刷新间隔（如 `slabtop -s 5`）来控制刷新速度。
- `-c`：显示缓存信息，帮助识别内存中各类缓存的使用情况。

 获取其他信息

- `slabtop` 命令的数据来自 `/proc/slabinfo` 文件。
- 您也可以通过 `vmstat -m` 命令获取一些相关的 slab 内存统计信息。

该系统有两个 NUMA 节点，每个节点连接到每个插槽的内存条。Linux 尝试在最近的 NUMA 节点上分配内存，`numastat(8)` 显示了这一策略的成功程度。

 主要统计数据

- **numa_hit**：目标 NUMA 节点上的内存分配。此项越高，表示内存分配越成功地命中本地 NUMA 节点。
- **numa_miss + numa_foreign**：内存分配未命中首选 NUMA 节点。 
  - `numa_miss`：显示本应在其他节点分配的本地分配。
  - `numa_foreign`：显示本应在本地节点分配的远程分配。
- **other_node**：当进程在其他节点上运行时，该节点的内存分配。

示例输出表明 NUMA 分配策略表现良好：`numa_hit` 数值远高于 `numa_miss` 和 `numa_foreign`。命中率较高意味着内存分配成功地在本地 NUMA 节点上完成，这对于性能优化非常重要。

如果 `numa_hit` 数值显著低于其他统计数据，可能需要采取以下措施：

1. **调整 NUMA 可调参数**：可以通过 `sysctl(8)` 来调整 NUMA 的相关参数。
2. **提高内存局部性**：例如，将工作负载或系统进行分区，或选择具有较少 NUMA 节点的不同系统。
3. **优化内存访问**：减少远程内存访问，提高内存访问的局部性和性能。

 命令选项
- **`-n`**：以 MB 为单位打印统计信息。适用于需要更直观的内存容量视图的场景。
- **`-m`**：以 `/proc/meminfo` 风格打印输出，这样可以与其他内存统计工具兼容。

在 `top` 命令的输出中，有一个顶部摘要部分显示了系统的总体内存使用情况，以及当前正在运行的进程列表。以下是摘要部分的解释：

- **Mem**：显示系统的主内存使用情况
  - `total`：总内存
  - `used`：已使用的内存
  - `free`：空闲内存
  - `buffers`：用于缓存的内存

- **Swap**：显示系统交换空间的使用情况
  - `total`：总交换空间
  - `used`：已使用的交换空间

  - `cached`：缓存的交换空间

- **PID**：进程 ID
- **USER**：进程所属用户
- **VIRT**：虚拟内存大小（Virtual memory size），包括所有映射的文件和共享库
- **RES**：驻留集大小（Resident memory size），即进程实际使用的物理内存
- **SHR**：共享内存（Shared memory），即进程与其他进程共享的内存
- **S**：进程状态（S = sleeping，R = running，Z = zombie等）

- **%MEM**：进程所占用的内存百分比
- **TIME+**：进程使用的 CPU 时间
- **COMMAND**：进程的命令名称

- 按 `%MEM` 排序：在 `top` 会话中，按 `M` 键。
- 按 CPU 使用率排序：在 `top` 会话中，按 `P` 键。

`top` 命令显示的内存相关列与 `ps` 命令类似，通常包括以下几列：

- **%MEM**：进程的内存使用百分比（相对于总物理内存）。
- **VIRT**：虚拟内存大小，包括进程分配的所有内存（如映射文件、共享库等）。
- **RES**：驻留集大小，实际驻留在物理内存中的内存量。
  
**注意：** 对于进程的内存使用情况，`VIRT` 和 `RES` 可能会有所不同，因为 `VIRT` 包括所有的虚拟内存，部分内存可能尚未被加载到物理内存中。

 使用 `top` 查看内存类型

`top` 命令还可以显示与内存相关的其他详细信息，例如缓存（`cached`）、缓冲区（`buffers`）和交换空间（`swap`）。关于这些内存统计信息的更多信息，请参阅 `top(1)` 的手册页中的 **“Linux 内存类型”** 部分。

在使用 `top(1)` 时，可以键入 `?` 查看交互式命令的摘要，了解更多关于如何自定义输出、排序和过滤等功能。

 输出字段解释

- **Address**：内存映射的起始地址。
- **Kbytes**：该映射的虚拟内存大小（以千字节为单位）。
- **RSS**：驻留集大小（Resident Set Size），即实际驻留在物理内存中的内存量。
- **Dirty**：脏页的大小，即内存中已修改但尚未写回磁盘的页面。
- **Mode**：内存映射的权限，显示如何访问该内存（如只读 `r--`，可写 `rw-`，可执行 `x` 等）。
- **Mapping**：映射的对象或文件（如程序、共享内存、匿名内存等）。`[ anon ]` 表示匿名内存，`[ stack ]` 表示堆栈内存，`[ shm ]` 表示共享内存等。

在此示例中，MySQL 数据库服务器进程 `/usr/sbin/mysqld` 使用了多个内存段，包括：

- 程序映射（`mysqld`），如 `r-x--` 表示可执行的只读映射。
- 共享库映射（如 `ld-2.30.so`），它们通常是只读映射（`r-x--`）。
- 匿名内存（`[ anon ]`），这些通常是堆或其他动态分配的内存区域。
- 堆栈内存（`[ stack ]`），它用于存储进程的局部变量和函数调用信息。

 扩展选项

- `-x` 选项：打印每个内存映射的扩展信息，如 `RSS`、`Dirty`、`Mode` 和映射的文件名等。
  
- `-X` 选项：显示更多详细信息，涵盖内核页大小、内存页标志等。
  
- `-XX` 选项：显示内核提供的“所有内容”，包括内存页的详细信息和映射的内存比例集大小（`Pss`）。`PSS` 显示映射中有多少私有内存（包括共享内存和用户数），它有助于提供更实际的内存使用情况。

 示例扩展输出

```bash
# pmap -X $(pgrep mysqld) | head -2
5187:   /usr/sbin/mysqld
         Address Perm   Offset Device   Inode    Size    Rss    Pss Referenced
Anonymous LazyFree ShmemPmdMapped Shared_Hugetlb Private_Hugetlb Swap SwapPss Locked
THPeligible ProtectionKey Mapping
```

```bash
# pmap -XX $(pgrep mysqld) | head -2
5187:   /usr/sbin/mysqld
         Address Perm   Offset Device   Inode    Size KernelPageSize MMUPageSize
Rss    Pss Shared_Clean Shared_Dirty Private_Clean Private_Dirty Referenced Anonymous
LazyFree AnonHugePages ShmemPmdMapped Shared_Hugetlb Private_Hugetlb Swap SwapPss
Locked THPeligible ProtectionKey VmFlags Mapping
```

重要字段解释

- **Pss**：比例共享内存（Proportional Set Size），它是共享内存中实际由该进程使用的内存。PSS 通过将共享内存的大小按共享进程数进行除法计算，提供了更精确的内存使用量，避免了重复计数。
- **Referenced**：该内存页面是否已被引用。
- **Private_Clean/Private_Dirty**：私有内存的干净页和脏页数量。
- **Shared_Clean/Shared_Dirty**：共享内存的干净页和脏页数量。

`pmap` 是一个非常有用的工具，用于分析进程的内存使用情况，尤其在多进程共享内存的情况下。通过不同的选项（如 `-x`、`-X` 和 `-XX`），可以获得详细的内存映射信息，帮助您深入了解进程如何分配和使用内存，特别是在高负载或内存瓶颈的场景下。此外，PSS 提供了一种衡量共享内存实际使用量的方式，有助于更准确地评估内存消耗。

### 7.5.10 perf

`perf(1)` 是 Linux 官方的性能分析工具，提供了多种功能，适用于分析系统的各个方面，包括内存使用。本节将重点介绍 `perf` 在内存分析中的应用，特别是页面错误采样和堆栈跟踪。

#### One-Liners

以下是一些 `perf` 命令示例，演示了内存分析的不同功能：

1. **示例页面错误（RSS 增长）以及系统范围内的堆栈跟踪，直到 Ctrl-C：**

    ```bash
    perf record -e page-faults -a -g
    ```

    该命令将在所有 CPU 上记录页面错误，并通过堆栈跟踪 (`-g`) 显示触发该事件的代码路径。

2. **记录 PID 1843 的所有页面错误以及堆栈跟踪，持续 60 秒：**

    ```bash
    perf record -e page-faults -c 1 -p 1843 -g -- sleep 60
    ```

    该命令会监控进程 1843 的页面错误，记录堆栈跟踪，并在 60 秒后停止。

3. **通过 `brk(2)` 记录堆增长，直到 Ctrl-C：**

    ```bash
    perf record -e syscalls:sys_enter_brk -a -g
    ```

    `brk(2)` 系统调用用于调整进程的堆大小，此命令可用于监控堆内存的增长。

4. **记录 NUMA 系统上的页面迁移事件：**

    ```bash
    perf record -e migrate:mm_migrate_pages -a
    ```

    该命令记录 NUMA 系统中的页面迁移事件，适用于分析内存跨节点迁移的情况。

5. **统计内核内存（kmem）事件，每秒打印一份报告：**

    ```bash
    perf stat -e 'kmem:*' -a -I 1000
    ```

6. **统计虚拟内存扫描（vmscan）事件，每秒打印一份报告：**

    ```bash
    perf stat -e 'vmscan:*' -a -I 1000
    ```

7. **统计内存压缩（compaction）事件，每秒打印一份报告：**

    ```bash
    perf stat -e 'compaction:*' -a -I 1000
    ```

8. **使用堆栈跟踪跟踪 `kswapd` 唤醒事件，直到 Ctrl-C：**

    ```bash
    perf record -e vmscan:mm_vmscan_wakeup_kswapd -ag
    ```

9. **分析给定命令的内存访问：**

    ```bash
    perf mem record _command_
    ```

10. **总结内存概况：**

    ```bash
    perf mem report
    ```

11. **使用 `perf report` 汇总事件的记录或采样配置文件，或使用 `perf script --header` 打印详细内容。**

    ```bash
    perf report
    ```

#### 页面错误采样

`perf` 可以记录页面错误的堆栈跟踪，显示触发此事件的代码路径。页面错误发生时，进程的驻留集大小 (RSS) 会增加，因此分析页面错误能够帮助解释为什么进程的内存使用在增长。

以下是一个记录页面错误并进行堆栈跟踪的示例：

```bash
perf record -e page-faults -a -g -- sleep 60
```

### 7.5.11 drsnoop

`drsnoop(8)` 是一个 BCC 工具，用于跟踪直接回收内存的过程，显示与内存回收相关的进程和延迟。它特别用于量化内存受限系统对应用程序性能的影响，能够帮助分析系统内存压力对进程的影响。

当执行 `drsnoop -T` 命令时，工具会显示有关内存回收的信息，其中包括：

- **TIME(s)**: 时间戳，表示事件发生的时间（秒）。
- **COMM**: 触发事件的进程名称。
- **PID**: 进程的 PID。
- **LAT(ms)**: 内存回收的延迟，单位为毫秒。
- **PAGES**: 被回收的内存页数。

例如：

```bash
drsnoop -T
TIME(s)       COMM           PID     LAT(ms) PAGES
0.000000000   java           11266      1.72    57
0.004007000   java           11266      3.21    57
0.011856000   java           11266      2.02    43
0.018315000   java           11266      3.09    55
0.024647000   acpid          1209       6.46    73
[...]
```

输出中显示了多个内存回收事件，这些事件通常是在内存紧张时发生的。`LAT(ms)` 列显示了回收所需的时间，通常在 1 到 7 毫秒之间，`PAGES` 列显示了回收的内存页数。

 工作原理
`drsnoop(8)` 通过跟踪内核中的两个关键事件：

- `vmscan:mm_vmscan_direct_reclaim_begin`：标记内存回收开始的时刻。
- `vmscan:mm_vmscan_direct_reclaim_end`：标记内存回收结束的时刻。

这些事件通常在内存压力较大的情况下发生，回收操作是由内核的 `vmscan` 机制执行的。由于这些事件发生频率较低，因此该工具的开销通常是可以忽略的。

`drsnoop` 支持以下选项：
- `-T`：在输出中包含时间戳。
- `-p PID`：匹配并只跟踪指定进程的回收事件。

### 7.5.12 WSS

`wss(8)` 是一个实验性工具，用于测量进程的工作集大小（WSS，Working Set Size）。它通过监控进程的页表条目（PTE）访问位来跟踪哪些内存页面被频繁访问，进而估算出进程的工作集大小。


以下是使用 `wss` 工具来监控 MySQL 数据库服务器（`mysqld`）工作集大小的输出：

```bash
./wss.pl $(pgrep -n mysqld) 1
Watching PID 423 page references grow, output every 1 seconds...
Est(s)     RSS(MB)    PSS(MB)    Ref(MB)
1.014       403.66     400.59      86.00
2.034       403.66     400.59      90.75
3.054       403.66     400.59      94.29
4.074       403.66     400.59      97.53
5.094       403.66     400.59     100.33
6.114       403.66     400.59     102.44
7.134       403.66     400.59     104.58
8.154       403.66     400.59     106.31
9.174       403.66     400.59     107.76
10.194      403.66     400.59     109.14
```

输出解释：
- **Est(s)**: 测量时间（秒），表示自工具启动以来的时间。
- **RSS(MB)**: 进程的常驻集大小（RSS，Resident Set Size），即该进程当前在物理内存中占用的内存量。
- **PSS(MB)**: 进程的比例共享内存（PSS，Proportional Set Size），它考虑了多个进程共享页面的情况，因此它比 RSS 更能反映进程实际使用的内存。
- **Ref(MB)**: 被引用的内存（Ref，Referenced Memory），表示进程访问的内存大小，单位为 MB。

 工作原理

该工具的工作原理是通过检查进程的页表条目（PTE）中的访问位来确定哪些页面被访问。它会：
1. 重置进程中每个页面的 PTE 访问位。
2. 暂停一段时间。
3. 检查哪些页面的访问位被设置，从而推测哪些页面被频繁访问。

该工具的分辨率基于页面大小，通常为 4 KB。通过这种方式，`wss` 工具可以估算出进程的工作集大小，即频繁访问的内存量。

警告
- `wss` 使用了 `/proc/PID/clear_refs` 和 `/proc/PID/smaps`，这些操作可能会稍微增加内核对页面结构的遍历时间（约 10%）。因此，对于较大的进程（例如大于 100 GB），这种延迟可能会持续超过一秒钟，并消耗一些系统 CPU 时间。
- 工具会重置 PTE 访问位，这可能会导致内核在回收页面时出现一些困惑，特别是在交换活动时。
- 该工具激活了一些可能未在当前系统中使用的旧内核代码，因此建议在实验环境中进行测试，确保您了解其带来的开销。

### 7.5.13 bpftrace

**bpftrace** 是一个基于 BPF（eBPF）的跟踪工具，它提供了一种灵活的高级编程语言，允许开发人员编写强大的单行脚本和短脚本，快速进行自定义应用程序分析，尤其是在性能调优和故障排查时。它被广泛应用于内存分析、CPU性能分析、系统调用分析等方面。该工具适用于根据其他工具提供的信息进行定制的应用程序分析，并且支持许多内存分析示例，适合进行内存、性能和系统健康的实时监控。

bpftrace 可以与其他 BPF 工具结合使用，提供更精确的内存使用情况数据，比如 **oomkill.bt**，可以帮助跟踪“Out of Memory”杀死的事件（由系统 OOM Killer 触发）。此外，bpftrace 支持内存分配的详细分析，例如 libc 中的 `malloc()` 函数调用。

#### 单行示例

bpftrace 提供了一些单行脚本，可以非常高效地进行内存分析，以下是一些有用的例子：

1. **按用户堆栈和进程对 libc malloc() 请求字节进行求和**：
   ```bash
   bpftrace -e 'uprobe:/lib/x86_64-linux-gnu/libc.so.6:malloc { @\[ustack, comm\] = sum(arg0); }'
   ```

2. **按用户堆栈对特定 PID（如 181）的 malloc() 请求字节求和**：
   ```bash
   bpftrace -e 'uprobe:/lib/x86_64-linux-gnu/libc.so.6:malloc /pid == 181/ { @\[ustack\] = sum(arg0); }'
   ```

3. **显示特定 PID 的用户堆栈的 malloc() 请求字节为 2 的幂直方图**：
   ```bash
   bpftrace -e 'uprobe:/lib/x86_64-linux-gnu/libc.so.6:malloc /pid == 181/ { @\[ustack\] = hist(arg0); }'
   ```

4. **求和内核中 kmem 缓存分配字节数**：
   ```bash
   bpftrace -e 't:kmem:kmem_cache_alloc { @bytes\[kstack\] = sum(args->bytes_alloc); }'
   ```

5. **按代码路径计算进程堆扩展 (brk(2))**：
   ```bash
   bpftrace -e 'tracepoint:syscalls:sys_enter_brk { @\[ustack, comm\] = count(); }'
   ```

6. **按进程计算页面错误**：
   ```bash
   bpftrace -e 'software:page-fault:1 { @\[comm, pid\] = count(); }'
   ```

7. **跟踪页迁移事件**：
   ```bash
   bpftrace -e 'tracepoint:migrate:mm_migrate_pages { @ = count(); }'
   ```

8. **跟踪压缩事件**：
   ```bash
   bpftrace -e 't:compaction:mm_compaction_begin { time(); }'
   ```

#### 用户级堆栈跟踪

bpftrace 可以非常方便地通过追踪 libc 的 `malloc()` 函数来观察进程的内存分配。例如，以下命令将显示 PID 4840（如 MySQL 数据库服务器）进程的分配请求的大小，并将其呈现为直方图：

```bash
bpftrace -e 'uprobe:/lib/x86_64-linux-gnu/libc.so.6:malloc /pid == 4840/ { @\[ustack\] = hist(arg0); }'
```

这会记录 malloc 请求的大小，并将它们按用户堆栈分类，便于查看哪些代码路径导致了内存分配。

#### 火焰图生成

火焰图是非常适合展示和理解内存分配模式的工具。你可以利用 bpftrace 和 Brendan Gregg 的 FlameGraph 工具生成基于 malloc() 或页面错误的火焰图。例如，生成 malloc 请求字节的火焰图：

```bash
bpftrace -e 'u:/lib/x86_64-linux-gnu/libc.so.6:malloc /pid == 4840/ { @\[ustack\] = hist(arg0); }' > out.stacks
git clone https://github.com/brendangregg/FlameGraph; cd FlameGraph
./stackcollapse-bpftrace.pl < ../out.stacks | ./flamegraph.pl --hash --bgcolor=green --count=bytes --title="malloc() Bytes Flame Graph" > out.svg
```

同样，也可以根据页面错误生成火焰图：

```bash
bpftrace -e 't:exceptions:page_fault_user { @\[ustack, comm\] = count(); }' > out.stacks
git clone https://github.com/brendangregg/FlameGraph; cd FlameGraph
./stackcollapse-bpftrace.pl < ../out.stacks | ./flamegraph.pl --hash --bgcolor=green --count=pages --title="Page Fault Flame Graph" > out.svg
```

#### 内存内部结构跟踪

bpftrace 还允许开发者深入内核内存事件，探索内存分配和内部结构。比如，你可以列出内核中与内存相关的 tracepoints（例如，kmem 分配和释放事件），如下所示：

```bash
bpftrace -l 'tracepoint:kmem:*'
```

你还可以列出涉及虚拟内存的事件：

```bash
bpftrace -l 't:*:mm_*'
```

这种深度内存分析能够帮助开发人员了解内存的具体使用情况，进而优化性能。

## 7.6 Tuning

您可以做的最重要的内存调整是确保应用程序保留在主内存中，并且分页和交换不会频繁发生。第 7.4 节“方法论”和第 7.5 节“可观测性工具”介绍了如何识别此问题。本节讨论其他内存调整：内核可调参数、配置大页面、分配器和资源控制。

调整的细节（可用选项以及将其设置为什么）取决于操作系统版本和预期工作负载。以下部分按调整类型组织，提供了可用的可调参数以及为什么需要调整它们的示例。

| `vm.percpu_pagelist_fraction`        | 0     | 覆盖可分配给每个 CPU 页面列表的默认最大分数（值 10 限制为页面的 1/10）。    |
| `虚拟机内存过量使用`                          | 0     | 允许合理的内存过度使用（0 = 启发式方法，1 = 总是过度使用，2 = 不允许过度使用）。 |
| `虚拟机交换性`                             | 60    | 有利于通过交换（分页）释放内存而不是从页面缓存中回收内存的程度。               |
| `vm.vfs_cache_pressure`              | 100   | 回收缓存目录和 inode 对象的程度。较低的值更保留它们。                 |
| `kernel.numa_balancing`              | 1     | 启用自动 NUMA 页面平衡。                                |
| `kernel.numa_balancing_scan_size_mb` | 256   | 每次 NUMA 平衡扫描时扫描的页面大小（以 MB 为单位）。                |

 说明

1. **`vm.dirty_background_bytes` 与 `vm.dirty_background_ratio`**：这两个参数是互斥的。`vm.dirty_background_bytes` 设置触发后台写回的脏内存量（字节数），而 `vm.dirty_background_ratio` 设置触发该操作的脏内存百分比。
2. **`vm.dirty_bytes` 与 `vm.dirty_ratio`**：这两个参数也互斥。`vm.dirty_bytes` 是触发写回的脏内存量，`vm.dirty_ratio` 是触发写回的脏内存的比例（百分比）。这些设置控制内存写回的时机。
3. **`vm.min_free_kbytes`**：此参数设置内核在任何时候都应该保持的最小可用内存。它是动态调整的，具体大小取决于主内存的大小，并通过算法进行非线性设置。减少此值可以为应用程序释放更多内存，但可能导致在内存压力较大时系统变得不稳定。
4. **`vm.watermark_scale_factor` 与 `vm.watermark_boost_factor`**：这些参数控制 `kswapd` 进程的行为，帮助它在内存回收时更好地平衡不同内存水位的扫描行为。`watermark_scale_factor` 控制水位之间的距离，而 `watermark_boost_factor` 会在内存碎片问题出现时增加 `kswapd` 的扫描范围。
5. **`vm.percpu_pagelist_fraction`**：此参数控制可分配给每个 CPU 页面列表的页面的最大比例。通过调整该值，系统可以更有效地管理多核 CPU 上的页面列表。
6. **虚拟机内存过量使用**：这个参数允许你控制虚拟机是否应该过度使用内存。默认情况下，内核使用启发式方法决定是否允许虚拟机内存过度使用。设置为 1 会强制过度使用，设置为 2 会禁止过度使用。
7. **`vm.vfs_cache_pressure`**：此参数控制文件系统缓存的回收程度。较低的值会保留更多的缓存，避免过早清理它们。如果设置为 0，则文件系统缓存将不会被回收，这可能会导致内存不足的情况。
8. **`kernel.numa_balancing` 与 `kernel.numa_balancing_scan_size_mb`**：这些参数与 NUMA（非一致性内存访问）相关。`kernel.numa_balancing` 控制是否启用自动 NUMA 页面平衡，`kernel.numa_balancing_scan_size_mb` 控制每次平衡时扫描的页面量。

 常见配置建议

- **避免 OOM 杀死（Out Of Memory Killer）**：可以通过调整 `vm.overcommit_memory` 参数来避免 OOM。设置为 2 可以禁用内存过度使用，这有助于避免 OOM 事件的发生。
- **控制交换行为**：`vm.swappiness` 控制系统的交换策略。较高的值意味着系统倾向于交换更多的应用程序内存以保留页面缓存。设置为较低值（如 0）有助于系统在内存紧张时更长时间地保留应用程序内存，而以牺牲页面缓存为代价进行交换。

内存压力与 NUMA
早期版本的 Linux 内核（例如 Linux 3.13）中，`kernel.numa_balancing` 默认关闭，因为过度激进的 NUMA 页面平衡会占用过多的 CPU 资源。随着内核的更新，这个问题得到了修复，并且增加了 `kernel.numa_balancing_scan_size_mb` 等参数来更精细地控制 NUMA 页面的扫描行为，避免不必要的性能损失。

通过合理配置这些可调参数，系统管理员可以根据实际情况优化内存的使用和管理，尤其是在高负载和大内存系统中，能够有效缓解内存压力和避免系统崩溃。
### 7.6.2 Multiple Page Sizes

在现代计算中，使用大页面（huge pages）来优化内存 I/O 性能是一个重要的手段。大页面通过增加 TLB（Translation Lookaside Buffer）的缓存命中率，从而提高了内存访问效率，尤其是在内存密集型应用中。现代处理器支持多种页面大小，包括传统的 4 KB 页面和更大的 2 MB 页面，甚至更大。

在 Linux 上，使用大页面的方式有多种选择，这些方法通常被称为“巨页”。以下是一些常见的配置和使用方法。

 创建和配置巨页

1. **配置巨页的数量**：可以通过修改 `/proc/sys/vm/nr_hugepages` 来指定巨页的数量。这告诉内核创建多少个大页面。例如：
    
    ```bash
    echo 50 > /proc/sys/vm/nr_hugepages
    ```
    
    这样，系统将分配 50 个巨页。
    
2. **查看巨页信息**：可以通过 `grep Huge /proc/meminfo` 来检查系统中巨页的状态。以下是一些示例输出：
    
    ```bash
    grep Huge /proc/meminfo
    AnonHugePages:         0 kB
    HugePages_Total:      50
    HugePages_Free:       50
    HugePages_Rsvd:        0
    HugePages_Surp:        0
    Hugepagesize:       2048 kB
    ```
    
    这些输出项的含义如下：
    
    - **HugePages_Total**：已分配的巨页总数。
    - **HugePages_Free**：当前空闲的巨页数量。
    - **HugePages_Rsvd**：保留但未使用的巨页数量。
    - **HugePages_Surp**：额外的超量巨页数量。
    - **Hugepagesize**：每个巨页的大小（通常为 2MB）。


应用程序可以通过多种方法来使用这些巨页，以下是一些常见的方式：

1. **使用共享内存**：可以通过共享内存段和 `shmget(2)` 系统调用的 `SHM_HUGETLB` 标志来请求巨页。通过这种方式，多个进程可以共享同一块巨页内存。
2. **使用 `hugetlbfs` 文件系统**：一种常见的做法是创建一个基于巨页的文件系统，供应用程序进行内存映射。例如：
    ```bash
    mkdir /mnt/hugetlbfs
    mount -t hugetlbfs none /mnt/hugetlbfs -o pagesize=2048K
    ```
    
    这将创建一个巨页文件系统挂载点 `/mnt/hugetlbfs`，其中每个页面的大小为 2MB，供应用程序使用。
    
3. **使用 `mmap()`**：应用程序可以通过 `mmap()` 系统调用的 `MAP_ANONYMOUS | MAP_HUGETLB` 标志来请求巨页。例如：
    ```c
    void* addr = mmap(NULL, length, PROT_READ | PROT_WRITE, MAP_ANONYMOUS | MAP_HUGETLB, -1, 0);
    ```
    
    这种方法使得应用程序可以直接从匿名内存中映射巨页。
4. **使用 libhugetlbfs API**：另一个方法是通过 `libhugetlbfs` API 来分配和管理巨页。这个库提供了一个便捷的接口，使得开发者可以在应用程序中使用巨页。

 透明大页面 (THP)

透明大页面（Transparent Huge Pages，简称 THP）是 Linux 内核的一项功能，它自动将普通页面提升为大页面，或将大页面降级为普通页面，而无需应用程序的干预。这种机制减少了开发者管理巨页的复杂性，并能自动根据系统负载调整内存页面的大小。

透明大页面的工作原理是，系统会自动检测内存页面的使用情况，并在合适的时候将 4 KB 的普通页面转换为 2 MB 的大页面。这种自动化的管理减少了手动配置的需求，但透明大页面的性能在某些情况下可能会受到影响（例如，频繁的页面拆分和合并可能会增加开销）。

尽管透明大页面（THP）为用户带来了便利，但在某些情况下，它可能会导致性能问题，特别是与内存碎片化和 CPU 缓存的交互有关。历史上，这些性能问题曾阻碍了透明大页面的广泛应用，但随着 Linux 内核的改进，这些问题已经得到了一定的解决。更多细节可以参考 `Documentation/vm/transhuge.txt` 和 `admin-guide/mm/transhuge.rst`。

### 7.6.3 Allocators

可以使用不同的用户级分配器，从而为多线程应用程序提供改进的性能。这些可以在编译时选择，或者在执行时通过设置 LD\_PRELOAD 环境变量来选择。

例如，可以使用以下方式选择 libtcmalloc 分配器：

export LD\_PRELOAD=/usr/lib/x86\_64-linux-gnu/libtcmalloc\_minimal.so.4

这可以放置在其启动脚本中。

#### 7.6.4 NUMA Binding

在 NUMA 系统上，numactl(8) 命令可用于将进程绑定到 NUMA 节点。这可以提高不需要多个主内存 NUMA 节点的应用程序的性能。用法示例：

\# **numactl --membind=0 3161**

这会将 PID 3161 绑定到 NUMA 节点 0。如果此节点无法满足这些要求，则此进程的未来内存分配将失败。使用此选项时，您还应该研究使用 `--physcpubind` 选项来限制连接到该 NUMA 节点的 CPU 的 CPU 使用率。我通常使用 NUMA 和 CPU 绑定来将进程限制到单个套接字，以避免 CPU 互连访问的性能损失。

使用 numastat(8)（第 7.5.6 节）列出可用的 NUMA 节点。

#### 7.6.5 资源控制

使用 ulimit(1) 可以实现基本资源控制，包括设置主内存限制和虚拟内存限制。

对于 Linux，容器组 (cgroups) 内存子系统提供各种附加控制。这些包括：

*   `memory.limit_in_bytes`：允许的最大用户内存，包括文件缓存使用量，以字节为单位
*   `memory.memsw.limit_in_bytes`：允许的最大内存和交换空间，以字节为单位（当使用交换时）
*   `memory.kmem.limit_in_bytes`：允许的最大内核内存，以字节为单位
*   `memory.tcp.limit_in_bytes`：最大 tcp 缓冲区内存，以字节为单位。
*   `memory.swappiness`：与前面描述的 vm.swappiness 类似，但可以为 cgroup 设置
*   `memory.oom_control`：可以设置为 0，以允许此 cgroup 的 OOM Killer，或设置为 1，以禁用它

Linux 还允许在 /etc/security/limits.conf 中进行系统范围的配置。

有关资源控制的更多信息，请参阅第 11 章，云计算。
