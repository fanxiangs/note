# 第一章

介绍

计算机性能是一门令人兴奋、多样化且具有挑战性的学科。本章向您介绍系统性能领域。本章的学习目标是：

*   了解系统性能、角色、活动和挑战。
    
*   了解可观察性和实验工具之间的区别。
    
*   对性能可观察性有基本的了解：统计、分析、火焰图、跟踪、静态检测和动态检测。
    
*   了解方法论和 Linux 60 秒清单的作用。
    

包括对后面章节的参考，因此这可以作为系统性能和本书的介绍。本章最后通过案例研究来展示系统性能在实践中如何发挥作用。

## 1.1 系统性能

系统性能研究整个计算机系统的性能，包括所有主要软件和硬件组件。数据路径中的所有内容（从存储设备到应用软件）都包含在内，因为它会影响性能。对于分布式系统，这意味着多个服务器和应用程序。如果您没有显示数据路径的环境图，请找到一个或自己绘制；这将帮助您理解组件之间的关系，并确保您不会忽略整个区域。

系统性能的典型目标是通过减少延迟来改善最终用户体验并降低计算成本。降低成本可以通过消除低效率、提高系统吞吐量和一般调整来实现。

图 1.1 显示了单个服务器上的通用系统软件堆栈，包括操作系统 (OS) 内核以及示例数据库和应用程序层。术语“完整堆栈”有时仅用于描述应用程序环境，包括数据库、应用程序和 Web 服务器。然而，当谈到系统性能时，我们使用完整堆栈来表示从应用程序到硬件（硬件）的整个软件堆栈，包括系统库、内核和硬件本身。系统性能研究整个堆栈。

## 1.2 角色

系统性能由各种工作角色来完成，包括系统管理员、站点可靠性工程师、应用程序开发人员、网络工程师、数据库管理员、Web 管理员和其他支持人员。对于其中许多角色来说，性能只是工作的一部分，性能分析侧重于该角色的职责范围：网络团队检查网络，数据库团队检查数据库，等等。对于某些性能问题，找到根本原因或影响因素需要多个团队的合作努力。

一些公司雇用性能工程师，他们的主要活动是性能。他们可以与多个团队合作对环境进行整体研究，这种方法对于解决复杂的性能问题可能至关重要。他们还可以充当中央资源，寻找和开发更好的工具，用于整个环境的性能分析和容量规划。

例如，Netflix 有一个云性能团队，我就是其中的成员。我们协助微服务和 SRE 团队进行性能分析并构建供每个人使用的性能工具。雇用多名性能工程师的公司可以让个人专注于一个或多个领域，从而提供更深层次的支持。例如，大型性能工程团队可能包括内核性能、客户端性能、语言性能（例如 Java）、运行时性能（例如 JVM）、性能工具等领域的专家。

## 1.3 活动

系统性能涉及多种活动。以下活动列表也是软件项目从构思到开发再到生产部署的生命周期中的理想步骤。本书涵盖了帮助执行这些活动的方法和工具。

1.  为未来产品设定性能目标和性能建模。
    
2.  原型软件和硬件的性能表征。
    
3.  在测试环境中对开发中的产品进行性能分析。
    
4.  新产品版本的非回归测试。
    
5.  对产品发布进行基准测试。
    
6.  在目标生产环境中进行概念验证测试。
    
7.  生产中的性能调整。
    
8.  监控正在运行的生产软件。
    
9.  生产问题的性能分析。
    
10.  生产问题的事件审查。
    
11.  性能工具开发以增强生产分析。
    

步骤 1 至 5 包括传统的产品开发，无论是销售给客户的产品还是公司内部服务。然后推出产品，可能首先在目标环境（客户或本地）中进行概念验证测试，或者可能直接进行部署和配置。如果在目标环境中遇到问题（步骤6到9），则意味着该问题在开发阶段没有检测到或修复。

理想情况下，性能工程应该在选择任何硬件或编写软件之前开始：第一步应该是设定目标并创建性能模型。然而，产品的开发通常没有这一步，从而将性能工程工作推迟到问题出现后的某个时间。随着开发过程的每一步，修复由于早期做出的架构决策而出现的性能问题会变得越来越困难。

云计算提供了概念验证测试（步骤 6）的新技术，鼓励跳过前面的步骤（步骤 1 到 5）。其中一种技术是使用生产工作负载的一小部分在单个实例上测试新软件：这称为金丝雀测试。另一种技术使这成为软件部署中的正常步骤：流量逐渐转移到新的实例池，同时使旧池保持在线状态作为备份；这称为蓝绿部署。[^1]有了此类安全无故障选项，新软件通常会在生产中进行测试，而无需事先进行任何性能分析，并在需要时快速恢复。我建议，在实际情况下，您还可以执行早期的活动，以便实现最佳性能（尽管可能存在上市时间方面的原因，需要更快地转向生产）。

[^1]Netflix 使用术语“红黑部署”。

术语“容量规划”可以指许多前述活动。在设计过程中，它包括研究开发软件的资源占用，以了解设计能够在多大程度上满足目标需求。部署后，它包括监视资源使用情况以在问题发生之前进行预测。

生产问题的性能分析（步骤 9）也可能涉及现场可靠性工程师 (SRE)；此步骤之后是事件审查会议（步骤 10），以分析发生的情况、共享调试技术并寻找避免将来发生相同事件的方法。此类会议类似于开发人员回顾（有关回顾及其反模式，请参阅 \[Corry 20\]）。

环境和活动因公司和产品而异，并且在许多情况下，并非执行所有十个步骤。您的工作也可能只专注于这些活动中的部分或其中一项。

## 1.5 性能具有挑战性

系统性能工程是一个具有挑战性的领域，原因有很多，包括它是主观的、复杂的、可能没有单一的根本原因，并且通常涉及多个问题。

### 1.5.1 主观性

技术学科往往是客观的，以至于业内人士以黑白分明的方式看待事物而闻名。软件故障排除也是如此，其中错误要么存在，要么不存在，要么已修复，要么未修复。此类错误通常表现为错误消息，可以轻松解释和理解为存在错误。

另一方面，性能往往是主观的。对于性能问题，可能不清楚是否存在问题，如果存在，何时修复。对于一个用户来说可能被认为是“坏”的性能，因此是一个问题，对于另一个用户来说可能被认为是“好的”性能。

考虑以下信息：

平均磁盘 I/O 响应时间为 1 毫秒。

这是“好”还是“坏”？虽然响应时间或延迟是可用的最佳指标之一，但解释延迟信息很困难。在某种程度上，给定的指标是“好”还是“坏”可能取决于应用程序开发人员和最终用户的性能期望。

通过定义明确的目标，例如设定目标平均响应时间，或要求一定比例的请求落入特定延迟范围内，可以使主观性能变得客观。第 2 章“方法论”中介绍了处理这种主观性的其他方法，包括延迟分析。

### 1.5.2 复杂性

除了主观性之外，由于系统的复杂性和缺乏明显的分析起点，性能也可能是一门具有挑战性的学科。在云计算环境中，您甚至可能不知道首先查看哪个服务器实例。有时我们从一个假设开始，例如归咎于网络或数据库，性能分析师必须弄清楚这是否是正确的方向。

性能问题也可能源于在单独分析时表现良好的子系统之间的复杂交互。当一个发生故障的组件导致其他组件出现性能问题时，可能会由于级联故障而发生这种情况。要理解由此产生的问题，您必须理清组件之间的关系并了解它们如何发挥作用。

瓶颈也可能很复杂并且以意想不到的方式相互关联；修复一个问题可能只是将瓶颈转移到系统的其他地方，而整体性能的改善并没有达到预期的程度。

除了系统的复杂性之外，性能问题也可能是由生产工作负载的复杂特征引起的。这些情况可能永远无法在实验室环境中重现，或者只能间歇性地重现。

解决复杂的性能问题通常需要采用整体方法。整个系统——包括其内部结构和外部相互作用——可能需要进行研究。这需要广泛的技能，并且可以使性能工程成为一项多样化且具有智力挑战性的工作。

正如第 2 章所述，可以使用不同的方法来指导我们解决这些复杂问题；第 6 章到第 10 章包括针对特定系统资源的特定方法：CPU、内存、文件系统、磁盘和网络。 （\[Dekker 18\] 研究了一般复杂系统的分析，包括石油泄漏和金融系统崩溃。）

在某些情况下，这些资源的交互可能会导致性能问题。

### 1.5.3 多重原因

有些性能问题没有单一的根本原因，而是有多个影响因素。想象一个场景，三个正常事件同时发生并组合起来导致性能问题：每个事件都是正常事件，单独来看并不是根本原因。

除了多种原因之外，还可能存在多种性能问题。

### 1.5.4 多重性能问题

发现性能问题通常不是问题；问题在于解决问题。在复杂的软件中通常有很多。为了说明这一点，请尝试查找您的操作系统或应用程序的错误数据库并搜索“性能”一词。你可能会感到惊讶！通常，即使在被认为具有高性能的成熟软件中，也会存在许多已知但尚未解决的性能问题。这在分析性能时带来了另一个困难：真正的任务不是发现问题，而是发现问题。它正在确定哪个或哪些问题最重要。

为此，性能分析师必须量化问题的严重程度。某些性能问题可能不适用于您的工作负载，或者可能仅适用于很小的程度。理想情况下，您不仅要量化问题，还要估计每个问题所获得的潜在加速。当管理层寻找花费工程或运营资源的理由时，此信息可能很有价值。

如果可用的话，一个非常适合性能量化的指标是延迟。

## 1.6 延迟

延迟是衡量等待时间的指标，也是一项重要的性能指标。广泛使用时，它可以表示任何操作完成的时间，例如应用程序请求、数据库查询、文件系统操作等。例如，延迟可以表示网站完全加载的时间，从链接点击到屏幕绘制。对于客户和网站提供商来说，这都是一个重要的指标：高延迟可能会导致沮丧，并且客户可能会将业务转移到其他地方。

作为一个指标，延迟可以允许估计最大加速。例如，图 1.3 描述了一个需要 100 毫秒（即延迟）的数据库查询，在此期间它花费 80 毫秒阻塞等待磁盘读取。可以计算通过消除磁盘读取（例如通过缓存）实现的最大性能改进：从 100 毫秒到 20 毫秒 (100 – 80) 速度提高了五倍 (5x)。这是估计的加速比，计算还量化了性能问题：磁盘读取导致查询运行速度慢了 5 倍。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/01fig03.jpg)

图 1.3 磁盘 I/O 延迟示例

当使用其他指标时，这样的计算是不可能的。例如，每秒 I/O 操作数 (IOPS) 取决于 I/O 的类型，并且通常无法直接比较。如果进行更改以使 IOPS 率降低 80%，则很难知道会对性能产生什么影响。 IOPS 可能会减少 5 倍，但如果每个 I/O 的大小（字节）增加 10 倍怎么办？

如果没有限定术语，延迟也可能是不明确的。例如，在网络中，延迟可以指建立连接的时间，而不是数据传输时间；或者它可以表示连接的总持续时间，包括数据传输（例如，DNS 延迟通常以这种方式测量）。在本书中，我将尽可能使用澄清术语：这些示例最好描述为连接延迟和请求延迟。每章开头还总结了延迟术语。

虽然延迟是一个有用的指标，但它并不总是在需要的时间和地点可用。某些系统区域仅提供平均延迟；有些根本不提供延迟测量。随着基于 BPF[^2] 的新可观测性工具的推出，现在可以从自定义的任意兴趣点测量延迟，并提供显示延迟完整分布的数据。

[^2]BPF 现在是一个名称，不再是缩写词（最初是 Berkeley Packet Filter）。

## 1.7 可观察性

可观察性是指通过观察来理解系统，并对实现这一目标的工具进行分类。这包括使用计数器、分析和跟踪的工具。它不包括基准测试工具，该工具通过执行工作负载实验来修改系统的状态。对于生产环境，应尽可能首先尝试可观察性工具，因为实验工具可能会通过资源争用扰乱生产工作负载。对于空闲的测试环境，您可能希望首先使用基准测试工具来确定硬件性能。

在本节中，我将介绍计数器、指标、分析和跟踪。我将在第 4 章中更详细地解释可观察性，涵盖系统范围与每个进程的可观察性、Linux 可观察性工具及其内部结构。第 5 章到第 11 章包括有关可观测性的特定章节，例如，第 6.6 节介绍了 CPU 可观测性工具。

### 1.7.1 计数器、统计数据和指标

应用程序和内核通常提供有关其状态和活动的数据：操作计数、字节计数、延迟测量、资源利用率和错误率。它们通常被实现为称为计数器的整数变量，这些变量被硬编码在软件中，其中一些是累积的并且总是递增的。这些累积计数器可以由性能工具在不同时间读取以计算统计数据：随时间的变化率、平均值、百分位数等。

### 1.7.2 分析

在系统性能中，术语分析通常指使用执行采样的工具：获取测量的子集（样本）来绘制目标的粗略图片。 CPU 是常见的分析目标。分析 CPU 的常用方法包括对 CPU 代码路径进行定时间隔采样。

CPU 配置文件的有效可视化是火焰图。除了指标之外，CPU 火焰图可以帮助您找到比任何其他工具更多的性能优势。它们不仅揭示了 CPU 问题，还揭示了其他类型的问题，这些问题是通过它们留下的 CPU 足迹发现的。可以通过在自旋路径中查找 CPU 时间来发现锁争用问题；可以通过查找内存分配函数 (malloc()) 中过多的 CPU 时间以及导致这些问题的代码路径来分析内存问题；通过查看缓慢或遗留代码路径中的 CPU 时间，可以发现涉及网络配置错误的性能问题；等等。

### 1.7.3 追踪

跟踪是基于事件的记录，其中捕获并保存事件数据以供以后分析或即时使用以进行自定义摘要和其他操作。有针对系统调用（例如 Linux strace(1)）和网络数据包（例如 Linux tcpdump(8)）的专用跟踪工具；以及可以分析所有软件和硬件事件执行情况的通用跟踪工具（例如 Linux Ftrace、BCC 和 bpftrace）。这些全视追踪器使用各种事件源，特别是静态和动态仪器，以及用于可编程性的 BPF。

#### 静态仪表

静态检测描述了添加到源代码中的硬编码软件检测点。 Linux 内核中有数百个这样的点，用于检测磁盘 I/O、调度程序事件、系统调用等。用于内核静态检测的 Linux 技术称为跟踪点。还有一种用于用户空间软件的静态检测技术，称为用户静态定义跟踪 (USDT)。 USDT 被库（例如 libc）用于检测库调用，并被许多应用程序用于检测服务请求。

作为使用静态检测的示例工具，execsnoop(8) 通过检测 execve(2) 系统调用的跟踪点来打印在跟踪（运行）时创建的新进程。下面显示了 execsnoop(8) 跟踪 SSH 登录的情况：

#### 动态仪表

动态检测在软件运行后通过修改内存指令来插入检测例程来创建检测点。这类似于调试器如何在运行软件的任何函数上插入断点。当断点被命中时，调试器将执行流传递给交互式调试器，而动态检测则运行例程，然后继续目标软件。此功能允许从任何正在运行的软件创建自定义性能统计数据。以前由于缺乏可观察性而无法解决或极其难以解决的问题现在可以得到解决。

动态仪器与传统观察如此不同，以至于一开始很难掌握其作用。考虑一个操作系统内核：分析内核内部结构就像冒险进入一个暗室，在内核工程师认为需要的地方放置蜡烛（系统计数器）。动态仪器就像有一个可以指向任何地方的手电筒。

动态仪器首次创建于 20 世纪 90 年代 \[Hollingsworth 94\]，以及使用它的工具称为动态跟踪器（例如 kerninst \[Tamches 99\]）。对于 Linux，动态检测首次于 2000 年开发 \[Kleen 08\]，并于 2004 年开始合并到内核中 (kprobes)。然而，这些技术并不为人所知并且难以使用。当 Sun Microsystems 在 2005 年推出自己的版本 DTrace 时，这种情况发生了变化，该版本易于使用且生产安全。我开发了许多基于 DTrace 的工具，这些工具显示了它对系统性能的重要性，这些工具得到了广泛使用，并帮助使 DTrace 和动态检测变得众所周知。

#### 带通滤波器

BPF 最初代表 Berkeley Packet Filter，现在为 Linux 的最新动态跟踪工具提供支持。 BPF 最初是一个迷你内核虚拟机，用于加速 tcpdump(8) 表达式的执行。自 2013 年以来，它已被扩展（因此有时称为 eBPF[^3]），成为一种通用的内核内执行环境，提供对资源的安全和快速访问。它的许多新用途包括跟踪工具，它为 BPF 编译器集合 (BCC) 和 bpftrace 前端提供了可编程性。前面所示的 execsnoop(8) 是一个 BCC 工具。[^4]

[^3]eBPF 最初用于描述这种扩展的 BPF；然而，该技术现在简称为 BPF。

[^4]我首先为 DTrace 开发了它，此后我又为其他跟踪器（包括 BCC 和 bpftrace）开发了它。

第 3 章介绍了 BPF，第 15 章介绍了 BPF 跟踪前端：BCC 和 bpftrace。其他章节在其可观察性部分介绍了许多基于 BPF 的跟踪工具；例如，CPU 跟踪工具包含在第 6 章“CPU”、第 6.6 节“可观测性工具”中。我之前还出版过有关跟踪工具的书籍（DTrace \[Gregg 11a\] 和 BPF \[Gregg 19\]）。

perf(1) 和 Ftrace 也是跟踪器，具有与 BPF 前端类似的功能。第 13 章和第 14 章介绍了 perf(1) 和 Ftrace。

## 1.8 实验

除了可观测性工具之外，还有实验工具，其中大多数是基准测试工具。它们通过向系统应用合成工作负载并测量其性能来执行实验。这必须小心地完成，因为实验工具可能会扰乱被测系统的性能。

有一些宏观基准测试工具可以模拟现实世界的工作负载，例如客户端发出应用程序请求；还有一些微基准测试工具可以测试特定组件，例如 CPU、磁盘或网络。打个比方：一辆汽车在拉古纳塞卡赛道的单圈时间可以被视为宏观基准，而其最高时速和 0 至 60 英里/小时的时间可以被视为微观基准。尽管微基准测试通常更容易调试、重复和理解，并且更稳定，但这两种基准测试类型都很重要。

考虑单独使用可观测性工具在生产服务器上调试此性能问题的缺点。由于客户端工作负载的自然变化，网络吞吐量可能会随时变化，并且网络的底层双模式行为可能并不明显。通过将 iperf(1) 与固定工作负载结合使用，您可以消除客户端差异，从而揭示由于其他因素（例如外部网络限制、缓冲区利用率等）造成的差异。

正如我之前建议的，在生产系统上，您应该首先尝试可观察性工具。然而，可观察性工具如此之多，您可能会花费数小时来研究它们，而实验性工具可以更快地得出结果。一位高级性能工程师（Roch Bourbonnais）多年前教给我的一个类比是：你有两只手，可观察性和实验性。仅使用一种类型的工具就像试图单手解决问题一样。

第 6 章至第 10 章包括实验工具部分；例如，第 6 章“CPU”、第 6.8 节“实验”中介绍了 CPU 实验工具。

## 1.9 云计算

云计算是一种按需部署计算资源的方法，通过支持在越来越多的称为实例的小型虚拟系统上部署应用程序，实现了应用程序的快速扩展。这减少了对严格容量规划的需求，因为可以在短时间内从云中添加更多容量。在某些情况下，它还增加了对性能分析的需求，因为使用更少的资源可能意味着更少的系统。由于云使用通常按分钟或小时收费，因此减少系统数量带来的性能提升意味着立即节省成本。将此场景与企业数据中心进行比较，您可能会多年被固定的支持合同所束缚，在合同结束之前无法实现成本节省。

云计算和虚拟化带来的新困难包括管理其他租户的性能影响（有时称为性能隔离）以及每个租户的物理系统可观察性。例如，除非系统正确管理，否则磁盘 I/O 性能可能会由于与邻居的争用而变差。在某些环境中，每个租户可能无法观察到物理磁盘的真实使用情况，这使得识别此问题变得困难。

这些主题将在第 11 章“云计算”中介绍。

## 1.11 案例研究

如果您对系统性能不熟悉，显示执行各种活动的时间和原因的案例研究可以帮助您将它们与当前环境联系起来。这里总结了两个假设的例子；一种是涉及磁盘 I/O 的性能问题，一种是软件更改的性能测试。

这些案例研究描述了本书其他章节中解释的活动。这里描述的方法也不是为了展示正确的方式或唯一的方式，而是为了您批判性地考虑可以进行这些绩效活动的方式。

### 1.11.1 慢磁盘

Sumit 是一家中型公司的系统管理员。数据库团队已提交一份支持票，抱怨其中一个数据库服务器上的“磁盘速度慢”。

Sumit 的首要任务是了解有关该问题的更多信息，收集详细信息以形成问题陈述。该票证声称磁盘速度很慢，但没有解释这是否会导致数据库问题。 Sumit 通过提出以下问题进行回应：

*   目前是否存在数据库性能问题？它是如何测量的？
    
*   这个问题已经存在多久了？
    
*   最近数据库有什么变化吗？
    
*   为什么这些磁盘受到怀疑？
    

数据库团队回复道：“我们有一个慢于 1000 毫秒的查询日志。这些通常不会发生，但在过去的一周里，它们已经增长到每小时几十个。 AcmeMon 显示磁盘正忙。”

这证实了确实存在数据库问题，但也表明磁盘假设可能只是猜测。 Sumit 想要检查磁盘，但他也想快速检查其他资源，以防猜测错误。

AcmeMon 是该公司的基本服务器监控系统，提供基于标准操作系统指标的历史性能图表、mpstat(1)、iostat(1) 和系统实用程序打印的相同指标。 Sumit 登录 AcmeMon 亲自查看。

Sumit 首先使用一种称为 USE 方法（在第 2 章方法论第 2.5.9 节中定义）的方法来快速检查资源瓶颈。正如数据库团队报告的那样，磁盘利用率很高，约为 80%，而其他资源（CPU、网络）利用率则低得多。历史数据显示，过去一周磁盘利用率一直在稳步上升，而CPU利用率则保持稳定。 AcmeMon 不提供磁盘的饱和度或错误统计信息，因此要完成 USE 方法 Sumit 必须登录到服务器并运行一些命令。

他从 /sys 检查磁盘错误计数器；他们是零。他以一秒的间隔运行 iostat(1)，并观察一段时间内的利用率和饱和度指标。 AcmeMon 报告利用率为 80%，但使用一分钟间隔。在一秒的粒度上，Sumit 可以看到磁盘利用率波动，通常达到 100%，并导致饱和度和磁盘 I/O 延迟增加。

为了进一步确认这会阻塞数据库，并且与数据库查询不是异步的，他使用名为 offcputime(8) 的 BCC/BPF 跟踪工具来捕获数据库在内核取消调度时的堆栈跟踪，以及离开 CPU 所花费的时间。堆栈跟踪显示数据库在文件系统读取和查询期间经常阻塞。这对于苏米特来说已经是足够的证据了。

下一个问题是为什么。磁盘性能统计数据似乎与高负载一致。 Sumit 执行工作负载表征以进一步了解这一点，使用 iostat(1) 测量 IOPS、吞吐量、平均磁盘 I/O 延迟和读/写比率。有关更多详细信息，Sumit 可以使用磁盘 I/O 跟踪；然而，他很满意这已经表明磁盘负载较高，而不是磁盘出现问题。

Sumit 在票证中添加了更多详细信息，说明了他检查的内容，并包括用于研究磁盘的命令的屏幕截图。到目前为止，他的总结是磁盘处于高负载状态，这增加了 I/O 延迟并减慢了查询速度。然而，磁盘对于负载来说似乎表现正常。他询问是否有一个简单的解释：数据库负载是否增加了？

数据库团队回应说没有，而且查询率（AcmeMon 没有报告）一直很稳定。这听起来与之前的发现一致，即 CPU 利用率也很稳定。

Sumit 思考在 CPU 没有明显增加的情况下还有什么可能导致更高的磁盘 I/O 负载，并与他的同事对此进行了快速讨论。其中之一建议文件系统碎片，当文件系统接近 100% 容量时，这是预料之中的。 Sumit 发现该比例仅为 30%。

Sumit 知道他可以执行深入分析[^7]来了解磁盘 I/O 的确切原因，但这可能非常耗时。他尝试根据自己对内核 I/O 堆栈的了解，想出其他可以首先快速检查的简单解释。他记得这个磁盘I/O很大程度上是由文件系统缓存（页面缓存）未命中引起的。

[^7]第 2 章“方法论”第 2.5.12 节“深入分析”对此进行了介绍。

Sumit 使用cachestat(8)[^8] 检查文件系统缓存命中率，发现当前为 91%。这听起来很高（很好），但他没有历史数据可以比较。他登录了其他具有类似工作负载的数据库服务器，发现它们的缓存命中率超过了 98%。他还发现其他服务器上的文件系统缓存大小要大得多。

[^8]第 8 章“文件系统”第 8.6.12 节“cachestat”中介绍的 BCC 跟踪工具。

将注意力转向文件系统缓存大小和服务器内存使用情况，他发现了一些被忽视的事情：一个开发项目的原型应用程序正在消耗越来越多的内存，即使它尚未处于生产负载下。该内存取自文件系统缓存的可用内存，从而降低其命中率并导致更多的文件系统读取变成磁盘读取。

Sumit 联系应用程序开发团队，要求他们关闭应用程序并将其移至其他服务器，并提及数据库问题。完成此操作后，Sumit 观察到随着文件系统缓存恢复到其原始大小，AcmeMon 中的磁盘利用率逐渐下降。缓慢的查询返回到零，他关闭了已解决的票证。

### 1.11.2 软件变更

Pamela 是一家小公司的性能和可扩展性工程师，负责所有与性能相关的活动。应用程序开发人员开发了一项新的核心功能，但不确定其引入是否会损害性能。 Pamela 决定在将新应用程序版本部署到生产环境之前对其执行非回归测试[^9]。

[^9]有人称之为回归测试，但它是一项旨在确认软件或硬件更改不会导致性能回归的活动，因此称为非回归测试。

Pamela 获取一台闲置服务器用于测试并搜索客户端工作负载模拟器。应用程序团队不久前编写了一个，尽管它有各种限制和已知的错误。她决定尝试一下，但想确认它是否与当前的生产工作负载足够相似。

她配置服务器以匹配当前的部署配置，并从与服务器不同的系统运行客户端工作负载模拟器。客户端工作负载可以通过研究访问日志来表征，并且她已经使用了一个公司工具来执行此操作。她还在一天中不同时间的生产服务器日志上运行该工具并比较工作负载。客户端模拟器似乎应用了平均生产工作负载，但没有考虑差异。她注意到这一点并继续她的分析。

帕梅拉知道此时可以使用多种方法。她选择了最简单的：增加客户端模拟器的负载直到达到限制（这有时称为压力测试）。客户端模拟器可以配置为每秒执行目标数量的客户端请求，她之前使用的默认值为 1,000。她决定从 100 开始增加负载，并以 100 为增量增加，直到达到限制，每个级别测试一分钟。她编写了一个 shell 脚本来执行测试，该脚本将结果收集到一个文件中以供其他工具进行绘图。

随着负载运行，她执行主动基准测试以确定限制因素是什么。服务器资源和服务器线程似乎大部分闲置。客户端模拟器显示请求吞吐量稳定在每秒 700 个左右。

她切换到新的软件版本并重复测试。这也达到了 700 大关并趋于平稳。她还分析服务器以寻找限制因素，但同样找不到任何限制因素。

她绘制了结果，显示完成的请求率与负载的关系，以直观地识别可扩展性配置文件。两者似乎都达到了突然的天花板。

虽然这些软件版本似乎具有相似的性能特征，但帕梅拉感到失望的是，她无法识别导致可扩展性上限的限制因素。她知道她只检查了服务器资源，而限制器可能是应用程序逻辑问题。它也可能在其他地方：网络或客户端模拟器。

Pamela 想知道是否需要采用不同的方法，例如运行固定速率的操作，然后表征资源使用情况（CPU、磁盘 I/O、网络 I/O），以便可以用单个客户端请求来表示。她以每秒 700 次的速度运行当前软件和新软件的模拟器，并测量资源消耗。当前软件使 32 个 CPU 对于给定负载的平均利用率达到 20%。对于相同的负载，新软件将相同的 CPU 利用率提高到 30%。看来这确实是一种回归，一种消耗更多CPU资源的回归。

Pamela 出于好奇了解 700 个限制，因此启动了更高的负载，然后研究了数据路径中的所有组件，包括网络、客户端系统和客户端工作负载生成器。她还对服务器和客户端软件进行深入分析。她记录了她检查过的内容，包括屏幕截图，以供参考。

为了调查客户端软件，她执行了线程状态分析，发现它是单线程的！该线程 100% 的时间都在 CPU 上执行。这让她相信这是测试的限制因素。

作为一项实验，她在不同的客户端系统上并行启动客户端软件。通过这种方式，她可以将服务器的当前软件和新软件的 CPU 利用率提高到 100%。当前版本达到 3,500 个请求/秒，新版本达到 2,300 个请求/秒，与之前的资源消耗调查结果一致。

Pamela 通知应用程序开发人员，新软件版本出现了回归，她开始使用 CPU 火焰图分析 CPU 使用情况，以了解原因：哪些代码路径在起作用。她指出，测试了平均生产工作负载，但未测试不同的工作负载。她还提交了一个错误，指出客户端工作负载生成器是单线程的，这可能成为瓶颈。
