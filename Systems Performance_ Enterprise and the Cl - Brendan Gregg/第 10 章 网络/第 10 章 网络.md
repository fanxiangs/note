# 第10章

网络

随着系统变得更加分布式，特别是在云计算环境中，网络在性能方面发挥着越来越大的作用。网络性能的常见任务包括改善网络延迟和吞吐量，以及消除可能由数据包丢失或延迟引起的延迟异常值。

网络分析涵盖硬件和软件。硬件是物理网络，包括网络接口卡、交换机、路由器和网关（这些通常也有软件）。该软件是内核网络堆栈，包括网络设备驱动程序、数据包队列和数据包调度程序以及网络协议的实现。较低层协议通常是内核软件（IP、TCP、UDP 等），高层协议通常是库或应用程序软件（例如 HTTP）。

考虑到潜在的拥塞及其固有的复杂性（归咎于未知因素），网络经常被指责性能不佳。本章将展示如何弄清楚到底发生了什么，这可能会证明网络无罪，以便继续进行分析。

本章的学习目标是：

*   了解网络模型和概念。
    
*   了解网络延迟的不同衡量标准。
    
*   具备常用网络协议的应用知识。
    
*   熟悉网络硬件内部结构。
    
*   熟悉套接字和设备的内核路径。
    
*   遵循不同的网络分析方法。
    
*   表征系统范围和每个进程的网络 I/O。
    
*   识别 TCP 重传引起的问题。
    
*   使用跟踪工具调查网络内部结构。
    
*   了解网络可调参数。
    

本章由六个部分组成，前三部分提供网络分析的基础，后三部分展示其在基于 Linux 的系统中的实际应用。各部分如下：

*   背景介绍了网络相关的术语、模型和关键网络性能概念。
    
*   体系结构提供物理网络组件和网络堆栈的通用描述。
    
*   方法论描述了观察性和实验性的性能分析方法。
    
*   可观测性工具展示了适用于基于 Linux 的系统的网络性能可观测性工具。
    
*   实验总结了网络基准和实验工具。
    
*   调整描述了示例可调参数。
    

网络基础知识（例如 TCP 和 IP 的作用）是本章假定的知识。

## 10.1 术语

作为参考，本章中使用的网络相关术语包括：

*   接口：术语接口端口是指物理网络连接器。术语“接口”或“链路”是指操作系统看到和配置的网络接口端口的逻辑实例。 （并非所有操作系统接口都由硬件支持：有些是虚拟的。）
    
*   数据包：术语“数据包”是指数据包交换网络中的消息，例如 IP 数据包。
    
*   帧：物理网络级消息，例如以太网帧。
    
*   Socket：源自 BSD 的用于网络端点的 API。
    
*   带宽：网络类型的最大数据传输速率，通常以每秒位数来衡量。 “100 GbE”是带宽为 100 Gbits/s 的以太网。每个方向可能存在带宽限制，因此 100 GbE 可能能够并行传输 100 Gbit/s 和接收 100 Gbit/s（总吞吐量为 200 Gbit/s）。
    
*   吞吐量：网络端点之间的当前数据传输速率，以每秒位数或每秒字节数为单位。
    
*   延迟：网络延迟可以指消息在端点之间往返所需的时间，或建立连接（例如 TCP 握手）所需的时间，不包括随后的数据传输时间。
    

本章将介绍其他术语。该术语表包含可供参考的基本术语，包括客户端、以太网、主机、IP、RFC、服务器、SYN、ACK。另请参阅第 2 章和第 3 章中的术语部分。

## 10.3 概念

以下是网络和网络性能方面的一些重要概念的精选。

### 10.3.1 网络和路由

网络是一组连接的主机，通过网络协议地址关联。出于多种原因，特别是可扩展性，拥有多个网络（而不是一个庞大的全球网络）是可取的。一些网络消息将广播到所有相邻主机。通过创建更小的子网，可以在本地隔离此类广播消息，这样它们就不会造成大规模的洪泛问题。这也是隔离常规消息仅传输到源和目的地之间的网络的基础，从而更有效地利用网络基础设施。

路由管理跨这些网络的消息（称为数据包）的传递。路由的作用如图 10.4 所示。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/10fig04.jpg)

图10.4 通过路由器连接的网络

从主机A的角度来看，localhost就是主机A本身。图中的所有其他主机都是远程主机。

主机 A 可以通过本地网络连接到主机 B，通常由网络交换机驱动（参见第 10.4 节，体系结构）。主机 A 可以通过路由器 1 连接到主机 C，并通过路由器 1、2 和 3 连接到主机 D。由于路由器等网络组件是共享的，来自其他流量（例如主机 C 到主机 E）的争用可能会损害性能。

主机对之间的连接涉及单播传输。多播传输允许发送方同时向多个目的地进行传输，这些目的地可能跨越多个网络。这必须得到路由器配置的支持才能允许传送。在公共云环境中，它可能会被阻止。

除了路由器之外，典型的网络还将使用防火墙来提高安全性，阻止主机之间不需要的连接。

路由数据包所需的地址信息包含在 IP 标头中。

### 10.3.2 协议

IP、TCP 和 UDP 等网络协议标准是系统和设备之间通信的必要要求。通信是通过传输称为数据包的可路由消息来执行的，通常通过有效负载数据的封装来执行。

网络协议具有不同的性能特征，这些特征源于原始协议设计、扩展或软件或硬件的特殊处理。例如，不同版本的 IP 协议 IPv4 和 IPv6 可以由不同的内核代码路径处理，并且可以表现出不同的性能特征。其他协议的设计表现不同，可以在适合工作负载时进行选择：示例包括流控制传输协议 (SCTP)、多路径 TCP (MPTCP) 和 QUIC。

通常，还有一些系统可调参数可以通过更改缓冲区大小、算法和各种计时器等设置来影响协议性能。特定协议的这些差异将在后面的部分中描述。

协议通常通过使用封装来传输数据。

### 10.3.3 封装

封装将元数据添加到有效负载的开头（标头）、结尾（页脚）或两者。这不会改变有效负载数据，尽管它确实稍微增加了消息的总大小，这会增加一些传输开销。

### 10.3.4 数据包大小

数据包的大小及其有效负载会影响性能，较大的数据包可提高吞吐量并减少数据包开销。对于 TCP/IP 和以太网，数据包可以在 54 到 9,054 字节之间，包括 54 字节（或更多，取决于选项或版本）的协议标头。

数据包大小通常受到网络接口最大传输单元 (MTU) 大小的限制，对于许多以太网网络，该最大传输单元配置为 1,500 字节。 1500 MTU 大小的起源来自以太网的早期版本，以及平衡 NIC 缓冲内存成本和传输延迟等因素的需要 \[Nosachev 20\]。主机竞相使用共享介质（同轴电缆或以太网集线器），并且较大的尺寸会增加主机等待轮到的延迟。

以太网现在支持高达约 9,000 字节的较大数据包（帧），称为巨型帧。这些可以通过需要更少的数据包来提高网络吞吐量性能以及数据传输的延迟。

两个组件的融合干扰了巨型帧的采用：较旧的网络硬件和配置错误的防火墙。不支持巨型帧的旧硬件可以使用 IP 协议对数据包进行分段（导致数据包重组的性能成本），或者以 ICMP“无法分段”错误进行响应，让发送方知道减小数据包大小。现在，错误配置的防火墙开始发挥作用：过去曾发生过基于 ICMP 的攻击（包括“死亡之ping”），一些防火墙管理员对此的反应是阻止所有 ICMP。这可以防止有用的“无法分段”消息到达发送者，并导致网络数据包在数据包大小增加超过 1,500 时被静默丢弃。如果收到 ICMP 消息并发生碎片，则还存在碎片数据包被不支持它们的设备丢弃的风险。为了避免这些问题，许多系统坚持使用 1,500 MTU 默认值。

网络接口卡功能（包括 TCP 卸载和大段卸载）提高了 1,500 MTU 帧的性能。它们将更大的缓冲区发送到网卡，然后网卡可以使用专用和优化的硬件将它们分成更小的帧。这在一定程度上缩小了1,500和9,000 MTU网络性能之间的差距。

### 10.3.5 延迟

延迟是网络性能的一个重要指标，可以通过不同的方式进行测量，包括名称解析延迟、ping 延迟、连接延迟、首字节延迟、往返时间和连接生命周期。这些被描述为由连接到服务器的客户端测量的。

#### 名称解析延迟

建立与远程主机的连接时，主机名通常会解析为 IP 地址，例如通过 DNS 解析。所花费的时间可以单独测量为名称解析延迟。这种延迟的最坏情况是名称解析超时，这可能需要数十秒。

操作系统通常提供提供缓存的名称解析服务，以便后续的 DNS 查找可以从缓存中快速解析。有时应用程序仅使用 IP 地址而不使用名称，因此完全避免了 DNS 延迟。

#### Ping 延迟

这是 ICMP 回显请求到回显响应的时间，由 ping(1) 命令测量。该时间用于测量主机之间的网络延迟，包括主机之间的跃点，并且测量为网络请求进行往返所需的时间。它之所以被广泛使用，是因为它简单且通常容易获得：许多操作系统默认都会响应 ping。它可能无法准确反映应用程序请求的往返时间，因为路由器可能以不同的优先级处理 ICMP。

#### 连接延迟

连接延迟是指在传输任何数据之前建立网络连接的时间。对于 TCP 连接延迟，这是 TCP 握手时间。从客户端测量，是从发送SYN到收到相应SYN-ACK的时间。连接延迟可能更好地称为连接建立延迟，以明确区分它与连接生命周期。

连接延迟与 ping 延迟类似，尽管它使用更多的内核代码来建立连接，并且包括重新传输任何丢弃的数据包的时间。特别是，如果 TCP SYN 数据包的积压已满，服务器可能会丢弃该数据包，从而导致客户端发送基于计时器的 SYN 重传。这发生在 TCP 握手期间，因此连接延迟可能包括重新传输延迟，增加一秒或多秒。

连接延迟之后是第一个字节延迟。

#### 第一个字节延迟

第一个字节延迟也称为第一个字节时间 (TTFB)，是指从建立连接到接收到第一个数据字节的时间。这包括远程主机接受连接、调度为其提供服务的线程以及该线程执行和发送第一个字节的时间。

虽然 ping 和连接延迟衡量的是网络产生的延迟，但首字节延迟包括目标服务器的思考时间。如果服务器过载并且需要时间来处理请求（例如 TCP 积压）和调度服务器（CPU 调度程序延迟），这可能包括延迟。

#### 往返时间

往返时间 (RTT) 描述网络请求在端点之间进行往返所需的时间。这包括信号传播时间和每个网络跃点的处理时间。预期用途是确定网络的延迟，因此理想情况下，RTT 主要由请求和回复数据包在网络上花费的时间（而不是远程主机服务请求的时间）决定。人们经常研究 ICMP 回显请求的 RTT，因为远程主机处理时间很短。

#### 连接寿命

连接生命周期是指从网络连接建立到关闭的时间。某些协议使用保持活动策略，延长连接的持续时间，以便将来的操作可以使用现有连接并避免连接建立（和 TLS 建立）的开销和延迟。

有关更多网络延迟测量，请参阅第 10.5.4 节“延迟分析”，其中描述了使用它们来诊断网络性能。

### 10.3.6 缓冲

尽管可能会遇到各种网络延迟，但通过在发送器和接收器上使用缓冲，可以以高速率维持网络吞吐量。较大的缓冲区可以通过在阻塞和等待确认之前继续发送数据来减轻较长往返时间的影响。

TCP 采用缓冲以及滑动发送窗口来提高吞吐量。网络套接字也有缓冲区，应用程序也可以使用自己的缓冲区，在发送之前聚合数据。

缓冲也可以由外部网络组件（例如交换机和路由器）执行，以提高其自身的吞吐量。不幸的是，在这些组件上使用大缓冲区可能会导致缓冲区膨胀，即数据包排队时间较长。这会导致主机上的 TCP 拥塞避免，从而限制性能。 Linux 3.x 内核中添加了一些功能来解决此问题（包括字节队列限制、CoDel 排队规则 \[Nichols 12\] 和 TCP 小队列）。还有一个讨论该问题的网站\[Bufferbloat 20\]。

缓冲（或大缓冲）的功能可能最好由端点（主机）而不是中间网络节点提供，遵循称为端到端参数的原则 \[Saltzer 84\]。

### 10.3.7 连接积压

另一种类型的缓冲用于初始连接请求。 TCP 实现了积压，SYN 请求可以在被用户进程接受之前在内核中排队。当 TCP 连接请求过多，进程无法及时接受时，积压达到限制，SYN 数据包将被丢弃，稍后由客户端重新传输。这些数据包的重传会导致客户端连接时间延迟。该限制是可调的：它是listen(2)系统调用的一个参数，并且内核还可以提供系统范围的限制。

积压丢弃和 SYN 重传是主机过载的指标。

### 10.3.8 接口协商

网络接口可以以不同的模式运行，在连接的收发器之间自动协商。一些例子是：

*   带宽：例如，10、100、1,000、10,000、40,000、100,000 Mbits/s
    
*   双工：半双工或全双工
    

这些示例来自以太网，以太网倾向于使用以 10 为基数的整数来限制带宽。其他物理层协议（例如 SONET）具有一组不同的可能带宽。

网络接口通常根据其最高带宽和协议进行描述，例如 1 Gbit/s 以太网 (1 GbE)。然而，如果需要，该接口可以自动协商到较低的速度。如果另一个端点无法更快地运行，或者无法适应连接介质的物理问题（接线不良），则可能会发生这种情况。

全双工模式允许双向同时传输，具有单独的发送和接收路径，每个路径都可以在全带宽下运行。半双工模式一次只允许一个方向。

### 10.3.9 拥塞避免

网络是共享资源，当流量负载较高时，网络可能会变得拥塞。这可能会导致性能问题：例如，路由器或交换机可能会丢弃数据包，从而导致导致延迟的 TCP 重新传输。当接收高数据包速率时，主机也会变得不堪重负，并且可能会自行丢弃数据包。

有很多机制可以避免这些问题；应研究这些机制，并在必要时进行调整，以提高负载下的可扩展性。不同协议的示例包括：

*   以太网：不堪重负的主机可能会向发送器发送暂停帧，请求它们暂停传输 (IEEE 802.3x)。每个类别还有优先级类别和优先级暂停帧。
    
*   IP：包括显式拥塞通知 (ECN) 字段。
    
*   TCP：包括拥塞窗口，并且可以使用各种拥塞控制算法。
    

后面的部分将更详细地描述 IP ECN 和 TCP 拥塞控制算法。

### 10.3.10 使用

网络接口利用率可以计算为当前吞吐量除以最大带宽。考虑到自动协商导致的可变带宽和双工，计算这一点并不像听起来那么简单。

对于全双工，利用率适用于每个方向，并以当前协商带宽上该方向的当前吞吐量来衡量。通常，最重要的只是一个方向，因为主机通常是不对称的：服务器传输量大，客户端接收量大。

一旦网络接口方向达到 100% 利用率，它就会成为瓶颈，限制性能。

一些性能工具仅以数据包而非字节的形式报告活动。由于数据包大小可能变化很大（如前所述），因此不可能将数据包计数与字节计数关联起来以计算吞吐量或（基于吞吐量的）利用率。

### 10.3.11 本地连接

网络连接可以发生在同一系统上的两个应用程序之间。这些是本地主机连接并使用虚拟网络接口：环回。

分布式应用程序环境通常分为通过网络进行通信的逻辑部分。这些可以包括 Web 服务器、数据库服务器、缓存服务器、代理服务器和应用程序服务器。如果它们在同一主机上运行，则它们的连接将连接到本地主机。

通过 IP 连接到本地主机是进程间通信 (IPC) 的 IP 套接字技术。另一种技术是 Unix 域套接字 (UDS)，它在文件系统上创建文件以进行通信。使用 UDS 的性能可能会更好，因为可以绕过内核 TCP/IP 堆栈，跳过内核代码和协议数据包封装的开销。

对于 TCP/IP 套接字，内核可能会在握手后检测本地主机连接，然后快捷地使用 TCP/IP 堆栈进行数据传输，从而提高性能。这是作为 Linux 内核功能开发的，称为 TCP Friends，但没有合并 \[Corbet 12\]。 BPF 现在可以在 Linux 上用于此目的，正如用于容器网络性能和安全性的 Cilium 软件 \[Cilium 20a\] 所做的那样。

## 10.4 架构

本节介绍网络架构：协议、硬件和软件。这些已被总结为性能分析和调优的背景，重点是性能特征。有关更多详细信息，包括一般网络主题，请参阅网络文本 \[Stevens 93\]\[Hassan 03\]、RFC 和网络硬件供应商手册。其中一些列在本章末尾。

### 10.4.1 协议

本节总结了IP、TCP、UDP和QUIC的性能特点和特点。这些协议如何在硬件和软件中实现（包括分段卸载、连接队列和缓冲等功能）将在后面的硬件和软件部分中介绍。

#### 知识产权

互联网协议 (IP) 版本 4 和 6 包括一个用于设置所需连接性能的字段：IPv4 中的服务类型字段和 IPv6 中的流量类别字段。这些字段后来被重新定义为包含差分服务代码点 (DSCP) (RFC 2474) \[Nichols 98\] 和显式拥塞通知 (ECN) 字段 (RFC 3168) \[Ramakrishnan 01\]。

DSCP 旨在支持不同的服务类别，每个服务类别具有不同的特征，包括丢包概率。示例服务类别包括：电话、广播视频、低延迟数据、高吞吐量数据和低优先级数据。

ECN 是一种机制，允许路径上的服务器、路由器或交换机通过在 IP 标头中设置一个位来明确表示存在拥塞，而不是丢弃数据包。接收器会将这个信号回传给发送器，然后发送器可以限制传输。这提供了避免拥塞的好处，而不会带来丢包的损失（前提是 ECN 位在网络中正确使用）。

#### 传输控制协议

传输控制协议 (TCP) 是一种常用的 Internet 标准，用于创建可靠的网络连接。 TCP 由 RFC 793 \[Postel 81\] 和后来的补充内容指定。

在性能方面，TCP 通过使用缓冲和滑动窗口，即使在高延迟网络上也可以提供高吞吐量。 TCP 还采用拥塞控制和发送方设置的拥塞窗口，以便它可以在不同和变化的网络上保持高且可靠的传输速率。拥塞控制可避免发送过多的数据包，否则会导致拥塞和性能下降。

以下是 TCP 性能特性的摘要，包括自原始规范以来的新增内容：

*   滑动窗口：这允许在收到确认之前在网络上发送最多达到窗口大小的多个数据包，即使在高延迟网络上也能提供高吞吐量。窗口的大小由接收方通告，以指示当时愿意接收多少个数据包。
    
*   拥塞避免：防止发送过多数据并导致饱和，从而导致数据包丢失和性能下降。
    
*   慢启动：TCP 拥塞控制的一部分，以较小的拥塞窗口开始，然后随着在一定时间内收到确认 (ACK) 而增加拥塞窗口。当它们不存在时，拥塞窗口就会减小。
    
*   选择性确认 (SACK)：允许 TCP 确认不连续的数据包，从而减少所需的重传次数。
    
*   快速重传：TCP 可以根据重复 ACK 的到达来重传丢弃的数据包，而不是等待计时器。这些是往返时间的函数，而不是通常慢得多的计时器。
    
*   快速恢复：在检测到重复的 ACK 后，通过重置连接以执行慢启动来恢复 TCP 性能。
    
*   TCP 快速打开：允许客户端将数据包含在 SYN 数据包中，以便服务器请求处理可以更早开始，而无需等待 SYN 握手 (RFC7413)。这可以使用加密 cookie 来验证客户端。
    
*   TCP 时间戳：包括在 ACK 中返回的已发送数据包的时间戳，以便可以测量往返时间 (RFC 1323) \[Jacobson 92\]。
    
*   TCP SYN cookie：在可能的 SYN 洪水攻击（完全积压）期间向客户端提供加密 cookie，以便合法客户端可以继续连接，并且服务器不需要为这些连接尝试存储额外的数据。
    

在某些情况下，这些功能是通过使用添加到协议标头的扩展 TCP 选项来实现的。

TCP 性能的重要主题包括三向握手、重复 ACK 检测、拥塞控制算法、Nagle、延迟 ACK、SACK 和 FACK。

##### 三次握手

连接是通过主机之间的三向握手建立的。一台主机被动地监听连接；另一个主动发起连接。澄清术语：被动和主动来自 RFC 793 \[Postel 81\]；但是，它们通常在套接字 API 之后分别称为“监听”和“连接”。对于客户端/服务器模型，服务器执行listen，客户端执行connect。

##### 状态和定时器

TCP 会话根据数据包和套接字事件在 TCP 状态之间切换。这些状态是 LISTEN、SYN-SENT、SYN-RECEIVED、ESTABLISHED、FIN-WAIT-1、FIN-WAIT-2、CLOSE-WAIT、CLOSING、LAST-ACK、TIME-WAIT 和 CLOSED \[Postal 80\]。性能分析通常关注处于 ESTABLISHED 状态的连接，即活动连接。此类连接可能正在传输数据，或者空闲等待下一个事件：数据传输或关闭事件。

完全关闭的会话会进入 TIME\_WAIT[^2] 状态，以便迟到的数据包不会与同一端口上的新连接错误关联。这可能会导致端口耗尽的性能问题，如第 10.5.7 节“TCP 分析”中所述。

[^2]虽然它经常被编写（和编程）为 TIME\_WAIT，但 RFC 793 使用 TIME-WAIT。

有些状态有与其关联的计时器。 TIME\_WAIT 通常为两分钟（某些内核，例如 Windows 内核，允许对其进行调整）。 ESTABLISHED 上还可能有一个“保持活动”计时器，设置为较长的持续时间（例如，两个小时），以触发探测数据包来检查远程主机是否仍然活动。

##### 重复ACK检测

快速重传和快速恢复算法使用重复 ACK 检测来快速检测发送的数据包（或其 ACK）何时丢失。其工作原理如下：

1.  发送方发送序列号为 10 的数据包。
    
2.  接收方回复序列号 11 的 ACK。
    
3.  发送方发送 11、12 和 13。
    
4.  数据包 11 被丢弃。
    
5.  接收方通过发送其仍在等待的 11 的 ACK 来回复 12 和 13。
    
6.  发送方收到 11 的重复 ACK。
    

各种拥塞避免算法也使用重复 ACK 检测。

##### 重传

TCP 检测和重传丢失数据包的两种常用机制是：

*   基于定时器的重传：当一段时间过去但尚未收到数据包确认时，就会发生这种情况。该时间是 TCP 重传超时，根据连接往返时间 (RTT) 动态计算。在 Linux 上，第一次重传 [^3] 至少需要 200 毫秒 (TCP\_RTO\_MIN)，随后的重传将慢得多，遵循指数退避算法，使超时加倍。
    
    [^3]这似乎违反了 RFC6298，其中规定 RTO 最小值为一秒 \[Paxson 11\]。
    
*   快速重传：当重复的 ACK 到达时，TCP 可以假设数据包已丢失并立即重传。
    

为了进一步提高性能，开发了额外的机制来避免基于计时器的重传。出现的一个问题是当最后传输的数据包丢失，并且没有后续数据包触发重复 ACK 检测时。 （考虑前面第 13 个数据包丢失的示例。）这是通过尾部丢失探测 (TLP) 解决的，它会在上次传输短暂超时后发送一个额外的数据包（探测），以帮助检测数据包丢失 \[Dukkipati 13\]。

拥塞控制算法还可能在存在重传的情况下限制吞吐量。

##### 拥塞控制

拥塞控制算法的开发是为了维持拥塞网络的性能。某些操作系统（包括基于 Linux 的操作系统）允许选择算法作为系统调整的一部分。这些算法包括：

*   Reno：三次重复ACK触发：拥塞窗口减半、慢启动阈值减半、快速重传和快速恢复。
    
*   Tahoe：三次重复 ACK 触发：快速重传、将慢启动阈值减半、拥塞窗口设置为一个最大分段大小 (MSS) 以及慢启动状态。 （Tahoe 与 Reno 一起最初是为 4.3BSD 开发的。）
    
*   CUBIC：使用三次函数（因此得名）来缩放窗口，并使用“混合启动”函数来退出慢启动。 CUBIC 往往比 Reno 更激进，并且是 Linux 中的默认设置。
    
*   BBR：BBR 不是基于窗口，而是使用探测阶段构建网络路径特征（RTT 和带宽）的显式模型。 BBR 可以在某些网络路径上提供显着更好的性能，同时会损害其他网络路径的性能。 BBRv2 目前正在开发中，有望修复 v1 的一些缺陷。
    
*   DCTCP：数据中心 TCP 依赖于配置为在非常浅的队列占用情况下发出显式拥塞通知 (ECN) 标记的交换机，以快速增加到可用带宽 (RFC 8257) \[Bensley 17\]。这使得 DCTCP 不适合跨 Internet 部署，但在适当配置的受控环境中，它可以显着提高性能。
    

之前未列出的其他算法包括 Vegas、New Reno 和 Hybla。

拥塞控制算法可以对网络性能产生很大的影响。例如，Netflix 云服务使用 BBR，发现它可以在严重丢包的情况下将吞吐量提高三倍 \[Ather 17\]。在分析 TCP 性能时，了解这些算法在不同网络条件下的反应是一项重要活动。

2020 年发布的 Linux 5.6 添加了对 BPF 中开发新拥塞控制算法的支持 \[Corbet 20\]。这使得它们可以由最终用户定义并按需加载。

##### 内格尔

该算法 (RFC 896) \[Nagle 84\] 通过延迟传输来减少网络上小数据包的数量，以允许更多数据到达并合并。仅当管道中有数据并且已经遇到延迟时，才会延迟数据包。

系统可能会提供一个可调参数或套接字选项来禁用 Nagle，如果其操作与延迟 ACK 冲突，这可能是必要的（请参见第 10.8.2 节，套接字选项）。

##### 延迟的 ACK

该算法 (RFC 1122) \[Braden 89\] 将 ACK 的发送延迟最多 500 ms，以便可以组合多个 ACK。还可以组合其他 TCP 控制消息，从而减少网络上的数据包数量。

与 Nagle 一样，系统可以提供可调参数来禁用此行为。

##### SACK、FACK 和 RACK

TCP 选择性确认 (SACK) 算法允许接收方通知发送方它收到了不连续的数据块。如果没有这个，数据包丢失最终将导致整个发送窗口被重新传输，以保留顺序确认方案。这会损害 TCP 性能，大多数支持 SACK 的现代操作系统都可以避免这种情况。

SACK 已通过前向确认 (FACK) 进行了扩展，Linux 默认支持 FACK。 FACK 跟踪额外的状态并更好地调节网络中未完成的数据量，从而提高整体性能 \[Mathis 96\]。

SACK和FACK都用于改善丢包恢复。一种较新的算法，Recent ACKnowledgment（RACK；现在称为 RACK-TLP，并结合了 TLP）使用来自 ACK 的时间信息来实现更好的丢失检测和恢复，而不是单独使用 ACK 序列 \[Cheng 20\]。对于 FreeBSD，Netflix 基于 RACK、TLP 和其他功能开发了一种新的重构 TCP 堆栈，称为 RACK \[Stewart 18\]。

##### 初始窗口

初始窗口 (IW) 是 TCP 发送方在连接开始时在等待发送方确认之前传输的数据包数量。对于短流，例如典型的 HTTP 连接，足够大的 IW 来跨越传输的数据可以大大减少完成时间，从而提高性能。然而，较大的 IW 可能会带来拥塞和数据包丢失的风险。当多个流同时启动时，这种情况尤其复杂。

Linux 默认值（10 个数据包，又名 IW10）在慢速链接或启动许多连接时可能过高；其他操作系统默认为 2 或 4 个数据包（IW2 或 IW4）。

#### UDP协议

用户数据报协议 (UDP) 是一种常用的互联网标准，用于通过网络发送消息（称为数据报）(RFC 768) \[Postel 80\]。在性能方面，UDP提供：

*   简单性：简单且小的协议头减少了计算和大小的开销。
    
*   无状态：降低连接和传输的开销。
    
*   无重传：这会显着增加 TCP 连接的延迟。
    

虽然 UDP 简单且性能通常很高，但它并不可靠，并且数据可能会丢失或乱序接收。这使得它不适合许多类型的连接。 UDP 也没有拥塞避免功能，因此可能会导致网络拥塞。

某些服务（包括 NFS 版本）可以根据需要配置为通过 TCP 或 UDP 运行。其他执行广播或多播数据的程序可能只能使用 UDP。

UDP 的一个主要用途是 DNS。由于 UDP 的简单性、缺乏拥塞控制和 Internet 支持（通常没有防火墙），现在有基于 UDP 构建的新协议来实现自己的拥塞控制和其他功能。一个例子是 QUIC。

#### QUIC 和 HTTP/3

QUIC 是由 Google 的 Jim Roskind 设计的网络协议，作为 TCP 的高性能、低延迟替代品，针对 HTTP 和 TLS 进行了优化 \[Roskind 12\]。 QUIC 基于 UDP 构建，并在其之上提供了多种功能，包括：

*   能够在同一“连接”之上多路复用多个应用程序定义的流。
    
*   类似 TCP 的可靠有序流传输，可以选择关闭各个子流。
    
*   当客户端更改其网络地址时，基于连接 ID 的加密身份验证恢复连接。
    
*   有效负载数据的完全加密，包括 QUIC 标头。
    
*   0-RTT 连接握手，包括加密（对于之前已通信的对等点）。
    

Chrome 网络浏览器大量使用 QUIC。

虽然 QUIC 最初由 Google 开发，但互联网工程任务组 (IETF) 正在标准化 QUIC 传输本身以及使用 HTTP over QUIC 的具体配置（后一种组合称为 HTTP/3）。

### 10.4.2 硬件

网络硬件包括接口、控制器、交换机、路由器和防火墙。了解他们的操作是有用的，即使他们是由其他员工（网络管理员）管理的。

#### 接口

物理网络接口在连接的网络上发送和接收消息（称为帧）。它们管理所涉及的电、光或无线信号，包括传输错误的处理。

接口类型基于第 2 层标准，每种类型都提供最大带宽。更高带宽的接口提供更低的数据传输延迟，但成本更高。在设计新服务器时，一个关键决策通常是如何平衡服务器的价格与所需的网络性能。

对于以太网，选择包括铜缆或光纤，最大速度为 1 Gbit/s (1 GbE)、10 GbE、40 GbE、100 GbE、200 GbE 和 400 GbE。许多供应商都生产以太网接口控制器，尽管您的操作系统可能不支持其中某些控制器的驱动程序。

接口利用率可以通过当前吞吐量除以当前协商带宽来检查。大多数接口都有单独的发送和接收通道，当在全双工模式下运行时，必须单独研究每个通道的利用率。

由于信号强度差和干扰，无线接口可能会出现性能问题。[^4]

[^4]我开发了 BPF 软件，可将 Linux Wi-Fi 信号强度转换为可听见的音调，并在 AWS re:Invent 2019 演讲中进行了演示 \[Gregg 19b\]。我会将其包含在本章中，但我尚未将其用于企业或云环境，到目前为止，这些环境都是有线的。

#### 控制器

物理网络接口通过控制器提供给系统，控制器可以内置于系统板中或通过扩展卡提供。

控制器由微处理器驱动，并通过 I/O 传输（例如 PCI）连接到系统。其中任何一个都可能成为网络吞吐量或 IOPS 的限制因素。

例如，双 10 GbE 网络接口卡连接到四通道 PCI Express (PCIe) Gen 2 插槽。该卡的最大发送或接收带宽为 2 × 10 GbE = 20 Gbits/s，双向带宽为 40 Gbit/s。该插槽的最大带宽为 4 × 4 Gbits/s = 16 Gbits/s。因此，两个端口上的网络吞吐量将受到 PCIe Gen 2 带宽的限制，并且不可能同时以线速驱动它们（我从实践中知道这一点！）。

#### 交换机和路由器

交换机在任意两个连接的主机之间提供专用通信路径，允许主机对之间进行多次传输而不会产生干扰。这项技术取代了集线器（以及之前的共享物理总线：常用的粗以太网同轴电缆），集线器与所有主机共享所有数据包。当主机同时传输时，这种共享会导致争用，接口使用“带冲突检测的载波侦听多路访问”(CSMA/CD) 算法将其识别为冲突。该算法将呈指数级后退并重新传输，直到成功为止，从而在负载下产生性能问题。通过使用开关，这已经成为过去，但一些可观察性工具仍然具有碰撞计数器 - 尽管这些通常仅由于错误（协商或接线不良）而发生。

路由器在网络之间传递数据包，并使用网络协议和路由表来确定有效的传递路径。在两个城市之间传递数据包可能涉及十几个或更多路由器，以及其他网络硬件。路由器和路由通常配置为动态更新，以便网络能够自动响应网络和路由器中断，并平衡负载。这意味着在给定的时间点，没有人可以确定数据包实际采用的路径。由于可能有多个路径，因此数据包也可能会无序传送，从而导致 TCP 性能问题。

网络上的这种神秘元素常常被归咎于性能不佳：也许来自其他不相关主机的大量网络流量正在使源和目标之间的路由器饱和？因此，网络管理团队经常被要求免除其基础设施的责任。他们可以使用先进的实时监控工具来检查所有路由器和其他涉及的网络组件。

路由器和交换机都包含缓冲区和微处理器，它们本身可能成为负载下的性能瓶颈。作为一个极端的例子，我曾经发现早期的 10 GbE 交换机由于其 CPU 容量有限，所有端口的总传输速度不能超过 11 Gb/s。

请注意，交换机和路由器也经常发生速率转换（从一种带宽切换到另一种带宽，例如，10 Gbps 链路转换到 1 Gbps 链路）。发生这种情况时，需要进行一些缓冲以避免过度丢弃，但许多交换机和路由器会过度缓冲（请参阅第 10.3.6 节“缓冲”中的缓冲膨胀问题），从而导致高延迟。更好的队列管理算法可以帮助消除这个问题，但并非所有网络设备供应商都支持它们。在源处进行调步也可以通过减少流量的突发性来缓解速率转换问题。

#### 防火墙

防火墙通常用于仅允许基于配置的规则集的授权通信，从而提高网络的安全性。它们可能以物理网络设备和内核软件的形式出现。

防火墙可能成为性能瓶颈，尤其是在配置为有状态时。有状态规则存储每个可见连接的元数据，防火墙在处理许多连接时可能会遇到过多的内存负载。发生这种情况的原因可能是拒绝服务 (DoS) 攻击试图用连接淹没目标。当出站连接频率很高时，也可能会发生这种情况，因为它们可能需要类似的连接跟踪。

由于防火墙是定制硬件或软件，因此可用于分析它们的工具取决于每个防火墙产品。请参阅各自的文档。

由于其性能、可编程性、易用性和最终成本，使用扩展 BPF 在商用硬件上实现防火墙的情况正在不断增长。采用 BPF 防火墙和 DDoS 解决方案的公司包括 Facebook \[Deepak 18\]、Cloudflare \[Majkowski 18\] 和 Cilium \[Cilium 20a\]。

防火墙在性能测试期间也可能是一个麻烦：在调试问题时执行带宽实验可能涉及修改防火墙规则以允许连接（并与安全团队进行协调）。

#### 其他的

您的环境可能包括其他物理网络设备，例如集线器、网桥、中继器和调制解调器。其中任何一个都可能成为性能瓶颈和丢包的根源。

#### TCP 连接队列

通过使用积压队列来处理突发的入站连接。有两种这样的队列，一种用于 TCP 握手完成时未完成的连接（也称为 SYN 积压），另一种用于等待应用程序接受的已建立会话（也称为侦听积压）。如图 10.9 所示。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/10fig09.jpg)

图 10.9 TCP 积压队列

早期内核中仅使用一个队列，并且很容易受到 SYN 洪水攻击。 SYN 洪水是一种 DoS 攻击，涉及从虚假 IP 地址向侦听 TCP 端口发送大量 SYN。当 TCP 等待完成握手时，这会填满积压队列，从而阻止真正的客户端连接。

对于两个队列，第一个队列可以充当潜在虚假连接的暂存区域，仅在建立连接后才会将其提升到第二个队列。第一个队列可以很长以吸收 SYN 洪泛并进行优化以仅存储必要的最少量元数据。

SYN cookies 的使用绕过了第一个队列，因为它们显示客户端已经被授权。

这些队列的长度可以独立调整（参见第 10.8 节，调整）。第二个也可以由应用程序设置为listen(2) 的backlog 参数。

#### TCP缓冲

通过使用与套接字关联的发送和接收缓冲区可以提高数据吞吐量。如图 10.10 所示。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/10fig10.jpg)

图10.10 TCP发送和接收缓冲区

发送和接收缓冲区的大小都是可调的。较大的大小可以提高吞吐量性能，但代价是每个连接花费更多的主内存。如果服务器期望执行更多发送或接收，则可以将一个缓冲区设置为大于另一个缓冲区。 Linux 内核还将根据连接活动动态增加这些缓冲区的大小，并允许调整其最小、默认和最大大小。

#### 分段卸载：GSO 和 TSO

网络设备和网络接受最大分段大小 (MSS) 的数据包，该最大分段大小可能小至 1500 字节。为了避免发送许多小数据包的网络堆栈开销，Linux 使用通用分段卸载 (GSO) 发送最大 64 KB 大小的数据包（“超级数据包”），这些数据包在传送到网络之前被分割成 MSS 大小的分段设备。如果 NIC 和驱动程序支持 TCP 分段卸载 (TSO)，GSO 会将拆分留给设备，从而提高网络堆栈吞吐量。[^5] GSO \[Linux 20i\] 还有一个通用接收卸载 (GRO) 补充。[^6] GRO和GSO由内核软件实现，TSO由NIC硬件实现。

[^5]某些网卡提供 TCP 卸载引擎 (TOE) 来卸载部分或全部 TCP/IP 协议处理。 Linux 出于多种原因不支持 TOE，包括安全性、复杂性甚至性能 \[Linux 16\]。

[^6]Linux 于 2018 年添加了对 GSO 和 GRO 的 UDP 支持，其中 QUIC 是一个关键用例 \[Bruijn 18\]。

#### 网络设备驱动程序

网络设备驱动程序通常有一个额外的缓冲区（环形缓冲区），用于在内核内存和 NIC 之间发送和接收数据包。这在图 10.8 中被描绘为驱动程序队列。

高速网络中越来越常见的一个性能特征是中断合并模式的使用。仅当达到计时器（轮询）或达到一定数量的数据包时才会发送中断，而不是为每个到达的数据包中断内核。这降低了内核与 NIC 通信的速率，允许缓冲更大的传输，从而提高吞吐量，但会付出一定的延迟代价。

Linux 内核使用新的 API (NAPI) 框架，该框架使用中断缓解技术：对于低数据包速率，使用中断（通过软中断调度处理）；对于高数据包速率，禁用中断，并使用轮询来允许合并 \[Corbet 03\]\[Corbet 06b\]。这可以提供低延迟或高吞吐量，具体取决于工作负载。 NAPI 的其他功能包括：

*   数据包限制，允许网络适配器尽早丢弃数据包，以防止系统被数据包风暴淹没。
    
*   接口调度，使用配额来限制轮询周期内处理的缓冲区，以确保繁忙网络接口之间的公平性。
    
*   支持 SO\_BUSY\_POLL 套接字选项，其中用户级应用程序可以通过请求套接字上的忙等待（在 CPU 上旋转，直到事件发生）来减少网络接收延迟 \[Dumazet 17a\]。
    

合并对于改进虚拟机网络尤其重要，并且由 AWS EC2 使用的 ena 网络驱动程序使用。

#### 网卡发送和接收

对于发送的数据包，NIC 会收到通知，并且通常使用直接内存访问 (DMA) 从内核内存读取数据包（帧）以提高效率。 NIC 提供用于管理 DMA 数据包的传输描述符；如果 NIC 没有空闲描述符，网络堆栈将暂停传输以允许 NIC 赶上。[^7]

[^7]字节队列限制 (BQL) 在其他优化标题下进行了总结，通常可以防止 TX 描述符耗尽。

对于接收到的数据包，NIC 可以使用 DMA 将数据包放入内核环形缓冲区内存中，然后使用中断通知内核（可以忽略该中断以允许合并）。中断触发软中断，将数据包传送到网络堆栈进行进一步处理。

#### CPU 扩展

通过使用多个 CPU 来处理数据包和 TCP/IP 堆栈，可以实现高数据包速率。 Linux 支持多种多 CPU 数据包处理方法（请参阅 Documentation/networking/scaling.txt）：

*   RSS：接收端扩展：对于支持多个队列的现代网卡，可以将数据包散列到不同的队列，然后由不同的 CPU 依次处理，直接中断它们。该散列可能基于 IP 地址和 TCP 端口号，因此来自同一连接的数据包最终由同一 CPU 处理。[^8]
    
    [^8] Netflix FreeBSD CDN 使用 RSS 来协助 TCP 大型接收卸载 (LRO)，允许聚合同一连接的数据包，即使被其他数据包分隔开也是如此 \[Gallatin 17\]。
    
*   RPS：接收数据包引导：RSS 的软件实现，适用于不支持多个队列的 NIC。这涉及一个短中断服务例程，将入站数据包映射到 CPU 进行处理。类似的散列可用于根据数据包标头中的字段将数据包映射到 CPU。
    
*   RFS：接收流控制：这与 RPS 类似，但与 CPU 上上次处理套接字的位置密切相关，以提高 CPU 缓存命中率和内存局部性。
    
*   加速接收流控制：对于支持此功能的 NIC，这可以在硬件中实现 RFS。它涉及使用流信息更新 NIC，以便它可以确定要中断哪个 CPU。
    
*   XPS：传输数据包引导：对于具有多个传输队列的 NIC，这支持多个 CPU 向队列传输。
    

如果没有针对网络数据包的 CPU 负载均衡策略，一块网卡可能只中断一个 CPU，而该 CPU 的利用率可能达到 100%，成为瓶颈。这可能表现为单个 CPU 上的高软中断 CPU 时间（例如，使用 Linux mpstat(1)：请参阅第 6 章，CPU，第 6.6.3 节，mpstat）。这种情况尤其可能发生在负载均衡器或代理服务器（例如 nginx）上，因为它们的预期工作负载是高速率的入站数据包。

根据缓存一致性等因素将中断映射到 CPU（如 RFS 所做的那样）可以显着提高网络性能。这也可以通过 irqbalance 进程来完成，该进程将中断请求 (IRQ) 线分配给 CPU。

#### 内核绕过

图 10.8 显示了 TCP/IP 堆栈中最常采用的路径。应用程序可以使用数据平面开发套件 (DPDK) 等技术绕过内核网络堆栈，以实现更高的数据包速率和性能。这涉及应用程序在用户空间中实现自己的网络协议，并通过 DPDK 库和内核用户空间 I/O (UIO) 或虚拟功能 I/O (VFIO) 驱动程序写入网络驱动程序。通过直接访问NIC上的内存可以避免复制数据包数据的开销。

eXpress 数据路径 (XDP) 技术为网络数据包提供了另一条路径：使用扩展 BPF 并集成到现有内核堆栈而不是绕过它的可编程快速路径 \[Høiland-Jørgensen 18\]。 （DPDK 现在支持 XDP 接收数据包，将一些功能移回内核 \[DPDK 20\]。）

通过内核网络堆栈旁路，使用传统工具和指标的检测不可用，因为它们使用的计数器和跟踪事件也被绕过。这使得性能分析变得更加困难。

除了完全堆栈旁路之外，还有一些功能可以避免复制数据的费用：MSG\_ZEROCOPY send(2) 标志，以及通过 mmap(2) 进行零复制接收 \[Linux 20c\]\[Corbet 18b\]。

Linux 网络堆栈中还使用其他算法来提高性能。图 10.11 显示了 TCP 发送路径的这些（其中许多是从 tcp\_write\_xmit() 内核函数调用的）。

![Images](/Systems%20Performance_%20Enterprise%20and%20the%20Cl%20-%20Brendan%20Gregg/images/10fig11.jpg)

图10.11 TCP发送路径

其中一些组件和算法已在前面描述过（套接字发送缓冲区、TSO、[^9] 拥塞控制、Nagle 和 qdisc）；其他包括：

[^9]请注意，TSO 在图中出现了两次：首先是在 Pacing 之后构建超级数据包，然后在 NIC 中进行最终分段。

*   调步：这控制何时发送数据包，分散传输（调步）以避免可能损害性能的突发（这可能有助于避免可能导致排队延迟，甚至导致网络交换机丢弃数据包的 TCP 微突发。它也可能当许多端点同时传输到一个端点时，有助于解决 incast 问题 \[Fritchie 12\]）。
    
*   TCP 小队列 (TSQ)：这控制（减少）网络堆栈排队的数量以避免包括缓冲区膨胀 \[Bufferbloat 20\] 在内的问题。
    
*   字节队列限制 (BQL)：这些自动将驱动程序队列的大小设置得足够大以避免饥饿，但也足够小以减少排队数据包的最大延迟，并避免耗尽 NIC TX 描述符 \[Hrubý 12\]。它的工作原理是在必要时暂停向驱动程序队列添加数据包，并在 Linux 3.3 \[Siemon 13\] 中添加。
    
*   最早出发时间 (EDT)：它使用定时轮而不是队列来对发送到 NIC 的数据包进行排序。根据策略和速率配置在每个数据包上设置时间戳。这是在 Linux 4.20 中添加的，并且具有类似 BQL 和 TSQ 的功能 \[Jacobson 18\]。
    

这些算法通常结合使用以提高性能。 TCP 发送的数据包在到达 NIC 之前可以通过任何拥塞控制、TSO、TSQ、调步和排队规则进行处理 \[Cheng 16\]。

### 10.5.1 工具方法

工具方法是迭代可用工具、检查它们提供的关键指标的过程。它可能会忽略工具提供的可见性较差或没有可见性的问题，并且执行起来可能非常耗时。

对于网络，工具方法可能涉及检查：

*   `nstat/netstat -s`：查找高重传率和无序数据包。什么构成“高”重传率取决于客户端：具有不可靠远程客户端的面向互联网的系统应该比客户端位于同一数据中心的内部系统具有更高的重传率。
    
*   `ip -s link/netstat -i`：检查接口错误计数器，包括“错误”、“丢弃”、“溢出”。
    
*   `ss -tiepm`：检查重要套接字的限制器标志，以了解其瓶颈是什么，以及显示套接字运行状况的其他统计信息。
    
*   `nicstat`/`ip -s link`：检查发送和接收的字节速率。高吞吐量可能受到协商的数据链路速度或外部网络限制的限制。它还可能导致系统上的网络用户之间发生争用和延迟。
    
*   `tcplife`：记录 TCP 会话，其中包含进程详细信息、持续时间（生命周期）和吞吐量统计信息。
    
*   `tcptop`：实时观看顶级 TCP 会话。
    
*   `tcpdump`：虽然就 CPU 和存储成本而言，使用该方法可能会很昂贵，但短期使用 tcpdump(8) 可能会帮助您识别异常的网络流量或协议标头。
    
*   perf(1)/BCC/bpftrace：检查应用程序和线路之间选定的数据包，包括检查内核状态。
    

如果发现问题，请检查可用工具中的所有字段以了解更多背景信息。有关每个工具的更多信息，请参阅第 10.6 节“可观察性工具”。其他方法可以识别更多类型的问题。

### 10.5.2 使用方法

USE 方法用于快速识别所有组件的瓶颈和错误。对于每个网络接口以及每个方向（发送 (TX) 和接收 (RX)），检查：

*   使用率：接口忙于发送或接收帧的时间
    
*   饱和度：由于充分利用接口而导致的额外排队、缓冲或阻塞的程度
    
*   错误：对于接收：错误的校验和、帧太短（小于数据链路头）或太长、冲突（不太可能使用交换网络）；对于传输：延迟冲突（接线不良）
    

可以首先检查错误，因为它们通常可以快速检查并且最容易解释。

操作系统或监控工具通常不直接提供利用率（nicstat(1) 是一个例外）。它可以计算为每个方向（RX、TX）的当前吞吐量除以当前协商速度。当前吞吐量应以网络上每秒的字节数来衡量，包括所有协议标头。

对于实施网络带宽限制（资源控制）的环境（如某些云计算环境中发生的情况），除了物理限制之外，还可能需要根据施加的限制来衡量网络利用率。

网络接口的饱和度很难测量。某些网络缓冲是正常的，因为应用程序发送数据的速度比接口传输数据的速度快得多。可以测量应用程序线程在网络发送上阻塞的时间，该时间应该随着饱和度的增加而增加。还要检查是否有其他与接口饱和度更密切相关的内核统计信息，例如 Linux“溢出”。请注意，Linux 使用 BQL 来调节 NIC 队列大小，这有助于避免 NIC 饱和。

TCP 级别的重传通常可以作为统计数据轻松获得，并且可以作为网络饱和度的指标。然而，它们是通过服务器与其客户端之间的网络进行测量的，并且可能是由任何跃点的问题引起的。

USE 方法还可以应用于网络控制器以及它们与处理器之间的传输。由于这些组件的可观察性工具很少，因此根据网络接口统计数据和拓扑推断指标可能更容易。例如，如果网络控制器 A 容纳端口 A0 和 A1，则网络控制器吞吐量可计算为接口吞吐量 A0 + A1 之和。有了已知的最大吞吐量，就可以计算网络控制器的利用率。
