第 9 章磁盘

第9章

磁盘

磁盘 I/O 可能会导致严重的应用程序延迟，因此是系统性能分析的重要目标。在高负载下，磁盘成为瓶颈，导致 CPU 在系统等待磁盘 I/O 完成时处于空闲状态。识别并消除这些瓶颈可以将性能和应用程序吞吐量提高几个数量级。

术语“磁盘”是指系统的主存储设备。它们包括基于闪存的固态磁盘 (SSD) 和磁性旋转磁盘。 SSD 的引入主要是为了提高磁盘 I/O 性能，而它们确实做到了这一点。然而，对容量、I/O 速率和吞吐量的需求也在不断增加，闪存设备也无法避免性能问题。

本章的学习目标是：

  • 了解磁盘模型和概念。

  • 了解磁盘访问模式如何影响性能。

  • 了解解释磁盘利用率的危险。

  • 熟悉磁盘设备的特性和内部结构。

  • 熟悉从文件系统到设备的内核路径。

  • 了解 RAID 级别及其性能。

  • 遵循不同的磁盘性能分析方法。

  • 表征系统范围和每个进程的磁盘 I/O。

  • 测量磁盘 I/O 延迟分布并识别异常值。

  • 识别请求磁盘 I/O 的应用程序和代码路径。

  • 使用跟踪器详细调查磁盘 I/O。

  • 了解磁盘可调参数。

本章由六部分组成，前三部分提供磁盘 I/O 分析的基础，后三部分展示其在基于 Linux 的系统中的实际应用。各部分如下：

  • 背景介绍了存储相关的术语、磁盘设备的基本模型以及磁盘性能的关键概念。

  • 架构提供了存储硬件和软件架构的通用描述。

  • 方法论描述了观察性和实验性的性能分析方法。

  • 可观察性工具展示了适用于基于 Linux 的系统的磁盘性能可观察性工具，包括跟踪和可视化。

  • 实验总结了磁盘基准测试工具。

  • 调整描述了示例磁盘可调参数。

上一章介绍了基于磁盘构建的文件系统的性能，是了解应用程序性能的更好的学习目标。

9.1 术语

本章中使用的与磁盘相关的术语包括：

  • 虚拟磁盘：存储设备的模拟。它在系统中显示为单个物理磁盘，但它可以由多个磁盘或磁盘的一部分构成。

  • 传输：用于通信的物理总线，包括数据传输 (I/O) 和其他磁盘命令。

  • 扇区：磁盘上的存储块，传统上为 512 字节大小，但现在通常为 4 KB。

  • I/O：严格来说，I/O只包括磁盘读写，不包括其他磁盘命令。 I/O 至少可以通过方向（读或写）、磁盘地址（位置）和大小（字节）来描述。

  • 磁盘命令：可以命令磁盘执行其他非数据传输命令（例如缓存刷新）。

  • 吞吐量：对于磁盘，吞吐量通常指当前数据传输速率，以每秒字节数为单位。

  • 带宽：这是存储传输或控制器的最大可能数据传输速率；它受到硬件的限制。

  • I/O 延迟：I/O 操作从开始到结束的时间。第 9.3.1 节“测量时间”定义了更精确的时间术语。请注意，网络使用术语“延迟”来指启动 I/O 所需的时间，后跟数据传输时间。

  • 延迟异常值：磁盘 I/O 延迟异常高。

本章将介绍其他术语。词汇表包含基本术语供需要时参考，包括磁盘、磁盘控制器、存储阵列、本地磁盘、远程磁盘和IOPS。另请参阅第 2 章和第 3 章中的术语部分。

9.2 模型

以下简单模型说明了磁盘 I/O 性能的一些基本原理。

9.2.1 简单磁盘

现代磁盘包含一个用于 I/O 请求的磁盘队列，如图 9.1 所示。

图 9.1 带队列的简单磁盘

磁盘接受的 I/O 可能正在队列中等待或正在接受服务。这个简单的模型类似于杂货店的收银台，顾客排队等待服务。它也非常适合使用排队论进行分析。

虽然这可能意味着先到先服务的队列，但磁盘控制器可以应用其他算法来优化性能。这些算法可能包括旋转磁盘的电梯搜索（请参阅第 9.4.1 节“磁盘类型”中的讨论），或用于读取和写入 I/O 的单独队列（尤其是基于闪存的磁盘）。

9.2.2 缓存磁盘

添加磁盘缓存允许通过更快的内存类型来满足某些读取请求，如图 9.2 所示。这可以被实现为包含在物理磁盘设备内的少量存储器(DRAM)。

图 9.2 带有磁盘缓存的简单磁盘

虽然缓存命中返回的延迟非常低（良好），但缓存未命中仍然很频繁，并且磁盘设备延迟很高。

磁盘缓存还可以通过将其用作回写缓存来提高写入性能。这表示写入已在数据传输到缓存之后、较慢的传输到永久磁盘存储之前完成。与之相对的术语是直写式缓存，它仅在完全传输到下一级后才完成写入。

在实践中，存储回写式缓存通常与电池配合使用，以便在断电时仍然可以保存缓冲的数据。此类电池可以位于磁盘或磁盘控制器上。

9.2.3 控制器

图 9.3 显示了一种简单类型的磁盘控制器，它将 CPU I/O 传输与存储传输和连接的磁盘设备桥接起来。这些也称为主机总线适配器 (HBA)。

图 9.3 简单磁盘控制器和连接的传输

性能可能受到这些总线、磁盘控制器或磁盘的限制。有关磁盘控制器的更多信息，请参见第 9.4 节“体系结构”。

9.3 概念

以下是磁盘性能中的重要概念。

9.3.1 测量时间

I/O 时间可以通过以下方式测量：

  • I/O 请求时间（也称为 I/O 响应时间）：从发出 I/O 到完成的整个时间

  • I/O等待时间：等待队列所花费的时间

  • I/O服务时间：处理I/O的时间（不等待）

如图 9.4 所示。

图 9.4 I/O 时间术语（通用）

服务时间一词起源于磁盘是由操作系统直接管理的简单设备，因此操作系统知道磁盘何时主动提供 I/O 服务。磁盘现在进行自己的内部排队，操作系统服务时间包括等待内核队列所花费的时间。

在可能的情况下，我使用澄清术语来说明正在测量的内容，从哪个开始事件到哪个结束事件。开始和结束事件可以是基于内核的或基于磁盘的，基于内核的时间是从磁盘设备的块 I/O 接口测量的（如图 9.7 所示）。

从内核：

  • 块 I/O 等待时间（也称为操作系统等待时间）是从创建新 I/O 并将其插入内核 I/O 队列到其离开最终内核队列并发送到磁盘设备所花费的时间。这可能跨越多个内核级队列，包括块I/O层队列和磁盘设备队列。

  • 块I/O服务时间是从向设备发出请求到设备完成中断的时间。

  • 块 I/O 请求时间既是块 I/O 等待时间又是块 I/O 服务时间：从创建 I/O 到完成的完整时间。

从磁盘：

  • 磁盘等待时间是在磁盘队列上花费的时间。

  • 磁盘服务时间是主动处理 I/O 所需的磁盘队列之后的时间。

  • 磁盘请求时间（也称为磁盘响应时间和磁盘 I/O 延迟）既是磁盘等待时间，也是磁盘服务时间，等于块 I/O 服务时间。

如图 9.5 所示，其中 DWT 是磁盘等待时间，DST 是磁盘服务时间。此图还显示了磁盘缓存，以及磁盘缓存命中如何导致磁盘服务时间 (DST) 更短。

图 9.5 内核和磁盘时间术语

I/O 延迟是另一个常用术语，在第 1 章中介绍。与其他术语一样，它的含义取决于测量的位置。 I/O 延迟单独可以指块 I/O 请求时间：整个 I/O 时间。应用程序和性能工具通常使用术语“磁盘 I/O 延迟”来指代磁盘请求时间：设备上的整个时间。如果您从设备的角度与硬件工程师交谈，他们可能会使用术语“磁盘 I/O 延迟”来指代磁盘等待时间。

块 I/O 服务时间通常被视为当前磁盘性能的衡量标准（这是旧版本的 iostat(1) 显示的）；但是，您应该意识到这是一种简化。图 9.7 描绘了一个通用 I/O 堆栈，其中显示了块设备接口下的三个可能的驱动程序层。其中任何一个都可能实现自己的队列，或者可能阻塞互斥体，从而增加 I/O 的延迟。该延迟包含在块 I/O 服务时间中。

计算时间

磁盘服务时间通常无法通过内核统计数据直接观察，但可以使用 IOPS 和利用率推断平均磁盘服务时间：

磁盘服务时间=利用率/IOPS

例如，利用率为 60% 且 IOPS 为 300 时，平均服务时间为 2 毫秒 (600 毫秒/300 IOPS)。这假设利用率反映的是单个设备（或服务中心），该设备一次只能处理一个 I/O。磁盘通常可以并行处理多个 I/O，因此计算不准确。

事件跟踪可以通过测量磁盘 I/O 的问题和完成的高分辨率时间戳来提供准确的磁盘服务时间，而不是使用内核统计数据。这可以使用本章后面描述的工具来完成（例如，第 9.6.6 节中的生物潜伏期（8），生物潜伏期）。

9.3.2 时间尺度

磁盘 I/O 的时间尺度可以有几个数量级的变化，从几十微秒到几千毫秒。在最慢的一端，单个缓慢的磁盘 I/O 可能会导致应用程序响应时间不佳；在最快的一端，只有大量磁盘 I/O 才可能成为问题（许多快速 I/O 的总和等于慢速 I/O）。

作为上下文，表 9.1 提供了磁盘 I/O 延迟可能范围的一般概念。有关精确的当前值，请查阅磁盘供应商文档，并执行您自己的微基准测试。另请参阅第 2 章“方法论”，了解磁盘 I/O 以外的时间尺度。

9.3.4 随机与顺序 I/O

磁盘 I/O 工作负载可以根据磁盘上 I/O 的相对位置（磁盘偏移），使用术语“随机”和“顺序”来描述。这些术语已在第 8 章“文件系统”中关于文件访问模式进行了讨论。

顺序工作负载也称为流式工作负载。术语“流”通常用在应用程序级别，描述“磁盘”（文件系统）的流式读取和写入。

在旋转磁盘时代，研究随机磁盘 I/O 模式与顺序磁盘 I/O 模式非常重要。对于这些，当磁盘头寻道和盘片在 I/O 之间旋转时，随机 I/O 会产生额外的延迟。如图 9.6 所示，其中磁头在扇区 1 和扇区 2 之间移动需要寻道和旋转（实际采取的路径将尽可能直接）。性能调优涉及识别随机 I/O 并尝试通过多种方式消除它，包括缓存、将随机 I/O 隔离到单独的磁盘以及磁盘放置以减少寻道距离。

其他磁盘类型（包括基于闪存的 SSD）在随机读取模式和顺序读取模式上的性能通常没有什么不同。根据驱动器的不同，可能会因其他因素而存在微小差异，例如，地址查找缓存可以跨越顺序访问，但不能跨越随机访问。由于读取-修改-写入周期，小于块大小的写入可能会遇到性能损失，特别是对于随机写入。

请注意，从操作系统看到的磁盘偏移量可能与物理磁盘上的偏移量不匹配。例如，硬件提供的虚拟磁盘可以跨多个磁盘映射连续的偏移范围。磁盘可以按照自己的方式重新映射偏移量（通过磁盘数据控制器）。有时随机 I/O 不是通过检查偏移量来识别的，但可以通过测量增加的磁盘服务时间来推断。

9.3.5 读/写比率

除了识别随机工作负载与顺序工作负载之外，另一个特征指标是读取与写入的比率，指的是 IOPS 或吞吐量。这可以表示为随时间变化的比率（百分比），例如，“系统自启动以来已以 80% 的读取速度运行”。

了解此比率有助于设计和配置系统。具有高读取率的系统可能会从添加缓存中受益最多。添加更多磁盘以提高最大可用吞吐量和 IOPS 可能会给写入速率较高的系统带来最大好处。

读取和写入本身可能会显示不同的工作负载模式：读取可能是随机 I/O，而写入可能是顺序的（特别是对于写入时复制文件系统）。它们还可能表现出不同的 I/O 大小。

9.3.6 I/O 大小

平均 I/O 大小（字节）或 I/O 大小的分布是另一个工作负载特征。较大的 I/O 大小通常会提供较高的吞吐量，但每个 I/O 的延迟会较长。

I/O 大小可以由磁盘设备子系统更改（例如，量化为 512 字节扇区）。由于 I/O 是由文件系统、卷管理器和设备驱动程序等内核组件在应用程序级别发出的，因此大小也可能会增加和减少。请参阅第 8 章“文件系统”、第 8.3.12 节“逻辑与物理 I/O”中的膨胀和收缩部分。

某些磁盘设备（尤其是基于闪存的磁盘设备）在读取和写入大小不同时，性能差异很大。例如，基于闪存的磁盘驱动器可以以 4 KB 读取和 1 MB 写入来实现最佳性能。理想的 I/O 大小可以由磁盘供应商记录或使用微基准测试来识别。当前使用的 I/O 大小可以使用观察工具找到（参见第 9.6 节，可观察性工具）。

9.3.7 IOPS 不相等

由于最后三个特征，IOPS 并不相同，并且无法在不同设备和工作负载之间进行直接比较。 IOPS 值本身并没有多大意义。

例如，对于旋转磁盘，5,000 个顺序 IOPS 的工作负载可能比 1,000 个随机 IOPS 的工作负载快得多。基于闪存的 IOPS 也很难比较，因为它们的 I/O 性能通常与 I/O 大小和方向（读或写）相关。

IOPS 对于应用程序工作负载来说甚至可能没有那么重要。由随机请求组成的工作负载通常对延迟敏感，在这种情况下需要高 IOPS 速率。流式（顺序）工作负载对吞吐量敏感，这可能会使较大 I/O 的较低 IOPS 速率更可取。

要了解 IOPS，请包括其他详细信息：随机或顺序、I/O 大小、读/写、缓冲/直接以及并行 I/O 数量。还可以考虑使用基于时间的指标，例如利用率和服务时间，它们反映了最终的性能并且可以更容易地进行比较。

9.3.8 非数据传输磁盘命令

除了 I/O 读取和写入之外，还可以向磁盘发送其他命令。例如，可以命令具有盘上高速缓存（RAM）的磁盘将高速缓存刷新到磁盘。这样的命令不是数据传输；而是数据传输。数据之前已通过写入发送到磁盘。

另一个示例命令用于丢弃数据：ATA TRIM 命令或 SCSI UNMAP 命令。这告诉驱动器不再需要某个扇区范围，并且可以帮助 SSD 驱动器保持写入性能。

这些磁盘命令会影响性能，并可能导致在其他 I/O 等待时使用磁盘。

9.3.9 使用

利用率可以计算为磁盘在某个时间间隔内忙于主动执行工作的时间。

利用率为 0% 的磁盘处于“空闲”状态，利用率为 100% 的磁盘持续忙于执行 I/O（和其他磁盘命令）。利用率为 100% 的磁盘很可能是性能问题的根源，尤其是当它们在一段时间内保持 100% 的状态时。然而，任何磁盘利用率都可能导致性能不佳，因为磁盘 I/O 通常是一项缓慢的活动。

在 0% 到 100%（例如 60%）之间也可能存在某个点，此时磁盘的性能不再令人满意，因为排队的可能性增加（无论是在磁盘上队列还是在操作系统中）。成为问题的确切利用率值取决于磁盘、工作负载和延迟要求。请参阅第 2 章“方法论”第 2.6.5 节“排队理论”中的 M/D/1 和 60% 利用率部分。

要确认高利用率是否导致应用程序问题，请研究磁盘响应时间以及应用程序是否在此 I/O 上阻塞。应用程序或操作系统可能正在异步执行 I/O，因此缓慢的 I/O 不会直接导致应用程序等待。

请注意，利用率是间隔汇总。磁盘 I/O 可能会突发发生，尤其是由于写入刷新，这在较长时间间隔内进行汇总时可能会被掩盖。有关利用率指标类型的进一步讨论，请参阅第 2 章“方法”、第 2.3.11 节“利用率”。

虚拟磁盘利用率

对于由硬件（例如，磁盘控制器或网络连接存储）提供的虚拟磁盘，操作系统可能知道虚拟磁盘何时繁忙，但对构建它的底层磁盘的性能一无所知。这会导致操作系统报告的虚拟磁盘利用率与实际磁盘上发生的情况显着不同（并且违反直觉）的情况：

  • 100% 繁忙且基于多个物理磁盘构建的虚拟磁盘可能能够接受更多工作。在这种情况下，100%可能意味着某些磁盘一直处于繁忙状态，但并非所有磁盘一直处于繁忙状态，因此有些磁盘处于空闲状态。

  • 包含回写式缓存的虚拟磁盘在写入工作负载期间可能不会显得非常繁忙，因为磁盘控制器会立即返回写入完成，即使底层磁盘随后会繁忙一段时间。

  • 由于硬件 RAID 重建，磁盘可能很忙，操作系统看不到相应的 I/O。

出于同样的原因，解释操作系统软件（软件 RAID）创建的虚拟磁盘的利用率可能很困难。但是，操作系统也应该公开物理磁盘的利用率，可以进行检查。

一旦物理磁盘达到 100% 利用率并请求更多 I/O，它就会饱和。

9.3.10 饱和度

饱和度是衡量排队工作超出资源可交付能力的指标。对于磁盘设备，可以计算为操作系统中设备等待队列的平均长度（假设它进行排队）。

这提供了超过 100% 利用率点的性能衡量标准。利用率为 100% 的磁盘可能没有饱和（排队），也可能有很多饱和，由于 I/O 排队而显着影响性能。

您可能会假设利用率低于 100% 的磁盘没有饱和，但这实际上取决于利用率间隔：间隔期间的 50% 磁盘利用率可能意味着一半时间利用率为 100%，其余时间空闲。任何区间总结都可能遇到类似的问题。当准确了解发生的情况非常重要时，可以使用跟踪工具来检查 I/O 事件。

9.3.11 I/O 等待

I/O 等待是每个 CPU 的性能指标，显示当 CPU 调度程序队列（处于睡眠状态）上有线程在磁盘 I/O 上被阻塞时，空闲时间。这将 CPU 空闲时间分为闲置时间和磁盘 I/O 阻塞时间。每个 CPU 的 I/O 等待率较高表明磁盘可能是瓶颈，导致 CPU 在等待磁盘时处于空闲状态。

I/O 等待可能是一个非常令人困惑的指标。如果出现另一个需要大量 CPU 的进程，I/O 等待值可能会下降：CPU 现在有事情可做，而不是闲置。然而，尽管 I/O 等待指标有所下降，但相同的磁盘 I/O 仍然存在并阻塞线程。当系统管理员升级应用程序软件并且新版本效率更高并且使用更少的 CPU 周期时，有时会发生相反的情况，从而显示 I/O 等待。这可能会让系统管理员认为升级导致了磁盘问题并使性能变差，而实际上磁盘性能相同，但 CPU 性能有所提高。

更可靠的指标是应用程序线程在磁盘 I/O 上被阻塞的时间。这捕获了由磁盘引起的应用程序线程所承受的痛苦，无论 CPU 可能正在执行什么其他工作。该指标可以使用静态或动态仪器来测量。

I/O 等待仍然是 Linux 系统上的一个流行指标，尽管其性质令人困惑，但它已成功用于识别一种磁盘瓶颈：磁盘繁忙，CPU 空闲。解释它的一种方法是将任何等待 I/O 视为系统瓶颈的迹象，然后调整系统以最大限度地减少它 — 即使 I/O 仍然与 CPU 使用率同时发生。并发 I/O 更有可能是非阻塞 I/O，并且不太可能导致直接问题。由 I/O 等待确定的非并发 I/O 更有可能是应用程序阻塞 I/O，并且是瓶颈。

9.3.12 同步与异步

重要的是要了解，如果应用程序 I/O 和磁盘 I/O 异步操作，则磁盘 I/O 延迟可能不会直接影响应用程序性能。这种情况通常发生在回写式缓存中，其中应用程序 I/O 提前完成，磁盘 I/O 稍后发出。

应用程序可以使用预读来执行异步读取，这在磁盘完成 I/O 时可能不会阻塞应用程序。文件系统可能会自行启动此操作以预热缓存（预取）。

即使应用程序同步等待 I/O，该应用程序代码路径也可能是非关键的并且与客户端应用程序请求异步。它可以是一个应用程序 I/O 工作线程，创建它是为了在其他线程继续处理工作时管理 I/O。

内核通常还支持异步或非阻塞 I/O，其中为应用程序提供 API 来请求 I/O 并在稍后收到其完成的通知。有关这些主题的更多信息，请参阅第 8 章“文件系统”、第 8.3.9 节“非阻塞 I/O”； 8.3.5、预读； 8.3.4、预取； 8.3.7，同步写入。

9.3.13 磁盘与应用程序 I/O

磁盘 I/O 是各种内核组件（包括文件系统和设备驱动程序）的最终结果。有很多原因导致此磁盘 I/O 的速率和容量可能与应用程序发出的 I/O 不匹配。这些包括：

  • 文件系统膨胀、紧缩和不相关的 I/O。请参见第 8 章“文件系统”、第 8.3.12 节“逻辑 I/O 与物理 I/O”。

  • 由于系统内存不足而进行分页。请参见第 7 章“内存”、第 7.2.2 节“分页”。

  • 设备驱动程序 I/O 大小：对 I/O 大小进行四舍五入，或对 I/O 进行分段。

  • RAID写入镜像或校验和块，或验证读取的数据。

当意想不到的时候，这种不匹配可能会令人困惑。可以通过学习架构并进行分析来理解。

9.4 架构

本节介绍磁盘架构，通常在容量规划期间研究磁盘架构以确定不同组件和配置选择的限制。在调查以后的性能问题期间还应该检查它，以防问题源于架构选择而不是当前的负载和调整。

9.4.1 磁盘类型

目前最常用的两种磁盘类型是磁旋转 SSD 和基于闪存的 SSD。这两者都提供永久存储；与易失性存储器不同，它们存储的内容在电源循环后仍然可用。

9.4.1.1 磁旋转

也称为硬盘驱动器 (HDD)，这种类型的磁盘由一个或多个磁盘（称为盘片）组成，其中浸渍有氧化铁颗粒。这些颗粒的一小部分区域可以在两个方向之一上被磁化；该方向用于存储位。盘片旋转，同时带有读写数据电路的机械臂到达表面。该电路包括磁盘头，并且一个臂可以具有多个磁头，从而允许它同时读取和写入多个位。数据以圆形磁道存储在盘片上，每个磁道又分为多个扇区。

作为机械设备，它们的执行速度相对较慢，尤其是对于随机 I/O。随着基于闪存的技术的进步，SSD 正在取代旋转磁盘，可以想象有一天旋转磁盘将被淘汰（以及其他较旧的存储技术：鼓磁盘和核心内存）。同时，旋转磁盘在某些场景下仍然具有竞争力，例如经济的高密度存储（每兆字节成本较低），尤其是数据仓库。2

2托管流媒体视频的 Netflix Open Connect Appliances (OCA) 听起来可能像是 HDD 的另一个用例，但每台服务器支持大量并发客户可能会导致随机 I/O。一些 OCA 已改用闪存驱动器 [Netflix 20]。

以下主题总结了旋转磁盘性能的因素。

寻找和旋转

磁旋转磁盘的 I/O 缓慢通常是由磁头的寻道时间和磁盘盘片的旋转时间引起的，这两者都可能需要几毫秒。最好的情况是下一个请求的 I/O 位于当前服务的 I/O 的末尾，这样磁盘头就不需要寻找或等待额外的旋转。如前所述，这称为顺序 I/O，而需要磁头查找或等待旋转的 I/O 称为随机 I/O。

有许多策略可以减少寻道和旋转等待时间，包括：

  • 缓存：完全消除 I/O。

  • 文件系统布局和行为，包括写入时复制（这使得写入顺序，但可能使以后的读取随机）。

  • 将不同的工作负载分离到不同的磁盘，以避免工作负载 I/O 之间的查找。

  • 将不同的工作负载转移到不同的系统（某些云计算环境可以这样做以减少多租户效应）。

  • 电梯寻道，由磁盘本身执行。

  • 更高密度的磁盘，以收紧工作负载位置。

  • 分区（或“切片”）配置，例如短行程。

减少轮转等待时间的另一个策略是使用更快的磁盘。磁盘有不同的转速，包括 5400、7200、10 K 和 15 K 转/分钟 (rpm)。请注意，由于热量和磨损增加，较高的速度可能会导致磁盘寿命缩短。

理论最大吞吐量

如果已知磁盘每磁道的最大扇区数，则可以使用以下公式计算磁盘吞吐量：

最大吞吐量 = 每磁道最大扇区 × 扇区大小 × rpm/60 s

此公式对于准确公开此信息的旧磁盘更有用。现代磁盘向操作系统提供磁盘的虚拟映像，并且仅公开这些属性的合成值。

短行程

短行程是指仅将磁盘的外磁道用于工作负载；其余部分未使用或用于低吞吐量工作负载（例如存档）。这减少了寻道时间，因为磁头移动受到较小范围的限制，并且磁盘可以使磁头在外边缘处静止，从而减少空闲后的第一次寻道。由于扇区分区，外部磁道通常也具有更好的吞吐量（请参阅下一节）。在检查已发布的磁盘基准时，请留意短冲程，尤其是不包括价格的基准，其中可能使用了许多短冲程磁盘。

部门分区

磁盘磁道的长度各不相同，磁盘中心最短，外边缘最长。扇区分区（也称为多区域记录）不是固定每个磁道的扇区数（和位数），而是增加较长磁道的扇区数，因为可以物理写入更多扇区。由于旋转速度是恒定的，因此较长的外边缘磁道比内磁道提供更高的吞吐量（兆字节/秒）。

扇区大小

存储行业为磁盘设备开发了一种新标准，称为“高级格式”，以支持更大的扇区大小，特别是 4 KB。这减少了 I/O 计算开销，提高了吞吐量，并减少了磁盘每个扇区存储元数据的开销。磁盘固件仍然可以通过称为高级格式 512e 的模拟标准提供 512 字节的扇区。根据磁盘的不同，这可能会增加写入开销，调用读取-修改-写入周期将 512 字节映射到 4 KB 扇区。其他需要注意的性能问题包括未对齐的 4 KB I/O，它跨越两个扇区，导致为它们提供服务的扇区 I/O 膨胀。

磁盘缓存

这些磁盘的一个常见组件是少量内存 (RAM)，用于缓存读取和缓冲区写入的结果。该内存还允许 I/O（命令）在设备上排队并更有效地重新排序。对于 SCSI，这就是标记命令队列 (TCQ)；对于 SATA，它称为本机命令队列 (NCQ)。

寻找电梯

电梯算法（也称为电梯搜索）是命令队列提高效率的一种方法。它根据 I/O 在磁盘上的位置重新排序，以最大限度地减少磁盘头的移动。结果类似于建筑物电梯，它不会根据按下楼层按钮的顺序来服务楼层，而是在建筑物上上下扫动，并在当前请求的楼层停止。

当检查磁盘 I/O 跟踪并发现按完成时间排序 I/O 与按开始时间排序不匹配时，此行为变得明显：I/O 未按顺序完成。

虽然这看起来是明显的性能优势，但请考虑以下场景：已向磁盘发送一批接近偏移量 1,000 的 I/O，以及在偏移量 2,000 处发送的单个 I/O。目前磁盘磁头数量为 1,000 个。偏移量 2,000 处的 I/O 何时得到服务？现在考虑一下，在为接近 1,000 个 I/O 提供服务的同时，更多的 I/O 到达接近 1,000 个，越来越多 — 足够的连续 I/O 使磁盘在偏移量 1,000 附近保持繁忙 10 秒。 2,000 个偏移 I/O 何时得到服务？其最终 I/O 延迟是多少？

数据完整性

磁盘在每个扇区的末尾存储一个纠错码 (ECC) 以确保数据完整性，以便驱动器可以验证数据是否正确读取，或纠正可能发生的任何错误。如果未正确读取扇区，则磁盘头可能会在下一次旋转时重试读取（并且可能会重试几次，每次都会稍微改变磁头的位置）。这可能是 I/O 异常缓慢的原因。驱动器可能会向操作系统提供软错误以解释发生的情况。监视软错误率可能很有用，因为软错误率的增加可能表明驱动器可能很快就会出现故障。

行业从 512 字节扇区切换到 4 KB 扇区的好处之一是，相同数据量所需的 ECC 位更少，因为 ECC 对于较大扇区大小更有效 [Smith 09]。

请注意，其他校验和也可能用于验证数据。例如，循环冗余校验（CRC）可以用于验证到主机的数据传输，并且文件系统可以使用其他校验和。

振动

虽然磁盘设备供应商很清楚振动问题，但这些问题并未被业界广泛了解或认真对待。 2008 年，在调查一个神秘的性能问题时，我进行了一项振动诱导实验，在执行写入基准测试时对磁盘阵列大喊大叫，导致 I/O 突发非常缓慢。我的实验立即被录制成视频并发布到 YouTube 上，并在 YouTube 上疯传，它被描述为振动对磁盘性能影响的首次演示 [Turner 10]。该视频的观看次数已超过 1,700,000 次，提高了人们对磁盘振动问题的认识 [Gregg 08]。根据我收到的电子邮件，我似乎还意外地催生了一个数据中心隔音行业：您现在可以聘请专业人士来分析数据中心的声级并通过阻尼振动来提高磁盘性能。

树懒盘

当前一些旋转磁盘的性能问题是所谓的懒惰磁盘的发现。这些磁盘有时会返回非常慢的 I/O，超过一秒，并且没有任何报告的错误。这比基于 ECC 的重试所需的时间要长得多。如果此类磁盘返回故障而不是花费这么长时间，实际上可能会更好，以便操作系统或磁盘控制器可以采取纠正措施，例如在冗余环境中使磁盘脱机并报告故障。懒惰磁盘很麻烦，尤其是当它们是存储阵列提供的虚拟磁盘的一部分时，操作系统无法直接看到它们，从而使它们更难以识别。3

3如果使用 Linux 分布式复制块设备 (DRBD) 系统，它会提供“磁盘超时”参数。

小型反应堆

叠瓦式磁记录 (SMR) 驱动器通过使用较窄的磁道提供更高的密度。这些轨道太窄，写入头无法记录，但（较小的）读取头无法读取，因此它通过部分重叠其他轨道来写入它们，其风格类似于屋顶瓦（因此得名）。使用 SMR 的驱动器密度增加了约 25%，但代价是写入性能下降，因为重叠数据被破坏并且还必须重新写入。这些驱动器适合写入一次然后大部分读取的归档工作负载，但不适合 RAID 配置中的写入密集型工作负载 [Mellor 20]。

磁盘数据控制器

机械磁盘为系统提供了一个简单的接口，意味着固定的扇区/磁道比率和连续的可寻址偏移范围。磁盘上实际发生的情况取决于磁盘数据控制器——一个由固件编程的磁盘内部微处理器。磁盘可以实现包括扇区分区在内的算法，从而影响偏移量的布局方式。这是需要注意的事情，但很难分析——操作系统无法看到磁盘数据控制器。

9.4.1.2 固态硬盘

这些也称为固态磁盘 (SSD)。术语“固态”指的是它们对固态电子器件的使用，它提供可编程非易失性存储器，其性能通常比旋转磁盘好得多。由于没有移动部件，这些磁盘在物理上也很耐用，并且不易受到振动引起的性能问题的影响。

这种磁盘类型的性能通常在不同偏移量下保持一致（无旋转或寻道延迟），并且对于给定的 I/O 大小是可预测的。工作负载的随机或顺序特征远不如旋转磁盘那么重要。所有这些都使它们更容易研究和进行容量规划。然而，如果它们确实遇到性能问题，由于它们内部的运行方式，理解它们可能与旋转磁盘一样复杂。

一些 SSD 使用非易失性 DRAM (NV-DRAM)。大多数使用闪存。

闪存

基于闪存的 SSD 提供很高的读取性能，特别是随机读取性能，可以比旋转磁盘高出几个数量级。大多数都是使用 NAND 闪存构建的，它使用基于电子的俘获电荷存储介质，可以在无电状态下持久存储电子4 [Cornwell 12]。 “闪存”这个名字与数据的写入方式有关，这需要一次擦除整个内存块（包括多个页面，通常每页 8 或 64 KB）并重写内容。由于这些写入开销，闪存具有不对称的读/写性能：快速读取和较慢写入。驱动器通常使用回写式缓存来提高写入性能，并使用小电容器作为电源故障时的备用电池来缓解这种情况。

4但不是无限期的。现代 MLC 的数据保留错误可能会在断电后的短短几个月内发生 [Cassidy 12][Cai 15]。

闪存有不同类型：

  • 单级单元 (SLC)：将数据位存储在各个单元中。

  • 多级单元 (MLC)：每个单元存储多个位（通常为两个，这需要四个电压电平）。

  • 企业多级单元 (eMLC)：具有适合企业使用的高级固件的 MLC。

  • 三电平单元 (TLC)：存储三位（八个电压电平）。

  • 四级单元 (QLC)：存储四位。

  • 3D NAND / 垂直 NAND (V-NAND)：通过堆叠闪存层（例如 TLC）来提高密度和存储容量。

此列表大致按时间顺序排列，最新技术列在最后：3D NAND 自 2013 年以来已投入商用。

与其他类型相比，SLC 往往具有更高的性能和可靠性，尽管成本较高，但更适合企业使用。 MLC 现在经常在企业中使用，尽管其可靠性较低，但其密度较高。闪存可靠性通常通过驱动器预期支持的块写入（编程/擦除周期）数量来衡量。对于 SLC，这一预期约为 50,000 至 100,000 个周期； MLC 大约 5,000 至 10,000 次循环； TLC 约 3,000 个循环；对于 QLC 来说大约是 1,000 个周期 [Liu 20]。

控制器

SSD 的控制器具有以下任务 [Leventhal 13]：

  • 输入：读取和写入按页进行（通常为 8 KB）；写入只能发生在已擦除的页上；页以 32 至 64（256-512 KB）为单位进行擦除。

  • 输出：模拟硬盘驱动器块接口：读取或写入任意扇区（512 字节或 4 KB）。

输入和输出之间的转换由控制器的闪存转换层 (FTL) 执行，该层还必须跟踪空闲块。它本质上使用自己的文件系统来完成此操作，例如日志结构文件系统。

写入特性对于写入工作负载来说可能是一个问题，特别是当写入小于闪存块大小（可能大至 512 KB）的 I/O 大小时。这可能会导致写入放大，即在擦除之前将块的剩余部分复制到其他地方，并且还会导致至少擦除-写入周期的延迟。一些闪存驱动器通过提供由电池支持的磁盘缓冲区（基于 RAM）来缓解延迟问题，这样即使在发生电源故障的情况下，也可以对写入进行缓冲并稍后写入。

由于闪存布局，我使用过的最常见的企业级闪存驱动器的最佳性能为 4 KB 读取和 1 MB 写入。这些值因不同的驱动器而异，并且可以通过 I/O 大小的微基准测试找到。

鉴于闪存的本机操作和公开的块接口之间的差异，操作系统及其文件系统还有改进的空间。 TRIM 命令就是一个例子：它通知 SSD 某个区域不再使用，从而使 SSD 能够更轻松地组装其空闲块池，从而减少写入放大。 （对于 SCSI，这可以使用 UNMAP 或 WRITE SAME 命令来实现；对于 ATA，可以使用 DATA SET MANAGEMENT 命令。Linux 支持包括丢弃安装选项和 fstrim(8) 命令。）

寿命

NAND闪存作为存储介质存在各种问题，包括烧毁、数据褪色和读取干扰[Cornwell 12]。这些都可以通过SSD控制器来解决，它可以移动数据以避免出现问题。它通常会采用磨损均衡（将写入分散到不同的块上，以减少各个块上的写入周期）和内存过度配置（保留额外的内存，以便在需要时映射到服务中）。

虽然这些技术可以延长使用寿命，但 SSD 每个块的写入周期数仍然有限，具体取决于闪存的类型和驱动器采用的缓解功能。企业级驱动器使用内存预留空间和最可靠的闪存类型 SLC，以实现 100 万次甚至更高的写入周期率。基于 MLC 的消费级驱动器可提供少至 1,000 个周期。

病理学

以下是一些需要注意的闪存 SSD 问题：

  • 由于老化而导致延迟异常，SSD 更努力地提取正确的数据（使用 ECC 检查）。

  • 由于碎片而导致更高的延迟（重新格式化可以通过清理 FTL 块映射来解决此问题）。

  • 如果 SSD 实施内部压缩，吞吐量性能会降低。

检查 SSD 性能功能的其他进展以及遇到的问题。

9.4.1.3 持久内存

持久内存采用电池供电的 5 DRAM 形式，用于存储控制器回写缓存。这种类型的性能比闪存快几个数量级，但其成本和有限的电池寿命使其仅限于特殊用途。

5电池或超级电容器。

由英特尔和美光开发的一种名为 3D XPoint 的新型持久内存将使持久内存能够以介于 DRAM 和闪存之间的极具吸引力的性价比用于更多应用。 3D XPoint 的工作原理是将位存储在可堆叠的交叉网格数据访问数组中，并且是字节可寻址的。英特尔性能比较报告显示，3D XPoint 的访问延迟为 14 微秒，而 3D NAND SSD 的访问延迟为 200 微秒 [Hady 18]。 3D XPoint 的测试还显示出一致的延迟，而 3D NAND 的延迟分布更广泛，高达 3 毫秒。

3D XPoint 自 2017 年起投入商用。英特尔使用 Optane 品牌，并将其作为 DIMM 封装中的英特尔 Optane 持久内存和英特尔 Optane SSD 发布。

9.4.2 接口

该接口是驱动器支持的用于与系统通信的协议，通常通过磁盘控制器。下面是 SCSI、SAS、SATA、FC 和 NVMe 接口的简要概述。您需要检查当前的接口和支持的带宽是什么，因为当开发和采用新规范时，它们会随着时间的推移而变化。

小型计算机系统接口

小型计算机系统接口最初是并行传输总线，使用多个电连接器并行传输位。第一个版本是 1986 年的 SCSI-1，数据总线宽度为 8 位，允许每个时钟传输一个字节，并提供 5 Mbytes/s 的带宽。这是使用 50 针 Centronics C50 连接的。后来的并行 SCSI 版本使用了更宽的数据总线和更多的连接器引脚（多达 80 个引脚），带宽达到数百兆字节。

由于并行 SCSI 是共享总线，因此它可能会因总线争用而遭受性能问题，例如，当计划的系统备份使低优先级 I/O 使总线饱和时。解决方法包括将低优先级设备放置在它们自己的 SCSI 总线或控制器上。

并行总线的时钟在更高速度下也会成为一个问题，这与其他问题（包括有限的设备和对 SCSI 终结器包的需求）一起导致转向串行版本：SAS。

SAS

串行连接SCSI接口被设计为高速点对点传输，避免了并行SCSI的总线争用问题。最初的 SAS-1 规范是 3 Gbits/s（2003 年发布），随后是支持 6 Gbits/s（2009 年）的 SAS-2、支持 12 Gbits/s（2012 年）的 SAS-3 和支持 22.5 Gbits 的 SAS-4 /秒（2017）。支持链路聚合，以便多个端口可以组合起来提供更高的带宽。由于 8b/10b 编码，实际数据传输率为带宽的 80%。

其他 SAS 功能包括与冗余连接器和架构一起使用的驱动器双端口、I/O 多路径、SAS 域、热插拔以及 SATA 设备的兼容性支持。这些功能使 SAS 在企业中广受欢迎，尤其是在冗余架构中。

SATA

由于与 SCSI 和 SAS 类似的原因，并行 ATA（又名 IDE）接口标准已发展成为串行 ATA 接口。 SATA 1.0 创建于 2003 年，支持 1.5 Gbits/s；后来的主要版本是支持 3.0 Gbits/s 的 SATA 2.0 (2004 年) 和支持 6.0 Gbits/s 的 SATA 3.0 (2008 年)。主要和次要版本中添加了其他功能，包括本机命令队列支持。 SATA采用8b/10b编码，因此数据传输率为带宽的80%。 SATA 已广泛用于消费类台式机和笔记本电脑。

FC

光纤通道 (FC) 是一种用于数据传输的高速接口标准，最初仅适用于光纤电缆（因此得名），后来也支持铜缆。 FC 通常在企业环境中用于创建存储区域网络 (SAN)，其中多个存储设备可以通过光纤通道结构连接到多个服务器。与其他接口相比，这提供了更大的可扩展性和可访问性，类似于通过网络连接多个主机。而且，与网络一样，FC 可以涉及使用交换机将多个本地端点（服务器和存储）连接在一起。光纤通道标准的开发始于 1988 年，第一个版本于 1994 年获得 ANSI 批准 [FICA 20]。此后出现了许多变体和速度改进，最近的 Gen 7 256GFC 标准达到了 51,200 MB/s 全双工 [FICA 18]。

NVMe

非易失性内存 Express (NVMe) 是存储设备的 PCIe 总线规范。 NVMe 设备本身不是将存储设备连接到存储控制器卡，而是直接连接到 PCIe 总线的卡。创建于 2011 年，第一个 NVMe 规范是 1.0e（2013 年发布），最新的是 1.4（2019 年）[NVMe 20]。较新的规范添加了各种功能，例如热管理功能和用于自测试、验证数据和清理数据（使恢复变得不可能）的命令。 NVMe卡的带宽受PCIe总线的限制；目前常用的 PCIe 4.0 版本对于 x16 卡（链路宽度）具有 31.5 GB/s 的单向带宽。

与传统 SAS 和 SATA 相比，NVMe 的优势在于它支持多个硬件队列。可以从同一个 CPU 使用这些队列来促进缓存预热（并且借助 Linux 多队列支持，还可以避免共享内核锁）。这些队列还允许更大的缓冲，每个队列支持多达 64,000 个命令，而典型的 SAS 和 SATA 分别限制为 256 个和 32 个命令。

NVMe 还支持 SR-IOV 以提高虚拟机存储性能（请参阅第 11 章“云计算”、第 11.2 节“硬件虚拟化”）。

NVMe 用于低延迟闪存设备，预计 I/O 延迟低于 20 微秒。

9.4.3 存储类型

可以通过多种方式向服务器提供存储；以下各节介绍四种通用架构：磁盘设备、RAID、存储阵列和网络附加存储 (NAS)。

磁盘设备

最简单的架构是带有内部磁盘的服务器，由操作系统单独控制。磁盘连接到磁盘控制器，磁盘控制器是主板或扩展卡上的电路，允许查看和访问磁盘设备。在这种架构中，磁盘控制器仅充当管道，以便系统可以与磁盘进行通信。典型的个人计算机或笔记本电脑具有以这种方式连接的磁盘作为主存储。

这种架构最容易使用性能工具进行分析，因为每个磁盘对于操作系统都是已知的，并且可以单独观察。

一些磁盘控制器支持这种架构，称为一堆磁盘（JBOD）。

袭击

高级磁盘控制器可以为磁盘设备提供独立磁盘冗余阵列（RAID）架构（最初是廉价磁盘冗余阵列[Patterson 88]）。 RAID 可以将多个磁盘呈现为一个大、快速且可靠的虚拟磁盘。这些控制器通常包含板载缓存 (RAM)，以提高读写性能。

通过磁盘控制卡提供的RAID称为硬件RAID。 RAID 也可以通过操作系统软件来实现，但硬件 RAID 一直是首选，因为 CPU 昂贵的校验和和奇偶校验计算可以在专用硬件上更快地执行，而且此类硬件可以包括电池备份单元 (BBU) 以提高弹性。然而，处理器的进步已经产生了具有过剩周期和内核的 CPU，从而减少了卸载奇偶校验计算的需要。许多存储解决方案已回归到软件 RAID（例如，使用 ZFS），这降低了复杂性和硬件成本，并提高了操作系统的可观察性。在发生重大故障的情况下，软件 RAID 也可能比硬件 RAID 更容易修复（想象一下损坏的 RAID 卡）。

以下部分描述 RAID 的性能特征。经常使用术语“条带”：这是指数据被分组为跨多个驱动器写入的块（就像在所有驱动器上绘制条带一样）。

类型

可观察性

正如前面有关虚拟磁盘利用率的部分所述，使用硬件提供的虚拟磁盘设备可能会使操作系统中的可观察性变得更加困难，因为操作系统不知道物理磁盘正在做什么。如果 RAID 通过软件提供，通常可以观察到各个磁盘设备，因为操作系统直接管理它们。

读-修改-写

当数据存储为包含奇偶校验的条带时（如 RAID-5），写入 I/O 可能会产生额外的读取 I/O 和计算时间。这是因为小于条带大小的写入可能需要读取整个条带、修改字节、重新计算奇偶校验，然后重写条带。可以使用对 RAID-5 的优化来避免这种情况：不是读取整个条带，而是仅读取包括修改数据以及奇偶校验的条带（条带）部分。通过一系列 XOR 运算，可以计算更新的奇偶校验并将其与修改的条带一起写入。

跨越整个条带的写入可以覆盖之前的内容，而无需先读取它们。通过平衡条带大小与写入的平均 I/O 大小，可以提高此环境中的性能，以减少额外的读取开销。

缓存

实施 RAID-5 的磁盘控制器可以通过使用回写高速缓存来降低读-写-修改性能。这些缓存必须由电池供电，以便在发生电源故障时它们仍然可以完成缓冲写入。

附加功能

请注意，高级磁盘控制器卡可以提供可能影响性能的高级功能。浏览供应商文档是个好主意，这样您至少可以了解可能发生的情况。例如，以下是 Dell PERC 5 卡 [Dell 20] 的一些功能：

  • 巡检读取：每隔几天，读取所有磁盘块并验证其校验和。如果磁盘忙于处理请求，则分配给巡检读取功能的资源就会减少，以避免与系统工作负载竞争。

  • 缓存刷新间隔：将缓存中的脏数据刷新到磁盘之间的时间（以秒为单位）。由于写入取消和更好的聚合写入，更长的时间可能会减少磁盘 I/O；然而，它们也可能在较大的刷新期间导致更高的读取延迟。

这两者都会对性能产生重大影响。

存储阵列

存储阵列允许将许多磁盘连接到系统。它们使用先进的磁盘控制器以便可以配置RAID，并且通常提供大缓存（千兆字节）以提高读写性能。这些缓存通常也由电池供电，允许它们在回写模式下运行。常见的策略是在电池出现故障时切换到直写模式，这可能会被注意到由于等待读取-修改-写入周期而导致写入性能突然下降。

另一个性能考虑因素是存储阵列如何连接到系统 - 通常通过外部存储控制器卡。卡以及卡与存储阵列之间的传输都会受到 IOPS 和吞吐量的限制。为了提高性能和可靠性，存储阵列通常是双连接的，这意味着它们可以使用两根物理电缆连接到一个或两个不同的存储控制器卡。

网络附加存储

NAS 通过网络协议（例如 NFS、SMB/CIFS 或 iSCSI）通过现有网络提供给系统，通常来自称为 NAS 设备的专用系统。这些是独立的系统，应该单独分析。可以在客户端上进行一些性能分析，以检查所应用的工作负载和 I/O 延迟。网络性能也成为一个因素，网络拥塞和多跳延迟可能会引起问题。

块设备接口

块设备接口是在早期的Unix中创建的，用于以512字节的块为单位访问存储设备，并提供缓冲区高速缓存以提高性能。 Linux 中存在该接口，尽管缓冲区高速缓存的作用随着其他文件系统高速缓存的引入而减弱，如第 8 章“文件系统”中所述。

I/O 调度程序

I/O 在块层中通过经典调度程序（仅存在于 5.0 之前的 Linux 版本中）或通过较新的多队列调度程序进行排队和调度。这些调度程序允许对 I/O 进行重新排序（或重新安排）以优化交付。这可以提高并更公平地平衡性能，特别是对于具有高 I/O 延迟的设备（旋转磁盘）。

经典的调度程序包括：

  • Noop：这不执行调度（noop 是 CPU 中无操作的意思），并且可以在调度开销被认为不必要时使用（例如，在 RAM 磁盘中）。

  • 截止日期：尝试强制执行延迟截止日期；例如，可以选择以毫秒为单位的读取和写入到期时间。这对于需要确定性的实时系统非常有用。它还可以解决饥饿问题：当新发出的 I/O 跳入队列时，I/O 请求会缺乏磁盘资源，从而导致延迟异常。饥饿的发生可能是由于写入导致读取不足，以及由于电梯寻道和磁盘一个区域的大量 I/O 导致另一区域的 I/O 不足而导致的。截止时间调度程序通过使用三个独立的 I/O 队列部分解决了这个问题：读 FIFO、写 FIFO 和排序 [Love 10]。

  • CFQ：完全公平的队列调度器，为进程分配I/O时间片，类似于CPU调度，公平使用磁盘资源。它还允许通过 ionice(1) 命令为用户进程设置优先级和类别。

经典调度程序的一个问题是它们使用由单个锁保护的单个请求队列，这在高 I/O 速率下成为性能瓶颈。多队列驱动程序（blk-mq，在 Linux 3.13 中添加）通过为每个 CPU 使用单独的提交队列以及为设备使用多个调度队列来解决此问题。与经典调度程序相比，这为 I/O 提供了更好的性能和更低的延迟，因为请求可以在启动 I/O 的同一 CPU 上并行处理。这对于支持基于闪存的设备和能够处理数百万 IOPS 的其他设备类型是必要的 [Corbet 13b]。

多队列调度程序包括：

  • 无：无需排队。

  • BFQ：预算公平排队调度器，与CFQ类似，但分配带宽以及I/O时间。它为每个执行磁盘 I/O 的进程创建一个队列，并为每个队列维护一个以扇区为单位的预算。还有一个系统范围的预算超时，以防止一个进程持有设备的时间过长。 BFQ 支持 cgroup。

  • mq-deadline：截止日期的 blk-mq 版本（前面已描述）。

  • Kyber：一种调度程序，可根据性能调整读取和写入调度队列长度，以便满足目标读取或写入延迟。它是一个简单的调度程序，只有两个可调参数：目标读取延迟 (read_lat_nsec) 和目标同步写入延迟 (write_lat_nsec)。 Kyber 在 Netflix 云中显示出改进的存储 I/O 延迟，默认情况下使用它。

从Linux 5.0开始，多队列调度程序是默认的（不再包括经典调度程序）。

I/O 调度程序在 Linux 源文件的 Documentation/block 下有详细记录。

I/O调度后，请求被放入块设备队列中，以便下发给设备。

9.5.1 工具方法

工具方法是迭代可用工具、检查它们提供的关键指标的过程。虽然这是一种简单的方法，但它可能会忽略工具提供的可见性较差或没有可见性的问题，并且执行起来可能非常耗时。

对于磁盘，工具方法可能涉及检查以下内容（对于 Linux）：

  • iostat：使用扩展模式查找繁忙的磁盘（利用率超过 60%）、高平均服务时间（超过 10 毫秒）和高 IOPS（视情况而定）

  • iotop/biotop：识别哪个进程导致磁盘 I/O

  • biolatency：以直方图的形式检查 I/O 延迟的分布，查找多模式分布和延迟异常值（例如超过 100 毫秒）

  • biosnoop：检查单个 I/O

  • perf(1)/BCC/bpftrace：用于自定义分析，包括查看发出 I/O 的用户和内核堆栈

  • 磁盘控制器特定工具（来自供应商）

如果发现问题，请检查可用工具中的所有字段以了解更多背景信息。有关每个工具的更多信息，请参阅第 9.6 节“可观测性工具”。还可以使用其他方法，这可以识别更多类型的问题。

9.5.2 使用方法

USE 方法用于在性能调查的早期识别所有组件的瓶颈和错误。接下来的部分描述了如何将 USE 方法应用于磁盘设备和控制器，而第 9.6 节“可观察性工具”则展示了用于测量特定指标的工具。

磁盘设备

对于每个磁盘设备，检查：

  • 使用率：设备繁忙的时间

  • 饱和度：I/O 在队列中等待的程度

  • 错误：设备错误

可以先检查错误。它们有时会被忽视，因为尽管出现磁盘故障，系统仍能正常运行（尽管速度较慢）：磁盘通常配置在旨在容忍某些故障的冗余磁盘池中。除了操作系统中的标准磁盘错误计数器之外，磁盘设备还可以支持更广泛的错误计数器，这些计数器可以通过特殊工具（例如 SMART 数据6）检索。

6在 Linux 上，请参阅 MegaCLI 和 smartctl（稍后介绍）、cciss-vol-status、cpqarrayd、varmon 和 dpt-i2o-raidutils 等工具。

如果磁盘设备是物理磁盘，则利用率应该很容易找到。如果它们是虚拟磁盘，则利用率可能无法反映底层物理磁盘正在执行的操作。有关于此的更多讨论，请参见第 9.3.9 节“利用率”。

磁盘控制器

对于每个磁盘控制器，检查：

  • 利用率：当前吞吐量与最大吞吐量，操作率相同

  • 饱和度：I/O 由于控制器饱和而等待的程度

  • 错误：控制器错误

这里的利用率指标不是根据时间来定义的，而是根据磁盘控制器卡的限制来定义的：吞吐量（每秒字节数）和操作速率（每秒操作数）。操作包括读/写和其他磁盘命令。吞吐量或操作速率也可能受到将磁盘控制器连接到系统的传输的限制，正如它也可能受到从控制器到各个磁盘的传输的限制一样。应以相同的方式检查每种传输：错误、利用率、饱和度。

您可能会发现可观察性工具（例如 Linux iostat(1)）不提供每个控制器的指标，而是仅提供每个磁盘的指标。对此有一些解决方法：如果系统只有一个控制器，您可以通过对所有磁盘的这些指标求和来确定控制器 IOPS 和吞吐量。如果系统有多个控制器，您将需要确定哪些磁盘属于哪个磁盘，并相应地对指标进行求和。

磁盘控制器和传输的性能经常被忽视。幸运的是，它们不是系统瓶颈的常见来源，因为它们的容量通常超过所连接磁盘的容量。如果即使在不同的工作负载下，总磁盘吞吐量或 IOPS 始终以一定的速率稳定下来，这可能表明磁盘控制器或传输实际上导致了问题。

9.5.3 性能监控

性能监控可以随着时间的推移识别活动问题和行为模式。磁盘 I/O 的关键指标包括：

  • 磁盘利用率

  • 响应时间

磁盘利用率持续多秒达到 100% 很可能是一个问题。根据您的环境，超过 60% 也可能因队列增加而导致性能不佳。 “正常”或“不良”的值取决于您的工作负载、环境和延迟要求。如果您不确定，可以执行已知良好与不良工作负载的微基准测试，以显示如何通过磁盘指标找到这些工作负载。请参见第 9.8 节，实验。

应针对每个磁盘检查这些指标，以查找不平衡的工作负载和单个性能不佳的磁盘。响应时间度量可以作为每秒平均值进行监控，并且可以包括其他值，例如最大值和标准偏差。理想情况下，可以检查响应时间的完整分布，例如通过使用直方图或热图来查找延迟异常值和其他模式。

如果系统实施磁盘 I/O 资源控制，则还可以收集统计数据以显示这些资源是否以及何时使用。由于施加的限制，磁盘 I/O 可能成为瓶颈，而不是磁盘本身的活动。

利用率和响应时间显示了磁盘性能的结果。可以添加更多指标来表征工作负载，包括 IOPS 和吞吐量，从而提供用于容量规划的重要数据（请参阅下一节和第 9.5.10 节“扩展”）。

9.5.4 工作负载特征

表征所应用的负载是容量规划、基准测试和模拟工作负载的重要练习。通过识别可以消除的不必要的工作，它还可以带来一些最大的性能提升。

以下是表征磁盘 I/O 工作负载的基本属性：

  • 输入/输出速率

  • 输入/输出吞吐量

  • 输入/输出尺寸

  • 读/写比率

  • 随机与顺序

随机与顺序、读/写比率和 I/O 大小在第 9.3 节概念中描述。 I/O 速率 (IOPS) 和 I/O 吞吐量在第 9.1 节“术语”中定义。

这些特征可能会随时变化，特别是对于定期缓冲和刷新写入的应用程序和文件系统。为了更好地描述工作负载的特征，请捕获最大值和平均值。更好的是，检查值随时间的完整分布。

系统盘的随机读取工作负载较轻，平均 350 IOPS，吞吐量为 3 Mbytes/s，读取率为 96%。偶尔会出现持续 2 到 5 秒的短突发顺序写入，这会将磁盘驱动到最大 4,800 IOPS 和 560 MB/s。读取大小约为 8 KB，写入大小约为 128 KB。

除了在系统范围内描述这些特征之外，它们还可以用于描述每个磁盘和每个控制器的 I/O 工作负载。

高级工作负载特征/清单

可能会包含其他详细信息来描述工作负载的特征。这些已在此处列出作为需要考虑的问题，也可以作为彻底研究磁盘问题时的清单：

  • 系统范围内的 IOPS 速率是多少？每个磁盘？每个控制器？

  • 系统范围内的吞吐量是多少？每个磁盘？每个控制器？

  • 哪些应用程序或用户正在使用这些磁盘？

  • 正在访问哪些文件系统或文件？

  • 是否遇到任何错误？是由于无效请求还是磁盘问题造成的？

  • 可用磁盘上的 I/O 平衡程度如何？

  • 所涉及的每条传输总线的 IOPS 是多少？

  • 所涉及的每条传输总线的吞吐量是多少？

  • 正在发出哪些非数据传输磁盘命令？

  • 为什么会发出磁盘I/O（内核调用路径）？

  • 磁盘 I/O 应用程序同步的程度如何？

  • I/O 到达时间的分布是怎样的？

IOPS 和吞吐量问题可以分别针对读取和写入提出。还可以随着时间的推移检查其中任何一个，以查找最大值、最小值和基于时间的变化。另请参阅第 2 章“方法论”、第 2.5.11 节“工作负载特征”，其中提供了要测量的特征的更高级别摘要（谁、为什么、什么、如何）。

性能表征

前面的工作负载特征列表检查了所应用的工作负载。下面检查最终的性能：

  • 每个磁盘的繁忙程度（利用率）？

  • 每个磁盘的 I/O（等待队列）饱和程度如何？

  • 平均 I/O 服务时间是多少？

  • 平均 I/O 等待时间是多少？

  • 是否存在高延迟的 I/O 异常值？

  • I/O 延迟的完整分布是怎样的？

  • 系统资源控制（例如 I/O 限制）是否存在且处于活动状态？

  • 非数据传输磁盘命令的延迟是多少？

事件追踪

跟踪工具可用于将所有文件系统操作和详细信息记录到日志中以供以后分析（例如，第 9.6.7 节，biosnoop）。这可以包括磁盘设备 ID、I/O 或命令类型、偏移量、大小、发出和完成时间戳、完成状态以及原始进程 ID 和名称（如果可能）。通过发出和完成时间戳，可以计算 I/O 延迟（或者可以直接包含在日志中）。通过研究请求和完成时间戳的顺序，还可以识别设备的 I/O 重新排序。虽然这可能是工作负载表征的终极工具，但实际上，根据磁盘操作的速率，捕获和保存可能会花费大量开销。如果事件跟踪的磁盘写入包含在跟踪中，则不仅可能污染跟踪，而且还会创建反馈循环和性能问题。

9.5.5 延迟分析

延迟分析涉及更深入地研究系统以找到延迟的根源。对于磁盘，这通常会在磁盘接口处结束：I/O 请求和完成中断之间的时间。如果这与应用程序级别的 I/O 延迟相匹配，通常可以安全地假设 I/O 延迟源自磁盘，从而使您可以将调查重点放在磁盘上。如果延迟不同，在操作系统堆栈的不同级别上测量延迟将确定其来源。

图 9.10 描绘了一个通用 I/O 堆栈，其中显示了两个 I/O 异常值 A 和 B 的不同级别的延迟。

图9.10 堆栈延迟分析

从应用程序到磁盘驱动程序的每个级别，I/O A 的延迟都是相似的。这种相关性表明磁盘（或磁盘驱动程序）是延迟的原因。如果根据层之间相似的延迟值独立测量各层，则可以推断出这一点。

B 的延迟似乎源自文件系统级别（锁定或排队？），较低级别的 I/O 延迟贡献的时间要少得多。请注意，堆栈的不同层可能会增加或减少 I/O，这意味着一层与下一层的大小、计数和延迟会有所不同。 B 示例可能是仅观察较低级别（10 毫秒）的一个 I/O，但未能考虑为同一文件系统 I/O（例如元数据）提供服务的其他相关 I/O 的情况。

每个级别的延迟可以表示为：

  • 每个时间间隔的 I/O 平均值：通常由操作系统工具报告。

  • 完整的 I/O 分布：作为直方图或热图；请参阅第 9.7.3 节，延迟热图。

  • 每个 I/O 延迟值：请参阅前面的事件跟踪部分。

最后两个对于跟踪异常值的起源很有用，并且可以帮助识别 I/O 已拆分或合并的情况。

9.5.6 静态性能调优

静态性能调优重点关注配置环境的问题。对于磁盘性能，请检查静态配置的以下方面：

  • 存在多少个磁盘？属于哪些类型（例如 SMR、MLC）？尺寸？

  • 磁盘固件是什么版本？

  • 有多少个磁盘控制器？属于哪几种接口类型？

  • 磁盘控制器卡是否连接到高速插槽？

  • 每个 HBA 连接了多少个磁盘？

  • 如果存在磁盘/控制器备用电池，它们的功率水平是多少？

  • 磁盘控制器固件是什么版本？

  • 是否配置了RAID？具体如何，包括条纹宽度？

  • 多路径是否可用并已配置？

  • 磁盘设备驱动程序是什么版本？

  • 服务器主内存大小是多少？正在被页面缓存和缓冲区缓存使用？

  • 任何存储设备驱动程序是否存在操作系统错误/补丁？

  • 是否有用于磁盘 I/O 的资源控制？

请注意，设备驱动程序和固件中可能存在性能错误，最好通过供应商的更新来修复这些错误。

回答这些问题可以揭示被忽视的配置选择。有时，系统已针对一种工作负载进行配置，然后重新用于另一种工作负载。该战略将重新审视这些选择。

在担任 Sun ZFS 存储产品的性能主管期间，我收到的最常见的性能投诉是由错误配置引起的：使用 RAID-Z2（宽条带）的半个 JBOD（12 个磁盘）。这种配置提供了良好的可靠性，但性能并不令人印象深刻，与单个磁盘的性能相似。我学会了先询问配置详细信息（通常通过电话），然后再花时间登录系统并检查 I/O 延迟。

9.5.7 缓存调优

系统中可能存在许多不同的缓存，包括应用程序级、文件系统、磁盘控制器以及磁盘本身。其中的列表包含在第 9.3.3 节“缓存”中，可以按照第 2 章“方法论”第 2.5.18 节“缓存调优”中的描述进行调优。总之，检查存在哪些缓存，检查它们是否正在工作，检查它们的工作情况，然后调整缓存的工作负载并针对工作负载调整缓存。

9.5.8 资源控制

操作系统可以提供用于将磁盘I/O资源分配给进程或进程组的控制。这些可能包括 IOPS 和吞吐量的固定限制，或者更灵活方法的份额。这些如何工作是特定于实现的，如第 9.9 节“调优”中所述。

9.5.9 微基准测试

第 8 章“文件系统”中介绍了磁盘 I/O 微基准测试，解释了测试文件系统 I/O 和测试磁盘 I/O 之间的区别。这里我们要测试磁盘 I/O，这通常意味着通过操作系统的设备路径进行测试，特别是原始设备路径（如果可用），以避免所有文件系统行为（包括缓存、缓冲、I/O 拆分、I/O合并、代码路径开销和偏移映射差异）。

微基准测试的因素包括：

  • 方向：读或写

  • 磁盘偏移模式：随机或顺序

  • 偏移量范围：整个磁盘或严格范围（例如，仅限偏移量 0）

  • I/O 大小：512 字节（典型最小值）最大 1 Mbyte

  • 并发：运行中的 I/O 数量，或执行 I/O 的线程数量

  • 设备数量：单个磁盘测试或多个磁盘（以探索控制器和总线限制）

接下来的两节将展示如何结合这些因素来测试磁盘和磁盘控制器的性能。有关可用于执行这些测试的特定工具的详细信息，请参阅第 9.8 节“实验”。

磁盘

可以在每个磁盘的基础上执行微基准测试，以确定以下内容以及建议的工作负载：

  • 最大磁盘吞吐量（兆字节每秒）：128 KB 或 1 MB 读取，顺序

  • 最大磁盘操作速率（IOPS）：512字节读取，仅7偏移0

    7此大小旨在匹配最小磁盘块大小。如今许多磁盘使用 4 KB。

  • 最大磁盘随机读取 (IOPS)：512 字节读取，随机偏移

  • 读取延迟配置文件（平均微秒）：顺序读取、重复 512 字节、1K、2K、4K 等

  • 随机 I/O 延迟配置文件（平均微秒）：512 字节读取、对整个偏移范围重复、仅开始偏移、仅结束偏移

可以针对写入重复这些测试。使用“仅偏移 0”的目的是将数据缓存在磁盘缓存中，以便测量缓存访问时间。8

8我听说一些驱动器制造商有固件例程来加速扇区 0 I/O，从而夸大了此类测试的性能。您可以通过测试扇区 0 与扇区 your_favorite_number 来进行验证。

磁盘控制器

磁盘控制器可以通过将工作负载应用于多个磁盘来进行微基准测试，旨在达到控制器中的限制。这些测试可以使用以下内容以及建议的磁盘工作负载来执行：

  • 最大控制器吞吐量（兆字节每秒）：128 KB，仅偏移 0

  • 最大控制器操作速率 (IOPS)：512 字节读取，仅偏移 0

将工作负载逐一应用到磁盘，观察是否有限制。可能需要十几块磁盘才能找到磁盘控制器中的限制。

9.5.10 缩放

磁盘和磁盘控制器具有吞吐量和 IOPS 限制，这可以通过前面所述的微基准测试来证明。调整只能将性能提高到这些限制。如果需要更高的磁盘性能，并且缓存等其他策略不起作用，则需要扩展磁盘。

这是一个基于资源容量规划的简单方法：

  1. 根据吞吐量和 IOPS 确定目标磁盘工作负载。如果这是一个新系统，请参阅第 2 章“方法”、第 2.7 节“容量规划”。如果系统已经有工作负载，则用当前磁盘吞吐量和 IOPS 来表示用户数量，并将这些数字扩展到目标用户数量。 （如果不同时扩展缓存，磁盘工作负载可能会增加，因为每个用户的缓存比率会变小。）

  2. 计算支持此工作负载所需的磁盘数量。 RAID 配置因素。不要使用每个磁盘的最大吞吐量和 IOPS 值，因为这会导致磁盘利用率达到 100%，从而因饱和和排队而立即出现性能问题。选择目标利用率（例如 50%）并相应地缩放值。

  3. 计算支持此工作负载所需的磁盘控制器数量。

  4. 检查是否超出运输限制，并在必要时调整运输规模。

  5. 计算每个磁盘 I/O 的 CPU 周期以及所需的 CPU 数量（这可能需要多个 CPU 和并行 I/O）。

使用的每个磁盘的最大吞吐量和 IOPS 数量取决于其类型和磁盘类型。请参见第 9.3.7 节“IOPS 不相等”。微基准测试可用于查找给定 I/O 大小和 I/O 类型的具体限制，工作负载特征可用于现有工作负载，以了解哪些大小和类型重要。

为了满足磁盘工作负载要求，需要数十个磁盘并通过存储阵列连接的服务器并不罕见。我们过去常说，“添加更多主轴。”我们现在可能会说，“添加更多闪光灯。”

9.6.1 iostat

iostat(1) 总结每个磁盘的 I/O 统计信息，提供工作负载特征、利用率和饱和度的指标。它可以由任何用户执行，并且通常是在命令行中用于调查磁盘 I/O 问题的第一个命令。它提供的统计信息通常也由监控软件显示，因此值得详细学习 iostat(1) 以加深您对监控统计信息的理解。这些统计信息默认由内核启用，9 因此该工具的开销可以忽略不计。

9可以通过 /sys/block//queue/iostats 文件禁用统计信息。我不知道有人这样做过。

“iostat”这个名称是“I/O stats”的缩写，尽管最好将其称为“diskiostat”以反映它报告的 I/O 类型。当用户知道应用程序正在执行 I/O（对文件系统）但想知道为什么无法通过 iostat(1)（磁盘）看到它时，这偶尔会导致混乱。

iostat(1) 是在 20 世纪 80 年代初为 Unix 编写的，不同的操作系统上有不同的版本。可以通过 sysstat 包将其添加到基于 Linux 的系统。下面介绍Linux版本。

iostat 默认输出

如果没有任何参数或选项，则会打印 CPU 和磁盘统计信息的启动后摘要。这里介绍了这个工具；但是，您不应该使用此模式，因为稍后介绍的扩展模式更有用。

$ iostat Linux 5.3.0-1010-aws (ip-10-1-239-218) 02/12/20 _x86_64_ (2 CPU)

Device tps kB_read/s kB_wrtn/s kB_read kB_wrtn loop0 0.00 0.05 0.00 1232 0 [...] nvme0n1 3.40 17.11 36.03 409902 863344

第一个输出行是系统的摘要，包括内核版本、主机名、日期、体系结构和 CPU 数量。后续行显示 CPU（avg-cpu；这些统计信息已在第 6 章 CPU 中介绍）和磁盘设备（在 Device: 下）的启动后统计信息摘要。每个磁盘设备显示为一行，基本详细信息显示在列中。我用粗体突出显示了列标题；他们是：

  • tps：每秒事务数 (IOPS)

  • kB_read/s, kB_wrtn/s：每秒读取的千字节数和每秒写入的千字节数

  • kB_read, kB_wrtn：读取和写入的总千字节数

iostat(1) 可能不会显示某些 SCSI 设备，包括 CD-ROM。可以使用 Tapestat(1) 来检查 SCSI 磁带驱动器，该命令也在 sysstat 包中。另请注意，虽然 iostat(1) 报告块设备读取和写入，但它可能会排除某些其他类型的磁盘设备命令，具体取决于内核（请参阅内核函数 blk_do_io_stat() 中的逻辑）。 iostat(1) 扩展模式包括这些设备命令的额外字段。

iostat选项

iostat(1) 可以使用各种选项来执行，后跟可选的间隔和计数。例如：

# iostat 1 10

将打印一秒钟的摘要十次。和：

# iostat 1

将无休止地打印一秒摘要（直到键入 Ctrl-C）。

常用的选项有：

  • -c：显示 CPU 报告

  • -d：显示磁盘报告

  • -k：使用千字节而不是（512 字节）块

  • -m：使用兆字节而不是（512 字节）块

  • -p：包括每个分区的统计信息

  • -t：时间戳输出

  • -x：扩展统计信息

  • -s：短（窄）输出

  • -z：跳过显示零活动摘要

还有一个环境变量 POSIXLY_CORRECT=1，用于输出块（每个块 512 字节）而不是千字节。一些旧版本包含 NFS 统计信息选项，-n。从 sysstat 版本 9.1.3 开始，它已移至单独的 nfsiostat 命令。

iostat 扩展短输出

扩展输出 (-x) 提供了对前面介绍的方法有用的额外列。这些额外的列包括用于工作负载特征的 IOPS 和吞吐量指标、用于 USE 方法的利用率和队列长度，以及用于性能特征和延迟分析的磁盘响应时间。

多年来，扩展输出获得了越来越多的领域，最新版本（2019 年 12 月 12.3.1）产生了 197 个字符宽的输出。这不仅不适合本书，也不适合许多宽终端，导致输出由于换行而难以阅读。 2017 年添加了一个解决方案，即 -s 选项，以提供“短”或窄的输出，旨在适应 80 个字符的宽度。

以下是简短 (-s) 扩展 (-x) 统计信息和跳过零活动设备 (-z) 的示例：

点击这里查看代码图片

$ iostat -sxz 1 [...] avg-cpu: %user %nice %system %iowait %steal %idle 15.82 0.00 10.71 31.63 1.53 40.31

Device tps kB/s rqm/s await aqu-sz areq-sz %util nvme0n1 1642.00 9064.00 664.00 0.44 0.00 5.52 100.00 [...]

磁盘列是：

  • tps：每秒发出的交易数 (IOPS)

  • kB/s：每秒千字节数

  • rqm/s：每秒排队和合并的请求数

  • await：平均I/O响应时间，包括在操作系统中排队的时间和设备的I/O响应时间（毫秒）

  • aqu-sz：在驱动程序请求队列中等待且在设备上处于活动状态的平均请求数

  • areq-sz：平均请求大小（以千字节为单位）

  • %util：设备忙于处理 I/O 请求的时间百分比（利用率）

交付性能最重要的指标是 await，显示 I/O 的平均总等待时间。什么构成“好”或“坏”取决于您的需求。在示例输出中，await 为 0.44 毫秒，这对此数据库服务器来说是令人满意的。它可能会因多种原因而增加：排队（负载）、较大的 I/O 大小、旋转设备上的随机 I/O 以及设备错误。

对于资源使用和容量规划，%util 很重要，但请记住，它只是繁忙度（非空闲时间）的衡量标准，对于多个磁盘支持的虚拟设备可能意义不大。通过施加的负载可以更好地理解这些设备：tps（IOPS）和kB/s（吞吐量）。

rqm/s 列中的非零计数表明连续请求在传送到设备之前已合并，以提高性能。该指标也是顺序工作负载的标志。

由于 areq-sz 是合并后的，因此小尺寸（8 KB 或更少）表明存在无法合并的随机 I/O 工作负载。大尺寸可能是大 I/O 或合并的顺序工作负载（如前面的列所示）。

9.6.3 PSI

Linux 4.20 中添加的 Linux 压力失速信息 (PSI) 包括 I/O 饱和度的统计信息。这些不仅显示是否存在 I/O 压力，还显示它在过去五分钟内的变化情况。输出示例：

点击这里查看代码图片

# cat /proc/pressure/io some avg10=63.11 avg60=32.18 avg300=8.62 total=667212021 full avg10=60.76 avg60=31.13 avg300=8.35 total=622722632

此输出显示 I/O 压力正在增加，10 秒平均值 (63.11) 高于 300 秒平均值 (8.62)。这些平均值是任务 I/O 停滞的时间百分比。 some 行显示某些任务（线程）何时受到影响，full 行显示何时所有可运行任务受到影响。

与平均负载一样，这可以是用于警报的高级指标。一旦意识到存在磁盘性能问题，您可以使用其他工具来查找根本原因，包括按进程进行磁盘统计的 pidstat(8)。

磁盘响应时间（通常称为磁盘 I/O 延迟）可以通过检测设备发出到完成事件来测量。 biolatency.bt 工具执行此操作，将磁盘 I/O 延迟显示为直方图。例如：

点击这里查看代码图片

# biolatency.bt Attaching 4 probes... Tracing block device I/O... Hit Ctrl-C to end. ^C

@usecs: [32, 64) 2 |@ | [64, 128) 1 | | [128, 256) 1 | | [256, 512) 27 |@@@@@@@@@@@@@@@@@@@@@@@@@@ | [512, 1K) 43 |@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@ | [1K, 2K) 54 |@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@| [2K, 4K) 41 |@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@ | [4K, 8K) 47 |@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@ | [8K, 16K) 16 |@@@@@@@@@@@@@@@ | [16K, 32K) 4 |@@@ |

此输出显示 I/O 通常在 256 微秒到 16 毫秒（16K 微秒）之间完成。

源代码是：

点击这里查看代码图片

!/usr/local/bin/bpftrace

BEGIN { printf("Tracing block device I/O... Hit Ctrl-C to end.\n"); }

tracepoint:block:block_rq_issue { @start[args->dev, args->sector] = nsecs; }

tracepoint:block:block_rq_complete /@start[args->dev, args->sector]/ { @usecs = hist((nsecs - @start[args->dev, args->sector]) / 1000); delete(@start[args->dev, args->sector]); } END { clear(@start); }

测量 I/O 延迟需要存储每个 I/O 开始的自定义时间戳，然后在 I/O 完成时引用它来计算经过的时间。当在第 8 章“文件系统”第 8.6.15 节“BPFtrace”中测量 VFS 延迟时，开始时间戳存储在由线程 ID 键入的 BPF 映射中：这有效，因为相同的线程 ID 将在 CPU 上用于启动和完成事件。磁盘 I/O 的情况并非如此，因为完成事件将中断 CPU 上的任何其他操作。 biolatency.bt 中的唯一 ID 是根据设备和扇区号构建的：它假设一次只有一个 I/O 会传输到给定扇区。

与 I/O 大小单行一样，您可以将 args->rwbs 添加到映射键以按 I/O 类型进行细分。

9.7 可视化

有许多类型的可视化可以帮助分析磁盘 I/O 性能。本节通过各种工具的屏幕截图来演示这些内容。有关可视化的一般讨论，请参阅第 2 章“方法”、第 2.10 节“可视化”。

9.7.1 折线图

性能监控解决方案通常将磁盘 IOPS、吞吐量和利用率测量值随时间的变化绘制为折线图。这有助于说明基于时间的模式，例如一天中负载的变化，或重复发生的事件（例如文件系统刷新间隔）。

请注意绘制图表的指标。平均延迟可以隐藏多模式分布和异常值。所有磁盘设备的平均值可以隐藏不平衡的行为，包括单个设备的异常值。长期平均值也可以隐藏短期波动。

9.7.2 延迟散点图

散点图对于可视化每个事件的 I/O 延迟非常有用，其中可能包括数千个事件。 x 轴可以显示完成时间，y 轴可以显示 I/O 响应时间（延迟）。图 9.11 中的示例绘制了来自生产 MySQL 数据库服务器的 1,400 个 I/O 事件，使用 iosnoop(8) 捕获并使用 R 绘制。

图9.11 磁盘读写延迟散点图

散点图以不同方式显示读取 (+) 和写入 (°)。可以绘制其他维度，例如 y 轴上的磁盘块地址而不是延迟。

此处可以看到几个读取异常值，延迟超过 150 毫秒。这些异常值的原因此前尚不清楚。该散点图以及其他包含类似异常值的散点图表明它们发生在大量写入之后。由于写入是从 RAID 控制器回写缓存返回的，因此写入延迟较低，在返回完成后会将它们写入设备。我怀疑读取在设备写入后面排队。

该散点图显示了单个服务器几秒钟的情况。多个服务器或更长的时间间隔可以捕获更多的事件，这些事件在绘制时合并在一起并变得难以阅读。此时，请考虑使用延迟热图。

热图可用于可视化延迟，将时间的推移放在 x 轴上，将 I/O 延迟放在 y 轴上，将特定时间和延迟范围内的 I/O 数量放在 z 轴上。轴，以颜色显示（颜色越深表示越多）。热图在第 2 章“方法论”第 2.10.3 节“热图”中介绍。图 9.12 显示了一个有趣的磁盘示例。

可视化的工作负载是实验性的：我对多个磁盘逐一应用顺序读取，以探索总线和控制器的限制。由此产生的热图是出乎意料的（它被描述为翼手龙），并显示了仅考虑平均值时会错过的信息。看到的每个细节都有技术原因：例如，“喙”以八个磁盘结束，等于连接的 SAS 端口数（两个 x4 端口），一旦这些端口开始出现争用，“头”就从九个磁盘开始。

9.8 实验

本节介绍用于主动测试磁盘 I/O 性能的工具。请参阅第 9.5.9 节“微基准测试”，了解要遵循的建议方法。

使用这些工具时，最好让 iostat(1) 持续运行，以便可以立即双重检查任何结果。一些微基准测试工具可能需要“直接”操作模式来绕过文件系统缓存，并专注于磁盘设备性能。

9.8.1 对此

dd(1) 命令（设备到设备复制）可用于执行顺序磁盘性能的临时测试。例如，测试使用 1 MB I/O 大小的顺序读取：

点击这里查看代码图片

# dd if=/dev/sda1 of=/dev/null bs=1024k count=1k 1024+0 records in 1024+0 records out 1073741824 bytes (1.1 GB) copied, 7.44024 s, 144 MB/s

由于内核可以缓存和缓冲数据，因此 dd(1) 测量的吞吐量可以是缓存和磁盘的吞吐量，而不仅仅是磁盘的吞吐量。要仅测试磁盘的性能，您可以为磁盘使用字符特殊设备：在 Linux 上，raw(8) 命令（如果可用）可以在 /dev/raw 下创建这些设备。顺序写入可以类似地测试；但是，请注意不要破坏磁盘上的所有数据，包括主引导记录和分区表！

更安全的方法是将直接 I/O 标志与 dd(1) 和文件系统文件一起使用，而不是磁盘设备。请记住，测试现在包括一些文件系统开销。例如，对名为 out1 的文件进行写入测试：

点击这里查看代码图片

# dd if=/dev/zero of=out1 bs=1024k count=1000 oflag=direct 1000+0 records in 1000+0 records out 1048576000 bytes (1.0 GB, 1000 MiB) copied, 1.79189 s, 585 MB/s

另一个终端会话中的 iostat(1) 确认磁盘 I/O 写入吞吐量约为 585 MB/秒。

使用 iflag=direct 与输入文件进行直接 I/O。

9.8.2 自定义负载生成器

要测试自定义工作负载，您可以编写自己的负载生成器并使用 iostat(1) 测量结果性能。自定义负载生成器可以是一个简短的 C 程序，用于打开设备路径并应用预期的工作负载。在 Linux 上，可以使用 O_DIRECT 打开块特殊设备文件，以避免缓冲。如果您使用高级语言，请尝试使用至少避免库缓冲的系统级接口（例如，Perl 中的 sysread()），并且最好也避免内核缓冲（例如，O_DIRECT）。

ioping(1) 是一个有趣的磁盘微基准测试工具，类似于 ICMP ping(8) 实用程序。在 nvme0n1 磁盘设备上运行 ioping(1)：

# ioping /dev/nvme0n1 4 KiB <<< /dev/nvme0n1 (block device 8 GiB): request=1 time=438.7 us (warmup) 4 KiB <<< /dev/nvme0n1 (block device 8 GiB): request=2 time=421.0 us 4 KiB <<< /dev/nvme0n1 (block device 8 GiB): request=3 time=449.4 us 4 KiB <<< /dev/nvme0n1 (block device 8 GiB): request=4 time=412.6 us 4 KiB <<< /dev/nvme0n1 (block device 8 GiB): request=5 time=468.8 us ^C

默认情况下，ioping(1) 每秒发出 4 KB 读取并以微秒为单位打印其 I/O 延迟。终止时，会打印各种统计数据。

ioping(1) 与其他基准测试工具的不同之处在于它的工作负载很轻。以下是 ioping(1) 运行时的一些 iostat(1) 输出：

点击这里查看代码图片

$ iostat -xsz 1 [...] Device tps kB/s rqm/s await aqu-sz areq-sz %util nvme0n1 1.00 4.00 0.00 0.00 0.00 4.00 0.40

磁盘利用率仅为 0.4%。 ioping(1) 可用于调试生产环境中的问题，而其他微基准测试不适用，因为它们通常会将目标磁盘驱动到 100% 利用率。

9.9 调优

第 9.5 节“方法论”中介绍了许多调优方法，包括缓存调优、扩展和工作负载特征，它们可以帮助您识别和消除不必要的工作。调优的另一个重要领域是存储配置，可以将其作为静态性能调优方法的一部分进行研究。

以下部分显示了可以调整的区域：操作系统、磁盘设备和磁盘控制器。可用的可调参数因操作系统版本、磁盘型号、磁盘控制器及其固件而异；请参阅各自的文档。虽然更改可调参数很容易，但默认设置通常是合理的，很少需要进行太多调整。

9.9.1 操作系统可调参数

其中包括 ionice(1)、资源控制和内核可调参数。

爱奥尼亚

在 Linux 上， ionice(1) 命令可用于设置进程的 I/O 调度类和优先级。调度类别以数字方式标识：

  • 0、none：未指定类，因此内核将选择默认值（尽力而为），其优先级基于进程的nice值。

  • 1、实时性：以最高优先级访问磁盘。如果误用，这可能会导致其他进程挨饿（就像 RT CPU 调度类一样）。

  • 2、besteffort：默认调度等级，支持优先级0~7，其中0为最高。

  • 3、空闲：仅在磁盘空闲一段宽限期后才允许磁盘 I/O。

用法示例：

# ionice -c 3 -p 1623

这会将进程 ID 1623 置于空闲 I/O 调度类中。这对于长时间运行的备份作业来说可能是理想的，这样它们就不太可能干扰生产工作负载。

资源控制

现代操作系统提供资源控制，用于以自定义方式管理磁盘或文件系统 I/O 使用情况。

对于 Linux，容器组 (cgroups) 块 I/O (blkio) 子系统为进程或进程组提供存储设备资源控制。这可以是比例权重（如份额）或固定限制。可以独立设置读取和写入以及 IOPS 或吞吐量（每秒字节数）的限制。有关更多详细信息，请参阅第 11 章，云计算。

可调参数

Linux 可调参数示例包括：

  • /sys/block/*/queue/scheduler：选择 I/O 调度程序策略：这些可能包括 noop、deadline、cfq 等。请参阅第 9.4 节“体系结构”中的早期描述。

  • /sys/block/*/queue/nr_requests：块层可以分配的读或写请求的数量。

  • /sys/block/*/queue/read_ahead_kb：文件系统请求的最大预读千字节数。

与其他内核可调参数一样，请查看文档以获取完整列表、描述和警告。在 Linux 源代码中，请参阅 Documentation/block/queue-sysfs.txt。

9.9.2 磁盘设备可调参数

在 Linux 上，hdparm(8) 工具可以设置各种磁盘设备可调参数，包括电源管理和旋转超时 [Archlinux 20]。使用此工具时要非常小心，并研究 hdparm(8) 手册页 - 各种选项都被标记为“危险”，因为它们可能会导致数据丢失。
