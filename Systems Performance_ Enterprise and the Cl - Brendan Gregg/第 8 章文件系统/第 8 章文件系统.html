第 8 章文件系统

第8章

文件系统

对于应用程序来说，文件系统性能通常比磁盘或存储设备性能更重要，因为应用程序是与文件系统交互并等待的。文件系统可以使用缓存、缓冲和异步 I/O 来避免应用程序遭受磁盘级（或远程存储系统）延迟。

系统性能分析和监控工具历来关注磁盘性能，而将文件系统性能视为盲点。本章重点介绍文件系统，展示它们的工作原理以及如何测量其延迟和其他细节。这通常可以排除文件系统及其底层磁盘设备作为性能不佳的根源，从而使调查可以转移到其他领域。

本章的学习目标是：

  • 了解文件系统模型和概念。

  • 了解文件系统工作负载如何影响性能。

  • 熟悉文件系统缓存。

  • 熟悉文件系统内部结构和性能特征。

  • 遵循文件系统分析的各种方法。

  • 测量文件系统延迟以识别模式和异常值。

  • 使用跟踪工具调查文件系统的使用情况。

  • 使用微基准测试文件系统性能。

  • 了解文件系统可调参数。

本章由六部分组成，前三部分提供文件系统分析的基础，后三部分展示其在基于 Linux 的系统中的实际应用。各部分如下：

  • 背景介绍文件系统相关术语和基本模型，阐释文件系统原理和文件系统关键性能概念。

  • 体系结构介绍了通用和特定的文件系统体系结构。

  • 方法论描述了观察性和实验性的性能分析方法。

  • 可观察性工具展示了适用于基于 Linux 的系统的文件系统可观察性工具，包括静态和动态检测。

  • 实验总结了文件系统基准测试工具。

  • 调优描述了文件系统可调参数。

8.1 术语

作为参考，本章中使用的文件系统相关术语包括：

  • 文件系统：将数据组织为文件和目录，具有用于访问它们的基于文件的接口以及用于控制访问的文件权限。附加内容可能包括设备、套接字和管道的特殊文件类型，以及包括文件访问时间戳的元数据。

  • 文件系统缓存：主内存（通常是 DRAM）的一个区域，用于缓存文件系统内容，其中可能包括针对各种数据和元数据类型的不同缓存。

  • 操作：文件系统操作是对文件系统的请求，包括read(2)、write(2)、open(2)、close(2)、stat(2)、mkdir(2)等操作。

  • I/O：输入/输出。文件系统 I/O 可以通过多种方式定义；这里用来表示仅直接读写（执行I/O）的操作，包括read(2)、write(2)、stat(2)（读取统计数据）和mkdir(2)（写入新目录）入口）。 I/O 不包括 open(2) 和 close(2)（尽管这些调用会更新元数据并可能导致间接磁盘 I/O）。

  • 逻辑I/O：应用程序向文件系统发出的I/O。

  • 物理 I/O：由文件系统（或通过原始 I/O）直接向磁盘发出的 I/O。

  • 块大小：也称为记录大小，是文件系统磁盘上数据组的大小。请参见第 8.4.4 节“文件系统功能”中的块与范围。

  • 吞吐量：应用程序和文件系统之间的当前数据传输速率，以每秒字节数为单位。

  • inode：索引节点（inode）是包含文件系统对象元数据的数据结构，包括权限、时间戳和数据指针。

  • VFS：虚拟文件系统，抽象并支持不同文件系统类型的内核接口。

  • 卷：存储实例，比使用整个存储设备提供更大的灵活性。卷可以是一个设备或多个设备的一部分。

  • 卷管理器：以灵活的方式管理物理存储设备的软件，创建供操作系统使用的虚拟卷。

本章将介绍其他术语。术语表包含供参考的基本术语，包括 fsck、IOPS、操作率和 POSIX。另请参阅第 2 章和第 3 章中的术语部分。

8.3 概念

以下是一些重要的文件系统性能概念的选择。

8.3.1 文件系统延迟

文件系统延迟是文件系统性能的主要指标，以从逻辑文件系统请求到完成的时间来衡量。它包括在文件系统和磁盘 I/O 子系统中花费的时间，以及等待磁盘设备（物理 I/O）的时间。应用程序线程通常会在应用程序请求期间阻塞，以等待文件系统请求完成，这样，文件系统延迟会直接按比例影响应用程序性能。

应用程序可能不会直接受到影响的情况包括使用非阻塞 I/O、预取（第 8.3.4 节）以及从异步线程（例如后台刷新线程）发出 I/O 时。如果应用程序提供其文件系统使用情况的详细指标，则可以从应用程序中识别这些情况。如果没有，通用方法是使用内核跟踪工具，该工具可以显示导致逻辑文件系统 I/O 的用户级堆栈跟踪。然后可以研究该堆栈跟踪以查看哪些应用程序例程发出了它。

操作系统历史上并没有使文件系统延迟易于观察，而是提供磁盘设备级统计数据。但在很多情况下，此类统计数据与应用程序性能无关，而且还具有误导性。一个例子是文件系统执行写入数据的后台刷新，这可能表现为高延迟磁盘 I/O 突发。从磁盘设备级别的统计来看，这看起来令人震惊；但是，没有应用程序正在等待这些完成。有关更多情况，请参见第 8.3.12 节“逻辑 I/O”与“物理 I/O”。

8.3.2 缓存

文件系统通常会使用主内存 (RAM) 作为缓存以提高性能。对于应用程序来说，这个过程是透明的：应用程序逻辑 I/O 延迟变得更低（更好），因为它可以从主内存而不是慢得多的磁盘设备提供服务。

随着时间的推移，缓存会增加，而操作系统的可用内存会减少。这可能会让新用户感到震惊，但这是完全正常的。原则是：如果有空闲的主内存，就用它做一些有用的事情。当应用程序需要更多内存时，内核应快速将其从文件系统缓存中释放以供使用。

文件系统使用缓存来提高读取性能，并使用缓冲（在缓存中）来提高写入性能。文件系统和块设备子系统通常使用多种类型的缓存，其中可能包括表 8.1 中的那些。

8.3.3 随机与顺序 I/O

根据每个 I/O 的文件偏移量，一系列逻辑文件系统 I/O 可以被描述为随机或顺序。对于顺序 I/O，每个 I/O 偏移量从前一个 I/O 的末尾开始。随机I/O之间没有明显的关系，偏移量随机变化。随机文件系统工作负载也可以指随机访问许多不同的文件。

图 8.4 顺序和随机文件 I/O

图 8.4 说明了这些访问模式，显示了一系列有序的 I/O 和示例文件偏移量。

由于某些存储设备的性能特征（在第 9 章“磁盘”中描述），文件系统历来尝试通过将文件数据顺序且连续地放置在磁盘上来减少随机 I/O。术语“碎片”描述了文件系统在这方面表现不佳所发生的情况，导致文件分散在驱动器上，从而使顺序逻辑 I/O 产生随机物理 I/O。

文件系统可以测量逻辑 I/O 访问模式，以便它们可以识别顺序工作负载，然后使用预取或预读来提高其性能。这对于旋转磁盘很有帮助；对于闪存驱动器来说则不然。

8.3.4 预取

常见的文件系统工作负载涉及顺序读取大量文件数据，例如，用于文件系统备份。该数据可能太大而无法放入缓存，或者可能只读取一次并且不太可能保留在缓存中（取决于缓存驱逐策略）。这样的工作负载性能相对较差，因为它的缓存命中率较低。

预取是解决此问题的常见文件系统功能。它可以根据当前和先前的文件 I/O 偏移量检测顺序读取工作负载，然后在应用程序请求磁盘读取之前预测并发出磁盘读取操作。这会填充文件系统缓存，因此，如果应用程序确实执行了预期的读取，则会导致缓存命中（所需的数据已在缓存中）。示例场景如下：

  1. 应用程序发出文件 read(2)，将执行传递给内核。

  2. 数据未缓存，因此文件系统将读取发出到磁盘。

  3. 将前一个文件偏移指针与当前位置进行比较，如果它们是连续的，则文件系统发出额外的读取（预取）。

  4. 第一次读取完成，内核将数据和执行传递回应用程序。

  5. 任何预取读取完成，填充缓存以供将来的应用程序读取。

  6. 未来的顺序应用程序通过 RAM 中的缓存快速完成读取。

图 8.5 也说明了这种情况，其中应用程序读取偏移量 1，然后读取偏移量 2 触发接下来三个偏移量的预取。

图8.5 文件系统预取

当预取检测工作良好时，应用程序会显着提高顺序读取性能；磁盘领先于应用程序请求（前提是它们有足够的带宽来执行此操作）。当预取检测效果不佳时，会发出应用程序不需要的不必要的 I/O，从而污染缓存并消耗磁盘和 I/O 传输资源。文件系统通常允许根据需要调整预取。

8.3.5 预读

从历史上看，预取也称为预读。 Linux 使用预读术语 readahead(2) 来表示系统调用，它允许应用程序显式预热文件系统缓存。

8.3.6 回写式缓存

文件系统通常使用回写式缓存来提高写入性能。它的工作原理是在传输到主内存后将写入视为已完成，并稍后将它们异步写入磁盘。将这些“脏”数据写入磁盘的文件系统过程称为刷新。示例序列如下：

  1. 应用程序发出文件 write(2)，将执行传递给内核。

  2. 来自应用程序地址空间的数据被复制到内核。

  3. 内核将 write(2) 系统调用视为已完成，并将执行传递回应用程序。

  4. 一段时间后，异步内核任务找到写入的数据并发出磁盘写入。

权衡是可靠性。基于 DRAM 的主存储器具有易失性，一旦发生电源故障，脏数据可能会丢失。数据也可能不完整地写入磁盘，从而留下损坏的磁盘状态。

如果文件系统元数据损坏，文件系统可能不再加载。这种状态可能只能通过系统备份来恢复，从而导致停机时间延长。更糟糕的是，如果损坏影响应用程序读取和使用的文件内容，则业务可能会陷入危险。

为了平衡速度和可靠性的需求，文件系统可以默认提供回写缓存，并提供同步写入选项来绕过此行为并直接写入持久存储设备。

8.3.7 同步写入

仅当完全写入持久存储（例如磁盘设备）时，同步写入才会完成，其中包括写入任何必要的文件系统元数据更改。这些比异步写入（回写缓存）慢得多，因为同步写入会导致磁盘设备 I/O 延迟（并且可能由于文件系统元数据而导致多个 I/O）。同步写入由某些应用程序使用，例如数据库日志编写器，其中异步写入导致数据损坏的风险是不可接受的。

同步写入有两种形式：同步写入的单个 I/O 和同步提交的先前写入组。

单独的同步写入

当使用标志 O_SYNC 或其变体之一 O_DSYNC 和 O_RSYNC（从 Linux 2.6.31 开始由 glibc 映射到 O_SYNC）打开文件时，写入 I/O 是同步的。某些文件系统具有挂载选项，可强制所有文件的所有写入 I/O 同步。

同步提交先前的写入

应用程序可以使用 fsync(2) 系统调用在其代码中的检查点同步提交先前的异步写入，而不是同步写入单个 I/O。这可以通过对写入进行分组来提高性能，并且还可以通过使用写入取消来避免多次元数据更新。

还有其他情况会提交以前的写入，例如关闭文件句柄，或者文件上有太多未提交的缓冲区。前者在解压许多文件的存档时通常会出现长时间的暂停，尤其是通过 NFS。

8.3.8 原始和直接 I/O

这些是应用程序可以使用的其他类型的 I/O（如果内核或文件系统支持的话）：

原始 I/O 直接发送到磁盘偏移量，完全绕过文件系统。它已经被一些应用程序使用，特别是数据库，它们可以比文件系统缓存更好地管理和缓存自己的数据。缺点是软件更加复杂，并且管理困难：常规文件系统工具集不能用于备份/恢复或可观察性。

直接 I/O 允许应用程序使用文件系统，但绕过文件系统缓存，例如，通过在 Linux 上使用 O_DIRECT open(2) 标志。这类似于同步写入（但没有 O_SYNC 提供的保证），并且它也适用于读取。它不像原始设备 I/O 那样直接，因为文件偏移到磁盘偏移的映射仍然必须由文件系统代码执行，并且 I/O 也可以调整大小以匹配文件系统用于 on- 的大小。磁盘布局（其记录大小），否则可能会出现错误（EINVAL）。根据文件系统的不同，这不仅可能禁用读缓存和写缓冲，还可能禁用预取。

8.3.9 非阻塞 I/O

通常，文件系统 I/O 将立即完成（例如，从缓存）或等待后完成（例如，对于磁盘设备 I/O）。如果需要等待，应用程序线程将阻塞并离开CPU，允许其他线程在其等待时执行。虽然被阻止的线程无法执行其他工作，但这通常不是问题，因为多线程应用程序可以在某些线程被阻止时创建额外的线程来执行。

在某些情况下，需要非阻塞 I/O，例如避免创建线程的性能或资源开销时。非阻塞 I/O 可以通过使用 open(2) 系统调用的 O_NONBLOCK 或 O_NDELAY 标志来执行，这会导致读取和写入返回 EAGAIN 错误而不是阻塞，从而告诉应用程序稍后重试。 （对此的支持取决于文件系统，该系统可能仅对建议性或强制文件锁支持非阻塞。）

操作系统还可以提供单独的异步I/O接口，例如aio_read(3)和aio_write(3)。 Linux 5.1 添加了一个名为 io_uring 的新异步 I/O 接口，提高了易用性、效率和性能 [Axboe 19]。

非阻塞 I/O 也在第 5 章“应用程序”第 5.2.6 节“非阻塞 I/O”中进行了讨论。

8.3.10 内存映射文件

对于某些应用程序和工作负载，可以通过将文件映射到进程地址空间并直接访问内存偏移量来提高文件系统 I/O 性能。这避免了调用 read(2) 和 write(2) 系统调用来访问文件数据时产生的系统调用执行和上下文切换开销。如果内核支持直接将文件数据缓冲区映射到进程地址空间，还可以避免数据的双重复制。

内存映射是使用 mmap(2) 系统调用创建的，并使用 munmap(2) 删除的。可以使用 madvise(2) 来调整映射，如第 8.8 节“调整”中所总结的那样。某些应用程序提供了在其配置中使用 mmap 系统调用（可能称为“mmap 模式”）的选项。例如，Riak 数据库可以使用 mmap 作为其内存数据存储。

我注意到一种倾向，即尝试使用 mmap(2) 来解决文件系统性能问题，而不首先对其进行分析。如果问题是来自磁盘设备的高 I/O 延迟，那么当高磁盘 I/O 延迟仍然存在并占主导地位时，使用 mmap(2) 避免小系统调用开销可能收效甚微。

在多处理器系统上使用映射的一个缺点是保持每个 CPU MMU 同步的开销，特别是删除映射的 CPU 交叉调用（TLB 关闭）。根据内核和映射，可以通过延迟 TLB 更新（惰性击落）来最小化这些问题 [Vahalia 96]。

8.3.11 元数据

数据描述文件和目录的内容，而元数据描述有关它们的信息。元数据可以指可以从文件系统接口（POSIX）读取的信息或实现文件系统磁盘布局所需的信息。这些分别称为逻辑元数据和物理元数据。

逻辑元数据

逻辑元数据是由消费者（应用程序）读取和写入文件系统的信息，可以是：

  • 显式：读取文件统计信息（stat(2)）、创建和删除文件（creat(2)、unlink(2)）和目录（mkdir(2)、rmdir(2)）、设置文件属性（chown(2)、修改(2))

  • 隐式：文件系统访问时间戳更新、目录修改时间戳更新、已用块位图更新、可用空间统计信息

“元数据密集”的工作负载通常是指逻辑元数据，例如，网络服务器对 stat(2) 文件进行统计，以确保它们自缓存以来没有发生更改，其速度比读取文件数据内容的速度要快得多。

物理元数据

物理元数据是指记录所有文件系统信息所需的磁盘布局元数据。使用的元数据类型取决于文件系统，可能包括超级块、索引节点、数据指针块（主、辅助...）和空闲列表。

逻辑和物理元数据是逻辑 I/O 和物理 I/O 之间存在差异的原因之一。

8.3.12 逻辑 I/O 与物理 I/O

尽管看起来违反直觉，但由于多种原因，应用程序向文件系统请求的 I/O（逻辑 I/O）可能与磁盘 I/O（物理 I/O）不匹配。

文件系统的作用远不止于将持久存储（磁盘）作为基于文件的接口。它们缓存读取、缓冲写入、将文件映射到地址空间，并创建额外的 I/O 来维护记录所有内容所需的磁盘上物理布局元数据。与应用程序 I/O 相比，这可能会导致磁盘 I/O 不相关、间接、隐式、膨胀或收缩。示例如下。

无关

这是与应用程序无关的磁盘 I/O，可能是由于：

  • 其他应用

  • 其他租户：磁盘I/O来自另一个云租户（通过某些虚拟化技术下的系统工具可见）。

  • 其他内核任务：例如，当内核正在重建软件 RAID 卷或执行异步文件系统校验和验证时（请参阅第 8.4 节，体系结构）。

  • 管理任务：例如备份。

间接

这是由应用程序引起的磁盘 I/O，但没有立即对应的应用程序 I/O。这可能是由于：

  • 文件系统预取：添加应用程序可能使用或不使用的附加 I/O。

  • 文件系统缓冲：使用回写式缓存来推迟和合并写入，以便稍后刷新到磁盘。某些系统可能会在写入之前缓冲写入数十秒，然后这些写入会显示为大的、不频繁的突发。

隐含的

这是由应用程序事件直接触发的磁盘 I/O，而不是显式文件系统读写，例如：

  • 内存映射加载/存储：对于内存映射 (mmap(2)) 文件，加载和存储指令可能会触发磁盘 I/O 来读取或写入数据。写入可能会被缓冲并稍后写入。在分析文件系统操作（读（2）、写（2））并且无法找到 I/O 源（因为它是由指令而不是系统调用触发）时，这可能会令人困惑。
瘪

磁盘 I/O 小于应用程序 I/O，甚至不存在。这可能是由于：

  • 文件系统缓存：满足从主内存而不是磁盘的读取。

  • 文件系统写入取消：相同的字节偏移量在刷新到磁盘之前会被多次修改。

  • 压缩：减少逻辑 I/O 到物理 I/O 的数据量。

  • 合并：在将顺序 I/O 发送到磁盘之前合并它们（这会减少 I/O 计数，但不会减少总大小）。

  • 内存中文件系统：可能永远不会写入磁盘的内容（例如 tmpfs）。

充气

大于应用程序 I/O 的磁盘 I/O。这是典型的案例，原因如下：

  • 文件系统元数据：添加额外的 I/O。

  • 文件系统记录大小：四舍五入 I/O 大小（膨胀字节）或分段 I/O（膨胀计数）。

  • 文件系统日志记录：如果采用，这可以使磁盘写入加倍，一次写入日志，另一次写入最终目的地。

  • 卷管理器奇偶校验：读取-修改-写入周期，添加额外的 I/O。

  • RAID 膨胀：将额外的奇偶校验数据或数据写入镜像卷。

例子

为了展示这些因素如何同时发生，以下示例描述了 1 字节应用程序写入时可能发生的情况：

  1. 应用程序对现有文件执行 1 字节写入。

  2. 文件系统将该位置标识为 128 KB 文件系统记录的一部分，该记录不会被缓存（但引用它的元数据会被缓存）。

  3. 文件系统请求从磁盘加载记录。

  4. 磁盘设备层将 128 KB 读取分解为适合设备的较小读取。

  5. 磁盘执行多次较小的读取，总计 128 KB。

  6. 文件系统现在用新字节替换记录中的 1 个字节。

  7. 一段时间后，文件系统或内核请求将 128 KB 脏记录写回磁盘。

  8. 磁盘写入 128 KB 记录（如果需要，则将其分解）。

  9. 文件系统写入新的元数据，例如，以更新引用（用于写入时复制）或访问时间。

  10. 磁盘执行更多写入操作。

因此，虽然应用程序仅执行一次 1 字节写入，但磁盘执行多次读取（总共 128 KB）和更多写入（超过 128 KB）。

8.3.13 运算不相等

从前面的部分可以清楚地看出，文件系统操作可以根据其类型表现出不同的性能。仅从速率无法了解“500 次操作/秒”的工作负载。某些操作可能会以主存速度从文件系统缓存返回；其他的可能会从磁盘返回并且速度要慢几个数量级。其他决定因素包括操作是随机的还是顺序的、读取还是写入、同步写入还是异步写入、它们的 I/O 大小、它们是否包含其他操作类型、它们的 CPU 执行成本（以及系统的 CPU 负载情况），以及存储设备的特性。

8.3.15 访问时间戳

许多文件系统都支持访问时间戳，它记录每个文件和目录被访问（读取）的时间。这会导致每次读取文件时都会更新文件元数据，从而产生消耗磁盘 I/O 资源的写入工作负载。第 8.8 节“调整”展示了如何关闭这些更新。

某些文件系统通过推迟和分组访问时间戳写入来优化访问时间戳写入，以减少对活动工作负载的干扰。

8.3.16 容量

当文件系统填满时，性能可能会因多种原因而降低。首先，写入新数据时，可能需要更多的 CPU 时间和磁盘 I/O 来查找磁盘上的空闲块。5其次，磁盘上的空闲空间区域可能更小且位置更稀疏，从而降低性能由于较小的 I/O 或随机 I/O。

5例如，当池存储超过阈值（最初为 80%，后来为 99%）时，ZFS 会切换到不同且较慢的空闲块查找算法。请参阅“当池非常满时，池性能可能会降低”[Oracle 12]。

这个问题的严重程度取决于文件系统类型、磁盘布局、写时复制的使用及其存储设备。下一节将介绍各种文件系统类型。

图8.8 Linux文件系统缓存

缓冲区高速缓存

Unix 在块设备接口上使用缓冲区高速缓存来缓存磁盘设备块。这是一个单独的、固定大小的缓存，随着后来添加的页面缓存，在平衡它们之间的不同工作负载以及双缓存和同步的开销时出现了调优问题。这些问题在很大程度上通过使用页缓存来存储缓冲区高速缓存来解决，这种方法是由 SunOS 引入的，称为统一缓冲区高速缓存。

Linux 最初与 Unix 一样使用缓冲区高速缓存。从 Linux 2.4 开始，缓冲区缓存也存储在页缓存中（因此图 8.8 中的虚线边框），避免了双重缓存和同步开销。缓冲区缓存功能仍然存在，提高了块设备 I/O 的性能，并且该术语仍然出现在 Linux 可观察性工具中（例如 free(1)）。

缓冲区高速缓存的大小是动态的，可以从 /proc 中观察到。

页面缓存

页缓存在 1985 年虚拟内存重写期间首次引入 SunOS，并添加到 SVR4 Unix [Vahalia 96]。它缓存虚拟内存页面，包括映射的文件系统页面，提高文件和目录 I/O 的性能。它的文件访问比缓冲区高速缓存更有效，缓冲区高速缓存需要每次查找时从文件偏移量转换为磁盘偏移量。多种文件系统类型可以使用页面缓存，包括原始消费者 UFS 和 NFS。大小是动态的：页面缓存将增长以使用可用内存，并在应用程序需要时再次释放它。

Linux 有一个具有相同属性的页缓存。 Linux 页面缓存的大小也是动态的，可以通过一个可调参数来设置从页面缓存中逐出和交换（swappiness；请参阅第 7 章，内存）之间的平衡。

脏（已修改）且文件系统需要使用的内存页面由内核线程刷新到磁盘。在 Linux 2.6.32 之前，有一个页面脏刷新 (pdflush) 线程池，根据需要在 2 到 8 个之间。此后，这些线程已被刷新器线程（称为刷新）取代，刷新器线程是为每个设备创建的，以更好地平衡每个设备的工作负载并提高吞吐量。页面刷新到磁盘的原因如下：

  • 间隔一段时间后（30秒）

  • sync(2)、fsync(2)、msync(2) 系统调用

  • 脏页太多（dirty_ratio 和 dirty_bytes 可调参数）

  • 页面缓存中没有可用页面

如果系统内存不足，另一个内核线程，即页调出守护进程（kswapd，也称为页面扫描器），也可能会查找并安排脏页写入磁盘，以便释放内存页以供重用（参见第 7 章，内存）。为了可观察性，kswapd 和flush 线程在操作系统性能工具中作为内核任务可见。

有关页面扫描仪的更多详细信息，请参阅第 7 章“内存”。

目录项缓存

dentry 缓存 (Dcache) 会记住从目录条目（struct dentry）到 VFS inode 的映射，类似于早期的 Unix 目录名称查找缓存 (DNLC)。 Dcache 提高了路径名查找的性能（例如，通过 open(2)）：当遍历路径名时，每个名称查找都可以检查 Dcache 的直接 inode 映射，而不是单步遍历目录内容。 Dcache 条目存储在哈希表中，以便进行快速且可扩展的查找（按父项和目录条目名称进行哈希处理）。

多年来，性能得到了进一步提高，包括使用读取-复制-更新-行走 (RCU-walk) 算法 [Corbet 10]。这尝试在不更新 dentry 引用计数的情况下遍历路径名，这会由于多 CPU 系统上路径名查找率较高的缓存一致性而导致可伸缩性问题。如果遇到不在缓存中的 dentry，RCU-walk 将恢复为较慢的引用计数遍历 (ref-walk)，因为在文件系统查找和阻塞期间需要引用计数。对于繁忙的工作负载，预计 dentry 数据可能会被缓存，并且 RCU-walk 方法将会成功。

Dcache 还执行负缓存，它会记住对不存在条目的查找。这提高了查找失败的性能，这种情况在搜索共享库时通常会发生。

当系统需要更多内存时，Dcache 会动态增长，并通过 LRU（最近最少使用）进行缩减。它的大小可以通过/proc看到。

索引节点缓存

该缓存包含 VFS inode（struct inode），每个索引节点描述文件系统对象的属性，其中许多属性是通过 stat(2) 系统调用返回的。文件系统工作负载经常访问这些属性，例如在打开文件时检查权限，或在修改期间更新时间戳。这些 VFS inode 存储在哈希表中，用于快速且可扩展的查找（按 inode 编号和文件系统超级块进行哈希处理），尽管大多数查找将通过 Dentry 缓存完成。

inode 缓存动态增长，至少保存 Dcache 映射的所有 inode。当存在系统内存压力时，索引节点缓存将收缩，删除没有关联目录项的索引节点。它的大小可以通过 /proc/sys/fs/inode* 文件查看。

8.4.4 文件系统特性

此处描述了影响性能的其他关键文件系统功能。

块与范围

基于块的文件系统将数据存储在固定大小的块中，由存储在元数据块中的指针引用。对于大文件，这可能需要许多块指针和元数据块，并且块的放置可能会变得分散，从而导致随机 I/O。一些基于块的文件系统尝试连续放置块以避免这种情况。另一种方法是使用可变的块大小，以便随着文件的增长可以使用更大的大小，这也减少了元数据开销。

基于范围的文件系统为文件（范围）预先分配连续空间，并根据需要增长它们。这些范围的长度是可变的，代表一个或多个连续的块。这可以提高流性能，并且可以在文件数据本地化时提高随机 I/O 性能。它还提高了元数据性能，因为要跟踪的对象更少，而不会牺牲一定范围内未使用块的空间。

写日记

文件系统日志（或日志）记录对文件系统的更改，以便在系统崩溃或电源故障时，可以原子地重播更改 — 要么完全成功，要么失败。这使得文件系统能够快速恢复到一致的状态。如果与更改相关的数据和元数据写入不完整，非日志文件系统可能会在系统崩溃期间损坏。从此类崩溃中恢复需要遍历所有文件系统结构，对于大型（TB）文件系统可能需要数小时。

日志同步写入磁盘，对于某些文件系统，可以将其配置为使用单独的设备。有些日志同时记录数据和元数据，这会消耗更多的存储 I/O 资源，因为所有 I/O 都会写入两次。其他人仅写入元数据并通过采用写入时复制来维护数据完整性。

有一种文件系统类型仅包含日志：日志结构的文件系统，其中所有数据和元数据更新都写入连续且循环的日志中。这优化了写入性能，因为写入始终是连续的，并且可以合并以使用更大的 I/O 大小。

写时复制

写时复制 (COW) 文件系统不会覆盖现有块，而是遵循以下步骤：

  1. 将块写入新位置（新副本）。

  2. 更新对新块的引用。

  3. 将旧块添加到空闲列表中。

这有助于在系统发生故障时保持文件系统的完整性，并且还可以通过将随机写入转变为顺序写入来提高性能。

当文件系统接近容量时，COW 可能会导致文件的磁盘数据布局碎片化，从而降低性能（尤其是 HDD）。文件系统碎片整理（如果可用）可能有助于恢复性能。

擦洗

这是一项文件系统功能，可异步读取所有数据块并验证校验和，以尽早检测故障驱动器，理想情况下是在由于 RAID 导致故障仍可恢复的情况下。但是，清理读取 I/O 可能会损害性能，因此应以低优先级或在低工作负载时发出。

其他特点

其他可能影响性能的文件系统功能包括快照、压缩、内置冗余、重复数据删除、修剪支持等。以下部分描述了特定文件系统的各种此类功能。

8.4.5 文件系统类型

本章的大部分内容描述了可应用于所有文件系统类型的通用特征。以下各节总结了常用文件系统的具体性能特征。它们的分析和调整将在后面的章节中介绍。

FFS

许多文件系统最终都是基于 Berkeley 快速文件系统 (FFS)，该系统旨在解决原始 Unix 文件系统的问题。6一些背景知识可以帮助解释当今文件系统的状态。

6原始 Unix 文件系统不应与后来称为 UFS 的文件系统混淆，后者基于 FFS。

原始 Unix 文件系统磁盘布局由 inode 表、512 字节存储块和分配资源时使用的信息超级块组成 [Ritchie 74][Lions 77]。 inode 表和存储块将磁盘分区分为两个范围，这会导致在它们之间查找时出现性能问题。另一个问题是使用较小的固定块大小（512 字节），这限制了吞吐量并增加了存储大文件所需的元数据（指针）数量。 [McKusick 84] 描述了将其加倍至 1024 字节的实验，然后遇到了瓶颈：

尽管吞吐量增加了一倍，但旧文件系统仍然只使用大约百分之四的磁盘带宽。主要问题是，尽管空闲列表最初是为了最佳访问而排序的，但随着文件的创建和删除，它很快就变得混乱。最终，空闲列表变得完全随机，导致文件的块在磁盘上随机分配。这迫使在每次块访问之前进行查找。尽管旧文件系统在首次创建时提供高达每秒 175 KB 的传输速率，但由于数据块放置的随机性，在适度使用几周后，该速率会下降到每秒 30 KB。

此摘录描述了空闲列表碎片，随着文件系统的使用，它会随着时间的推移降低性能。

FFS 的另一个性能特征是块交错：将连续的文件块放置在磁盘上，它们之间有一个或多个块的间距 [Doeppner 10]。这些额外的块为内核和处理器提供了时间来发出下一个顺序文件读取。如果没有交错，下一个块可能会在准备好发出读取之前通过（旋转）磁盘头，从而在等待完整旋转时导致延迟。

外部3

Linux 扩展文件系统 (ext) 于 1992 年开发，作为 Linux 及其 VFS 的第一个文件系统，基于原始 Unix 文件系统。第二个版本 ext2 (1993) 包括来自 FFS 的多个时间戳和柱面组。第三个版本 ext3 (1999) 包括文件系统增长和日志功能。

主要性能特性（包括自发布以来添加的特性）包括：

  • 日记：仅适用于元数据的有序模式，或适用于元数据和数据的日记模式。日志记录可提高系统崩溃后的启动性能，从而无需运行 fsck。它还可以通过合并元数据写入来提高某些写入工作负载的性能。

  • 日志设备：可以使用外部日志设备，这样日志工作负载就不会与读取工作负载发生冲突。

  • Orlov 块分配器：这会将顶级目录分布在柱面组中，以便子目录和内容更有可能位于同一位置，从而减少随机 I/O。

  • 目录索引：这些索引将散列 B 树添加到文件系统中，以加快目录查找速度。

mke2fs(8) 手册页中记录了可配置的功能。

外部4

Linux ext4 文件系统于 2008 年发布，通过新功能和性能改进扩展了 ext3：范围、大容量、使用 Fallocate(2) 进行预分配、延迟分配、日志校验和、更快的 fsck、多块分配器、纳秒时间戳和快照。

主要性能特性（包括自发布以来添加的特性）包括：

  • 扩展区：扩展区改进了连续放置，减少了随机 I/O 并增加了顺序 I/O 的 I/O 大小。第 8.4.4 节“文件系统功能”中介绍了它们。

  • 预分配：通过 Fallocate(2) 系统调用，这允许应用程序预分配可能连续的空间，从而提高以后的写入性能。

  • 延迟分配：块分配被延迟，直到它被刷新到磁盘，允许写入组（通过多块分配器），减少碎片。

  • 更快的 fsck：标记未分配的块和 inode 条目，减少 fsck 时间。

XFS

XFS 是 Silicon Graphics 于 1993 年为其 IRIX 操作系统创建的，旨在解决先前 IRIX 文件系统 EFS（基于 FFS）中的可扩展性限制 [Sweeney 96]。 XFS 补丁在 2000 年代初被合并到 Linux 内核中。如今，大多数 Linux 发行版都支持 XFS，并且可以将其用作根文件系统。例如，Netflix 将 XFS 用于其 Cassandra 数据库实例，因为它对于该工作负载具有高性能（并使用 ext4 作为根文件系统）。

主要性能特性（包括自发布以来添加的特性）包括：

  • 分配组：分区被分为大小相等的分配组 (AG)，可以并行访问。为了限制争用，每个 AG 的 inode 和空闲块列表等元数据是独立管理的，而文件和目录可以跨 AG。

  • 范围：（参见 ext4 中先前的描述。）

  • 日志记录：日志记录可提高系统崩溃后的启动性能，从而无需运行 fsck(8)。它还可以通过合并元数据写入来提高某些写入工作负载的性能。

  • 日志设备：可以使用外部日志设备，这样日志工作负载就不会与数据工作负载发生冲突。

  • 条带分配：如果文件系统是在条带 RAID 或 LVM 设备上创建的，则可以提供数据和日志的条带单元，以确保针对底层硬件优化数据分配。

  • 延迟分配：盘区分配被延迟，直到数据刷新到磁盘，从而允许写入分组并减少碎片。块是为内存中的文件保留的，以便在刷新发生时有可用空间。

  • 在线碎片整理：XFS 提供了碎片整理实用程序，可以在主动使用的情况下对文件系统进行操作。虽然 XFS 使用范围和延迟分配来防止碎片，但某些工作负载和条件可能会使文件系统产生碎片。

mkfs.xfs(8) 手册页中记录了可配置的功能。 XFS 的内部性能数据可以通过 /prov/fs/xfs/stat 查看。该数据专为高级分析而设计：有关更多信息，请参阅 XFS 网站 [XFS 06][XFS 10]。

ZFS

ZFS 由 Sun Microsystems 开发并于 2005 年发布，它将文件系统与卷管理器相结合，并包含众多企业功能，使其成为文件服务器（文件管理器）的有吸引力的选择。 ZFS 以开源方式发布，并被多个操作系统使用，但通常作为附加组件，因为 ZFS 使用 CDDL 许可证。大多数开发都发生在 OpenZFS 项目中，该项目于 2019 年宣布支持 Linux 作为主要操作系统 [Ahrens 19]。虽然它在 Linux 中得到越来越多的支持和使用，但由于源许可证（包括 Linus Torvalds [Torvalds 20a]）的原因，仍然存在阻力。

ZFS 的主要性能特性（包括自发布以来添加的特性）包括：

  • 池化存储：所有分配的存储设备都放置在池中，并从中创建文件系统。这允许并行使用所有设备以获得最大吞吐量和 IOPS。可以使用不同的 RAID 类型：0、1、10、Z（基于 RAID-5）、Z2（双奇偶校验）和 Z3（三奇偶校验）。

  • COW：复制修改的块，然后将它们分组并按顺序写入。

  • 日志记录：ZFS 将更改的事务组 (TXG) 批量刷新，这些更改作为一个整体成功或失败，以便磁盘上的格式始终保持一致。

  • ARC：自适应替换缓存通过同时使用多种缓存算法来实现高缓存命中率：最近使用（MRU）和最频繁使用（MFU）。主内存根据性能在它们之间进行平衡，这是通过维护额外的元数据（幽灵列表）来了解每个元数据在统治所有主内存时的执行情况而得知的。

  • 智能预取：ZFS 根据需要应用不同类型的预取：元数据、znode（文件内容）和 vdev（虚拟设备）。

  • 多个预取流：一个文件上的多个流读取器可以在文件系统在它们之间查找时创建随机 I/O 工作负载。 ZFS 跟踪各个预取流，允许新流加入它们。

  • 快照：由于 COW 架构，快照几乎可以立即创建，从而将新块的复制推迟到需要时。

  • ZIO 管道：设备 I/O 由阶段管道处理，每个阶段由线程池提供服务以提高性能。

  • 压缩：支持多种算法，这通常会因 CPU 开销而降低性能。 lzjb (Lempel-Ziv Jeff Bonwick) 选项是轻量级的，可以通过减少 I/O 负载（因为它被压缩）来略微提高存储性能，但会消耗一些 CPU。

  • SLOG：ZFS 单独意图日志允许将同步写入写入单独的设备，从而避免与池磁盘工作负载发生争用。在系统发生故障时，对 SLOG 的写入是只读的，以便重放。

  • L2ARC：2 级 ARC 是主内存之后的第二级缓存，用于在基于闪存的固态磁盘 (SSD) 上缓存随机读取工作负载。它不缓冲写入工作负载，并且仅包含已驻留在存储池磁盘上的干净数据。它还可以复制 ARC 中的数据，以便系统在发生主内存刷新扰动时能够更快地恢复。

  • 重复数据删除：一种文件系统级功能，可避免记录相同数据的多个副本。此功能具有显着的性能影响，既有好处（减少设备 I/O）也有坏处（当哈希表不再适合主内存时，设备 I/O 可能会显着增加）。初始版本仅适用于哈希表应始终适合主内存的工作负载。

与其他文件系统相比，ZFS 的一种行为可能会降低性能：默认情况下，ZFS 向存储设备发出缓存刷新命令，以确保在断电的情况下写入已完成。这是 ZFS 完整性功能之一；然而，这是有代价的：它可能会导致必须等待缓存刷新的 ZFS 操作的延迟。

BTFS

B 树文件系统 (btrfs) 基于写时复制 B 树。这是一个现代文件系统和卷管理器组合架构，类似于 ZFS，预计最终将提供类似的功能集。当前的功能包括池化存储、大容量、扩展、COW、卷增长和收缩、子卷、块设备添加和删除、快照、克隆、压缩和各种校验和（包括 crc32c、xxhash64、sha256 和 blake2b）。 Oracle 于 2007 年开始开发。

主要性能特点包括：

  • 池化存储：存储设备放置在组合卷中，从中创建文件系统。这允许并行使用所有设备以获得最大吞吐量和 IOPS。可以使用 RAID 0、1 和 10。

  • COW：按顺序分组和写入数据。

  • 在线平衡：对象可以在存储设备之间移动以平衡其工作负载。

  • 范围：改进顺序布局和性能。

  • 快照：由于 COW 架构，快照几乎可以立即创建，从而将新块的复制推迟到需要时。

  • 压缩：支持 zlib 和 LZO。

  • 日志记录：可以创建每个子卷的日志树来记录同步 COW 工作负载。

计划的性能相关功能包括 RAID-5 和 6、对象级 RAID、增量转储和重复数据删除。

8.4.6 卷和池

从历史上看，文件系统是构建在单个磁盘或磁盘分区上的。卷和池允许在多个磁盘上构建文件系统，并且可以使用不同的 RAID 策略进行配置（请参阅第 9 章“磁盘”）。

卷将多个磁盘呈现为一个虚拟磁盘，在该虚拟磁盘上构建文件系统。当基于整个磁盘（而不是切片或分区）构建时，卷会隔离工作负载，从而减少争用的性能问题。

卷管理软件包括用于基于 Linux 的系统的逻辑卷管理器 (LVM)。卷或虚拟磁盘也可以由硬件 RAID 控制器提供。

池化存储包括存储池中的多个磁盘，可以从中创建多个文件系统。图 8.11 中显示了这一点以及用于比较的体积。池存储比卷存储更灵活，因为无论支持设备如何，文件系统都可以增长和收缩。现代文件系统（包括 ZFS 和 btrfs）都使用这种方法，也可以使用 LVM。

池存储可以将所有磁盘设备用于所有文件系统，从而提高性能。工作负载不是孤立的；在某些情况下，考虑到一定的灵活性，可以使用多个池来分离工作负载，因为磁盘设备最初必须放置在一个池或另一个池中。请注意，池磁盘可能具有不同的类型和大小，而卷可能仅限于卷内的统一磁盘。

使用软件卷管理器或池存储时的其他性能注意事项包括：

  • 条带宽度：将其与工作负载相匹配。

  • 可观察性：虚拟设备利用率可能会令人困惑；检查单独的物理设备。

  • CPU 开销：尤其是在执行 RAID 奇偶校验计算时。对于现代、更快的 CPU 来说，这个问题已经不再是问题了。 （奇偶校验计算也可以卸载到硬件 RAID 控制器。）

  • 重建：也称为重新同步，这是指将空磁盘添加到 RAID 组（例如，更换故障磁盘）并填充加入该组所需的数据。这会显着影响性能，因为它会消耗 I/O 资源，并且可能持续数小时甚至数天。

重建是一个日益恶化的问题，因为存储设备的容量增长速度快于其吞吐量的增长速度，从而增加了重建时间，并使重建期间发生故障或介质错误的风险更大。如果可能，对未安装的驱动器进行离线重建可以缩短重建时间。

8.5.1 磁盘分析

常见的故障排除策略是忽略文件系统并关注磁盘性能。这假设最差的 I/O 是磁盘 I/O，因此通过仅分析磁盘，您可以方便地关注预期的问题根源。

对于更简单的文件系统和更小的缓存，这通常是有效的。如今，这种方法变得令人困惑，并且错过了整类问题（请参见第 8.3.12 节，逻辑 I/O 与物理 I/O）。

8.5.2 延迟分析

操作延迟=时间（操作完成）-时间（操作请求）

  • 应用程序文档：某些应用程序已经提供文件系统延迟指标或启用其收集的能力。

发现高延迟后，继续深入分析文件系统以确定来源。

交易成本

表示文件系统延迟的另一种方法是应用程序事务（例如数据库查询）期间等待文件系统所花费的总时间：

文件系统中的时间百分比 = 100 * 总阻塞文件系统延迟/应用程序事务时间

这使得文件系统操作的成本可以根据应用程序性能进行量化，并可以预测性能改进。该度量可以表示为一个时间间隔内所有事务或单个事务的平均值。

图 8.12 显示了为事务提供服务的应用程序线程所花费的时间。该事务发出单个文件系统读取；应用程序阻塞并等待其完成，转换到脱离 CPU。这种情况下的总阻塞时间是单个文件系统读取的时间。如果在事务期间调用了多个阻塞 I/O，则总时间是它们的总和。

图 8.12 应用程序和文件系统延迟

作为一个具体示例，应用程序事务需要 200 毫秒，在此期间它在多个文件系统 I/O 上总共等待 180 毫秒。应用程序被文件系统阻止的时间为 90% (100 * 180 ms/200 ms)。消除文件系统延迟可将性能提高多达 10 倍。

再举一个例子，如果一个应用程序事务需要 200 毫秒，其中在文件系统中只花费了 2 毫秒，则文件系统以及整个磁盘 I/O 堆栈只为事务运行时间贡献了 1%。这个结果非常有用，因为它可以引导性能调查找到延迟的真正来源。

如果应用程序以非阻塞方式发出 I/O，则应用程序可以在文件系统响应时继续在 CPU 上执行。在这种情况下，阻塞文件系统延迟仅测量应用程序被阻止离开 CPU 的时间。

8.5.3 工作负载特征

在进行容量规划、基准测试和模拟工作负载时，表征所应用的负载是一项重要的工作。通过识别可以消除的不必要的工作，它还可以带来一些最大的性能提升。

以下是表征文件系统工作负载的基本属性：

  • 开工率及开工类型

  • 文件 I/O 吞吐量

  • 文件输入/输出大小

  • 读/写比率

  • 同步写入比例

  • 随机与顺序文件偏移访问

操作率和吞吐量在第 8.1 节“术语”中定义。同步写入以及随机写入与顺序写入在第 8.3 节“概念”中进行了描述。

这些特征可能会随时变化，特别是对于间隔执行的定时应用程序任务。为了更好地描述工作负载的特征，请捕获最大值和平均值。更好的是，检查值随时间的完整分布。

下面是一个工作负载描述示例，展示了如何一起表达这些属性：

在金融交易数据库上，文件系统具有随机读取工作负载，平均读取次数为 18,000 次/秒，平均读取大小为 4 KB。总操作速率为 21,000 次操作/秒，其中包括读取、统计、打开、关闭以及大约 200 次同步写入/秒。写入速率稳定，而读取速率变化，最高可达 39,000 次读取/秒。

这些特征可以用单个文件系统实例或同一类型系统上的所有实例来描述。

高级工作负载特征/清单

可能会包含其他详细信息来描述工作负载的特征。这些已在此处列出作为需要考虑的问题，也可以作为彻底研究文件系统问题时的清单：

  • 文件系统缓存命中率是多少？小姐率？

  • 文件系统缓存容量和当前使用情况是多少？

  • 还存在哪些其他缓存（目录、索引节点、缓冲区），以及它们的统计数据是什么？

  • 过去是否尝试过调整文件系统？是否有任何文件系统参数设置为默认值以外的值？

  • 哪些应用程序或用户正在使用该文件系统？

  • 正在访问哪些文件和目录？创建和删除？

  • 是否遇到任何错误？这是由于无效请求还是文件系统问题？

  • 为什么要发出文件系统I/O（用户级调用路径）？

  • 应用程序直接（同步）请求文件系统 I/O 的程度如何？

  • I/O 到达时间的分布是怎样的？

其中许多问题可以针对每个应用程序或每个文件提出。它们中的任何一个也可以随着时间的推移进行检查，以查找最大值、最小值和基于时间的变化。另请参阅第 2 章“方法论”中的第 2.5.10 节“工作负载特征描述”，其中提供了要测量的特征的更高级别摘要（谁、为什么、什么、如何）。

性能表征

前面的工作负载特征列表检查了所应用的工作负载。下面检查最终的性能：

  • 文件系统操作的平均延迟是多少？

  • 是否存在高延迟异常值？

  • 操作延迟的完全分布是怎样的？

  • 文件系统或磁盘 I/O 的系统资源控制是否存在且处于活动状态？

可以针对每种操作类型分别询问前三个问题。

事件追踪

跟踪工具可用于将所有文件系统操作和详细信息记录到日志中以供以后分析。这可以包括每个 I/O 的操作类型、操作参数、文件路径名、开始和结束时间戳、完成状态以及进程 ID 和名称。虽然这可能是工作负载表征的终极工具，但实际上，由于文件系统操作的速率，它可能会花费大量开销，除非经过严格过滤（例如，仅在日志中包含慢速 I/O：请参阅 ext4slower (8) 第 8.6.14 节中的工具。

8.5.4 性能监控

性能监控可以随着时间的推移识别活动问题和行为模式。文件系统性能的关键指标包括：

  • 开工率

  • 操作延迟

运行速率是应用的工作负载最基本的特征，而延迟则是由此产生的性能。正常或严重延迟的值取决于您的工作负载、环境和延迟要求。如果您不确定，可以执行已知良好与不良工作负载的微基准来调查延迟（例如，通常会命中文件系统缓存的工作负载与通常会错过的工作负载）。请参见第 8.7 节“实验”。

操作等待时间度量可以作为每秒平均值来监控，并且可以包括其他值，例如最大值和标准偏差。理想情况下，可以检查延迟的完整分布，例如通过使用直方图或热图来查找异常值和其他模式。

还可以记录每种操作类型（读、写、统计、打开、关闭等）的速率和延迟。通过识别特定操作类型的差异，这样做将极大地帮助调查工作负载和性能变化。

对于实施基于文件系统的资源控制的系统，可以包含统计信息以显示是否以及何时使用限制。

不幸的是，在 Linux 中，通常没有现成的文件系统操作统计信息（例外情况包括，对于 NFS，通过 nfsstat(8)）。

8.5.5 静态性能调优

静态性能调优重点关注配置环境的问题。对于文件系统性能，请检查静态配置的以下方面：

  • 有多少文件系统已安装并正在使用？

  • 文件系统记录大小是多少？

  • 是否启用访问时间戳？

  • 启用了哪些其他文件系统选项（压缩、加密...）？

  • 文件系统缓存是如何配置的？最大尺寸？

  • 其他缓存（目录、索引节点、缓冲区）是如何配置的？

  • 二级缓存是否存在并正在使用？

  • 有多少个存储设备存在并且正在使用？

  • 存储设备配置如何？袭击？

  • 使用哪些文件系统类型？

  • 文件系统（或内核）的版本是什么？

  • 是否存在应考虑的文件系统错误/补丁？

  • 文件系统 I/O 是否使用资源控制？

回答这些问题可以揭示被忽视的配置选择。有时，系统已针对一种工作负载进行配置，然后重新用于另一种工作负载。此方法会提醒您重新审视这些选择。

8.5.6 缓存调优

内核和文件系统可以使用许多不同的高速缓存，包括缓冲区高速缓存、目录高速缓存、索引节点高速缓存和文件系统（页）高速缓存。第 8.4 节“架构”中描述了各种缓存。根据可用的可调选项，可以检查并经常调整这些。

8.5.7 工作负载分离

某些类型的工作负载在配置为使用其自己的专用文件系统和磁盘设备时性能会更好。这种方法被称为使用“单独的主轴”，因为通过在两个不同的工作负载位置之间查找来创建随机 I/O 对于旋转磁盘来说尤其糟糕（请参阅第 9 章“磁盘”）。

例如，数据库可以受益于为其日志文件和数据库文件拥有单独的文件系统和磁盘。数据库的安装指南经常包含有关其数据存储的放置的建议。

8.5.8 微基准测试

文件系统和磁盘基准测试（有很多）的基准测试工具可用于测试给定工作负载的不同文件系统类型或文件系统内设置的性能。可以测试的典型因素包括

  • 操作类型：读取、写入和其他文件系统操作的速率

  • I/O 大小：1 字节至 1 MB 及以上

  • 文件偏移模式：随机或顺序

  • 随机访问模式：均匀、随机或帕累托分布

  • 写入类型：异步或同步（O_SYNC）

  • 工作集大小：它在文件系统缓存中的适合程度

  • 并发：并行 I/O 数量或执行 I/O 的线程数量

  • 内存映射：通过 mmap(2) 进行文件访问，而不是 read(2)/write(2)

  • 缓存状态：文件系统缓存是“冷”（未填充）还是“热”

  • 文件系统可调参数：可能包括压缩、重复数据删除等

常见的组合包括随机读取、顺序读取、随机写入和顺序写入。我没有在此列表中包含直接 I/O，因为其微基准测试的目的是绕过文件系统并测试磁盘设备性能（请参阅第 9 章“磁盘”）。

对文件系统进行微基准测试时的一个关键因素是工作集大小 (WSS)：访问的数据量。根据基准测试，这可能是正在使用的文件的总大小。除非使用直接 I/O 标志，否则较小的工作集大小可能会完全从主存储器 (DRAM) 中的文件系统缓存返回。较大的工作集大小可能主要从存储设备（磁盘）返回。性能差异可能是多个数量级。针对新安装的文件系统运行基准测试，然后在填充缓存后再次运行基准测试，并比较两次的结果，这通常是 WSS 的一个很好的说明。 （另请参阅第 8.7.3 节“缓存刷新”。）

8.6.7 跟踪

可以使用跟踪工具（包括适用于 Linux 的 strace(1)）在系统调用接口处测量文件系统延迟。然而，当前基于 ptrace(2) 的 strace(1) 实现可能会严重损害性能，并且仅当性能开销可以接受并且无法使用其他方法来分析延迟时才适合使用。有关 strace(1) 的更多信息，请参阅第 5 章第 5.5.4 节“strace”。

此示例显示 strace(1) 在 ext4 文件系统上的计时读取：

点击这里查看代码图片

$ strace -ttT -p 845 [...] 18:41:01.513110 read(9, "\334\260/\224\356k..."..., 65536) = 65536 <0.018225> 18:41:01.531646 read(9, "\371X\265|\244\317..."..., 65536) = 65536 <0.000056> 18:41:01.531984 read(9, "\357\311\347\1\241..."..., 65536) = 65536 <0.005760> 18:41:01.538151 read(9, "*\263\264\204|\370..."..., 65536) = 65536 <0.000033> 18:41:01.538549 read(9, "\205q\327\304f\370..."..., 65536) = 65536 <0.002033> 18:41:01.540923 read(9, "\6\2738>zw\321\353..."..., 65536) = 65536 <0.000032>

-tt 选项在左侧打印相对时间戳，而 -T 在右侧打印系统调用时间。每次读取 (2) 的数据量为 64 KB，第一次需要 18 毫秒，接下来是 56 微秒（可能已缓存），然后是 5 毫秒。读取的内容是文件描述符 9。要检查这是否是文件系统（而不是套接字），open(2) 系统调用将在早期的 strace(1) 输出中可见，或者使用其他工具（例如 lsof）可以使用(8)。您还可以在 /proc 文件系统中找到有关 FD 9 的信息：/proc/845/fd{,info}/9}。

考虑到 strace(1) 当前的开销，测量的延迟可能会因观察者效应而产生偏差。查看更新的跟踪工具，包括 ext4slower(8)，它使用每 CPU 缓冲跟踪和 BPF 来大大减少开销，提供更准确的延迟测量。

点击这里查看代码图片

每行显示进程名称、PID、事件类型、完整路径和可选状态。事件类型可以是打开 (O)、读取 (R)、写入 (W) 和关闭 (C)。 fatrace(1) 可用于工作负载表征：了解访问的文件，并查找可以消除的不必要的工作。

系统调用追踪

系统调用是跟踪的一个很好的目标，也是许多跟踪工具的检测源。然而，一些系统调用缺乏文件系统上下文，这使得它们使用起来很混乱。我将提供一个工作（openat(2) 跟踪）和不工作（read(2) 跟踪）的示例，以及建议的补救措施。

打开(2)

跟踪系统调用的 open(2) 系列会显示打开的文件。如今 openat(2) 变体更常用。追踪它：

点击这里查看代码图片

阅读(2)

read(2) 应该是了解文件系统读取延迟的有用跟踪目标。但是，请考虑跟踪点参数（看看是否可以发现问题）：

点击这里查看代码图片

read(2) 可以针对文件系统、套接字、/proc 和其他目标调用，并且参数不区分它们。为了显示这是多么令人困惑，以下按进程名称对 read(2) 系统调用进行计数：

点击这里查看代码图片

# bpftrace -e 't:syscalls:sys_enter_read { @[comm] = count(); }' Attaching 1 probe... ^C

@[systemd-journal]: 13 @[sshd]: 141 @[java]: 3472

在跟踪时，Java 执行了 3,472 个 read(2) 系统调用，但它们是来自文件系统、套接字还是其他东西？ （sshd 读取可能是套接字 I/O。）

read(2) 确实提供了作为整数的文件描述符 (FD)，但它只是一个数字并且不显示 FD 类型（并且 bpftrace 运行在受限内核模式下：它无法查找 FD 信息在 /proc 中）。对此至少有四种解决方案：

  • 从 bpftrace 打印 PID 和 FD，然后使用 lsof(8) 或 /proc 查找 FD 以了解它们是什么。

  • 即将推出的 BPF 助手 get_fd_path() 可以返回 FD 的路径名。这将有助于区分文件系统读取（具有路径名）与其他类型。

  • 相反，从 VFS 进行跟踪，其中有更多数据结构可用。

  • 直接跟踪文件系统功能，排除其他 I/O 类型。 ext4dist(8) 和 ext4slower(8) 使用此方法。

8.6.18 可视化

应用于文件系统的负载可以随着时间的推移绘制为折线图，以帮助识别基于时间的使用模式。为读取、写入和其他文件系统操作绘制单独的图表可能很有用。

文件系统延迟的分布预计是双峰的：一种模式是文件系统缓存命中的低延迟模式，另一种模式是缓存未命中的高延迟模式（存储设备 I/O）。因此，将分布表示为单个值（例如均值、众数或中位数）会产生误导。

解决此问题的一种方法是使用显示完整分布的可视化效果，例如热图。热图在第 2 章“方法论”第 2.10.3 节“热图”中介绍。文件系统延迟热图示例如图 8.13 所示：它在 x 轴上显示时间的推移，在 y 轴上显示 I/O 延迟 [Gregg 09a]。

此热图显示了启用 L2ARC 设备对 NFSv3 延迟的影响。 L2ARC 设备是主内存之后的辅助 ZFS 缓存，通常使用闪存（在第 8.3.2 节“缓存”中提到过）。图 8.13 中的系统具有 128 GB 的主内存 (DRAM) 和 600 GB 的 L2ARC（读取优化 SSD）。热图的左半部分显示没有 L2ARC 设备（L2ARC 已禁用），右半部分显示 L2ARC 设备的延迟。

图8.13 文件系统延迟热图

对于左半部分，文件系统延迟要么低要么高，中间有一个间隙。低延迟是底部的蓝线，大约 0 毫秒，这可能是主内存缓存命中。高延迟从大约 3 毫秒开始，一直延伸到顶部，显示为“云”，这可能是旋转磁盘延迟。当由旋转磁盘支持时，这种双模式延迟分布是文件系统延迟的典型特征。

对于右半部分，启用了 L2ARC，延迟现在通常低于 3 毫秒，并且较高的磁盘延迟较少。您可以看到 L2ARC 的延迟如何填补热图左侧空白的范围，从而总体上减少文件系统延迟。

8.7.2 微基准测试工具

有许多可用的文件系统基准测试工具，包括 Bonnie、Bonnie++、iozone、tiobench、SysBench、fio 和 FileBench。这里讨论了一些内容，按照复杂性递增的顺序。另请参阅第 12 章，基准测试。我个人的建议是使用fio。

邦妮，邦妮++

Bonnie 工具是一个简单的 C 程序，用于从单个线程测试单个文件上的多个工作负载。它最初由 Tim Bray 于 1989 年编写 [Bray 90]。用法很简单，不需要参数（将使用默认值）：

输出包括每次测试期间的 CPU 时间，100% 表明 Bonnie 从未在磁盘 I/O 上阻塞，而是始终从缓存命中并停留在 CPU 上。原因是目标文件大小为 100 MB，该文件完全缓存在该系统上。您可以使用 -s size 更改文件大小。

有一个名为 Bonnie-64 的 64 位版本，它允许测试更大的文件。还有一个 C++ 重写版本，名为 Bonnie++，由 Russell Coker [Coker 01] 编写。

不幸的是，除非您清楚地了解正在测试的内容，否则像 Bonnie 这样的文件系统基准测试工具可能会产生误导。第一个结果是 putc(3) 测试，它可能会根据系统库实现而有所不同，然后系统库实现将成为测试目标而不是文件系统。请参阅第 12 章“基准测试”、第 12.3.2 节“主动基准测试”中的示例。

菲奥

Jens Axboe 的灵活 IO 测试器 (fio) 是一款可定制的文件系统基准测试工具，具有许多高级功能 [Axboe 20]。促使我使用它而不是其他基准测试工具的两个原因是：

  • 非均匀随机分布，可以更准确地模拟现实世界的访问模式（例如，-random_distribution=pareto:0.9）

  • 报告延迟百分位数，包括 99.00、99.50、99.90、99.95、99.99

文件台

FileBench 是一个可编程文件系统基准测试工具，可以通过使用其工作负载模型语言描述应用程序工作负载来模拟它们。这允许模拟具有不同行为的线程，并指定同步线程行为。它附带了各种称为个性的配置，其中包括模拟 Oracle 数据库 I/O 模型的配置。不幸的是，FileBench 并不是一个易于学习和使用的工具，可能只有那些全职从事文件系统工作的人才会感兴趣。

8.7.3 缓存刷新

Linux 提供了一种刷新（删除条目）文件系统缓存的方法，这对于从一致且“冷”的缓存状态进行性能基准测试可能很有用，就像系统启动后那样。这种机制在内核源文档（Documentation/sysctl/vm.txt）中非常简单地描述为：

点击这里查看代码图片

To free pagecache: echo 1 > /proc/sys/vm/drop_caches To free reclaimable slab objects (includes dentries and inodes): echo 2 > /proc/sys/vm/drop_caches To free slab objects and pagecache: echo 3 > /proc/sys/vm/drop_caches

在其他基准测试运行之前释放所有内容 (3) 特别有用，以便系统以一致的状态（冷缓存）开始，有助于提供一致的基准测试结果。

8.8 调优

第 8.5 节“方法论”中已经介绍了许多调优方法，包括缓存调优和工作负载特征。后者可以通过识别和消除不必要的工作来获得最高的调整胜利。本节包括特定的调整参数（可调参数）。

调整的细节（可用选项以及将其设置为什么）取决于操作系统版本、文件系统类型和预期工作负载。以下部分提供了可用内容以及为何需要对其进行调整的示例。我介绍了应用程序调用和两个示例文件系统类型：ext4 和 ZFS。有关页面缓存的调整，请参阅第 7 章“内存”。

8.9 练习

  1. 回答以下有关文件系统术语的问题：

    • 逻辑 I/O 和物理 I/O 有什么区别？

    • 随机 I/O 和顺序 I/O 有什么区别？

    • 什么是直接 I/O？

    • 什么是非阻塞 I/O？

    • 工作集大小是多少？

  2. 回答以下概念性问题：

    • VFS的作用是什么？

    • 描述文件系统延迟，特别是可以从哪里测量延迟。

    • 预取（预读）的目的是什么？

    • 直接 I/O 的目的是什么？

  3. 回答以下更深层次的问题：

    • 描述使用 fsync(2) 相对于 O_SYNC 的优点。

    • 描述 mmap(2) 相对于 read(2)/write(2) 的优缺点。

    • 描述逻辑 I/O 在变为物理 I/O 时变得膨胀的原因。

    • 描述逻辑 I/O 在变为物理 I/O 时变得紧缩的原因。

    • 解释文件系统写时复制如何提高性能。

  4. 为您的操作系统开发以下过程：

    • 文件系统缓存调整清单。这应该列出存在的文件系统缓存、如何检查它们当前的大小和使用情况以及命中率。

    • 文件系统操作的工作负载特征检查表。包括如何获取每个细节，并首先尝试使用现有的操作系统可观察性工具。

  5. 执行这些任务：

    • 选择一个应用程序，并测量文件系统操作和延迟。包括：

      • 文件系统操作延迟的完整分布，而不仅仅是平均值。

      • 每个应用程序线程每秒花费在文件系统操作上的部分。

    • 使用微基准测试工具，通过实验确定文件系统缓存的大小。解释您在使用该工具时的选择。当工作集不再缓存时，还显示性能下降（使用任何指标）。

  6. （可选，高级）开发一个可观察性工具，为同步与异步文件系统写入提供指标。这应该包括它们的速率和延迟，并且能够识别发出它们的进程 ID，使其适合工作负载表征。

  7. （可选，高级）开发一个工具来提供间接和膨胀的文件系统 I/O 的统计信息：额外的字节和不是由应用程序直接发出的 I/O。该工具应该将这些额外的 I/O 分解为不同的类型以解释其原因。
